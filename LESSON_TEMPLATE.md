# Шаблон урока

Каждый урок находится в отдельной папке:

```text
NN-lesson-name/
├── code/
│   └── main.py
├── notebook/
│   └── lesson.ipynb
├── docs/
│   └── ru.md
├── tests/
│   └── test_main.py
├── outputs/
│   ├── artifact.json
│   └── <named-artifact>
├── quiz.json
└── lesson.json
```

Не все уроки обязаны использовать ноутбук, но каждый урок обязан содержать документацию,
исполняемый пример, проверку результата и итоговый артефакт.

## Метаданные `lesson.json`

```json
{
  "title": "Название урока",
  "type": "build",
  "tracks": ["core"],
  "prerequisites": ["03-pandas/05-groupby"],
  "time_minutes": 75,
  "outcome": "Рассчитывает метрику и проверяет ее на контрольном наборе",
  "artifact": {
    "name": "Функция расчета метрики",
    "type": "function",
    "path": "outputs/metric.py"
  }
}
```

## Типы уроков

Тип определяет форму ручной реализации, но не отменяет ее.

| Тип | Что студент строит сам | Что появляется в `Use It` |
|---|---|---|
| `build` | Минимальный механизм без целевой библиотеки или фреймворка | Production API и сравнение поведения |
| `learn` | Наблюдаемый эксперимент или уменьшенную модель устройства инструмента | Рабочий инструмент и его границы |
| `case` | Прозрачный baseline, формулу или последовательность аналитического решения | Масштабируемый расчет и оформление результата |

`Build It` нельзя заменять копированием вызовов изучаемой библиотеки. Для pandas-урока это
может быть расчет на списках и словарях, для SQL - ручная таблица результата и реляционная
операция, для статистики - формула и симуляция, для dbt - минимальный граф преобразований.

## Используйте это

Решение той же задачи рабочей библиотекой или платформой. Сравните результат, API,
ограничения и стоимость поддержки с ручной реализацией.

## Сломайте это

Намеренно проблемные входы: пропуски, дубликаты, неверный grain, новые категории,
временная утечка, несогласованные типы или другой релевантный failure mode.

## Проверьте это

Тесты, инварианты, контрольный расчет и критерии корректной интерпретации.

## Поставьте результат

Описание именованного артефакта из `outputs/`, команды запуска и сценария повторного
использования вне урока.

3. Расширение с новой ошибкой или ограничением.

## Требования к коду

- Код запускается из корня урока и не зависит от скрытого состояния ноутбука.
- `Build It` демонстрирует механизм до использования целевой библиотеки.
- Все случайные процессы воспроизводимы.
- Внешние данные загружаются скриптом с проверкой размера или checksum.
- Секреты не хранятся в репозитории.
- Основной расчет покрыт тестом или контрольным примером.
- Число без методологии и интерпретации не считается результатом.

## Допустимые артефакты

- функция расчета метрики;
- SQL-запрос или dbt-модель;
- схема или контракт данных;
- набор тестов качества;
- шаблон EDA или эксперимента;
- воспроизводимый отчет;
- CLI, приложение или dashboard;
- model card;
- prompt или agent skill для проверки аналитического решения.

## Контракт артефакта

`outputs/artifact.json`:

```json
{
  "name": "retention-calculator",
  "type": "function",
  "path": "outputs/retention.py",
  "description": "Рассчитывает cohort retention с явным grain",
  "usage": "python outputs/retention.py --input data/events.parquet"
}
```

Путь должен указывать на реальный файл. Артефакт обязан иметь собственную инструкцию
использования и работать без чтения текста урока.

## Контракт квиза

`quiz.json` содержит объект `questions`. Для завершенного урока обязательны минимум два
входных и три итоговых вопроса. Каждый вопрос содержит:

- стабильный `id`;
- `stage`: `pre` или `post`;
- формулировку и четыре варианта ответа;
- индекс правильного варианта;
- объяснение, почему ответ правильный.

Квиз проверяет понимание и перенос метода, а не запоминание имени функции.
