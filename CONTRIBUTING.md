# Участие в разработке

Исправления уроков, новые упражнения, улучшения сайта и tooling приветствуются. Один pull
request должен решать одну связанную задачу: так проще проверить аналитическую корректность
и не потерять полезное изменение среди побочных правок.

Участие в проекте означает согласие с [`CODE_OF_CONDUCT.md`](CODE_OF_CONDUCT.md).

## Перед началом

1. Для ошибки используйте [шаблон bug report](.github/ISSUE_TEMPLATE/bug_report.md).
2. Новый урок сначала предложите через
   [шаблон lesson proposal](.github/ISSUE_TEMPLATE/new_lesson_proposal.md).
3. Прочитайте [`docs/course-design.md`](docs/course-design.md) и
   [`LESSON_TEMPLATE.md`](LESSON_TEMPLATE.md).
4. Для изменения программы изучите `curriculum.json` и зависимости маршрутов.

## Источник правды

Фазы, зависимости и список уроков редактируются в `curriculum.json`. Файлы `ROADMAP.md` и
`phases/*/README.md`, `outputs/index.json` и `site/data.js` генерируются и не редактируются
вручную.

После изменения программы:

```bash
uv sync --locked --dev
uv run --locked python scripts/render_curriculum.py
uv run --locked python scripts/render_outputs.py
uv run --locked python scripts/render_site.py
uv run --locked python scripts/validate_course.py
uv run --locked python -m unittest discover -s tests
uv run --locked python scripts/run_lesson_tests.py
```

## Добавление урока

1. Добавьте урок в соответствующую фазу `curriculum.json`.
2. Перегенерируйте дорожную карту.
3. Создайте папку урока:

```bash
python3 scripts/scaffold_lesson.py 03-pandas 05-groupby "GroupBy и единица анализа"
```

4. Заполните документацию по `LESSON_TEMPLATE.md`.
5. Добавьте код, тесты, квиз и артефакт.
6. Добавьте 3-5 развивающих материалов по контракту из `LESSON_TEMPLATE.md`.
7. Переведите статус в `complete` только после удаления всех `TODO`.
8. Перегенерируйте индексы и запустите проверки из корня репозитория.

## Другие способы помочь

- Исправить код, который не запускается или дает неверный результат.
- Уточнить объяснение, failure mode или контрольный расчет.
- Добавить упражнение без утечки готового ответа.
- Обновить устаревшую ссылку или API по официальной документации.
- Улучшить доступность, мобильную версию или навигацию standalone-сайта.
- Предложить переиспользуемый аналитический артефакт для существующего урока.

## Правила изменения программы

- Новый инструмент добавляется только вместе с задачей, которую он решает лучше уже
  присутствующих средств.
- Факультатив не становится обязательным пререквизитом.
- Урок должен иметь один главный результат и укладываться примерно в 45-120 минут.
- Большая библиотека делится по рабочим задачам, а не повторяет оглавление документации.
- Ссылки ведут преимущественно на официальную документацию и первичные источники.
- Каждая развивающая ссылка объясняет, что читать и какой навык она углубляет.
- Данные и ожидаемые ответы не должны раскрывать решение упражнения.
- `Build It` показывает механизм до production API; обзор методов библиотеки не считается
  ручной реализацией.
- Артефакт имеет имя, manifest, команду использования и работает вне контекста урока.

## Pull request

1. Создайте отдельную ветку.
2. Внесите связанное изменение без постороннего форматирования файлов.
3. Запустите проверки и ручной сценарий, относящийся к изменению.
4. Заполните шаблон pull request, включая ограничения и способ проверки.
5. Не переводите урок в `complete`, пока не выполнен весь контракт.

Для заметного изменения программы, статуса фаз или следующего шага обновите
`docs/PROJECT_STATUS.md`. Пользовательские данные, секреты и большие локальные датасеты в
репозиторий не добавляются.
