GroupBy и единица анализа

groupby не просто считает несколько чисел. Он превращает исходные строки в таблицу с другой единицей наблюдения. Поэтому до кода нужно назвать grain входа, популяцию, смысл агрегатов и grain результата.

Тип: Case

Треки: Core

Пререквизиты: 03/04

Время: ~105 минут

Результат: на типизированной таблице товарных позиций объявляет входной и выходной grain, выбирает агрегаты по смыслу метрик, сохраняет неизвестность неполной суммы и сверяет результат с ручным контрольным расчётом.

Цели обучения

После урока вы сможете:

  • объяснить агрегацию как переход от одного grain к другому;
  • до groupby назвать входной ключ, популяцию, ключи результата и смысл каждой метрики;
  • вручную воспроизвести механизм split -> apply -> combine на нескольких строках;
  • различать size, count, nunique, sum и mean по вопросу, на который они отвечают;
  • использовать именованные агрегации и получать ключ группы обычным столбцом;
  • отличать пропуск в ключе группы от пропуска в измеряемом значении;
  • не превращать полностью или частично неизвестную сумму в доказанный ноль;
  • проверять новый grain, полноту, dtype, число строк и контрольные totals;
  • объяснить, почему GroupBy.apply не нужен для обычных агрегатов.

Проблема

После предыдущего урока у нас есть уже типизированные товарные позиции:

Метка order_id product_id line_total
item-a O1001 P01 800.0
item-b O1001 P02 800.0
item-c O1002 P03 пропуск
item-d O1003 P04 75.5
item-e O1004 P05 0.0

Входной grain:

text
одна строка -> одна товарная позиция заказа
кандидатный ключ -> order_id + product_id

Для следующего объединения нужна таблица на уровне заказа:

text
одна строка -> один заказ
кандидатный ключ -> order_id

Значит, две строки O1001 должны превратиться в одну. Повтор order_id во входе здесь не является дублем: он ожидается на grain товарной позиции. Дублем будет повтор всей пары (order_id, product_id).

До вызова groupby нужно принять четыре решения:

  1. Входной grain: что означает каждая строка и какой ключ его поддерживает?
  2. Популяция: все ли проверенные строки участвуют в расчёте?
  3. Метрики: что именно считаем внутри каждой группы и как трактуем пропуски?
  4. Выходной grain: что означает строка результата и какой ключ должен стать уникальным?

Если одно из решений не принято, pandas всё равно вернёт таблицу. Именно поэтому ошибка агрегации часто выглядит правдоподобным числом, а не исключением.

Концепция

Агрегация меняет grain

Преобразование из 03/04 добавляло значение каждой исходной строке:

text
5 товарных позиций -> 5 товарных позиций

Агрегация сворачивает несколько строк одной группы в одну:

text
5 товарных позиций -> 4 заказа

order_id повторяется во входе и становится уникальным в результате. Это не побочный эффект groupby, а целевое изменение единицы наблюдения.

Число строк после агрегации определяется не названием функции sum или mean, а числом уникальных комбинаций ключей группировки в выбранной популяции.

Ключи группировки являются обещанием о результате

Сравните два вызова:

python
items.groupby("order_id")
items.groupby(["order_id", "product_id"])

Первый создаёт группу для каждого заказа и действительно уменьшает детализацию. Второй использует исходный составной ключ, поэтому при корректном входном grain почти каждая строка останется отдельной группой. Код выполнится, но требуемого перехода к заказам не произойдёт.

Поэтому список ключей читается как человеческое утверждение:

Одна строка результата будет описывать одну уникальную комбинацию этих значений.

Популяция определяется до группировки

Фильтрация и агрегация решают разные задачи:

Операция Вопрос Что меняется
Фильтрация Какие наблюдения участвуют? Состав строк
Преобразование Что вычислить для каждой строки? Столбцы, но не grain
Агрегация Какой итог нужен для каждой группы? Grain результата

Если бизнес-вопрос относится только к выбранной популяции, сначала примените проверенную маску из 03/03, сохраните отчёт о True, False и NA, а уже затем группируйте разрешённые строки. В этом уроке входная таблица уже является нужной популяцией, поэтому мы не прячем дополнительный фильтр внутрь функции агрегации.

split -> apply -> combine — модель механизма

GroupBy удобно понимать как три шага.

Split — разделить. Значение order_id направляет каждую строку в группу:

text
O1001 -> item-a, item-b
O1002 -> item-c
O1003 -> item-d
O1004 -> item-e

Apply — применить агрегаты. Внутри каждой группы отдельно считаются число строк, число известных сумм и сумма известных значений.

Combine — собрать. Результаты групп соединяются в новую таблицу, где одна строка соответствует одному order_id.

Слово apply в названии модели не требует метода GroupBy.apply. Для обычных сумм, счётчиков, минимумов и средних pandas предоставляет специализированные агрегаты. Они яснее показывают намерение и не заставляют самостоятельно определять форму результата произвольной callback-функции.

Один универсальный count не существует

Разные агрегаты отвечают на разные вопросы:

Агрегат Что считает Как относится к пропуску
size все строки группы включает строки с пропущенным значением
count известные значения выбранного столбца исключает пропуски этого столбца
nunique разные известные значения по умолчанию исключает пропуск
sum сумму известных значений обычно пропускает NA; полностью пустая сумма может стать нулём
mean среднее известных значений исключает NA и меняет знаменатель

На группе из трёх строк со значениями P01, P01, NA получатся разные ответы:

text
size                 = 3  # строки
count(product_id)    = 2  # известные записи
nunique(product_id)  = 1  # разные известные продукты

Ни один ответ не «правильнее» остальных без вопроса. Имя выходной метрики должно фиксировать смысл: line_count, known_amount_lines, distinct_products, а не безымянный count.

Пропуск в ключе и пропуск в значении — разные проблемы

Пропуск в обязательном order_id означает, что строку нельзя связать с заказом. Для нашего входного контракта это ошибка до groupby; параметр dropna=False не превращает неизвестный идентификатор в корректный заказ.

У необязательного аналитического разреза политика может быть другой. Например, при группировке по необязательному segment:

  • dropna=True исключит строки с неизвестным сегментом из групп;
  • dropna=False сохранит отдельную группу неизвестного сегмента.

Это решение о популяции отчёта. Его нельзя автоматически переносить на обязательные бизнес-ключи.

Пропуск в line_total имеет другой смысл: заказ известен, но одна из его частей пока не оценена. Такой заказ должен остаться в результате вместе с evidence о неполноте.

Сумма известных частей не всегда является суммой целого

Для O1002 единственная line_total неизвестна. Обычный sum может вернуть 0.0, хотя данные не доказывают нулевую стоимость. При частично известном заказе проблема ещё опаснее:

text
известная позиция:     100.0
неизвестная позиция:   NA
обычная сумма:         100.0
полная сумма заказа:   неизвестна

Поэтому артефакт хранит два разных результата:

  • known_amount_total — сумма известных позиций; NA, если не известно ни одной;
  • order_amount — строгая сумма заказа; NA, если неизвестна хотя бы одна позиция.

Рядом сохраняются:

  • line_count — все позиции;
  • known_amount_lines — позиции с известной суммой;
  • missing_amount_lines — позиции с неизвестной суммой.

Так частичная сумма не получает слишком сильное название, а пропуск не превращается в ноль без предметного основания.

Именованная агрегация связывает столбец, функцию и смысл

В pandas рабочий контракт удобно записать именованными агрегациями:

python
grouped = items.groupby("order_id", as_index=False, sort=False).agg(
    line_count=("product_id", "size"),
    known_amount_lines=("line_total", "count"),
    known_amount_total=("line_total", "sum"),
)

Каждая строка внутри .agg(...) отвечает на три вопроса:

text
как назвать результат?
из какого столбца взять значения?
какой агрегат применить?

as_index=False оставляет order_id обычным столбцом. Это удобно для проверки бизнес-ключа и для merge в следующем уроке. Технический индекс результата не становится бизнес-ключом только потому, что pandas умеет помещать group keys в индекс.

sort=False сохраняет порядок первого появления групп в учебном примере. На смысл метрик порядок не влияет; если публикации нужна определённая сортировка, её задают отдельным явным шагом.

Среднее требует правильной единицы анализа

Средняя стоимость товарной позиции и средняя сумма заказа — разные метрики:

text
mean(line_total) до агрегации       -> средняя стоимость позиции
mean(order_amount) после агрегации  -> средняя сумма полного заказа

Нельзя назвать первое значение average_order_value: заказы с большим числом позиций получат больший вес.

Опасно и простое среднее групповых средних. Если одна группа содержит два значения 10, 20, а другая одно значение 100, то:

text
среднее первой группы       = 15
среднее второй группы       = 100
среднее двух средних        = 57.5
среднее всех трёх значений  = 43.33...

Для повторной агрегации среднего нужны как минимум исходная сумма и знаменатель. В этом уроке достаточно распознать ловушку; полноценные продуктовые метрики появятся позже.

Граница урока

Здесь мы изучаем только уменьшение детализации через встроенные агрегаты.

  • GroupBy.transform сохраняет число строк и будет нужен для групповых признаков позже.
  • GroupBy.filter отбирает целые группы и не заменяет построчную маску 03/03.
  • GroupBy.apply нужен для нестандартных задач, но не является стартовой точкой.
  • observed относится к категориальным ключам; категории появятся в 03/09.
  • временные группировки появятся после дат и часовых поясов в 03/08.
  • объединение order-level результата с orders относится к следующему уроку 03/06.

Соберите это

Сначала воспроизведите агрегацию без pandas. Используйте те же пять строк, но представьте их списком словарей:

python
rows = [
    {"order_id": "O1001", "line_total": 800.0},
    {"order_id": "O1001", "line_total": 800.0},
    {"order_id": "O1002", "line_total": None},
    {"order_id": "O1003", "line_total": 75.5},
    {"order_id": "O1004", "line_total": 0.0},
]

До кода предскажите:

  1. сколько групп получится;
  2. сколько строк будет у O1001;
  3. чем должны различаться O1002 и O1004;
  4. какой столбец станет ключом результата.

Шаг 1. Разделите строки по ключу

python
groups = {}

for row in rows:
    order_id = row["order_id"]
    groups.setdefault(order_id, []).append(row["line_total"])

Теперь структура выглядит так:

text
O1001 -> [800.0, 800.0]
O1002 -> [None]
O1003 -> [75.5]
O1004 -> [0.0]

Этот словарь делает наблюдаемым шаг split. pandas построит группы эффективнее, но их предметный смысл останется тем же.

Шаг 2. Посчитайте полноту и известную сумму

python
control = []

for order_id, values in groups.items():
    known = [value for value in values if value is not None]
    line_count = len(values)
    known_count = len(known)
    missing_count = line_count - known_count
    known_total = sum(known) if known else None
    order_total = known_total if missing_count == 0 else None

    control.append(
        {
            "order_id": order_id,
            "line_count": line_count,
            "known_amount_lines": known_count,
            "missing_amount_lines": missing_count,
            "known_amount_total": known_total,
            "order_amount": order_total,
        }
    )

Получится четыре записи:

text
O1001 -> lines=2, known=2, missing=0, known_total=1600.0, order=1600.0
O1002 -> lines=1, known=0, missing=1, known_total=None,   order=None
O1003 -> lines=1, known=1, missing=0, known_total=75.5,  order=75.5
O1004 -> lines=1, known=1, missing=0, known_total=0.0,   order=0.0

Ручная версия медленнее groupby, но показывает все решения: ключ, состав групп, знаменатели и правило неизвестной суммы. Готовая функция manual_order_rollup находится в артефакте и используется как независимый контроль pandas-результата.

Используйте это

Шаг 1. Создайте уже типизированный вход

Не читайте и не очищайте CSV повторно. В реальном pipeline это результат 03/02 и 03/04:

Шаг 2. Выполните именованные агрегации

python
result = (
    items.groupby("order_id", as_index=False, sort=False)
    .agg(
        line_count=("product_id", "size"),
        known_amount_lines=("line_total", "count"),
        known_amount_total=("line_total", "sum"),
    )
    .astype(
        {
            "order_id": "string",
            "line_count": "Int64",
            "known_amount_lines": "Int64",
            "known_amount_total": "Float64",
        }
    )
)

На этом шаге pandas уже создал одну строку на заказ, но политика неизвестной суммы ещё не завершена.

Шаг 3. Сделайте неполноту наблюдаемой

python
result["missing_amount_lines"] = (
    result["line_count"] - result["known_amount_lines"]
).astype("Int64")

Эта разность связывает size и count:

text
все строки - строки с известной суммой = строки с неизвестной суммой

Шаг 4. Не выдавайте пустую или частичную сумму за полный итог

Сначала исправьте полностью пустую сумму:

python
result["known_amount_total"] = result["known_amount_total"].mask(
    result["known_amount_lines"].eq(0),
    pd.NA,
)

Затем опубликуйте строгую сумму заказа только для полных групп:

python
result["order_amount"] = result["known_amount_total"].mask(
    result["missing_amount_lines"].gt(0),
    pd.NA,
).astype("Float64")

Здесь mask из 03/04 используется уже после агрегации: он не меняет группы, а явно задаёт предметную политику выходного значения.

Шаг 5. Проверьте новый grain и reconciliation

python

assert result["known_amount_lines"].sum() == items["line_total"].notna().sum()
assert result["missing_amount_lines"].sum() == items["line_total"].isna().sum()

Для пяти учебных строк подтвердите четыре группы и значения:

text
O1001 -> 2 строки, 2 известны, 0 пропущено, полный итог 1600.0
O1002 -> 1 строка, 0 известны, 1 пропущена, полный итог неизвестен
O1003 -> 1 строка, 1 известна, 0 пропущено, полный итог 75.5
O1004 -> 1 строка, 1 известна, 0 пропущено, полный итог 0.0

Ручной manual_order_rollup и pandas-функция должны совпасть по каждой группе, а не только по общей сумме.

Входной контракт

  • объект является DataFrame;
  • столбцы order_id, product_id, line_total существуют ровно по одному разу;
  • идентификаторы имеют dtype string, не пропущены и не пусты;
  • line_total уже имеет dtype Float64;
  • пара (order_id, product_id) уникальна;
  • повтор отдельного order_id разрешён;
  • входной объект не изменяется.

Выходной контракт

  • одна строка соответствует одному order_id;
  • order_id уникален и остаётся обычным столбцом;
  • счётчики имеют dtype Int64;
  • суммы имеют dtype Float64;
  • нулевая сумма остаётся 0.0;
  • полностью или частично неизвестный итог заказа остаётся pd.NA;
  • пустой типизированный вход возвращает пустой типизированный результат без фиктивной группы.

Reconciliation-инварианты

text
sum(line_count) == число входных строк
sum(known_amount_lines) == число известных line_total
sum(missing_amount_lines) == число пропущенных line_total
line_count == known_amount_lines + missing_amount_lines для каждой группы
число строк результата == число уникальных order_id

Для полных заказов дополнительно:

text
known_amount_total == order_amount

Для неполных заказов:

text
missing_amount_lines > 0
order_amount is NA

Поставьте результат

Готовый артефакт находится в phases/03-pandas/05-groupby/outputs/order_item_rollup.py.

Его публичный контракт:

python
aggregate_order_items(frame: pd.DataFrame) -> pd.DataFrame

Функция:

  • принимает уже типизированный результат построчных преобразований 03/04;
  • проверяет входной grain (order_id, product_id);
  • разрешает законный повтор order_id;
  • сворачивает позиции до одной строки на заказ;
  • сохраняет число всех, известных и неизвестных позиций;
  • разделяет сумму известных частей и строгую полную сумму заказа;
  • возвращает явные nullable-dtype;
  • проверяет reconciliation строк;
  • не изменяет входной объект;
  • не читает CSV, не очищает данные, не фильтрует популяцию и не использует GroupBy.apply.

Такой order-level результат можно безопасно передать следующему уроку. В 03/06 он будет объединён с orders, а cardinality one-to-one станет проверяемым обещанием, а не предположением.

Ключевые термины

Термин Распространённое заблуждение Точное значение
Group key Любой удобный столбец Поле или набор полей, задающий группы и grain результата
Aggregation Любая операция после groupby Операция, уменьшающая каждую группу до одного или нескольких итогов
size Число известных значений Число строк группы независимо от пропусков измеряемого столбца
count Число строк Число непропущенных значений выбранного столбца
nunique То же, что count Число разных известных значений
Named aggregation Косметическое переименование Контракт «имя результата -> исходный столбец -> агрегат»
Reconciliation Повторить ту же формулу Независимо сверить строки, знаменатели и totals до и после изменения grain
Complete total Сумма известных частей Итог, для которого известны все обязательные составляющие

Quiz · 5 вопроса

Проверьте себя

Ответьте после практики. Объяснение появится сразу после выбора.

01

В order_items две строки имеют order_id O1001, но разные product_id. Что это означает до groupby?

Повтор ключа оценивают относительно объявленного grain. Для товарной позиции отдельный order_id не является ключом строки.

02

В группе две позиции с line_total [100.0, pd.NA]. Чему равны size и count(line_total)?

size считает все строки группы, а count выбранного столбца — только его непропущенные значения.

03

Что произойдёт, если корректные товарные позиции сгруппировать одновременно по order_id и product_id?

Составной ключ уже идентифицирует исходную позицию, поэтому группы почти совпадут со строками и перехода к grain заказа не произойдёт.

04

У заказа одна известная позиция 100.0 и одна неизвестная. Какой результат соответствует строгой политике полноты?

Сумму известных частей можно показать как evidence, но полная сумма заказа неизвестна, пока отсутствует хотя бы одна обязательная часть.

05

Какой инвариант проверяет, что позиции не потерялись и не размножились при переходе к grain заказа?

Каждая входная позиция должна войти ровно в одну группу, поэтому суммы размеров групп и число исходных строк обязаны совпасть.

from __future__ import annotations

import importlib.util
import unittest
from pathlib import Path

import pandas as pd

ROOT = Path(__file__).resolve().parents[1]
ARTIFACT = ROOT / "outputs" / "order_item_rollup.py"
SPEC = importlib.util.spec_from_file_location("order_item_rollup", ARTIFACT)
if SPEC is None or SPEC.loader is None:
    raise RuntimeError(f"cannot load {ARTIFACT}")
ROLLUP = importlib.util.module_from_spec(SPEC)
SPEC.loader.exec_module(ROLLUP)


def typed_items() -> pd.DataFrame:
    return pd.DataFrame(
        {
            "order_id": ["O1001", "O1001", "O1002", "O1003", "O1004"],
            "product_id": ["P01", "P02", "P03", "P04", "P05"],
            "line_total": [800.0, 800.0, pd.NA, 75.5, 0.0],
        },
        index=["item-a", "item-b", "item-c", "item-d", "item-e"],
    ).astype(
        {
            "order_id": "string",
            "product_id": "string",
            "line_total": "Float64",
        }
    )


class OrderItemRollupTest(unittest.TestCase):
    def setUp(self) -> None:
        self.items = typed_items()

    def test_repeated_order_id_is_valid_at_line_item_grain(self) -> None:
        result = ROLLUP.aggregate_order_items(self.items)
        self.assertEqual(result["order_id"].tolist(), ["O1001", "O1002", "O1003", "O1004"])

    def test_result_has_one_row_per_order(self) -> None:
        result = ROLLUP.aggregate_order_items(self.items)
        self.assertFalse(result["order_id"].duplicated().any())
        self.assertEqual(len(result), self.items["order_id"].nunique())

    def test_complete_order_gets_strict_total(self) -> None:
        result = ROLLUP.aggregate_order_items(self.items).set_index("order_id")
        self.assertEqual(result.loc["O1001", "line_count"], 2)
        self.assertEqual(result.loc["O1001", "known_amount_lines"], 2)
        self.assertEqual(result.loc["O1001", "missing_amount_lines"], 0)
        self.assertEqual(result.loc["O1001", "known_amount_total"], 1600.0)
        self.assertEqual(result.loc["O1001", "order_amount"], 1600.0)

    def test_all_missing_amounts_do_not_become_zero(self) -> None:
        result = ROLLUP.aggregate_order_items(self.items).set_index("order_id")
        self.assertEqual(result.loc["O1002", "line_count"], 1)
        self.assertEqual(result.loc["O1002", "known_amount_lines"], 0)
        self.assertEqual(result.loc["O1002", "missing_amount_lines"], 1)
        self.assertTrue(pd.isna(result.loc["O1002", "known_amount_total"]))
        self.assertTrue(pd.isna(result.loc["O1002", "order_amount"]))

    def test_partial_known_sum_is_visible_but_order_total_is_unknown(self) -> None:
        partial = pd.DataFrame(
            {
                "order_id": pd.Series(["O1", "O1"], dtype="string"),
                "product_id": pd.Series(["P1", "P2"], dtype="string"),
                "line_total": pd.Series([100.0, pd.NA], dtype="Float64"),
            }
        )
        row = ROLLUP.aggregate_order_items(partial).iloc[0]
        self.assertEqual(row["known_amount_total"], 100.0)
        self.assertEqual(row["missing_amount_lines"], 1)
        self.assertTrue(pd.isna(row["order_amount"]))

    def test_zero_is_known_amount_not_missing_value(self) -> None:
        result = ROLLUP.aggregate_order_items(self.items).set_index("order_id")
        self.assertEqual(result.loc["O1004", "known_amount_lines"], 1)
        self.assertEqual(result.loc["O1004", "order_amount"], 0.0)

    def test_manual_control_matches_pandas_values(self) -> None:
        manual = ROLLUP.manual_order_rollup(self.items.to_dict("records"))
        pandas_result = ROLLUP.aggregate_order_items(self.items).to_dict("records")
        self.assertEqual(
            [row["order_id"] for row in manual],
            [row["order_id"] for row in pandas_result],
        )
        for expected, actual in zip(manual, pandas_result, strict=True):
            self.assertEqual(expected["line_count"], actual["line_count"])
            self.assertEqual(expected["known_amount_lines"], actual["known_amount_lines"])
            self.assertEqual(expected["missing_amount_lines"], actual["missing_amount_lines"])
            for column in ("known_amount_total", "order_amount"):
                if expected[column] is None:
                    self.assertTrue(pd.isna(actual[column]))
                else:
                    self.assertEqual(expected[column], actual[column])

    def test_counts_reconcile_with_source_rows(self) -> None:
        result = ROLLUP.aggregate_order_items(self.items)
        self.assertEqual(int(result["line_count"].sum()), len(self.items))
        self.assertEqual(
            int(result["known_amount_lines"].sum()),
            int(self.items["line_total"].notna().sum()),
        )
        self.assertEqual(
            int(result["missing_amount_lines"].sum()),
            int(self.items["line_total"].isna().sum()),
        )

    def test_output_dtypes_are_explicit(self) -> None:
        result = ROLLUP.aggregate_order_items(self.items)
        self.assertEqual(str(result["order_id"].dtype), "string")
        self.assertEqual(str(result["line_count"].dtype), "Int64")
        self.assertEqual(str(result["known_amount_lines"].dtype), "Int64")
        self.assertEqual(str(result["missing_amount_lines"].dtype), "Int64")
        self.assertEqual(str(result["known_amount_total"].dtype), "Float64")
        self.assertEqual(str(result["order_amount"].dtype), "Float64")

    def test_input_is_not_modified(self) -> None:
        before = self.items.copy(deep=True)
        ROLLUP.aggregate_order_items(self.items)
        pd.testing.assert_frame_equal(self.items, before)

    def test_duplicate_line_item_key_is_rejected(self) -> None:
        duplicate = pd.concat([self.items, self.items.iloc[[0]]], ignore_index=True)
        with self.assertRaisesRegex(ROLLUP.AggregationContractError, "one row per"):
            ROLLUP.aggregate_order_items(duplicate)

    def test_missing_or_blank_business_key_is_rejected(self) -> None:
        for column, value in (("order_id", pd.NA), ("product_id", "  ")):
            with self.subTest(column=column, value=value):
                broken = self.items.copy()
                broken.loc["item-a", column] = value
                with self.assertRaisesRegex(ROLLUP.AggregationContractError, column):
                    ROLLUP.aggregate_order_items(broken)

    def test_wrong_dtype_is_rejected_instead_of_reparsed(self) -> None:
        broken = self.items.astype({"line_total": "float64"})
        with self.assertRaisesRegex(ROLLUP.AggregationContractError, "dtype contract"):
            ROLLUP.aggregate_order_items(broken)

    def test_missing_column_is_rejected(self) -> None:
        with self.assertRaisesRegex(ROLLUP.AggregationContractError, "missing columns"):
            ROLLUP.aggregate_order_items(self.items.drop(columns="product_id"))

    def test_empty_typed_input_returns_empty_typed_result(self) -> None:
        result = ROLLUP.aggregate_order_items(self.items.iloc[:0])
        self.assertEqual(result.columns.tolist(), ROLLUP.OUTPUT_COLUMNS)
        self.assertTrue(result.empty)
        self.assertEqual(str(result["order_amount"].dtype), "Float64")

from __future__ import annotations

from collections.abc import Iterable, Mapping
from typing import Any

import pandas as pd


class AggregationContractError(ValueError):
    """Raised when line items cannot be rolled up without changing their meaning."""


REQUIRED_DTYPES = {
    "order_id": "string",
    "product_id": "string",
    "line_total": "Float64",
}

OUTPUT_COLUMNS = [
    "order_id",
    "line_count",
    "known_amount_lines",
    "missing_amount_lines",
    "known_amount_total",
    "order_amount",
]


def _is_missing_scalar(value: object) -> bool:
    missing = pd.isna(value)
    if not isinstance(missing, bool):
        raise AggregationContractError("line_total must contain scalar values")
    return missing


def manual_order_rollup(
    rows: Iterable[Mapping[str, Any]],
) -> list[dict[str, Any]]:
    """Build a transparent order-level control result without ``groupby``."""

    groups: dict[Any, dict[str, Any]] = {}
    for row in rows:
        order_id = row["order_id"]
        group = groups.setdefault(
            order_id,
            {
                "order_id": order_id,
                "line_count": 0,
                "known_amount_lines": 0,
                "missing_amount_lines": 0,
                "known_amount_total": 0.0,
            },
        )
        group["line_count"] += 1

        value = row["line_total"]
        if _is_missing_scalar(value):
            group["missing_amount_lines"] += 1
        else:
            group["known_amount_lines"] += 1
            group["known_amount_total"] += float(value)

    result: list[dict[str, Any]] = []
    for group in groups.values():
        known_amount = (
            group["known_amount_total"] if group["known_amount_lines"] > 0 else None
        )
        order_amount = known_amount if group["missing_amount_lines"] == 0 else None
        result.append(
            {
                "order_id": group["order_id"],
                "line_count": group["line_count"],
                "known_amount_lines": group["known_amount_lines"],
                "missing_amount_lines": group["missing_amount_lines"],
                "known_amount_total": known_amount,
                "order_amount": order_amount,
            }
        )
    return result


def _validate_input(frame: object) -> pd.DataFrame:
    if not isinstance(frame, pd.DataFrame):
        raise AggregationContractError("frame must be a pandas DataFrame")
    if not frame.columns.is_unique:
        raise AggregationContractError("column labels must be unique")

    missing_columns = sorted(set(REQUIRED_DTYPES) - set(frame.columns))
    if missing_columns:
        raise AggregationContractError(f"missing columns: {missing_columns}")

    wrong_dtypes = {
        column: {"expected": expected, "actual": str(frame[column].dtype)}
        for column, expected in REQUIRED_DTYPES.items()
        if str(frame[column].dtype) != expected
    }
    if wrong_dtypes:
        raise AggregationContractError(
            f"input must already satisfy the dtype contract: {wrong_dtypes}"
        )

    for column in ("order_id", "product_id"):
        blank = frame[column].str.strip().eq("").fillna(False)
        invalid = frame[column].isna() | blank
        if invalid.any():
            labels = frame.index[invalid].tolist()
            raise AggregationContractError(
                f"{column} must be non-missing and non-blank at rows: {labels}"
            )

    duplicated_items = frame.duplicated(["order_id", "product_id"], keep=False)
    if duplicated_items.any():
        labels = frame.index[duplicated_items].tolist()
        raise AggregationContractError(
            "source grain must be one row per (order_id, product_id); "
            f"duplicate rows: {labels}"
        )
    return frame


def aggregate_order_items(frame: pd.DataFrame) -> pd.DataFrame:
    """Roll typed line items up to one row per order.

    ``known_amount_total`` contains the sum of observed ``line_total`` values and remains
    missing when none are known. ``order_amount`` is stricter: it remains missing when at
    least one line amount is unknown. The input object is never modified.
    """

    source = _validate_input(frame)
    result = (
        source.groupby("order_id", as_index=False, sort=False)
        .agg(
            line_count=("product_id", "size"),
            known_amount_lines=("line_total", "count"),
            known_amount_total=("line_total", "sum"),
        )
        .astype(
            {
                "order_id": "string",
                "line_count": "Int64",
                "known_amount_lines": "Int64",
                "known_amount_total": "Float64",
            }
        )
    )

    result["missing_amount_lines"] = (
        result["line_count"] - result["known_amount_lines"]
    ).astype("Int64")
    result["known_amount_total"] = result["known_amount_total"].mask(
        result["known_amount_lines"].eq(0),
        pd.NA,
    )
    result["order_amount"] = result["known_amount_total"].mask(
        result["missing_amount_lines"].gt(0),
        pd.NA,
    ).astype("Float64")
    result = result.loc[:, OUTPUT_COLUMNS]

    if result["order_id"].duplicated().any():
        raise AggregationContractError("result grain must be one row per order_id")
    if int(result["line_count"].sum()) != len(source):
        raise AggregationContractError("line-count reconciliation failed")
    if int(result["known_amount_lines"].sum()) != int(source["line_total"].notna().sum()):
        raise AggregationContractError("known-amount reconciliation failed")
    return result
