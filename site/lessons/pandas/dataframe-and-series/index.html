На что опирается урок

В фазе NumPy вы уже работали с предметными осями, shape, dtype и позиционной индексацией. Вы также сохраняли соответствие связанных массивов одной перестановкой и считали частоты уникальных значений.

Главное ограничение NumPy выглядело так:

text
order_ids[i] <-> amounts[i] <-> segments[i]

NumPy знает позицию i, но не знает, что она означает заказ. Если один массив переставить отдельно от других, значения начнут относиться не к тем наблюдениям.

pandas решает другую задачу: хранит разнородную таблицу с именами столбцов и добавляет метки строк. Эти метки становятся частью вычислений. Но предметный смысл строки pandas по-прежнему не угадывает — его объявляет аналитик.

Проблема

Аналитик открывает orders.csv и видит семь строк:

text
   order_id  user_id  status     amount
0      O1001    U001  paid      1200.00
1      O1002    U001  refunded   800.00
2      O1003    U002   paid      5500.00
...

Числа слева 0, 1, 2, ... выглядят как идентификаторы, но это только автоматически созданные метки строк. Они отвечают на вопрос «где находится строка в этом объекте», а не «какой заказ описывает строка».

Предположим, что при повторной выгрузке заказ O1001 попал в CSV дважды. pandas всё равно создаст уникальные метки 0 и 1 для двух строк. Таблица успешно загрузится, RangeIndex останется уникальным, а сумма amount может оказаться завышенной.

Есть и второй риск. Два ряда значений могут иметь одинаковую длину, но относиться к разным заказам. Если соединить их по позиции, результат будет выглядеть правдоподобно. pandas умеет сопоставлять такие значения по меткам, но аналитик должен понимать, какие именно метки участвуют в операции.

Обе ошибки сводятся к одному вопросу:

Как определить, какие значения относятся друг к другу?

Концепция

DataFrame — двумерная таблица с метками двух осей

У DataFrame две оси:

text
ось строк    -> frame.index
ось столбцов -> frame.columns

shape по-прежнему хранит длины осей в том же порядке:

text
orders.shape == (число строк, число столбцов)

Но теперь у осей есть явные метки. Для таблицы заказов:

  • orders.index содержит метки строк;
  • orders.columns содержит order_id, user_id, ordered_at, status, currency, amount;
  • каждый столбец может иметь собственный dtype, потому что идентификаторы, даты, категории и суммы имеют разный смысл.

В этом уроке мы только наблюдаем физические типы после чтения CSV. Как объявлять семантические типы и что делать с пропусками, разберём в 03/02.

Один столбец DataFrame — это Series

Выражение:

python
amount = orders["amount"]

возвращает не новый DataFrame, а одномерный объект Series. У него есть:

  • значения сумм;
  • собственный dtype;
  • имя amount;
  • индекс строк, унаследованный от orders.

Связь можно представить так:

text
orders.index:     0       1       2
amount.index:     0       1       2
amount.values: 1200.0   800.0  5500.0

Индекс связывает каждое значение Series со строкой таблицы. Это важное отличие от обычного списка и от массива, где соответствие задаётся только позицией.

Index — координата pandas, но не обязательно бизнес-ключ

Index — набор меток оси. После обычного read_csv pandas создаёт RangeIndex:

text
0, 1, 2, ..., n - 1

Он удобен для адресации строк внутри объекта, но не сообщает, что означает одна строка. В частности, уникальный RangeIndex не доказывает уникальность order_id.

Бизнес-ключ и индекс могут совпадать, но не обязаны. В этом уроке order_id остаётся обычным столбцом. pandas не догадается использовать его для выравнивания только потому, что название похоже на идентификатор.

Grain сначала объявляется, затем проверяется

Grain — человеческое описание смысла одной строки. Для orders договорённость звучит так:

text
одна строка -> один заказ
кандидатный ключ -> order_id

Порядок здесь важен:

  1. Из предметной модели или контракта источника мы узнаём, что строка означает заказ.
  2. Выбираем столбец order_id как кандидатный ключ этой сущности.
  3. Проверяем, что ключ существует, не пуст и не повторяется.

Уникальность ключа поддерживает объявленный grain, но не доказывает весь смысл данных. Например, она не обнаружит потерянные при выгрузке заказы или ошибочно выбранную сущность. Для первого preflight достаточно честно назвать проверяемое обещание:

В этой таблице каждая имеющаяся строка получает один непустой уникальный order_id.

Для order_items одна строка означает уже товарную позицию заказа. Поэтому отдельный order_id там повторяется законно, а кандидатный ключ составной:

text
order_id + product_id

Арифметика Series выравнивает значения по меткам

Рассмотрим две Series:

text
price:    order-a=100, order-b=200
discount: order-b=10,  order-c=20

pandas не вычитает первый элемент из первого только потому, что оба стоят первыми. Сначала библиотека сопоставляет метки индекса:

Метка price discount price - discount
order-a 100 нет значения пропуск
order-b 200 10 190
order-c нет значения 20 пропуск

Индекс результата содержит объединение меток:

text
order-a, order-b, order-c

Число определено только для общей метки order-b. Метки без пары не удаляются молча: они остаются в результате с пропуском. Политику обработки таких пропусков мы выберем в следующем уроке; здесь важно заметить причину их появления.

Граница этого урока

Сейчас мы читаем объекты и выполняем арифметику. Правила выбора строк, loc, iloc, булевы маски, изменение таблицы и Copy-on-Write будут центральной темой 03/03. До этого не рассчитывайте изменить исходный DataFrame через побочный эффект операции над производным объектом.

Соберите это

Сначала воспроизведите два механизма без pandas.

Шаг 1. Отделите метку строки от идентификатора заказа

Запишите две строки с одним бизнес-ключом:

Метка строки order_id amount
0 O1001 1200
1 O1001 1200

Метки 0 и 1 уникальны, но заказ один и тот же. Следовательно:

text
уникальный технический индекс != уникальный бизнес-ключ

Подсчёт частот из фазы NumPy даст O1001 -> 2. Именно это наблюдение должна сохранить автоматическая проверка.

Шаг 2. Сопоставьте значения по меткам

Представьте два ряда как словари:

python
price_by_order = {"order-a": 100, "order-b": 200}
discount_by_order = {"order-b": 10, "order-c": 20}

Сначала получите все встречающиеся метки, затем для каждой проверьте наличие пары:

python
labels = sorted(price_by_order.keys() | discount_by_order.keys())
manual_result = {}

for label in labels:
    if label in price_by_order and label in discount_by_order:
        manual_result[label] = price_by_order[label] - discount_by_order[label]
    else:
        manual_result[label] = None

Ожидаемый результат:

python
assert manual_result == {
    "order-a": None,
    "order-b": 190,
    "order-c": None,
}

Этот прозрачный цикл показывает правило. Series выполняет то же сопоставление автоматически и сохраняет метки в результате.

python

Проверьте не только единственное число, но и структуру результата:

python
assert net.index.tolist() == ["order-a", "order-b", "order-c"]

Сначала мы проверили порядок меток, поэтому позиции в tolist() имеют однозначный смысл. Выбор по меткам через loc и изменение строк будут подробно разобраны в 03/03.

Шаг 4. Проверьте объявленный бизнес-ключ

python
key = orders["order_id"]

null_key_rows = int(key.isna().sum())
duplicate_key_rows = int(
    orders.duplicated(["order_id"], keep=False).sum()
)

assert null_key_rows == 0
assert duplicate_key_rows == 0

keep=False отмечает все строки, участвующие в повторе. Если один ключ встретился дважды, duplicate_key_rows будет равен 2, а не 1.

Для составного ключа передайте оба столбца одной проверке:

python
items = pd.read_csv(DATA / "order_items.csv")
duplicate_item_rows = int(
    items.duplicated(["order_id", "product_id"], keep=False).sum()
)
assert duplicate_item_rows == 0

Отдельный order_id в order_items повторяется правильно: один заказ содержит несколько товарных позиций. Проверяется именно ключ заявленного grain.

Сломайте это

Ошибка 1. Принять RangeIndex за ключ

Добавьте второй O1001. pandas выдаст новой строке новую техническую метку, поэтому orders.index.is_unique останется True. Проверка order_id должна стать красной.

Ошибка 2. Сложить значения по позиции

Списки [100, 200] и [10, 20] дают [90, 180], если вычитать элементы по позиции. Но Series из примера описывают разные наборы заказов. Единственное осмысленное совпадение — order-b, поэтому результат по меткам другой.

Ошибка 3. Посмотреть только на длину

Две Series могут иметь одинаковую длину и разные метки. Совпадающий shape не гарантирует, что операция сопоставит нужные наблюдения.

Ошибка 4. Считать любой повтор ключа дефектом

В order_items повтор order_id ожидаем: один заказ имеет несколько товаров. Ошибкой будет повтор составной пары order_id, product_id при объявленном grain товарной позиции.

Ошибка 5. Объявить grain по результату проверки

Инспектор может подтвердить непустоту и уникальность переданных столбцов, но не знает предметную область. Сначала получите описание таблицы от источника, затем передайте кандидатный ключ в проверку.

Поставьте результат

outputs/dataframe_inspector.py — небольшой preflight таблицы перед анализом. Он:

  • принимает CSV и обязательный список ключевых столбцов;
  • показывает форму, столбцы и ограниченный preview технического индекса;
  • считает строки с пустым или повторяющимся объявленным ключом;
  • поддерживает составной ключ;
  • возвращает код 0 для корректного контракта, 1 для нарушения данных и 2 для невалидного запуска.

Артефакт намеренно не приводит типы, не исправляет пропуски, не меняет индекс и не выбирает grain за аналитика. Эти решения относятся к следующим урокам.

Ключевые термины

Термин Распространённое заблуждение Точное значение

RangeIndex Номер бизнес-сущности Автоматическая последовательность технических меток строк

Что верно для amount = orders["amount"]?

Выбор одного столбца возвращает Series. Её индекс связывает значения с теми же строками, что и в исходном DataFrame.

02

У price есть метки A,B, а у discount — B,C. Какова структура price - discount?

Series выравниваются по меткам. Результат сохраняет объединение индексов, а метки без пары получают пропуск.

03

Проверка показала, что order_id существует, не пуст и уникален. Какой вывод обоснован?

Непустота и уникальность поддерживают контракт выбранного ключа, но не определяют предметный смысл строки и не обнаруживают потерянные сущности.

def load_module(name: str, path: Path):
    spec = importlib.util.spec_from_file_location(name, path)
    if spec is None or spec.loader is None:
        raise RuntimeError(f"cannot load {path}")
    module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(module)
    return module


INSPECTOR = load_module("dataframe_inspector", ARTIFACT)
LESSON = load_module("dataframe_series_lesson", EXAMPLE)


class DataFrameInspectorTest(unittest.TestCase):
    def test_report_exposes_table_axes_and_limited_index_preview(self) -> None:
        frame = pd.DataFrame(
            {
                "order_id": [f"O{index}" for index in range(7)],
                "amount": list(range(7)),
            }
        )
        report = INSPECTOR.inspect_dataframe(frame, ["order_id"])

        self.assertEqual(report["table"]["shape"], [7, 2])
        self.assertEqual(report["table"]["columns"], ["order_id", "amount"])
        self.assertEqual(report["table"]["index"]["type"], "RangeIndex")
        self.assertEqual(report["table"]["index"]["preview"], [0, 1, 2, 3, 4])
        self.assertNotIn("alignment_example", report)
        self.assertNotIn("copy_on_write", report)

    def test_selected_column_is_series_with_dataframe_index(self) -> None:
        frame = pd.DataFrame({"order_id": ["A", "B"], "amount": [10, 20]})
        relationship = LESSON.dataframe_series_relationship(frame, "amount")

        self.assertEqual(relationship["frame_type"], "DataFrame")
        self.assertEqual(relationship["series_type"], "Series")
        self.assertEqual(relationship["series_name"], "amount")
        self.assertTrue(relationship["same_row_index"])

    def test_series_alignment_preserves_union_of_labels(self) -> None:
        result = LESSON.build_alignment_example()

        self.assertEqual(result.index.tolist(), ["order-a", "order-b", "order-c"])
        values = result.tolist()
        self.assertTrue(pd.isna(values[0]))
        self.assertEqual(values[1], 190)
        self.assertTrue(pd.isna(values[2]))

    def test_unique_business_key_passes_declared_grain_contract(self) -> None:
        frame = pd.DataFrame({"order_id": ["A", "B"]})
        report = INSPECTOR.inspect_dataframe(frame, ["order_id"])

        self.assertTrue(report["declared_grain"]["valid"])

    def test_unique_range_index_does_not_hide_duplicate_business_key(self) -> None:
        frame = pd.DataFrame({"order_id": ["A", "A"]})
        report = INSPECTOR.inspect_dataframe(frame, ["order_id"])

        self.assertTrue(report["table"]["index"]["is_unique"])
        self.assertFalse(report["declared_grain"]["valid"])
        self.assertEqual(report["declared_grain"]["duplicate_key_rows"], 2)

    def test_null_business_key_is_invalid(self) -> None:
        frame = pd.DataFrame({"order_id": ["A", None]})
        report = INSPECTOR.inspect_dataframe(frame, ["order_id"])

        self.assertFalse(report["declared_grain"]["valid"])
        self.assertEqual(report["declared_grain"]["null_key_rows"], 1)

    def test_composite_key_is_checked_as_one_contract(self) -> None:
        valid = pd.DataFrame(
            {
                "order_id": ["A", "A"],
                "product_id": ["P1", "P2"],
            }
        )
        duplicate_pair = pd.DataFrame(
            {
                "order_id": ["A", "A", "A"],
                "product_id": ["P1", "P2", "P1"],
            }
        )

        valid_report = INSPECTOR.inspect_dataframe(valid, ["order_id", "product_id"])
        invalid_report = INSPECTOR.inspect_dataframe(
            duplicate_pair,
            ["order_id", "product_id"],
        )

        self.assertTrue(valid_report["declared_grain"]["valid"])
        self.assertFalse(invalid_report["declared_grain"]["valid"])
        self.assertEqual(invalid_report["declared_grain"]["duplicate_key_rows"], 2)

    def test_missing_or_empty_key_contract_is_rejected(self) -> None:
        frame = pd.DataFrame({"id": [1]})

        with self.assertRaisesRegex(INSPECTOR.TableContractError, "at least one"):
            INSPECTOR.inspect_dataframe(frame, [])

            INSPECTOR.inspect_dataframe(frame, ["order_id"])

    if spec is None or spec.loader is None:

def build_alignment_example() -> pd.Series:
    price = pd.Series(
        [100, 200],
        index=["order-a", "order-b"],
        name="price",
    )
    discount = pd.Series(
        [10, 20],
        index=["order-b", "order-c"],
        name="discount",
    )
    return price - discount


def dataframe_series_relationship(frame: pd.DataFrame, column: str) -> dict[str, Any]:
    series = frame[column]
    return {
        "frame_type": type(frame).__name__,
        "series_type": type(series).__name__,
        "series_name": series.name,
        "same_row_index": bool(series.index.equals(frame.index)),
    }


def main() -> None:
    inspector = load_artifact()
    orders = inspector.load_table(DATA)

    print("DataFrame -> Series")
    print(dataframe_series_relationship(orders, "amount"))
    print()

    print("Label alignment")
    print(build_alignment_example().to_string())
    print()

    print("Declared grain preflight")
    report = inspector.inspect_dataframe(orders, ["order_id"])
    print(inspector.render_report(report), end="")


if __name__ == "__main__":
    main()

from __future__ import annotations

import argparse
import json
from pathlib import Path
from typing import Any

import pandas as pd


class TableContractError(ValueError):
    """Raised when the declared table contract cannot be checked."""


def load_table(path: str | Path) -> pd.DataFrame:
    source = Path(path)
    if not source.is_file():
        raise TableContractError(f"input file does not exist: {source}")
    return pd.read_csv(source)


def inspect_dataframe(frame: pd.DataFrame, keys: list[str]) -> dict[str, Any]:
    if not isinstance(frame, pd.DataFrame):
        raise TableContractError("expected a pandas DataFrame")
    if not keys:
        raise TableContractError("at least one grain key is required")

    missing_keys = [key for key in keys if key not in frame.columns]
    if missing_keys:
        raise TableContractError(f"missing key columns: {missing_keys}")

    duplicate_key_rows = int(frame.duplicated(keys, keep=False).sum())
    null_key_rows = int(frame[keys].isna().any(axis=1).sum())
    return {
        "table": {
            "object_type": type(frame).__name__,
            "shape": list(frame.shape),
            "columns": frame.columns.tolist(),
            "index": {
                "type": type(frame.index).__name__,
                "name": frame.index.name,
                "is_unique": bool(frame.index.is_unique),
                "preview": frame.index[:5].tolist(),
            },
        },
        "declared_grain": {
            "keys": keys,
            "null_key_rows": null_key_rows,
            "duplicate_key_rows": duplicate_key_rows,
            "valid": duplicate_key_rows == 0 and null_key_rows == 0,
        },
    }


def render_report(report: dict[str, Any]) -> str:
    return json.dumps(report, ensure_ascii=False, indent=2) + "\n"


def main(argv: list[str] | None = None) -> int:
    parser = argparse.ArgumentParser(
        description="Inspect a DataFrame and its declared grain",
    )
    parser.add_argument("input", type=Path)
    parser.add_argument(
        "--keys",
        required=True,
        help="Comma-separated columns that identify one observation",
    )
    args = parser.parse_args(argv)

    try:
        frame = load_table(args.input)
        keys = [item.strip() for item in args.keys.split(",") if item.strip()]
        report = inspect_dataframe(frame, keys)
    except (OSError, TableContractError, pd.errors.ParserError) as error:
        parser.error(str(error))

    print(render_report(report), end="")
    return 0 if report["declared_grain"]["valid"] else 1


if __name__ == "__main__":
    raise SystemExit(main())
