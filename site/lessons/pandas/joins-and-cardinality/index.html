Проблема

Есть таблица заказов:

order_id amount
O1001 1200.0
O1002 800.0
O1003 5500.0

Её grain — один заказ. order_id уникален.

Есть и сырые товарные позиции:

order_id product_id line_total
O1001 P01 800.0
O1001 P02 800.0
O1002 P03 300.0

Их grain — одна позиция заказа. Отдельный order_id закономерно повторяется.

Если сразу соединить эти таблицы, O1001 найдёт две подходящие строки справа:

order_id amount product_id line_total
O1001 1200.0 P01 800.0
O1001 1200.0 P02 800.0
O1002 800.0 P03 300.0
O1003 5500.0 пропуск пропуск

merge сработал правильно: одна строка заказа O1001 соответствует двум позициям. Но grain результата теперь — товарная позиция, а amount=1200.0 повторён дважды. Если после этого посчитать result["amount"].sum(), получится правдоподобная, но завышенная сумма.

Ошибка не в pandas и не обязательно в самой связи one-to-many. Ошибка в несогласованных решениях:

  • хотели итог на уровне заказа;
  • объединили заказ с таблицей более детального grain;
  • затем использовали order-level метрику так, будто она не размножилась.

Исключения может не быть, все ключи могут существовать, а число будет неверным. Поэтому проверка объединения начинается до вызова merge.

Концепция

Что делает join обычными словами

У двух таблиц есть поле или набор полей с общим смыслом. Для каждого значения этого ключа объединение ищет подходящие строки с другой стороны и складывает их столбцы в одну строку результата.

В выражении:

python
left.merge(right, on=["order_id"], ...)
  • left — левая таблица;
  • right — правая таблица;
  • order_id — правило сопоставления;
  • how — какие ключи сохранить при отсутствии пары;
  • validate — какие повторы ключа разрешены с каждой стороны.

Технический индекс строк не участвует в сопоставлении, если его явно не выбрали как ключ. В этом уроке ключи остаются обычными столбцами, как в результате 03/05.

Cardinality — это отношение частот ключа

Cardinality, или кардинальность связи, отвечает на вопрос: сколько строк с одним значением ключа может находиться слева и справа.

Направление всегда читается относительно конкретного вызова left.merge(right):

Cardinality Слева Справа Пример
one-to-one ключ уникален ключ уникален orders + order_rollup по order_id
one-to-many ключ уникален ключ может повторяться orders + сырые order_items
many-to-one ключ может повторяться ключ уникален orders + users по user_id
many-to-many ключ может повторяться ключ может повторяться события + кампании по слишком грубому ключу

Если поменять таблицы местами, one-to-many превратится в many-to-one. Сама предметная связь не изменилась; изменилось направление чтения.

Cardinality не является типом join. one-to-one можно выполнить и как left, и как inner, и как outer. Параметры отвечают на разные вопросы:

text
cardinality -> сколько строк может соответствовать одному ключу с каждой стороны
how         -> какие ключи без пары должны остаться в результате
grain       -> что означает одна строка результата

Частоты ключа позволяют предсказать число строк

Обозначим:

  • f_left(k) — сколько раз ключ k встречается слева;
  • f_right(k) — сколько раз ключ k встречается справа.

Для общего ключа число строк inner join равно произведению частот:

text
f_left(k) * f_right(k)

А общее число строк:

text
sum по общим ключам: f_left(k) * f_right(k)

Почему произведение? Каждая строка слева соединяется с каждой подходящей строкой справа. Если ключ встретился два раза слева и три раза справа, он создаст 2 * 3 = 6 строк. Это маленькое декартово произведение внутри одного ключа.

Для unmatched-ключа произведение равно нулю, но результат зависит от how:

  • inner не сохранит такой ключ;
  • left сохранит каждую unmatched-строку слева;
  • right сохранит каждую unmatched-строку справа;
  • outer сохранит unmatched-строки обеих сторон.

Many-to-many особенно опасен, но не магически неверен

Связь many-to-many создаёт произведение частот по каждому ключу. Например:

text
2 строки слева * 3 строки справа = 6 строк результата

Иногда это и есть требуемая предметная операция. Но для аналитической витрины такая связь часто означает, что:

  • выбран слишком грубый ключ;
  • пропущена часть составного ключа;
  • одна из сторон должна была быть агрегирована;
  • в справочнике появился дубликат;
  • нужна промежуточная таблица связей.

validate="many_to_many" в pandas не проверяет уникальность ни одной стороны. Поэтому наш безопасный артефакт сознательно не принимает этот режим: сначала нужно сформулировать более сильный контракт или отдельно доказать ожидаемое размножение строк.

how определяет сохраняемое множество ключей

Пусть слева есть O1, O2, O3, а справа — O2, O3, O4.

how Какие ключи попадут в результат Типичный вопрос
inner только O2, O3 Какие объекты присутствуют в обеих таблицах?
left все O1, O2, O3 Как обогатить основную левую таблицу?
right все O2, O3, O4 Как сохранить полный список правой таблицы?
outer O1, O2, O3, O4 Как диагностировать расхождение покрытий?

Для сборки order-level витрины orders является основной популяцией, поэтому нужен left join. Заказ без свёртки позиций остаётся видимым как left_only; его судьба решается явно после диагностики.

inner здесь опасен: он тихо удалит заказы без позиций и изменит популяцию. outer полезен для разовой диагностики, но добавит строки неизвестных заказов в рабочий результат. Артефакт получает покрытие ключей отдельным диагностическим объединением, поэтому умеет увидеть обе стороны, сохраняя основной left-результат.

Target grain объявляется независимо от допустимой cardinality

one-to-many не является ошибкой сам по себе. Например, если задача — получить одну строку на товарную позицию вместе со статусом заказа, такой результат естественен.

Но если целевой grain — один заказ, возможны два безопасных пути:

  1. оставить order-level и присоединить таблицу, где ключ заказа уникален;
  2. временно перейти на detail-level, но не суммировать повторённые order-level метрики и явно вернуться к нужному grain.

В этом уроке выбран первый путь. order_items уже агрегирован в 03/05, поэтому справа находится order_rollup с уникальным order_id. Ожидаемая cardinality становится one-to-one.

validate превращает ожидание в исполняемый контракт

Параметр pandas читается относительно левой и правой таблиц:

validate Что проверяется
one_to_one ключ уникален слева и справа
one_to_many ключ уникален слева; справа повторы разрешены
many_to_one слева повторы разрешены; ключ уникален справа

Например:

python
orders.merge(
    order_rollup,
    on="order_id",
    how="left",
    validate="one_to_one",
)

Если в order_rollup случайно появятся две строки O1001, объединение остановится до размножения заказа. Проверка не доказывает правильность бизнес-ключа, но не позволяет данным нарушить уже объявленное ожидание.

indicator показывает происхождение строки

Именованный индикатор:

python
indicator="items_match"

добавляет одно из трёх значений:

  • both — ключ найден с обеих сторон;
  • left_only — строка есть только слева;
  • right_only — строка есть только справа.

У индикатора есть важное ограничение: он описывает строки конкретного результата. После how="left" в нём не будет right_only, потому что правые ключи без пары не попали в результат. Кроме того, при повторяющемся ключе несколько строк могут описывать один unmatched-ключ.

Поэтому безопасный отчёт отдельно сопоставляет уникальные ключи обеих сторон и считает:

  • число уникальных matched-ключей;
  • число уникальных left_only ключей;
  • число уникальных right_only ключей;
  • несколько примеров каждого типа.

Unmatched-ключ — наблюдение, а политика зависит от смысла

Одинаковый технический статус может требовать разных решений.

Ситуация Возможная политика
Новый заказ ещё не получил позиции Сохранить, пометить left_only, не подставлять ноль
Свёртка позиций ссылается на неизвестный заказ Остановить сборку и исправить источник
Событие не имеет необязательной маркетинговой метки Сохранить и отнести к unknown
Заказ не нашёл обязательного пользователя Сохранить evidence и заблокировать публикацию

Нельзя автоматически удалять unmatched-строки или заполнять их нулями. Ноль утверждает, что значение известно и равно нулю; unmatched означает, что соответствующая запись не найдена.

Пропуски в ключе нужно запретить до merge

В отличие от обычной семантики SQL join, pandas может сопоставить пропущенный ключ слева с пропущенным ключом справа. Две неизвестности при этом выглядят как найденная пара.

Для обязательных бизнес-ключей это слишком опасно. Артефакт до merge отклоняет:

  • pd.NA, NaN и другие missing-значения;
  • пустые строки;
  • строки только из пробелов.

Пропуск в неключевой метрике разрешается её собственным контрактом. Например, order_amount из 03/05 остаётся pd.NA, если сумма заказа неполна; объединение не должно превращать её в ноль или отбрасывать заказ.

Типы ключей являются частью контракта

Строка "1001" и число 1001 могут выглядеть одинаково человеку, но представляют разные типы. Автоматическое приведение внутри merge скрывает, на какой стороне нарушен контракт.

Поэтому обе таблицы сначала проходят 03/02, а артефакт требует одинаковый dtype ключа. Он не исправляет типы на лету. В учебном примере order_id имеет nullable dtype string с обеих сторон.

Одноимённые неключевые столбцы требуют решения до merge

Если обе стороны содержат amount, pandas может создать amount_x и amount_y. Технически данные сохранены, но смысл столбцов стал неясным: какой из них заказан, рассчитан или опубликован?

Безопаснее заранее назвать происхождение:

text
booked_amount
calculated_item_amount

Артефакт отклоняет пересечение неключевых названий вместо автоматических suffixes.

Составной ключ нельзя сокращать ради удобства

Если строка описывает товар в заказе, ключом может быть (order_id, product_id). Если объединить только по order_id, каждая позиция слева получит все позиции того же заказа справа. Возникнет many-to-many внутри заказа.

Список on=[...] должен выражать реальное правило идентичности, а не минимальный набор столбцов, на котором код запускается.

Соберите это

Сначала предскажите результат без pandas. Так становится виден механизм, который merge выполняет автоматически.

Шаг 1. Назовите grain и частоты ключа

Возьмём уменьшенный пример:

text
orders:      O1, O2, O3
order_items: O1, O1, O2, O4

Запишите частоты:

order_id f_left в orders f_right в order_items
O1 1 2
O2 1 1
O3 1 0
O4 0 1

До вычислений сформулируйте:

text
left grain     -> одна строка на заказ
right grain    -> одна строка на товарную позицию
key            -> order_id
cardinality    -> one-to-many
target grain   -> пока не определён самим merge

Шаг 2. Предскажите строки inner join

Для общих ключей:

text
O1 -> 1 * 2 = 2 строки
O2 -> 1 * 1 = 1 строка

Итого inner join даст три строки. O3 и O4 не имеют пары и не попадут в результат.

Шаг 3. Предскажите строки left join

Совпавшие ключи по-прежнему дадут три строки. O3 сохранится как одна unmatched-строка:

text
2 + 1 + 1 = 4 строки

O4 отсутствует в самом left-результате, но это не означает, что правый orphan исчез из данных. Диагностический отчёт должен всё равно показать один right_only ключ.

Шаг 4. Покажите размножение метрики вручную

Пусть суммы заказов равны:

text
O1 -> 100
O2 -> 50
O3 -> 20

До объединения control total равен 170. После one-to-many left join значения будут:

text
100, 100, 50, 20

Новая сумма равна 270. Join не менял amount арифметически; он повторил всю строку O1 для двух соответствий. Именно поэтому контрольная сумма важна рядом с row count.

Шаг 5. Приведите правую таблицу к целевому grain

Эту работу уже выполнил 03/05. Две позиции O1 превратились в одну строку order_rollup:

text
order_rollup: O1, O2, O4

Теперь частоты любого ключа не превышают единицу, а объявленный контракт становится:

text
left grain     -> одна строка на заказ
right grain    -> одна строка на заказ
key            -> order_id
cardinality    -> one-to-one
how            -> left, потому что основная популяция находится в orders
target grain   -> одна строка на заказ
left_only      -> сохранить и исследовать
right_only     -> ошибка: свёртка ссылается на неизвестный заказ
control        -> число заказов, уникальность order_id и значения orders не меняются

Шаг 6. Не смешивайте подготовку и join

Поломка 2. Объявить one_to_one, не проверив фактические дубликаты

Дубликат в правом order_rollup означает две свёртки одного заказа. Без validate заказ размножится; с one_to_one артефакт остановится и покажет пример дублированного ключа.

Поломка 3. Считать любой one-to-many ошибкой

Для таблицы на grain товарной позиции связь заказа с позициями корректна. Ошибка возникает только тогда, когда результат должен сохранить grain заказа или когда повторённую order-level метрику используют как построчную.

Поломка 4. Разрешить many_to_many ради успешного запуска

Поломка 5. Оставить пропуски в ключах

Missing order_id слева может сопоставиться с missing order_id справа в pandas. Такая строка получит both, хотя настоящий заказ неизвестен. Обязательный ключ проверяется до merge, а не интерпретируется после.

Поломка 6. Использовать inner, чтобы избавиться от unmatched

Исчезновение left_only из результата не исправляет качество данных. Оно меняет популяцию и может занизить число заказов или сумму. Сначала сохраните основную сторону, посчитайте unmatched и только затем применяйте обоснованную политику.

Поломка 7. Искать right_only только в индикаторе left join

В самом left-результате правые orphan-ключи отсутствуют по определению. Артефакт сравнивает уникальные ключи отдельно от рабочего join, поэтому right_only остаётся видимым.

Поломка 8. Соединить строковый и числовой идентификатор

Приведение order_id внутри функции может скрыть лидирующие нули или источник нарушения. Обе стороны должны заранее соответствовать одному dtype-контракту.

Поломка 9. Положиться на автоматические suffixes

amount_x и amount_y не объясняют происхождение метрик. Переименуйте поля до merge и зафиксируйте их смысл. Артефакт отклоняет неоднозначное пересечение.

Поломка 10. Убрать часть составного ключа

Если реальный ключ (order_id, product_id), объединение только по order_id соединит каждую позицию со всеми позициями того же заказа. Успешный merge по существующему столбцу не доказывает правильность правила сопоставления.

Поломка 11. Считать строки вместо уникальных unmatched-ключей

Три unmatched-события одного пользователя — это три строки, но один проблемный user_id. Для оценки покрытия обычно нужны оба числа; отчёт текущего артефакта намеренно считает уникальные ключи и сохраняет число строк отдельно.

Поломка 12. Заполнить отсутствие свёртки нулём

left_only заказ может ещё не получить позиции, потеряться в источнике или действительно быть пустым. fillna(0) смешивает эти причины. Ноль допустим только после отдельного предметного доказательства.

01

В вызове orders.merge(order_items, ...) order_id уникален в orders и повторяется в order_items. Какую cardinality нужно объявить относительно левой и правой таблиц?

Ключ уникален слева и может повторяться справа, поэтому направление относительно left.merge(right) — one-to-many.

02

Один ключ встречается два раза слева и три раза справа. Сколько строк этого ключа создаст inner join?

Каждая левая строка соединяется с каждой правой: 2 × 3 = 6 строк.

03

Нужно сохранить все заказы и добавить к ним готовый order_rollup с уникальным order_id. Какой основной контракт подходит?

Left join сохраняет популяцию orders, а уникальный order_id с обеих сторон соответствует one-to-one.

04

Почему одного indicator в результате how='left' недостаточно для полного отчёта о покрытии ключей?

Left join не содержит правые orphan-строки, а row-level статусы могут повторять один ключ; уникальные ключи нужно сравнить отдельно.

05

Обе таблицы содержат строку с пропущенным обязательным order_id. Какое поведение безопасно для аналитического контракта?

Неизвестные идентификаторы не доказывают равенство сущностей; обязательные ключи должны быть non-missing и non-blank до merge.

06

После корректного one-to-many merge результат имеет grain товарной позиции. Как безопасно работать с повторившимся orders.amount?

One-to-many может быть структурно корректным, но order-level значение повторено; дальнейшая агрегация должна учитывать его исходный grain.

from __future__ import annotations

from typing import Any

import pandas as pd


class MergeContractError(ValueError):
    """Raised when a merge would violate the declared data contract."""


ALLOWED_HOW = {"inner", "left", "right", "outer"}
ALLOWED_VALIDATE = {"one_to_one", "one_to_many", "many_to_one"}
ROLLUP_DTYPES = {
    "order_id": "string",
    "line_count": "Int64",
    "known_amount_lines": "Int64",
    "missing_amount_lines": "Int64",
    "known_amount_total": "Float64",
    "order_amount": "Float64",
}


def _normalise_keys(on: object) -> list[str]:
    if not isinstance(on, list) or not on:
        raise MergeContractError("on must be a non-empty list of column names")
    if any(not isinstance(column, str) or not column for column in on):
        raise MergeContractError("every merge key must be a non-empty string")
    if len(on) != len(set(on)):
        raise MergeContractError("merge key names must not repeat")
    return on


def _validate_frame(frame: object, *, label: str, keys: list[str]) -> pd.DataFrame:
    if not isinstance(frame, pd.DataFrame):
        raise MergeContractError(f"{label} must be a pandas DataFrame")
    if not frame.columns.is_unique:
        raise MergeContractError(f"{label} column labels must be unique")

    missing_columns = sorted(set(keys) - set(frame.columns))
    if missing_columns:
        raise MergeContractError(f"{label} misses key columns: {missing_columns}")

    missing_key = frame[keys].isna().any(axis=1)
    if missing_key.any():
        labels = frame.index[missing_key].tolist()
        raise MergeContractError(
            f"{label} merge key must be non-missing at rows: {labels}"
        )

    blank_key = pd.Series(False, index=frame.index, dtype="boolean")
    for column in keys:
        blank_key |= frame[column].map(
            lambda value: isinstance(value, str) and value.strip() == ""
        ).astype("boolean")
    if blank_key.any():
        labels = frame.index[blank_key].tolist()
        raise MergeContractError(
            f"{label} merge key must be non-blank at rows: {labels}"
        )
    return frame


def _assert_unique_key(frame: pd.DataFrame, keys: list[str], *, label: str) -> None:
    duplicated = frame.duplicated(keys, keep=False)
    if duplicated.any():
        examples = frame.loc[duplicated, keys].drop_duplicates().head(5).to_dict("records")
        raise MergeContractError(
            f"{label} key must be unique for the declared cardinality; "
            f"duplicate key examples: {examples}"
        )


def _validate_cardinality(
    left: pd.DataFrame,
    right: pd.DataFrame,
    *,
    keys: list[str],
    validate: str,
) -> None:
    if validate not in ALLOWED_VALIDATE:
        if validate == "many_to_many":
            raise MergeContractError(
                "many_to_many performs no uniqueness check; declare a safer cardinality"
            )
        raise MergeContractError(
            f"validate must be one of {sorted(ALLOWED_VALIDATE)}"
        )
    if validate in {"one_to_one", "one_to_many"}:
        _assert_unique_key(left, keys, label="left")
    if validate in {"one_to_one", "many_to_one"}:
        _assert_unique_key(right, keys, label="right")


def _frequency_table(
    left: pd.DataFrame,
    right: pd.DataFrame,
    *,
    keys: list[str],
) -> pd.DataFrame:
    left_frequency = (
        left.groupby(keys, as_index=False, sort=False, dropna=False)
        .size()
        .rename(columns={"size": "left_frequency"})
    )
    right_frequency = (
        right.groupby(keys, as_index=False, sort=False, dropna=False)
        .size()
        .rename(columns={"size": "right_frequency"})
    )
    coverage = left_frequency.merge(
        right_frequency,
        on=keys,
        how="outer",
        validate="one_to_one",
        indicator="key_status",
        sort=False,
    )
    coverage["left_frequency"] = coverage["left_frequency"].fillna(0).astype("Int64")
    coverage["right_frequency"] = coverage["right_frequency"].fillna(0).astype("Int64")
    return coverage


def _predicted_rows(coverage: pd.DataFrame, *, how: str) -> int:
    left_frequency = coverage["left_frequency"]
    right_frequency = coverage["right_frequency"]
    matched = left_frequency * right_frequency

    if how == "inner":
        contribution = matched
    elif how == "left":
        contribution = matched.mask(right_frequency.eq(0), left_frequency)
    elif how == "right":
        contribution = matched.mask(left_frequency.eq(0), right_frequency)
    else:
        contribution = matched.mask(
            right_frequency.eq(0), left_frequency
        ).mask(left_frequency.eq(0), right_frequency)
    return int(contribution.sum())


def _key_examples(
    coverage: pd.DataFrame,
    *,
    keys: list[str],
    status: str,
) -> list[dict[str, Any]]:
    return coverage.loc[coverage["key_status"].eq(status), keys].head(5).to_dict("records")


def merge_with_contract(
    left: pd.DataFrame,
    right: pd.DataFrame,
    *,
    on: list[str],
    how: str,
    validate: str,
    indicator: str = "merge_status",
) -> tuple[pd.DataFrame, dict[str, Any]]:
    """Merge two tables only after checking the declared relationship.

    Business keys are mandatory: missing and blank values are rejected before pandas can
    match null keys to each other. The report counts distinct unmatched keys independently
    of ``how``, so a left join still reveals keys that exist only on the right.
    """

    keys = _normalise_keys(on)
    if how not in ALLOWED_HOW:
        raise MergeContractError(f"how must be one of {sorted(ALLOWED_HOW)}")
    if not isinstance(indicator, str) or not indicator.strip():
        raise MergeContractError("indicator must be a non-blank column name")

    left_frame = _validate_frame(left, label="left", keys=keys)
    right_frame = _validate_frame(right, label="right", keys=keys)

    if indicator in left_frame.columns or indicator in right_frame.columns:
        raise MergeContractError(f"indicator column already exists: {indicator}")

    dtype_mismatches = {
        key: {"left": str(left_frame[key].dtype), "right": str(right_frame[key].dtype)}
        for key in keys
        if str(left_frame[key].dtype) != str(right_frame[key].dtype)
    }
    if dtype_mismatches:
        raise MergeContractError(
            f"merge key dtypes must match before merge: {dtype_mismatches}"
        )

    overlaps = sorted((set(left_frame.columns) & set(right_frame.columns)) - set(keys))
    if overlaps:
        raise MergeContractError(
            "non-key columns overlap; rename them before merge so provenance stays "
            f"explicit: {overlaps}"
        )

    _validate_cardinality(
        left_frame,
        right_frame,
        keys=keys,
        validate=validate,
    )
    coverage = _frequency_table(left_frame, right_frame, keys=keys)
    expected_rows = _predicted_rows(coverage, how=how)

    try:
        merged = left_frame.merge(
            right_frame,
            on=keys,
            how=how,
            validate=validate,
            indicator=indicator,
            sort=False,
        )
    except pd.errors.MergeError as error:
        raise MergeContractError(str(error)) from error

    if len(merged) != expected_rows:
        raise MergeContractError(
            f"row-count prediction failed: expected {expected_rows}, got {len(merged)}"
        )

    key_counts = coverage["key_status"].value_counts()
    report: dict[str, Any] = {
        "on": keys.copy(),
        "how": how,
        "validate": validate,
        "left_rows": len(left_frame),
        "right_rows": len(right_frame),
        "left_distinct_keys": len(coverage.loc[coverage["left_frequency"].gt(0)]),
        "right_distinct_keys": len(coverage.loc[coverage["right_frequency"].gt(0)]),
        "matched_key_count": int(key_counts.get("both", 0)),
        "left_only_key_count": int(key_counts.get("left_only", 0)),
        "right_only_key_count": int(key_counts.get("right_only", 0)),
        "left_only_key_examples": _key_examples(
            coverage,
            keys=keys,
            status="left_only",
        ),
        "right_only_key_examples": _key_examples(
            coverage,
            keys=keys,
            status="right_only",
        ),
        "predicted_result_rows": expected_rows,
        "result_rows": len(merged),
    }
    return merged, report


def _validate_order_rollup(frame: object) -> pd.DataFrame:
    rollup = _validate_frame(frame, label="order_rollup", keys=["order_id"])
    missing_columns = sorted(set(ROLLUP_DTYPES) - set(rollup.columns))
    if missing_columns:
        raise MergeContractError(f"order_rollup misses columns: {missing_columns}")

    wrong_dtypes = {
        column: {"expected": expected, "actual": str(rollup[column].dtype)}
        for column, expected in ROLLUP_DTYPES.items()
        if str(rollup[column].dtype) != expected
    }
    if wrong_dtypes:
        raise MergeContractError(
            f"order_rollup must satisfy the 03/05 dtype contract: {wrong_dtypes}"
        )
    return rollup


def attach_order_rollup(
    orders: pd.DataFrame,
    order_rollup: pd.DataFrame,
) -> tuple[pd.DataFrame, dict[str, Any]]:
    """Attach the typed 03/05 rollup while preserving one row per order."""

    orders_frame = _validate_frame(orders, label="orders", keys=["order_id"])
    if str(orders_frame["order_id"].dtype) != "string":
        raise MergeContractError("orders.order_id must already have dtype string")
    rollup_frame = _validate_order_rollup(order_rollup)

    merged, report = merge_with_contract(
        orders_frame,
        rollup_frame,
        on=["order_id"],
        how="left",
        validate="one_to_one",
        indicator="items_match",
    )
    if report["right_only_key_count"]:
        raise MergeContractError(
            "order_rollup references unknown orders: "
            f"{report['right_only_key_examples']}"
        )
    if len(merged) != len(orders_frame) or merged["order_id"].duplicated().any():
        raise MergeContractError("result grain must remain one row per order_id")

    expected_left = orders_frame.reset_index(drop=True)
    actual_left = merged.loc[:, orders_frame.columns].reset_index(drop=True)
    try:
        pd.testing.assert_frame_equal(actual_left, expected_left)
    except AssertionError as error:
        raise MergeContractError("left-table values changed during merge") from error

    report["target_grain"] = "one row per order_id"
    report["grain_preserved"] = True
    report["left_values_preserved"] = True
    return merged, report
