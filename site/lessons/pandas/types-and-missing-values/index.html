Типы данных и пропуски

Сначала объявите, что означает столбец и допустим ли в нём пропуск; неразбираемое непустое значение не должно незаметно становиться пропуском.

Цели обучения

После урока вы сможете:

  • описать столбец через его смысл, целевой dtype и правило допустимости пропуска;
  • различить пустое значение и непустую строку, которую не смог разобрать конвертер;
  • осознанно использовать nullable-типы Int64, Float64, boolean и string;
  • находить пропуски через isna() и предсказывать их поведение в сравнении;
  • применять errors="coerce" как диагностический этап, не принимая появившийся NA за исправление данных;
  • получить отчёт со строками-нарушителями до дальнейших вычислений.

Проблема

Рассмотрим три исходных значения суммы заказа:

Строка Исходное значение amount Смысл
0 "1200.00" корректная сумма, пока записанная текстом
1 "" значение отсутствует
2 "oops" непустое значение, не являющееся числом

Удобная команда превращает их в числа:

python
pd.to_numeric(amount, errors="coerce")

Результат выглядит так:

text
0    1200.0
1       NaN
2       NaN

У строк 1 и 2 теперь одинаковое техническое представление, хотя причины разные:

  • в строке 1 факта нет;
  • в строке 2 факт, возможно, был, но его запись нарушает ожидаемый формат.

Если посмотреть только на результат преобразования, ошибка oops станет похожа на разрешённый пропуск. Последующий dropna() удалит обе строки, а fillna(0) превратит обе в реальные нулевые суммы. Код продолжит выполняться и сможет дать правдоподобный, но неверный итог.

Поэтому задача урока — не просто «поменять dtype». Нужно сначала объявить контракт, затем отделить пропуски от ошибок разбора и лишь после этого разрешить вычисления.

Концепция

У столбца есть физический dtype и предметный смысл

В DataFrame каждый столбец является Series со своим dtype. Но dtype отвечает только на часть вопросов.

Для рабочего контракта нужны как минимум три характеристики:

Характеристика Вопрос Пример для amount
Семантический тип Что означает значение? денежная сумма заказа
Физический dtype В каком представлении работаем? Float64
Политика пропусков Может ли факт отсутствовать? да, например пока заказ pending

Строка "1200.00" может быть технически текстом, но по смыслу является суммой. Число 1007 может иметь integer dtype, но по смыслу быть идентификатором, над которым нельзя вычислять среднее. Dtype помогает библиотеке хранить и обрабатывать данные, но не определяет их бизнес-смысл за аналитика.

В простом контракте этого урока будем задавать два машинно-проверяемых поля:

json
{
  "amount": {
    "dtype": "Float64",
    "nullable": true
  }
}

dtype задаёт целевое представление. nullable отвечает, разрешено ли отсутствие значения. Сам dtype не заменяет это решение: техническая возможность хранить pd.NA ещё не означает, что пропуск допустим по смыслу.

Пропуск и невалидное значение — разные состояния

В рамках учебного артефакта используем простое явное правило:

  • None, уже существующий NA, пустая строка и строка из пробелов считаются пропуском;
  • непустая строка, которую не удаётся привести к целевому dtype, считается невалидной;
  • допустимость пропуска определяется nullable, а не самим конвертером.

Получаются четыре полезных случая:

Значение Разобралось Пропуск разрешён Решение
"1200.00" да любое принять
"" не требуется да сохранить как пропуск
"" не требуется нет остановить проверку
"oops" нет любое остановить проверку как invalid

Невалидное значение нельзя «разрешить» флагом nullable=true: это не отсутствие факта, а нарушение формата. Если бизнес решит, что строка "unknown" действительно является допустимым маркером отсутствия, её нужно явно добавить в контракт чтения источника. Это уже отдельное решение, а не побочный эффект errors="coerce".

Почему pandas нужны nullable-типы

Обычные NumPy-типы хорошо подходят для плотных однородных массивов, но некоторые из них не имеют отдельного значения пропуска. pandas предлагает nullable-типы, которые хранят маску отсутствующих значений рядом с данными.

Обычный dtype Nullable-вариант pandas Пропуск nullable-варианта
int64 Int64 pd.NA
float64 Float64 pd.NA
bool boolean pd.NA
str в pandas 3 string с явной nullable-семантикой pd.NA

Регистр букв имеет значение: int64 и Int64 — разные dtype.

python
quantity = pd.Series([1, None], dtype="Int64")

print(quantity)
text
0       1
1    <NA>
dtype: Int64

Это не означает, что каждый столбец курса нужно автоматически переводить в nullable- вариант. Выбор остаётся частью контракта. В этом уроке мы используем nullable-типы, потому что хотим явно наблюдать пропуски, не превращать целые числа во float и получить согласованное поведение сравнений.

str, string и object в pandas 3 — не одно и то же

В pandas 3 текст, dtype которого библиотека выводит автоматически, обычно получает новый dtype str:

python
inferred = pd.Series(["paid", None])
inferred_values = inferred.tolist()

print(inferred.dtype)
print(repr(inferred_values[1]))
text
str
nan

Явный string в используемой курсом версии pandas хранит пропуск как pd.NA:

python
nullable_text = inferred.astype("string")
nullable_values = nullable_text.tolist()

print(nullable_text.dtype)
print(repr(nullable_values[1]))
text
string
<NA>

object — общий контейнер Python-объектов. Он способен одновременно хранить строки, числа и None, поэтому сам по себе не является точным утверждением «этот столбец — текст». Старый код часто использует object для строк, но новый контракт не должен выбирать его только по привычке.

Запоминать все внутренние варианты строкового хранения не нужно. Для этого урока важны два правила:

  1. dtype текста проверяем явно, не предполагаем, что он обязательно равен object;
  2. пропуск ищем через isna(), а не сравнением с None, np.nan или pd.NA.

Пропуск нельзя надёжно найти через равенство

python
quantity = pd.Series([1, None], dtype="Int64")

print(quantity == pd.NA)
text
0    <NA>
1    <NA>
dtype: boolean

Такое сравнение не отвечает на вопрос, где находится пропуск. Используйте специальную операцию:

python
missing = quantity.isna()

assert missing.tolist() == [False, True]

isna() скрывает различия между техническими sentinel-значениями и даёт обычную булеву маску мест, которые pandas считает отсутствующими.

pd.NA распространяет неопределённость

Nullable-тип влияет не только на хранение:

python
quantity = pd.Series([1, None], dtype="Int64")

print(quantity + 1)
print(quantity > 0)
text
0       2
1    <NA>
dtype: Int64

0    True
1    <NA>
dtype: boolean

Если количество неизвестно, pandas не утверждает, что оно положительное или неположительное. Результат сравнения тоже неизвестен. В 03/03 такая boolean Series станет маской выбора строк, и потребуется явно решить, исключать NA, считать его ошибкой или обрабатывать отдельно.

errors="coerce" — диагностический инструмент, а не очистка

У pd.to_numeric есть два особенно важных режима:

  • errors="raise" останавливается на неразбираемом значении;
  • errors="coerce" ставит на его месте пропуск.

Первый режим строгий, но обычно сообщает только о первой проблеме. Второй позволяет получить маску всех неразбираемых строк — при условии, что мы сравним результат с исходными значениями.

В следующем фрагменте встречаются две небольшие операции pandas, нужные именно для парсера:

  • .str применяет строковую операцию к каждому значению Series, сохраняя её метки;
  • series.mask(condition, replacement) заменяет значения только там, где условие истинно, также сохраняя метки;
  • .map(dictionary) сопоставляет каждому исходному токену явно объявленное значение.

Это локальный мост, а не полный урок по строкам или выборке. Систематическая нормализация текста будет в 03/09, а выбор строк по маске — в 03/03.

python
text = raw.astype("string")
missing = raw.isna() | text.str.strip().eq("").fillna(False)
prepared = text.mask(missing, pd.NA)
parsed = pd.to_numeric(prepared, errors="coerce")
non_finite = parsed.isin([float("inf"), float("-inf")])
invalid = ((~missing) & parsed.isna()) | non_finite

Читается это так:

  1. missing отмечает значения, которые наш контракт считает пустыми;
  2. parsed.isna() отмечает всё, что после преобразования оказалось пропуском;
  3. ~missing исключает уже известные допустимые пустоты;
  4. пересечение оставляет непустые значения, потерянные конвертером;
  5. non_finite дополнительно отмечает бесконечности через проверку принадлежности объявленному набору inf, -inf.

Только после проверки invalid можно приводить результат к целевому dtype и передавать дальше.

Разобрать число недостаточно: проверьте диапазон и конечность

pd.to_numeric отвечает на вопрос «можно ли прочитать эту запись как число», но не знает контракт будущего dtype. После разбора нужны дополнительные проверки:

  • Int64 допускает только целые значения от -9223372036854775808 до 9223372036854775807;
  • Float64 в артефакте допускает только конечные числа: inf, -inf и значение вроде 1e309, переполнившееся до бесконечности, считаются invalid;
  • конечное Float64 не обещает точного десятичного хранения каждого исходного числа.

Последний пункт особенно важен для суммы заказа. Бинарный Float64 удобен для учебных вычислений, но некоторые десятичные значения представляет приближённо. Поэтому этот урок проверяет формат, конечность и nullable-политику, а не объявляет Float64 универсальным типом бухгалтерских денег. Контракт точности из 02/10 по-прежнему действует: если задача требует точного десятичного представления, тип хранения нужно выбирать отдельно.

Минимальный алгоритм dtype-контракта

Для каждого нужного столбца:

  1. Назовите его предметный смысл.
  2. Выберите целевой dtype по будущим операциям, а не только по текущим значениям.
  3. Явно задайте nullable.
  4. Сохраните исходную Series до преобразования.
  5. Постройте отдельные маски missing и invalid.
  6. Не публикуйте преобразованный результат, пока есть нарушения.
  7. После успешной проверки подтвердите фактический dtype и число пропусков.

Что делать с настоящими пропусками

После отделения missing от invalid решение о пропуске всё ещё остаётся за аналитиком. Есть четыре понятные ветки:

  1. Отклонить данные, если поле обязательно по контракту.
  2. Сохранить pd.NA, если отсутствие допустимо и последующие операции умеют с ним работать.
  3. Исключить или отправить строки на разбор, предварительно посчитав их количество и долю, чтобы не скрыть изменение выборки.
  4. Заполнить значение, только если предметное правило однозначно задаёт замену; полезно сохранить отдельный признак того, что исходное значение отсутствовало.

Ни одна ветка не выбирается автоматически по названию dtype. В частности, fillna(0) не становится корректным только потому, что столбец числовой.

Соберите это

Сначала построим прозрачный конвертер без pandas. Его задача — показать различие между двумя ветками, которые errors="coerce" позже представит одинаково.

Шаг 1. Объявите контракт столбца

Для суммы заказа договоримся:

text
semantic type: money
target dtype: Float64
nullable: true
blank values: missing
non-blank values that are not numbers: invalid

Обратите внимание: nullable=true разрешает пустоту, но не разрешает oops.

Шаг 2. Классифицируйте одно исходное значение

python
import math


def classify_number(value: str | None) -> tuple[str, float | None]:
    if value is None or value.strip() == "":
        return "missing", None

    try:
        parsed = float(value)
    except ValueError:
        return "invalid", None

    if not math.isfinite(parsed):
        return "invalid", None
    return "valid", parsed

Проверьте три ветки:

python
assert classify_number("1200.00") == ("valid", 1200.0)
assert classify_number("") == ("missing", None)
assert classify_number("oops") == ("invalid", None)
assert classify_number("1e309") == ("invalid", None)

Шаг 3. Сохраните позиции нарушений

Одного количества ошибок недостаточно: источник или коллега должны понять, какие строки исправлять.

python
raw_values = ["1200.00", "", "oops"]
diagnostics = []

for row, raw_value in enumerate(raw_values):
    status, parsed_value = classify_number(raw_value)
    diagnostics.append(
        {
            "row": row,
            "raw_value": raw_value,
            "status": status,
            "parsed_value": parsed_value,
        }
    )

Ожидаем:

text
row 0 -> valid   -> 1200.0
row 1 -> missing -> None
row 2 -> invalid -> None

Прозрачная версия не оптимальна для большой таблицы, но она делает видимым ключевой механизм: решение принимается по исходному значению, а не только по результату parser.

Используйте это

Теперь повторим тот же процесс средствами pandas.

Шаг 1. Создайте исходную таблицу

Пример самодостаточен:

python
import pandas as pd

orders = pd.DataFrame(
    {
        "order_id": ["O1001", "O1002", "O1003"],
        "amount": ["1200.00", "", "oops"],
    },
    dtype="string",
)

dtype="string" здесь выбран специально, чтобы сохранить исходные текстовые значения и pd.NA до проверки.

Шаг 2. Разделите missing и invalid

python
raw = orders["amount"]
text = raw.astype("string")

missing = raw.isna() | text.str.strip().eq("").fillna(False)
prepared = text.mask(missing, pd.NA)
parsed = pd.to_numeric(prepared, errors="coerce")
non_finite = parsed.isin([float("inf"), float("-inf")])
invalid = ((~missing) & parsed.isna()) | non_finite

assert missing.tolist() == [False, True, False]
assert invalid.tolist() == [False, False, True]

parsed содержит пропуск и для пустой строки, и для oops, но две исходные маски не дают спутать причины. Отдельная маска non_finite не позволяет принять бесконечность за пригодное аналитическое число. Пока мы только строим маску и проверяем её значения; выбор строк pandas по этой маске будет темой следующего урока.

Шаг 3. Остановитесь при invalid

python
if invalid.any():
    examples = [
        {"row": row, "value": value}
        for row, value, is_invalid in zip(
            raw.index.tolist(),
            raw.tolist(),
            invalid.tolist(),
            strict=True,
        )
        if is_invalid
    ]
    raise ValueError(f"amount contains invalid values: {examples}")

Здесь доказательства собираются знакомым Python-циклом по трём согласованным спискам. Мы не вводим раньше времени синтаксис pandas для выбора строк; важно лишь не потерять метку строки и исходное значение.

На исходном примере код должен остановиться. Это ожидаемое поведение, а не неудача упражнения. Нельзя публиковать сумму, пока непонятно, что означает oops.

Предположим, владелец источника подтвердил, что в строке 2 должно быть "75.50":

python
corrected = pd.Series(["1200.00", "", "75.50"], dtype="string")

missing = corrected.isna() | corrected.str.strip().eq("").fillna(False)
parsed = pd.to_numeric(corrected.mask(missing, pd.NA), errors="coerce")
non_finite = parsed.isin([float("inf"), float("-inf")])
invalid = ((~missing) & parsed.isna()) | non_finite

assert not invalid.any()

amount = parsed.astype("Float64")
assert str(amount.dtype) == "Float64"
amount_values = amount.tolist()
assert amount_values[0] == 1200.0
assert pd.isna(amount_values[1])
assert amount_values[2] == 75.5

Мы не угадали исправление и не заменили ошибку нулём. Изменение появилось только после явного решения о конкретном исходном значении. Новая Series создаётся явно: изменение существующего объекта и способы выбора строки появятся в следующем уроке.

Шаг 4. Проверьте nullable integer и boolean

Для целого количества важны две дополнительные проверки: "1.5" является числом, но не целым, а "9223372036854775808" является целым, но не помещается в Int64. Нельзя молча отбросить дробную часть или позволить переполнению изменить значение.

python
from decimal import Decimal, InvalidOperation

INT64_MIN = -(2**63)
INT64_MAX = 2**63 - 1


def parse_int64(value: str) -> int | None:
    try:
        parsed = Decimal(value)
    except InvalidOperation:
        return None

    if not parsed.is_finite() or parsed != parsed.to_integral_value():
        return None
    if not INT64_MIN <= parsed <= INT64_MAX:
        return None
    return int(parsed)

assert parse_int64("2") == 2
assert parse_int64("1.5") is None
assert parse_int64("9223372036854775808") is None

Артефакт использует десятичный промежуточный разбор для Int64, чтобы не потерять точность сначала во float и только потом проверять границы. Пустоту он, как и раньше, обрабатывает отдельной маской до вызова такого parser.

Для логического столбца нужен явный словарь допустимых представлений:

python
boolean_values = {
    "true": True,
    "false": False,
    "1": True,
    "0": False,
    "yes": True,
    "no": False,
}

normalized = raw_opt_in.str.strip().str.lower()
parsed_opt_in = normalized.map(boolean_values).astype("boolean")

opt_in_values = parsed_opt_in.tolist()
assert opt_in_values[:2] == [True, False]
assert pd.isna(opt_in_values[2])
assert pd.isna(opt_in_values[3])

По одному результату последние две строки снова неразличимы. Для строгой проверки нужна та же пара масок missing и invalid, которую использует артефакт.

Сломайте это

Ошибка 1. Использовать coerce и смотреть только на результат

python
amount = pd.to_numeric(raw, errors="coerce")

Команда полезна, но без исходной маски она объединяет допустимый пропуск и ошибку разбора. Проверяйте непустые позиции, которые стали NA.

Ошибка 2. Заменить все пропуски нулём

python

Ноль — реальная сумма, а не универсальное обозначение отсутствия. После такой замены невозможно отличить бесплатный заказ от заказа с неизвестной суммой.

Ошибка 3. Превратить идентификатор в число

text
"0007" -> 7

Числовой dtype уничтожает ведущие нули и разрешает бессмысленные арифметические операции. Бизнес-ключ из 03/01 обычно должен оставаться строкой, даже если состоит только из цифр.

Ошибка 4. Молча превратить 1.5 в integer

Количество 1.5 может быть ошибкой единиц или источника. Округление и усечение — разные бизнес-решения. Строгий конвертер отклоняет дробь до получения явного правила.

Ошибка 5. Считать любое распознанное число допустимым

9223372036854775808 является целым, но выходит за верхнюю границу Int64. 1e309 синтаксически похоже на число, но в Float64 становится бесконечностью. Строгий контракт проверяет дробную часть, диапазон и конечность до публикации результата.

Ошибка 6. Искать пропуск через == pd.NA

Результатом сравнения с неизвестным снова является неизвестность. Для обнаружения используйте isna() или notna().

Ошибка 7. Считать NA в условии значением False

amount > 0 для отсутствующей суммы не доказывает ни истину, ни ложь. В следующем уроке политика маски будет выбираться явно; не вставляйте fillna(False) автоматически.

Ошибка 8. Доверить контракт dtype inference

Если в маленьком файле нет пропусков, библиотека может вывести обычный int64. Следующий файл с пустым значением изменит dtype или завершит преобразование ошибкой. Контракт описывает ожидаемые данные, а не только текущую выборку.

Ошибка 9. Считать object, str и string взаимозаменяемыми

Старый код может проверять dtype == "object" и перестать находить текстовые столбцы в pandas 3. Проверяйте реальный dtype и используйте pd.api.types.is_string_dtype, если нужна совместимая проверка разных строковых реализаций.

Quiz · 4 вопроса

Проверьте себя

Ответьте после практики. Объяснение появится сразу после выбора.

01

После pd.to_numeric(errors="coerce") и пустая строка, и "oops" стали NA. Как отличить допустимый пропуск от ошибки разбора?

Маска invalid строится как непустые исходные позиции, которые после преобразования стали NA. Поэтому причина пропуска остаётся наблюдаемой.

02

Каков результат выражения pd.Series([1, None], dtype="Int64") > 0?

Сравнение известного значения даёт True, а сравнение неизвестного сохраняет неопределённость как NA. Политика для такого NA понадобится при фильтрации.

03

Схема запрещает пропуск order_id, но разрешает пропуск amount. В строке A order_id пуст, а в строке B amount равен "oops". Что должен показать строгий аудитор?

Пустой обязательный ключ нарушает nullable-политику. Непустая неразбираемая сумма остаётся invalid даже при nullable=true, потому что nullable разрешает отсутствие, а не ошибочный формат.

04

Строка "9223372036854775808" разбирается как целое число, но схема требует Int64. Как должен поступить строгий аудитор?

Успешный синтаксический разбор ещё не гарантирует совместимость с целевым dtype. Строгий контракт проверяет диапазон до приведения и не допускает циклической замены или насыщения значения.

from __future__ import annotations

import argparse
import json
from collections.abc import Mapping, Sequence
from decimal import Decimal, InvalidOperation
from pathlib import Path
from typing import Any

import pandas as pd


class DtypeContractError(ValueError):
    """Raised when the schema or CLI invocation is invalid."""


SUPPORTED_DTYPES = {"string", "Int64", "Float64", "boolean"}
BOOLEAN_VALUES = {
    "true": True,
    "false": False,
    "1": True,
    "0": False,
    "yes": True,
    "no": False,
}
INT64_MIN = -(2**63)
INT64_MAX = 2**63 - 1


def validate_schema(schema: object) -> dict[str, dict[str, Any]]:
    if not isinstance(schema, Mapping) or not schema:
        raise DtypeContractError("schema must be a non-empty JSON object")

    normalized: dict[str, dict[str, Any]] = {}
    for column, raw_spec in schema.items():
        if not isinstance(column, str) or not column:
            raise DtypeContractError("schema column names must be non-empty strings")
        if not isinstance(raw_spec, Mapping):
            raise DtypeContractError(
                f"schema for {column!r} must contain dtype and nullable"
            )
        unknown = set(raw_spec) - {"dtype", "nullable"}
        if unknown:
            raise DtypeContractError(
                f"schema for {column!r} has unknown fields: {sorted(unknown)}"
            )
        target = raw_spec.get("dtype")
        nullable = raw_spec.get("nullable")
        if target not in SUPPORTED_DTYPES:
            raise DtypeContractError(
                f"unsupported dtype for {column!r}: {target!r}; "
                f"expected one of {sorted(SUPPORTED_DTYPES)}"
            )
        if not isinstance(nullable, bool):
            raise DtypeContractError(
                f"nullable for {column!r} must be true or false"
            )
        normalized[column] = {"dtype": target, "nullable": nullable}
    return normalized


def _as_nullable_text(series: pd.Series) -> pd.Series:
    return series.astype("string")


def _missing_mask(series: pd.Series) -> pd.Series:
    text = _as_nullable_text(series)
    blank = text.str.strip().eq("").fillna(False)
    return (series.isna() | blank).astype(bool)


def _examples(series: pd.Series, mask: pd.Series) -> list[dict[str, str]]:
    examples: list[dict[str, str]] = []
    rows = zip(series.index.tolist(), series.tolist(), mask.tolist(), strict=True)
    for label, value, selected in rows:
        if not selected:
            continue
        examples.append({"row": str(label), "value": str(value)})
        if len(examples) == 5:
            break
    return examples


def _convert_int64(clean: pd.Series) -> tuple[pd.Series, pd.Series]:
    values: list[Any] = []
    invalid_values: list[bool] = []

    for value in clean.tolist():
        if pd.isna(value):
            values.append(pd.NA)
            invalid_values.append(False)
            continue

        try:
            parsed = Decimal(str(value).strip())
        except InvalidOperation:
            values.append(pd.NA)
            invalid_values.append(True)
            continue

        is_valid = (
            parsed.is_finite()
            and parsed == parsed.to_integral_value()
            and INT64_MIN <= parsed <= INT64_MAX
        )
        values.append(int(parsed) if is_valid else pd.NA)
        invalid_values.append(not is_valid)

    converted = pd.Series(
        pd.array(values, dtype="Int64"),
        index=clean.index,
        name=clean.name,
    )
    invalid = pd.Series(invalid_values, index=clean.index, dtype=bool)
    return converted, invalid


def convert_series(
    series: pd.Series,
    *,
    target: str,
    nullable: bool,
) -> tuple[pd.Series, dict[str, Any]]:
    if target not in SUPPORTED_DTYPES:
        raise DtypeContractError(f"unsupported target dtype: {target}")

    source = series.copy()
    text = _as_nullable_text(source)
    missing = _missing_mask(source)
    clean = text.mask(missing, pd.NA)
    invalid = pd.Series(False, index=source.index, dtype=bool)

    if target == "string":
        converted = clean.astype("string")
    elif target == "Int64":
        converted, invalid = _convert_int64(clean)
    elif target == "Float64":
        parsed = pd.to_numeric(clean, errors="coerce")
        invalid = ((~missing) & parsed.isna()).astype(bool)
        non_finite = parsed.isin([float("inf"), float("-inf")])
        invalid = (invalid | non_finite).astype(bool)
        converted = parsed.mask(invalid, pd.NA).astype("Float64")
    elif target == "boolean":
        normalized = clean.str.strip().str.lower()
        parsed = normalized.map(BOOLEAN_VALUES)
        invalid = ((~missing) & parsed.isna()).astype(bool)
        converted = parsed.mask(invalid, pd.NA).astype("boolean")
    else:  # pragma: no cover - guarded by SUPPORTED_DTYPES
        raise DtypeContractError(f"unsupported target dtype: {target}")

    missing_rows = int(missing.sum())
    invalid_rows = int(invalid.sum())
    violations: list[str] = []
    if missing_rows and not nullable:
        violations.append("missing_not_allowed")
    if invalid_rows:
        violations.append("invalid_values")

    report = {
        "source_dtype": str(source.dtype),
        "target_dtype": target,
        "result_dtype": str(converted.dtype),
        "nullable": nullable,
        "missing_rows": missing_rows,
        "missing_examples": _examples(source, missing),
        "invalid_rows": invalid_rows,
        "invalid_examples": _examples(source, invalid),
        "violations": violations,
        "valid": not violations,
    }
    return converted, report


def audit_and_convert(
    frame: pd.DataFrame,
    schema: Mapping[str, Mapping[str, Any]],
) -> tuple[pd.DataFrame, dict[str, Any]]:
    normalized_schema = validate_schema(schema)
    result = frame.copy()
    missing_columns = sorted(set(normalized_schema) - set(frame.columns))
    columns: dict[str, Any] = {}

    for column, spec in normalized_schema.items():
        if column in missing_columns:
            continue
        converted, column_report = convert_series(
            result[column],
            target=spec["dtype"],
            nullable=spec["nullable"],
        )
        result[column] = converted
        columns[column] = column_report

    valid = not missing_columns and all(
        column_report["valid"] for column_report in columns.values()
    )
    report = {
        "valid": valid,
        "rows": len(result),
        "missing_columns": missing_columns,
        "columns": columns,
    }
    return result, report


def _parse_schema(raw_schema: str) -> dict[str, dict[str, Any]]:
    try:
        parsed = json.loads(raw_schema)
    except json.JSONDecodeError as error:
        raise DtypeContractError(f"schema is not valid JSON: {error.msg}") from error
    return validate_schema(parsed)


def run_cli(argv: Sequence[str] | None = None) -> int:
    parser = argparse.ArgumentParser(
        description="Audit pandas dtypes and per-column missing-value policies"
    )
    parser.add_argument("input", type=Path)
    parser.add_argument(
        "--schema",
        required=True,
        help='JSON object: column -> {"dtype": ..., "nullable": ...}',
    )
    args = parser.parse_args(argv)

    try:
        schema = _parse_schema(args.schema)
        frame = pd.read_csv(
            args.input,
            dtype="string",
            keep_default_na=False,
        )
    except (DtypeContractError, OSError) as error:
        parser.error(str(error))

    _, report = audit_and_convert(frame, schema)
    print(json.dumps(report, ensure_ascii=False, indent=2))
    return 0 if report["valid"] else 1


def main() -> None:
    raise SystemExit(run_cli())


if __name__ == "__main__":
    main()
