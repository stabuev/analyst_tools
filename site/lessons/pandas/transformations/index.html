03/03: какие строки оставить?       -> выбор может изменить состав строк

Проблема

Таблица order_items имеет grain «одна строка на товарную позицию заказа». Пара order_id, product_id служит кандидатным бизнес-ключом в учебных данных. Для каждой позиции нужно рассчитать стоимость:

text
line_total = quantity * unit_price

Затем позиции с суммой не меньше заданного порога нужно отметить для ручной проверки и сохранить их сумму в отдельном столбце:

text
requires_review = line_total >= review_threshold

review_amount = line_total, если requires_review истинно
review_amount = 0,          если requires_review ложно
review_amount = неизвестно, если requires_review неизвестно

Первая реализация часто начинается так:

python
items["line_total"] = items.apply(
    lambda row: row["quantity"] * row["unit_price"],
    axis=1,
)

Код похож на чтение одной строки таблицы, поэтому кажется естественным. Но исходное правило уже выражается обычным умножением двух столбцов. Построчная функция добавляет скрытый механизм, не добавляя смысла:

  • pandas создает представление строки и вызывает Python-функцию для каждого наблюдения;
  • внутри callback смешиваются получение значений, арифметика и обработка пропусков;
  • dtype результата может быть труднее предсказать;
  • неизвестное логическое решение легко случайно превратить в False или получить ошибку о неоднозначной истинности pd.NA;
  • проверять отдельные правила сложнее, потому что они спрятаны в одной функции.

Главный вопрос урока поэтому не «как запретить apply». Вопрос звучит так:

Как выразить расчет над столбцами так, чтобы видеть единицу наблюдения, индекс, dtype, пропуски и правило каждой ветки?

Концепция

Производный столбец отвечает на вопрос о каждой исходной строке

Рассмотрим три класса табличных операций:

Операция Вопрос Что происходит со строками Пример
Выбор Какие наблюдения нужны? часть строк может исчезнуть оставить оплаченные заказы
Преобразование Какое новое значение относится к каждому наблюдению? строки и grain сохраняются рассчитать стоимость позиции
Агрегация Какой итог нужен для каждой группы? несколько строк превращаются в одну сумма позиций по заказу

После преобразования order_items по-прежнему содержит одну строку на товарную позицию. В таблице появляются новые переменные, но не новые виды наблюдений.

Для входа из n строк ожидается:

text
input.shape[0] == n
line_total.shape[0] == n
output.shape[0] == n
output.index.equals(input.index)

Совпадение числа строк необходимо, но недостаточно. Если значения переставились между метками, длина останется прежней, а деньги окажутся приписаны не тем позициям. Поэтому мы отдельно проверяем индекс и исходные идентификаторы.

Производный столбец — это маркированная Series

Выражение над столбцами возвращает не безымянный список, а Series:

python
line_total = items["quantity"] * items["unit_price"]

У результата есть:

  • одно значение на каждую согласованную метку;
  • собственный dtype;
  • индекс, связывающий значение с исходной строкой;
  • пропуск там, где результат нельзя определить из входных значений.

Если оба операнда взяты из одного DataFrame, они уже используют один индекс. Если же одна Series пришла из другого объекта, pandas сначала выровняет операнды по меткам. Одинаковая длина не отменяет это правило:

python
quantity.index.equals(unit_price.index)

проверяет метки и порядок. Но, как и в 03/03, совпадение индексов не доказывает общее происхождение данных. В production-коде связанный столбец лучше брать из той же проверенной таблицы либо объединять по объявленному бизнес-ключу в уроке о joins.

Сначала предскажите результат на нескольких строках

Используем таблицу с пятью товарными позициями и осмысленными метками индекса:

Индекс order_id product_id quantity unit_price
item-a O1001 P01 2 400.00
item-b O1001 P02 1 800.00
item-c O1002 P03 3 NA
item-d O1003 P04 1 75.50
item-e O1004 P05 1 0.00

Пусть review_threshold = 800. Тогда ожидаемый подробный результат:

Индекс line_total requires_review review_amount Почему
item-a 800.00 True 800.00 значение равно включительной границе
item-b 800.00 True 800.00 значение равно включительной границе
item-c NA NA NA неизвестная цена не позволяет принять решение
item-d 75.50 False 0.00 сумма известна и ниже порога
item-e 0.00 False 0.00 известный ноль отличается от пропуска

Эта таблица задает контракт лучше одной итоговой суммы. Если item-a и item-b переставить местами, общий итог не изменится. Построчная проверка по меткам обнаружит ошибку, а проверка одной суммы — нет.

Арифметика сохраняет неизвестность, если dtype ее поддерживает

Входные столбцы уже прошли урок 03/02:

text
quantity.dtype   == Int64
unit_price.dtype == Float64

Мы намеренно не вызываем здесь pd.to_numeric. Если функция снова разбирает сырой текст, становится невозможно понять, на какой стадии появилось невалидное значение. Более того, errors="coerce" может смешать ошибку входа с разрешенным пропуском.

Для nullable-типов pandas поэлементное умножение ведет себя ожидаемо:

text
2 * 400.0   -> 800.0
3 * pd.NA   -> pd.NA

В учебном контракте Int64 * Float64 дает Float64. Пропуск цены сохраняется как неизвестная сумма. Мы не заменяем его нулем: «цена неизвестна» не означает «товар бесплатный».

У арифметики есть и предметные предусловия. Для этого примера:

  • известное quantity должно быть положительным;
  • известное unit_price должно быть неотрицательным;
  • неизвестное значение разрешено, чтобы увидеть его распространение;
  • порог проверки должен быть конечным неотрицательным числом.

Эти проверки не заменяют полный dtype-аудит. Они защищают только предположения, необходимые именно этому расчету.

Сравнение тоже создает производный столбец

Следующий шаг:

python
requires_review = line_total.ge(review_threshold)

также является поэлементным преобразованием. Для каждого значения line_total pandas строит логическое решение с тем же индексом:

text
800.0 >= 800.0 -> True
75.5  >= 800.0 -> False
pd.NA >= 800.0 -> pd.NA

Здесь pd.NA — не ошибка вычисления и не сокращение для False. Это честный ответ: из неизвестной суммы нельзя определить, достигла ли позиция порога.

Так появляется nullable-маска dtype boolean, знакомая по 03/03. Но теперь мы не используем ее для удаления строк. Она становится самостоятельным производным признаком и условием выбора значений.

where выбирает значения, а не строки

Метод Series.where(condition, other) сохраняет форму исходной Series:

text
если condition истинно -> оставить исходное значение
иначе                   -> взять other

Например:

python
review_amount = line_total.where(requires_review, 0.0)

Это не аналог items.loc[requires_review]:

  • loc выбирает строки и может уменьшить таблицу;
  • where сохраняет все метки и выбирает значение для каждой из них.

Метод mask выражает противоположное условие:

text
series.where(condition, other) -> заменить там, где условие не True
series.mask(condition, other)  -> заменить там, где условие не False

Для явного True и False удобно использовать такую таблицу:

Условие value.where(condition, other) value.mask(condition, other)
True оставить value взять other
False взять other оставить value

Но nullable-условие добавляет третий случай.

pd.NA в условии не сохраняется автоматически

В текущем окружении курса pandas 3.0.3 и where, и mask используют замену other на позиции с pd.NA в nullable-условии:

Условие amount.where(condition, 0) amount.mask(condition, 0)
True amount 0
False 0 amount
pd.NA 0 0

Поэтому эта короткая запись:

python
review_amount = line_total.where(requires_review, 0.0)

незаметно превращает неизвестное решение в известный ноль. Код исполняется, dtype выглядит правдоподобно, итоговая сумма может пройти поверхностную проверку. Но предметный смысл изменился.

Нужна явная политика. В этом уроке неизвестное решение должно остаться неизвестным:

python
review_amount = line_total.where(requires_review, 0.0)
review_amount = review_amount.mask(requires_review.isna(), pd.NA)

Вторая строка использует обычную двухзначную маску requires_review.isna():

  • True ровно там, где исходное решение неизвестно;
  • False во всех остальных строках;
  • собственного pd.NA у результата isna() нет.

Другие задачи могут выбрать другую политику:

  • остановиться, если условие содержит NA;
  • отправить неизвестные строки в отдельный отчет;
  • осознанно считать неизвестное условие ложным.

Последний вариант допустим только как названное бизнес-правило. Метод where не должен принимать это решение за аналитика.

Добавляйте признаки в независимый результат

Функция преобразования обычно не должна неожиданно менять объект, который передал вызывающий код. Поэтому сначала создаем результат:

python
result = frame.copy()

Затем добавляем каждую проверенную Series:

python
result["line_total"] = line_total
result["requires_review"] = requires_review
result["review_amount"] = review_amount

Такой порядок делает промежуточные значения наблюдаемыми. Если расчет сломался, можно отдельно проверить арифметику, условие и выбор значений.

В 03/10 мы соберем несколько преобразований через assign и pipe. Здесь длинная цепочка преждевременна: сначала нужно научиться видеть контракт каждого нового столбца.

apply(axis=1) — не ошибка сам по себе, а более дорогая абстракция

Проблема не в слове apply и не в нарушении стилевого запрета. Построчный callback может быть оправдан, когда правило действительно требует совместно разобрать несколько разнородных полей и не выражается понятной композицией доступных операций.

Перед DataFrame.apply(axis=1) задайте четыре вопроса:

  1. Есть ли готовая операция Series для каждого отдельного шага?
  2. Можно ли разложить правило на именованные арифметические выражения и маски?
  3. Станут ли dtype, NA и границы понятнее после такого разложения?
  4. Если callback все еще необходим, проверили ли мы его на минимальных строках и измерили ли стоимость на рабочем объеме?

Для quantity * unit_price ответы на первые три вопроса очевидны, поэтому apply не нужен. Производительность здесь является дополнительным аргументом, а не главным доказательством. Даже на маленькой таблице выражение над Series лучше показывает смысл расчета.

Не все методы с названием apply одинаковы, и не всякая короткая операция pandas гарантированно быстрее callback. Полный benchmark уже был в 02/09; в этом уроке мы не повторяем его и не обещаем универсального ускорения.

Урок отвечает на один вопрос: как безопасно добавить значения к тем же наблюдениям.

Соберите это

До pandas сделайте механизм видимым на обычных значениях Python. Используйте None как обозначение неизвестного значения:

python
quantities = [2, 1, 3, 1, 1]
unit_prices = [400.0, 800.0, None, 75.5, 0.0]
review_threshold = 800.0

line_totals = []
requires_review = []
review_amounts = []

for quantity, unit_price in zip(quantities, unit_prices, strict=True):
    if quantity is None or unit_price is None:
        line_total = None
        review = None
        review_amount = None
    else:
        line_total = quantity * unit_price
        review = line_total >= review_threshold
        review_amount = line_total if review else 0.0

    line_totals.append(line_total)
    requires_review.append(review)
    review_amounts.append(review_amount)

Проверьте наиболее подробный результат:

python
assert line_totals == [800.0, 800.0, None, 75.5, 0.0]
assert requires_review == [True, True, None, False, False]
assert review_amounts == [800.0, 800.0, None, 0.0, 0.0]

Этот цикл не является рекомендуемой реализацией pandas. Он нужен как прозрачная контрольная модель:

  • видно, что вычисляется для одной позиции;
  • явно названа включительная граница >=;
  • неизвестная цена проходит через все зависимые результаты;
  • список не становится короче;
  • каждое выходное значение остается на позиции своего входа.

Используйте это

Шаг 1. Создайте уже типизированный вход

Урок начинается после dtype-аудита. Поэтому учебный DataFrame создается сразу с контрактными типами:

python
import pandas as pd

items = pd.DataFrame(
    {
        "order_id": ["O1001", "O1001", "O1002", "O1003", "O1004"],
        "product_id": ["P01", "P02", "P03", "P04", "P05"],
        "quantity": [2, 1, 3, 1, 1],
        "unit_price": [400.0, 800.0, pd.NA, 75.5, 0.0],
    },
    index=pd.Index(
        ["item-a", "item-b", "item-c", "item-d", "item-e"],
        name="row_label",
    ),
).astype(
    {
        "order_id": "string",
        "product_id": "string",
        "quantity": "Int64",
        "unit_price": "Float64",
    }
)

В реальном pipeline это результат предыдущей проверенной стадии, а не повод вызывать astype перед каждым расчетом.

До преобразования зафиксируйте структурные ожидания:

python
input_index = items.index.copy()
input_rows = len(items)
input_keys = items[["order_id", "product_id"]].copy()

Шаг 2. Рассчитайте сумму каждой позиции

python
line_total = (items["quantity"] * items["unit_price"]).astype("Float64")

Проверьте не только числа:

python
assert line_total.index.equals(items.index)
assert str(line_total.dtype) == "Float64"
assert line_total.loc["item-a"] == 800.0
assert pd.isna(line_total.loc["item-c"])

Шаг 3. Постройте nullable-условие

python
review_threshold = 800.0
requires_review = line_total.ge(review_threshold).astype("boolean")
python

Типичные ошибки

«Операция в одну строку не требует контракта»

Короткое выражение может сохранить неправильный dtype, выровнять несвязанные индексы или заменить неизвестность нулем. Длина кода не определяет цену ошибки.

«Векторизация означает только скорость»

Для аналитика не менее важна наблюдаемость: отдельные Series позволяют проверить формулу, условие, dtype и пропуски независимо. Конкретное ускорение зависит от операции и данных.

«where фильтрует таблицу»

where выбирает значения и сохраняет форму. Для выбора строк используется маска вместе с loc, как в 03/03.

«pd.NA сам пройдет через любую условную операцию»

Арифметика nullable-типов обычно распространяет pd.NA, но where и mask с заданным other используют замену и для неизвестного условия. Политику нужно проверить для конкретной операции.

«Если функция принимает сырые строки, она удобнее»

Функция, которая одновременно разбирает типы и считает признаки, скрывает границу ответственности. Неудобный отказ на неправильном dtype полезнее правдоподобного расчета по испорченным данным.

«Новый столбец автоматически доказывает сохранение grain»

Внутри функции могли появиться фильтрация, сортировка или агрегация. Проверяйте число строк, индекс и исходные идентификаторы результата.

from __future__ import annotations

import importlib.util
import unittest
from pathlib import Path

import pandas as pd

ROOT = Path(__file__).resolve().parents[1]
ARTIFACT = ROOT / "outputs" / "order_transforms.py"
SPEC = importlib.util.spec_from_file_location("order_transforms", ARTIFACT)
if SPEC is None or SPEC.loader is None:
    raise RuntimeError(f"cannot load {ARTIFACT}")
TRANSFORMS = importlib.util.module_from_spec(SPEC)
SPEC.loader.exec_module(TRANSFORMS)


def make_items() -> pd.DataFrame:
    return pd.DataFrame(
        {
            "order_id": ["O1001", "O1001", "O1002", "O1003", "O1004"],
            "product_id": ["P01", "P02", "P03", "P04", "P05"],
            "quantity": [2, 1, 3, 1, 1],
            "unit_price": [400.0, 800.0, pd.NA, 75.5, 0.0],
        },
        index=pd.Index(
            ["item-a", "item-b", "item-c", "item-d", "item-e"],
            name="row_label",
        ),
    ).astype(
        {
            "order_id": "string",
            "product_id": "string",
            "quantity": "Int64",
            "unit_price": "Float64",
        }
    )


class LineItemFeaturesTest(unittest.TestCase):
    def setUp(self) -> None:
        self.items = make_items()

    def transform(self) -> pd.DataFrame:
        return TRANSFORMS.add_line_item_features(
            self.items,
            review_threshold=800.0,
        )

    def test_line_total_matches_manual_row_calculation(self) -> None:
        result = self.transform()
        expected = pd.Series(
            [800.0, 800.0, pd.NA, 75.5, 0.0],
            index=self.items.index,
            dtype="Float64",
            name="line_total",
        )
        pd.testing.assert_series_equal(result["line_total"], expected)

    def test_threshold_is_inclusive(self) -> None:
        result = self.transform()
        self.assertEqual(result.loc["item-a", "requires_review"], True)
        self.assertEqual(result.loc["item-b", "requires_review"], True)
        self.assertEqual(result.loc["item-d", "requires_review"], False)

    def test_unknown_condition_remains_unknown(self) -> None:
        result = self.transform()
        self.assertTrue(pd.isna(result.loc["item-c", "requires_review"]))
        self.assertTrue(pd.isna(result.loc["item-c", "review_amount"]))

    def test_known_false_condition_becomes_zero(self) -> None:
        result = self.transform()
        self.assertEqual(result.loc["item-d", "review_amount"], 0.0)
        self.assertEqual(result.loc["item-e", "review_amount"], 0.0)

    def test_known_true_condition_keeps_line_total(self) -> None:
        result = self.transform()
        self.assertEqual(result.loc["item-a", "review_amount"], 800.0)
        self.assertEqual(result.loc["item-b", "review_amount"], 800.0)

    def test_output_preserves_row_count_index_and_existing_columns(self) -> None:
        result = self.transform()
        self.assertEqual(len(result), len(self.items))
        self.assertTrue(result.index.equals(self.items.index))
        pd.testing.assert_frame_equal(result[self.items.columns], self.items)

    def test_input_frame_is_not_modified(self) -> None:
        before = self.items.copy()
        self.transform()
        pd.testing.assert_frame_equal(self.items, before)

    def test_derived_columns_use_nullable_dtypes(self) -> None:
        result = self.transform()
        self.assertEqual(str(result["line_total"].dtype), "Float64")
        self.assertEqual(str(result["requires_review"].dtype), "boolean")
        self.assertEqual(str(result["review_amount"].dtype), "Float64")

    def test_missing_required_column_is_rejected(self) -> None:
        with self.assertRaisesRegex(
            TRANSFORMS.TransformContractError,
            "missing columns.*unit_price",
        ):
            TRANSFORMS.add_line_item_features(
                self.items.drop(columns="unit_price"),
                review_threshold=800.0,
            )

from __future__ import annotations

import math
from numbers import Real

import pandas as pd


class TransformContractError(ValueError):
    """Raised when a table cannot be transformed without changing its meaning."""


def _validate_review_threshold(review_threshold: Real) -> float:
    if isinstance(review_threshold, bool) or not isinstance(review_threshold, Real):
        raise TransformContractError("review_threshold must be a finite non-negative number")
    threshold = float(review_threshold)
    if not math.isfinite(threshold) or threshold < 0:
        raise TransformContractError("review_threshold must be a finite non-negative number")
    return threshold


def _validate_input(frame: pd.DataFrame) -> None:
    required = {"quantity", "unit_price"}
    missing = sorted(required - set(frame.columns))
    if missing:
        raise TransformContractError(f"missing columns: {missing}")

    produced = {"line_total", "requires_review", "review_amount"}
    collisions = sorted(produced & set(frame.columns))
    if collisions:
        raise TransformContractError(f"output columns already exist: {collisions}")

    expected_dtypes = {"quantity": "Int64", "unit_price": "Float64"}
    wrong_dtypes = {
        column: {"expected": expected, "actual": str(frame[column].dtype)}
        for column, expected in expected_dtypes.items()
        if str(frame[column].dtype) != expected
    }
    if wrong_dtypes:
        raise TransformContractError(
            f"input must already satisfy the dtype contract: {wrong_dtypes}"
        )

    quantity_is_invalid = frame["quantity"].notna() & frame["quantity"].le(0)
    price_is_invalid = frame["unit_price"].notna() & frame["unit_price"].lt(0)
    if quantity_is_invalid.any():
        labels = frame.index[quantity_is_invalid].tolist()
        raise TransformContractError(f"quantity must be positive at rows: {labels}")
    if price_is_invalid.any():
        labels = frame.index[price_is_invalid].tolist()
        raise TransformContractError(f"unit_price must be non-negative at rows: {labels}")


def add_line_item_features(
    frame: pd.DataFrame,
    *,
    review_threshold: Real,
) -> pd.DataFrame:
    """Add row-preserving line-item features to an already typed DataFrame.

    Missing quantity or price remains unknown in every dependent feature. The input
    object is not modified; its row count, index and existing columns are preserved.
    """

    threshold = _validate_review_threshold(review_threshold)
    _validate_input(frame)

    result = frame.copy()

    line_total = (result["quantity"] * result["unit_price"]).astype("Float64")
    requires_review = line_total.ge(threshold).astype("boolean")

    # Series.where uses ``other`` not only for False, but also for pd.NA in a nullable
    # condition. Restore unknown decisions explicitly instead of turning them into zero.
    review_amount = line_total.where(requires_review, 0.0).astype("Float64")
    review_amount = review_amount.mask(requires_review.isna(), pd.NA)

    result["line_total"] = line_total
    result["requires_review"] = requires_review
    result["review_amount"] = review_amount.astype("Float64")
    return result
