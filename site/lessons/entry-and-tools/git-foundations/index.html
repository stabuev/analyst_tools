Проверка перед уроком

Ответьте до чтения: результат поможет заметить исходные предположения.

01

Аналитик изменил формулу метрики, но через неделю не помнит, что именно и зачем менялось. Какую проблему здесь прежде всего должен решать Git?

Git хранит версии проекта и позволяет сравнивать их. Он не проверяет смысл формулы, не заменяет документацию и сам по себе ничего не публикует в облако.

02

Какая группа файлов обычно требует исключения из истории аналитического проекта?

Секреты, локальные выгрузки и воспроизводимые временные файлы обычно не должны попадать в Git. Код, документация и небольшие безопасные fixtures, наоборот, часто являются частью проекта.

Git: история аналитического проекта

Git помогает не просто хранить файлы, а сохранять понятную историю решений: что изменилось, зачем это сделали и к какому состоянию можно вернуться.

Тип: Build Треки: Core Пререквизиты: 00/01 — Терминал и файловая система Время: ~75 минут Результат: создаёт локальный Git-репозиторий аналитического проекта, осознанно подготавливает изменения, делает понятные коммиты и не добавляет локальные данные в историю

Цели обучения

  • Объяснять человеческим языком, что такое Git, репозиторий и коммит.
  • Различать файлы в рабочей папке, подготовленные изменения и сохранённую историю.
  • Пользоваться базовым циклом status → diff → add → diff --cached → commit.
  • Собирать в один коммит связанные изменения, даже если они находятся в нескольких файлах.
  • Настраивать .gitignore до появления выгрузок, секретов и временных результатов.
  • Читать локальную историю и безопасно убирать случайно подготовленный файл.

Проблема

Представьте обычную аналитическую задачу. В понедельник запрос считал выручку по всем заказам. Во вторник вы добавили условие status = 'paid', поправили определение метрики и пересчитали отчёт. В пятницу значение перестало совпадать с дашбордом.

Если проект существует только как папка analysis_final_2_really_final, трудно ответить на простые вопросы:

  • какая формула использовалась в понедельник;
  • почему появилось новое условие;
  • менялись ли одновременно запрос и описание метрики;
  • можно ли вернуть рабочее состояние, не отменяя остальные правки;
  • не попала ли в папку с кодом локальная выгрузка или .env с паролем.

Git сохраняет последовательность проверяемых состояний проекта. Каждое такое состояние называется коммитом. Коммит можно прочитать, сравнить с предыдущим и использовать как точку возврата.

Проблемы с Git обычно возникают не из-за сложных команд. Чаще человек не понимает, с какой из трёх областей он сейчас работает: с файлами на диске, с подготовленным набором изменений или с уже сохранённой историей. Поэтому сначала построим простую модель, а затем несколько раз повторим один и тот же рабочий цикл.

Концепция

Что такое Git

Git — программа для управления версиями файлов. Она особенно полезна для текстовых материалов, где можно увидеть отдельные строки: Python-кода, SQL-запросов, Markdown, конфигурации и небольших тестовых данных.

Папка проекта, в которой Git хранит историю, называется репозиторием. После команды git init внутри неё появляется скрытый служебный каталог .git. В нём находятся коммиты и настройки репозитория. Не редактируйте и не удаляйте .git вручную.

Git не является:

  • автоматическим сохранением каждого нажатия клавиши;
  • облачным диском или резервной копией компьютера;
  • GitHub, GitLab или другим сайтом;
  • хранилищем для произвольно больших выгрузок;
  • защитой секрета, который уже попал в историю.

В этом уроке всё происходит локально, на вашем компьютере. Удалённые репозитории, ветки и pull request появятся в следующем уроке.

Репозиторий и его корень

У репозитория есть корневая папка. Команда:

bash
git rev-parse --show-toplevel

показывает её полный путь. Это полезная диагностика: если Git показывает неожиданный проект, вы выполняете команды не там, где думаете.

git status работает для всего репозитория, даже если команда запущена из вложенного каталога. Поэтому перед первым git add всегда полезно проверить и текущий каталог через pwd, и корень репозитория через git rev-parse --show-toplevel.

Три области, между которыми движутся изменения

Для начала достаточно такой модели:

text
файлы на диске  -- git add -->  подготовленные изменения  -- git commit -->  история
рабочая папка                  staging area                           репозиторий
  1. Рабочая папка — файлы, которые вы видите и редактируете.
  2. Staging area, или область подготовленных изменений, — точное содержимое, которое вы выбрали для следующего коммита. В документации Git её также называют index.
  3. История — уже сохранённые коммиты.

Staging area нужна, потому что не все текущие правки обязательно относятся к одному решению. Например, вы можете одновременно исправить SQL-запрос и набросать черновик совсем другого анализа. git add позволяет подготовить только SQL-запрос.

Главное значение команд:

Команда Что делает Чего не делает
git status Показывает состояние файлов Не меняет файлы и историю
git diff Показывает ещё не подготовленные изменения Не показывает содержимое новых untracked-файлов
git add путь Копирует текущую версию пути в staging area Не создаёт коммит и не отправляет файл в интернет
git diff --cached Показывает подготовленные изменения Не показывает более новые правки после git add
git commit -m "..." Сохраняет подготовленное состояние локально Не включает всё содержимое папки автоматически
git log --oneline Показывает краткую историю Не изменяет историю

Состояния файла

Новый файл сначала untracked: он существует на диске, но Git ещё не знает, нужно ли его хранить. После git add файл становится staged, то есть подготовленным. После коммита он становится tracked — Git знает его последнюю сохранённую версию.

Если изменить tracked-файл, он станет modified. Чтобы новое содержимое попало в следующий коммит, его снова нужно подготовить через git add.

text
новый файл:       untracked -- git add --> staged -- git commit --> tracked
известный файл:   tracked   -- изменение --> modified -- git add --> staged

git add запоминает версию файла именно в момент выполнения команды. Если после этого снова отредактировать файл, подготовленная и текущая версии будут различаться. Это не поломка: Git честно хранит обе версии и показывает их в разных разделах git status.

Что именно сохраняет коммит

Коммит — не команда «сохранить последний файл». Это согласованное состояние всех отслеживаемых файлов проекта, в котором новые изменения взяты из staging area. Коммит содержит сообщение, автора, время и связь с предыдущим коммитом.

Хороший коммит отвечает на вопрос: какое одно законченное изменение появилось в проекте? Например:

text
Add daily revenue query
Count only paid orders in revenue
Document cancellation rule
Exclude local exports from Git

Сообщения update, fix, правки и final почти ничего не объясняют. Сам факт изменения Git и так хранит.

Атомарный коммит не обязан содержать ровно один файл. Если изменение бизнес-правила требует одновременно поправить SQL и документацию метрики, эти два файла логично сохранить вместе. Один файл тоже не гарантирует атомарность: в нём могут быть смешаны несколько несвязанных решений.

Что хранить в Git в аналитическом проекте

Обычно храним Обычно не храним Почему
SQL, Python и небольшие shell-скрипты Сырые выгрузки из хранилища Выгрузки могут быть большими, чувствительными и быстро устаревать
Определения метрик и README .env и файлы с токенами Секрет нельзя делать частью истории
Конфигурацию без секретов .venv/, кэш Python и Jupyter Это локальные или автоматически создаваемые файлы
Тесты и маленькие обезличенные fixtures Временные графики и локальные отчёты Их можно пересоздать из кода и данных
Ноутбуки, если они являются частью результата .ipynb_checkpoints/ Checkpoints создаёт Jupyter, они не описывают решение

Не стоит автоматически игнорировать все *.csv: маленький обезличенный CSV может быть полезным тестовым примером. Безопаснее явно отделить локальные выгрузки каталогом вроде data/raw/.

Соберите это

Создадим небольшой проект расчёта дневной выручки. Все команды выполняются из корня курса. Учебный репозиторий будет находиться в work/, который не входит в историю самого курса.

Шаг 1. Проверьте Git и рабочую папку

bash
git --version
pwd
mkdir -p "work/00-02-git/revenue-project"
cd "work/00-02-git/revenue-project"
pwd

Первая команда должна напечатать версию Git. Если shell сообщает command not found, Git ещё не установлен и продолжать практику пока нельзя.

Команда pwd должна заканчиваться на work/00-02-git/revenue-project. Эта проверка важнее, чем кажется: большинство команд Git применяются ко всему найденному репозиторию, а не только к файлу, который открыт в редакторе.

Шаг 2. Создайте локальный репозиторий

bash
git init
git rev-parse --show-toplevel
git status

git init не меняет содержимое аналитических файлов и не публикует их. Команда создаёт служебный каталог .git. git status сообщит, что коммитов пока нет.

Настройте имя и email только для учебного репозитория:

bash
git config user.name "Course Student"
git config user.email "student@example.com"

Эти значения попадут в автора коммита. Отсутствие --global означает, что урок не изменяет настройки других проектов. В рабочем репозитории используйте настоящее имя и корпоративный email, принятые в вашей команде.

Шаг 3. Сначала определите, что нельзя сохранять

Создайте файл .gitignore со следующим содержимым:

gitignore
.venv/
__pycache__/
*.py[cod]
.ipynb_checkpoints/
.env
data/raw/
outputs/local/

Правила читаются сверху вниз, по одному на строку. Завершающий / означает каталог. Правило .env относится к файлу с таким именем в любой вложенности, а data/raw/ — к каталогу сырых данных внутри проекта.

Теперь создайте README.md:

markdown
# Revenue project

Учебный проект расчёта дневной выручки.

Проверьте состояние:

bash
git status

Оба файла появятся в разделе Untracked files. Git видит их, но пока не планирует сохранять. Подготовьте только эти пути:

bash
git add README.md .gitignore
git status
git diff --cached

Теперь файлы находятся в разделе Changes to be committed. Команда git diff --cached показывает точное содержимое первого коммита. Прочитайте его, даже если только что сами создали файлы: привычка важнее размера примера.

Сохраните первый коммит:

bash
git commit -m "Initialize revenue project"
git status
git log --oneline

После коммита git status должен сообщить, что рабочее дерево чистое. Это означает, что текущие tracked-файлы совпадают с последним коммитом. Это не означает, что проект синхронизирован с облаком: удалённого репозитория у нас пока нет.

Шаг 4. Добавьте аналитическое решение

Создайте каталоги queries и docs:

bash
mkdir -p queries docs

В queries/revenue_by_day.sql сохраните запрос:

sql
SELECT
    order_date,
    SUM(amount) AS revenue
FROM orders
GROUP BY order_date;

В docs/metric-definition.md сохраните определение:

markdown
# Дневная выручка

Сумма `amount` всех заказов за календарный день.

Сначала посмотрите состояние:

bash
git status
git diff

git status покажет два новых untracked-файла. Обычный git diff может ничего не напечатать: пока Git не отслеживает новый файл, ему не с чем сравнивать его построчно. Это частый источник недоумения у начинающих и нормальное поведение.

Подготовьте оба файла, потому что они вместе вводят одно аналитическое решение — запрос и определение результата:

bash
git add queries/revenue_by_day.sql docs/metric-definition.md
git status
git diff --cached
git commit -m "Add daily revenue calculation"

Несколько файлов в одном коммите здесь не являются ошибкой. Без определения метрики запрос неоднозначен, а определение без реализующего его запроса непроверяемо.

Шаг 5. Измените бизнес-правило

Заказчик уточнил: в выручку должны входить только оплаченные заказы. Добавьте в запрос условие перед GROUP BY:

sql
WHERE status = 'paid'

Одновременно измените определение метрики:

markdown
Сумма `amount` оплаченных заказов (`status = 'paid'`) за календарный день.

Теперь файлы уже tracked, поэтому обычный diff покажет изменения до staging:

bash
git status
git diff

Проверьте две вещи: SQL и документация описывают одно правило, а случайных изменений в diff нет. Затем подготовьте оба пути и проверьте уже staging area:

bash
git add queries/revenue_by_day.sql docs/metric-definition.md
git diff --cached
git commit -m "Count only paid orders in revenue"

Получилась история из трёх осмысленных шагов:

bash
git log --oneline
git show --stat

git show --stat без дополнительных аргументов показывает последний коммит и список затронутых файлов. Полный git show дополнительно покажет изменения строк.

Шаг 6. Убедитесь, что локальные данные игнорируются

Создайте маленькую имитацию локальной выгрузки:

bash
mkdir -p data/raw
printf 'order_id,order_date,status,amount\n101,2026-01-10,paid,120\n' \
  > data/raw/orders.csv
git status

data/raw/orders.csv не должен появиться в статусе. Git не удалил файл: он по-прежнему лежит на диске, но правило просит не предлагать его как untracked.

Если нужно понять, какое правило сработало, используйте:

bash
git check-ignore -v data/raw/orders.csv

В выводе будут имя ignore-файла, номер строки, правило и проверенный путь.

Используйте это

В реальной работе повторяется один короткий цикл.

Перед изменением

bash
git status

Убедитесь, что понимаете уже существующие незавершённые правки. Не начинайте новую задачу поверх неизвестного состояния.

После изменения

bash
git status
git diff

Сначала выясните, какие файлы изменились, затем прочитайте разницу. Для новых файлов откройте содержимое в редакторе: обычный diff ещё не покажет их.

Перед коммитом

bash
git add путь/к/связанному-файлу другой/связанный-файл
git status
git diff --cached

На этом шаге вы отвечаете не на вопрос «что я сегодня редактировал?», а на вопрос «какое одно законченное изменение я сейчас сохраняю?».

После коммита

bash
git commit -m "Describe the completed change"
git status
git log -1 --oneline

Проверьте, что коммит появился, а оставшиеся изменения ожидаемы. Чистый статус удобен, но не является самоцелью: иногда вы намеренно оставляете черновик для другого коммита.

Запустите поставляемую проверку из корня курса:

bash
cd ../../..
python3 \
  phases/00-entry-and-tools/02-git-foundations/outputs/git_project_check.py \
  work/00-02-git/revenue-project

Проверка подтверждает наблюдаемые свойства учебного репозитория: наличие истории, tracked-файла .gitignore, необходимых ignore-правил, отсутствие локальных данных в истории и понятное текущее состояние. Она выводит сообщения коммитов для человеческой проверки, но не пытается автоматически угадать их смысл.

Готовая демонстрация создаёт аналогичный временный проект и запускает ту же проверку:

bash
python3 phases/00-entry-and-tools/02-git-foundations/code/main.py

Python здесь не является темой урока. Скрипт нужен только для воспроизводимой проверки Git-workflow курса.

Сломайте это

Случайно подготовьте лишний файл

Предположим, вместе с нужным SQL вы выполнили git add notes/draft.md, но черновик не должен входить в коммит. Уберите только его из staging area:

bash
git restore --staged notes/draft.md
git status

Файл останется на диске со всем содержимым. Команда меняет только состав следующего коммита. Это безопаснее, чем удалять файл или откатывать работу.

Если нужно очистить весь staging area и заново выбрать файлы:

bash
git restore --staged .

После этого обязательно снова выполните git status.

Измените файл после git add

Если подготовить файл, а затем снова его отредактировать, git status покажет один путь и среди подготовленных, и среди ещё не подготовленных изменений. В следующий коммит попадёт версия на момент последнего git add, а более свежая правка останется на диске.

Сравните две области:

bash
git diff --cached
git diff

Если в коммит должна войти последняя версия, повторите git add путь. Если вы не уверены в составе коммита, выполните git restore --staged путь и соберите его заново.

Добавьте .gitignore слишком поздно

.gitignore влияет только на файлы, которые Git ещё не отслеживает. Если data/raw/orders.csv уже был закоммичен, новое правило не удалит его из истории и не заставит Git забыть файл.

Проверить, отслеживается ли конкретный путь, можно так:

bash
git ls-files --error-unmatch data/raw/orders.csv

Если команда напечатала путь, файл tracked. Для обычного несекретного файла можно перестать отслеживать будущие версии через git rm --cached, но старые коммиты сохранят содержимое. Если в историю попал пароль, токен или чувствительная выгрузка, не продолжайте как ни в чём не бывало: секрет нужно отозвать, а инцидент разобрать. Это рассматривается в уроке 00/05.

Выполните git add . без проверки

git add . не является запрещённой командой. Она удобна, когда все изменения в текущем каталоге относятся к одной задаче и вы уже прочитали git status и git diff.

Риск появляется, когда точка используется вместо выбора: вместе с SQL в staging area могут попасть черновик, временный отчёт и несвязанная правка README. После git add . всегда проверяйте:

bash
git status
git diff --cached

Назовите коммит update

Сообщение update не помогает понять историю. Хорошее короткое сообщение обычно называет завершённое изменение и его объект:

text
Плохо:    update
Плохо:    final fix
Лучше:    Count only paid orders in revenue
Лучше:    Document timezone used for daily metrics

Причину, ограничения и ссылку на задачу команда при необходимости добавляет в расширенное описание коммита или pull request. На первом занятии достаточно научиться ясно называть само изменение.

Quiz · 4 вопроса

Проверьте себя

Ответьте после практики. Объяснение появится сразу после выбора.

01

Вы изменили tracked SQL-файл и хотите сначала увидеть ещё не подготовленные изменения. Какая команда отвечает на этот вопрос?

Обычный git diff сравнивает текущие файлы с подготовленным состоянием и показывает ещё не staged-изменения. git diff --cached используется после git add для проверки следующего коммита.

02

После git add analyst.sql вы снова отредактировали этот файл. Какая версия попадёт в коммит, если сразу выполнить git commit?

git add помещает в staging area содержимое на момент команды. Более новая правка останется unstaged, пока файл не будет подготовлен повторно.

03

В staging area случайно попал notes/draft.md. Как убрать его из следующего коммита, сохранив содержимое на диске?

git restore --staged меняет состав staging area, но не удаляет рабочий файл и не отменяет сделанные в нём правки.

04

Одно изменение бизнес-правила требует поправить SQL и определение метрики. Как лучше сформировать коммит?

Атомарность определяется одной причиной изменения, а не числом файлов. SQL и согласованное определение метрики образуют одно проверяемое решение.

        with TemporaryDirectory() as directory:
            root = Path(directory)
            build_valid_repository(root)
            raw_extract = root / "data" / "raw" / "orders.csv"

        self.assertTrue(report["ready"])
        self.assertEqual(len(report["history"]), 2)

        with TemporaryDirectory() as directory:

                {"README.md": "# Project\n", ".gitignore": full_gitignore()},
                "Initialize project",

        check = next(item for item in report["checks"] if item["id"] == "history")
        self.assertFalse(check["passed"])
        self.assertFalse(report["ready"])

from __future__ import annotations

import argparse
import json
import subprocess
from pathlib import Path
from typing import Any


REQUIRED_IGNORED_PATHS = (
    ".env",
    "data/raw/orders.csv",
    "outputs/local/report.html",
)
PROTECTED_TRACKED_PATHS = (
    ".env",
    "data/raw/",
    "outputs/local/",
)


def run_git(
    root: Path,
    *arguments: str,
    check: bool = True,
) -> subprocess.CompletedProcess[str]:
    result = subprocess.run(
        ["git", "-C", str(root), "--no-pager", *arguments],
        capture_output=True,
        text=True,
        check=False,
    )
    if check and result.returncode != 0:
        message = result.stderr.strip() or result.stdout.strip() or "unknown Git error"
        raise RuntimeError(message)
    return result


def resolve_repository(path: Path) -> Path:
    candidate = path.expanduser()
    if not candidate.exists():
        raise ValueError(f"path does not exist: {candidate}")
    result = run_git(candidate, "rev-parse", "--show-toplevel", check=False)
    if result.returncode != 0:
        raise ValueError(f"not a Git repository: {candidate}")
    return Path(result.stdout.strip()).resolve()


def split_nul(value: str) -> list[str]:
    return [item for item in value.split("\0") if item]


def commit_history(root: Path) -> list[dict[str, str]]:
    has_commit = run_git(
        root,
        "rev-parse",
        "--verify",
        "HEAD",
        check=False,
    ).returncode == 0
    if not has_commit:
        return []

    result = run_git(root, "log", "--reverse", "--format=%h%x1f%s%x1e")
    history: list[dict[str, str]] = []
    for record in result.stdout.split("\x1e"):
        record = record.strip()
        if not record:
            continue
        short_hash, subject = record.split("\x1f", 1)
        history.append({"hash": short_hash, "subject": subject})
    return history


def is_ignored(root: Path, relative_path: str) -> bool:
    result = run_git(
        root,
        "check-ignore",
        "--no-index",
        "--quiet",
        "--",
        relative_path,
        check=False,
    )
    return result.returncode == 0


def is_protected_path(path: str) -> bool:
    for protected in PROTECTED_TRACKED_PATHS:
        if protected.endswith("/") and path.startswith(protected):
            return True
        if path == protected:
            return True
    return False


def evaluate_repository(path: Path, min_commits: int = 2) -> dict[str, Any]:
    if min_commits < 1:
        raise ValueError("min_commits must be positive")

    root = resolve_repository(path)
    history = commit_history(root)
    status = split_nul(
        run_git(
            root,
            "status",
            "--porcelain=v1",
            "-z",
            "--untracked-files=all",
        ).stdout
    )
    tracked = split_nul(run_git(root, "ls-files", "-z").stdout)
    protected_tracked = [item for item in tracked if is_protected_path(item)]
    ignore_results = {
        relative_path: is_ignored(root, relative_path)
        for relative_path in REQUIRED_IGNORED_PATHS
    }
    gitignore_tracked = ".gitignore" in tracked

    missing_ignore_rules = [
        path for path, ignored in ignore_results.items() if not ignored
    ]
    checks = [
        {
            "id": "history",
            "passed": len(history) >= min_commits,
            "message": (
                f"Коммитов в истории: {len(history)}; требуется минимум {min_commits}."
            ),
        },
        {
            "id": "clean-tree",
            "passed": not status,
            "message": (
                "Рабочая папка и staging area не содержат незавершённых изменений."
                if not status
                else f"Незавершённых путей: {len(status)}. Проверьте git status."
            ),
        },
        {
            "id": "gitignore-tracked",
            "passed": gitignore_tracked,
            "message": (
                ".gitignore хранится в репозитории."
                if gitignore_tracked
                else ".gitignore не отслеживается Git."
            ),
        },
        {
            "id": "ignore-rules",
            "passed": not missing_ignore_rules,
            "message": (
                "Локальная конфигурация, сырые данные и локальные отчёты игнорируются."
                if not missing_ignore_rules
                else "Не покрыты ignore-правилом: " + ", ".join(missing_ignore_rules)
            ),
        },
        {
            "id": "protected-paths",
            "passed": not protected_tracked,
            "message": (
                "Защищаемые локальные пути не отслеживаются."
                if not protected_tracked
                else "В истории находятся локальные пути: "
                + ", ".join(protected_tracked)
            ),
        },
    ]
    return {
        "repository": str(root),
        "ready": all(check["passed"] for check in checks),
        "checks": checks,
        "history": history,
        "working_tree": status,
        "protected_tracked": protected_tracked,
        "ignore_results": ignore_results,
    }


def markdown_escape(value: str) -> str:
    return value.replace("\\", "\\\\").replace("|", "\\|").replace("`", "'")


def render_markdown(report: dict[str, Any]) -> str:
    lines = [
        "# Git project check",
        "",
        f"- Repository: `{markdown_escape(report['repository'])}`",
        f"- Result: **{'ready' if report['ready'] else 'needs work'}**",
        "",
        "## Checks",
        "",
        "| Check | Result | Details |",
        "|---|---|---|",
    ]
    for check in report["checks"]:
        result = "PASS" if check["passed"] else "FAIL"
        lines.append(
            f"| `{check['id']}` | {result} | "
            f"{markdown_escape(check['message'])} |"
        )

    lines.extend(
        [
            "",
            "## History for human review",
            "",
            "| Commit | Subject |",
            "|---|---|",
        ]
    )
    if report["history"]:
        for commit in report["history"]:
            lines.append(
                f"| `{commit['hash']}` | {markdown_escape(commit['subject'])} |"
            )
    else:
        lines.append("| - | _No commits_ |")
    return "\n".join(lines) + "\n"


def main() -> None:
    parser = argparse.ArgumentParser(
        description="Check the basic safety of a learning analytics Git repository"
    )
    parser.add_argument("repository", nargs="?", default=".")
    parser.add_argument("--min-commits", type=int, default=2)
    parser.add_argument("--format", choices=("markdown", "json"), default="markdown")
    parser.add_argument("--output", type=Path)
    args = parser.parse_args()

    try:
        report = evaluate_repository(
            Path(args.repository),
            min_commits=args.min_commits,
        )
    except (RuntimeError, ValueError) as error:
        parser.exit(2, f"git-project-check: {error}\n")

    rendered = (
        json.dumps(report, ensure_ascii=False, indent=2) + "\n"
        if args.format == "json"
        else render_markdown(report)
    )
    if args.output:
        args.output.write_text(rendered, encoding="utf-8")
    else:
        print(rendered, end="")
    raise SystemExit(0 if report["ready"] else 1)


if __name__ == "__main__":
    main()
