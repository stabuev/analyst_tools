Проверка перед уроком

Ответьте до чтения: результат поможет заметить исходные предположения.

01

Зачем аналитическое изменение делать в отдельной ветке, а не сразу в main?

Ветка отделяет предлагаемую линию изменений от основной и даёт точную границу для проверки; корректность самого расчёта всё равно проверяют люди и тесты.

02

Запрос выполнился без ошибки, но автор не указал grain таблиц и контрольное значение метрики. Какое действие reviewer наиболее полезно?

Технически успешный SQL может размножить строки и исказить метрику. Reviewer должен запросить проверяемые доказательства корректности.

Ветки, pull request и ревью

В предыдущем уроке вы научились сохранять собственные решения. Теперь нужно научиться предлагать изменение команде так, чтобы другой человек мог его понять и проверить.

Тип: Build Треки: Core Пререквизиты: 00/03 — Git: история аналитического проекта Время: ~90 минут Результат: проводит изменение аналитического проекта через отдельную ветку, готовит описание pull request и формулирует доказательное ревью дефектного SQL

Цели обучения

  • Продолжить работу в репозитории revenue-project, созданном в предыдущем уроке.
  • Создавать, просматривать, переключать и завершать короткую рабочую ветку.
  • Объяснять различие между локальной веткой, remote, push и pull request.
  • Проверять коммиты и полный diff ветки относительно main до публикации.
  • Писать описание PR с задачей, изменением, проверкой и ограничениями.
  • Проверять аналитический diff по grain, ключам, кардинальности и контрольным значениям.
  • Различать обычный комментарий, одобрение и блокирующий Request changes.
  • Объяснять полный цикл: ветка → PR → ревью → исправление → merge → удаление ветки.

Проблема

В уроке 00/03 вы создали локальную историю расчёта дневной выручки. В ней видно, когда появился запрос и когда в выручке остались только оплаченные заказы. Это уже лучше папки final_really_final, но пока историю проверял только её автор.

Теперь заказчик просит добавить к дневной выручке количество оплаченных заказов. Правка кажется небольшой: одна строка в SQL и несколько строк в определении метрики. Но даже в таком изменении можно:

  • случайно считать строки после размножающего JOIN, а не заказы;
  • изменить SQL и забыть изменить документацию;
  • включить в коммит несвязанный черновик;
  • написать «всё работает» без контрольного расчёта;
  • отправить изменение в общую ветку до независимой проверки.

Ветка позволяет собрать изменение отдельно от основной линии истории. Pull request, или PR, превращает эту ветку в проверяемое предложение: показывает цель, diff, способ проверки, ограничения и обсуждение. Ревью нужно не для оценки автора, а для снижения риска ошибочного решения.

Для аналитического проекта особенно важно, что успешно выполнившийся SQL ещё не является доказательством корректности. Запрос может вернуть таблицу и одновременно завысить сумму из-за неверного grain. Поэтому в этом уроке недостаточно создать ветку: вы проведёте отдельное ревью намеренно дефектного изменения.

Концепция

На что опирается этот урок

Мы не начинаем Git заново. Вы уже умеете выполнять цикл:

text
status → diff → add → diff --cached → commit → log

У вас уже есть репозиторий:

text
work/00-03-git/revenue-project/
├── .gitignore
├── README.md
├── data/raw/orders.csv          # локальный игнорируемый файл
├── docs/metric-definition.md
└── queries/revenue_by_day.sql

В обязательной практике предыдущего урока появились три решения:

text
Initialize revenue project
Add daily revenue calculation
Count only paid orders in revenue

Если вы выполнили дополнительные упражнения и создали ещё несколько понятных коммитов, это нормально. Новый урок продолжает текущее чистое состояние проекта и не требует точного количества коммитов.

Ветка — не копия папки

Ветка — имя, указывающее на один из коммитов. Представьте текущую историю так:

text
A --- B --- C  main

После создания ветки обе метки сначала указывают на один коммит:

text
A --- B --- C  main, feature/paid-order-count

Новый коммит, сделанный в рабочей ветке, перемещает только её метку:

text
A --- B --- C  main
             \
              D  feature/paid-order-count

Ветка не создаёт вторую физическую папку проекта. Вы по-прежнему работаете в одном revenue-project. Команда git switch выбирает линию истории и приводит tracked-файлы в состояние выбранного коммита.

Отсюда следует важная граница: untracked и игнорируемые файлы не принадлежат отдельной ветке. Локальный data/raw/orders.csv останется на диске при переключении между main и feature-веткой. Ветка изолирует историю tracked-файлов, но не является отдельным окружением или защищённой копией данных.

Основная и рабочая ветка

В этом курсе основная ветка называется main. Она содержит состояние проекта, которое команда считает общим и пригодным для дальнейшей работы.

Рабочая, или feature-ветка, создаётся для одного ограниченного изменения:

text
feature/paid-order-count
fix/order-join-cardinality
docs/revenue-definition

Имя должно описывать задачу, а не автора и не календарную дату. Ветка обычно живёт недолго: её создают от актуального main, проверяют через PR, объединяют и удаляют.

Локальный Git, remote и GitHub

До этого момента Git работал только на вашем компьютере. GitHub добавляет удалённую копию репозитория и интерфейс совместной работы.

text
локальный репозиторий                  GitHub

main ---------------------- push ----> main
feature/paid-order-count --- push ----> feature/paid-order-count
                                      │
                                      └─ pull request в main

Remote — сохранённый в локальном Git адрес другого репозитория. origin — обычное, но не обязательное имя первого remote. Команда git remote -v показывает известные адреса.

git push отправляет коммиты и ветку в remote. Он не создаёт PR автоматически. Pull request существует на GitHub: там выбирают ветку назначения, ветку с изменением, приглашают reviewer и принимают решение о merge.

В PR используются два направления:

  • base — куда предлагается добавить изменение; в уроке это main;
  • head — откуда берётся изменение; это feature/paid-order-count.

Полное устройство remote-tracking веток, протоколов, fetch, rebase и конфликтов не входит в минимальный результат занятия.

Что показывает diff будущего PR

Пока вы редактируете файлы, обычный git diff показывает незавершённую работу. После коммита рабочая папка снова чиста, поэтому обычный diff может быть пустым. Чтобы увидеть всё предложение ветки относительно main, используйте:

bash
git diff main...HEAD

Здесь HEAD означает выбранный сейчас коммит. На практическом уровне команда отвечает на вопрос: «Что текущая ветка предлагает добавить в main?»

Список коммитов предложения показывает:

bash
git log --oneline main..HEAD

Почему в одной команде три точки, а в другой две, станет важно при расходящейся истории. На этом уроке достаточно запомнить две целые команды и проверяемый вопрос каждой. Команды git merge-base, rebase и подробная алгебра диапазонов остаются за пределами обязательной практики.

Pull request — не замена коммита

Коммит и PR отвечают на разные вопросы:

Объект Главный вопрос
Коммит Какое законченное изменение сохранено в истории?
Ветка Какие связанные коммиты пока отделены от main?
Pull request Почему это изменение нужно объединить и чем доказана его корректность?
Review Какие риски проверены и можно ли безопасно выполнить merge?

Описание PR не должно пересказывать список файлов. Reviewer и так увидит его в diff. Полезное описание содержит:

  1. задачу и ожидаемое решение;
  2. что изменилось на уровне поведения;
  3. как получено проверяемое доказательство;
  4. какие ограничения остаются.

Ревью аналитического изменения

Проверка стиля не заменяет проверку расчёта. Для аналитического SQL порядок вопросов обычно такой:

  1. Какова единица одной строки до и после изменения?
  2. Какие ключи используются и уникальны ли они там, где ожидается уникальность?
  3. Какова кардинальность каждого JOIN?
  4. Могут ли строки размножиться или исчезнуть?
  5. Что происходит с NULL, дубликатами и фильтрами?
  6. Сходится ли результат с маленьким ручным примером?
  7. Обновлены ли определение метрики и ограничения вывода?

Хорошее замечание reviewer содержит четыре части:

text
наблюдение → риск → требуемое исправление → способ повторной проверки

Например:

text
После JOIN одна строка заказа повторяется для каждого товара. Поэтому SUM(orders.amount)
завышает выручку. Нужно агрегировать order_items до order_id или считать выручку до JOIN,
а затем сверить сумму с контролем на исходной таблице orders.

Три решения reviewer

  • Comment — вопрос, уточнение или необязательное улучшение. Merge не обязательно блокировать.
  • Approve — обязательные риски проверены, изменение можно объединить.
  • Request changes — найден дефект или отсутствует доказательство, без которого merge небезопасен.

Request changes относится к состоянию изменения, а не к способностям автора. После исправления reviewer должен повторно проверить diff и изменить решение.

Команда должна напечатать main. Переименование ветки не переписывает её коммиты.

Шаг 2. Создайте ветку задачи

Перед ответвлением ещё раз убедитесь, что main выбран и рабочая папка чиста:

bash
git switch main
git status
git switch -c feature/paid-order-count

В выводе git branch звёздочка отмечает выбранную ветку:

text

  main

Сразу после создания обе ветки указывают на один коммит. Никаких изменений в файлах ещё нет.

Шаг 3. Добавьте количество оплаченных заказов

Откройте queries/revenue_by_day.sql. Добавьте COUNT(*) рядом с выручкой:

sql
SELECT
    order_date,
    SUM(amount) AS revenue,
    COUNT(*) AS paid_order_count
FROM orders
WHERE status = 'paid'
GROUP BY order_date;

Такой расчёт корректен при предположении, что таблица orders содержит одну строку на заказ. Зафиксируйте это предположение в docs/metric-definition.md:

markdown
# Дневная выручка

Grain результата — одна строка на календарный день.

- `revenue` — сумма `amount` оплаченных заказов за день;
- `paid_order_count` — число строк оплаченных заказов за день.

Расчёт предполагает одну строку на `order_id` в таблице `orders`.

Если вы выполнили упражнения прошлого урока и уже расширили запрос, не удаляйте полезные изменения. Добавьте новый показатель в текущую версию и сохраните только новый законченный шаг.

Повторите знакомый цикл:

bash
git status
git diff
git add queries/revenue_by_day.sql docs/metric-definition.md
git status
git diff --cached
git commit -m "Add paid order count to daily revenue"
git status

Перед коммитом ответьте себе:

  • в diff есть только изменение текущей задачи;
  • SQL и документация согласованы;
  • COUNT(*) считает заказы только при заявленном grain источника;
  • локальная выгрузка не вошла в staging area.

Шаг 4. Посмотрите предложение целиком

Коммит уже сохранён, поэтому теперь смотрите не обычный diff рабочей папки, а разницу ветки с main:

bash
git log --oneline main..HEAD
git diff --stat main...HEAD
git diff main...HEAD

Ожидается один новый коммит и два изменённых файла. Прочитайте полный diff как reviewer: видно ли из него, что показатель и его определение меняются вместе?

Теперь на минуту переключитесь на main:

bash
git switch main
cat queries/revenue_by_day.sql
ls data/raw

В SQL на main ещё нет paid_order_count, потому что новый коммит принадлежит рабочей ветке. При этом локальный data/raw/orders.csv остался на месте: он игнорируется и не является частью ни одной ветки.

Вернитесь к изменению:

bash
git switch feature/paid-order-count

Шаг 5. Подготовьте описание PR

Вернитесь в корень курса и создайте отдельный каталог учебных документов:

bash
cd ../../..
mkdir -p work/00-04-git

  phases/00-entry-and-tools/04-branches-and-review/outputs/pull_request_template.md \

Заполните work/00-04-git/pull-request.md:

markdown
## Задача

Добавить число оплаченных заказов к дневной выручке, чтобы вместе видеть объём денег и
количество заказов.

## Что изменено

В `revenue_by_day.sql` добавлен `paid_order_count`. В определении метрики зафиксированы
grain результата и предположение об уникальности `order_id`.

## Как проверено

Просмотрены `git diff --cached` перед коммитом и полный `git diff main...HEAD` после него.
Для входных строк с тремя оплаченными заказами ожидается `paid_order_count = 3`.

## Ограничения

Запрос не проверяет уникальность `order_id`. Если источник содержит дубликаты или строки
размножены JOIN, `COUNT(*)` перестаёт означать число заказов.

Описание не утверждает, что задача полностью решена для любых данных. Оно явно называет предположение, которое должен проверить reviewer.

Шаг 6. Разберите связь с GitHub

Посмотрите, настроен ли remote:

bash
git -C work/00-03-git/revenue-project remote -v

В локальном учебном проекте команда, скорее всего, ничего не напечатает. Это ожидаемо: предыдущий урок не создавал удалённый репозиторий.

Для необязательной реальной публикации сначала создайте пустой репозиторий на GitHub и добавьте его адрес. Не выполняйте команды с буквальным <USER> и <REPOSITORY>:

bash
cd work/00-03-git/revenue-project
git remote add origin https://github.com/<USER>/<REPOSITORY>.git
git push -u origin main
git push -u origin feature/paid-order-count
cd ../../..

Первая команда сохраняет адрес под именем origin. Первый push публикует base, второй — рабочую ветку. Флаг -u связывает локальную ветку с одноимённой удалённой, чтобы следующие push могли быть короче.

Публикация не является обязательной частью проверки урока. Не отправляйте рабочие данные, чужой репозиторий или проект, который вы не имеете права публиковать. Используйте только синтетический revenue-project и сначала перечитайте git status и историю tracked-файлов.

После push на GitHub создайте pull request:

text
base: main
compare/head: feature/paid-order-count

Вставьте подготовленное описание. Сам git push PR не создаёт.

Шаг 7. Проведите настоящее аналитическое ревью

Откройте учебный кейс:

bash
cat phases/00-entry-and-tools/04-branches-and-review/outputs/review_case_context.md
cat phases/00-entry-and-tools/04-branches-and-review/outputs/review_case.sql

Задача автора — добавить число товаров к дневному отчёту, не изменив выручку. Исходные данные:

text
orders:                         order_items:
order_id  amount               order_id  item_id
101       120                  101       A
102        75                  101       B
                                102       C

Контрольная выручка на grain заказа равна 120 + 75 = 195. После JOIN в предложенном SQL заказ 101 появляется дважды, поэтому SUM(orders.amount) даёт 120 + 120 + 75 = 315.

Скопируйте шаблон ревью:

bash
cp \
  phases/00-entry-and-tools/04-branches-and-review/outputs/analytics_review_template.md \
  work/00-04-git/review.md

Заполните его. Для данного дефекта решение должно быть блокирующим:

markdown
## Решение

Request changes

## Файл и строки

`review_case.sql`, выражение `SUM(orders.amount)` после `JOIN order_items`.

## Наблюдение

Один заказ повторяется для каждого товара. Заказ `101` после JOIN представлен двумя
строками.

## Риск

Выручка завышается: контрольное значение 195 превращается в 315. Любой отчёт с разным
числом товаров в заказе будет систематически искажён.

## Что исправить

Посчитать выручку на grain `order_id` до JOIN либо сначала агрегировать `order_items` до
одной строки на заказ и только затем соединять таблицы.

## Как проверить

Сверить итоговую выручку с независимой суммой из `orders` и добавить случай с заказом из
двух товаров. После исправления результат должен остаться равным 195.

Фраза «неправильный JOIN» была бы недостаточна: она не показывает механизм ошибки, её масштаб и критерий исправления.

Используйте это

Проверяющий артефакт собирает локальный пакет будущего PR. Запускайте его до merge, пока выбрана feature-ветка:

bash
python3 \
  phases/00-entry-and-tools/04-branches-and-review/outputs/pr_review_packet.py \
  work/00-03-git/revenue-project \
  --base main \
  --body work/00-04-git/pull-request.md \
  --review work/00-04-git/review.md

Сохраните отчёт:

bash
python3 \
  phases/00-entry-and-tools/04-branches-and-review/outputs/pr_review_packet.py \
  work/00-03-git/revenue-project \
  --base main \
  --body work/00-04-git/pull-request.md \
  --review work/00-04-git/review.md \
  --output work/00-04-git/pr-review-packet.md

CLI проверяет только наблюдаемую структуру:

  • выбрана отдельная ветка с коммитами и изменёнными файлами;
  • рабочая папка не содержит забытых изменений;
  • PR описывает задачу, изменение, проверку и ограничения;
  • ревью содержит решение, точное наблюдение, риск, исправление и повторную проверку;
  • решение reviewer принадлежит допустимому набору.

Программа не может доказать, что замечание методологически верно. Именно поэтому учебный кейс содержит ручное контрольное значение 195.

Готовая демонстрация создаёт временное продолжение revenue-project и печатает такой же пакет:

bash
python3 phases/00-entry-and-tools/04-branches-and-review/code/main.py

Python здесь служит только воспроизводимой проверкой Git-workflow и не является темой занятия.

Что происходит после замечания

В настоящем PR автор не создаёт новый PR для каждого исправления. Он меняет файлы в той же ветке, повторяет проверки, создаёт новый коммит и делает push:

bash
git status
git diff
git add путь/к/исправленному-файлу
git diff --cached
git commit -m "Fix order grain before item join"
git push

PR обновится автоматически, потому что его head остаётся той же веткой. Reviewer должен:

  1. прочитать новый diff и ответ автора;
  2. повторить контрольный расчёт;
  3. убедиться, что исправление не создало новый риск;
  4. выбрать Approve, только если блокирующее замечание действительно закрыто.

Как завершается ветка

При работе на GitHub merge выполняют через интерфейс PR после обязательных проверок и одобрений. Затем локальный main обновляют и удаляют уже объединённую рабочую ветку:

bash
git switch main
git pull --ff-only
git branch -d feature/paid-order-count

Для полностью локальной практики без GitHub после сохранения PR-пакета имитируйте одобренный merge:

bash
cd work/00-03-git/revenue-project
git switch main
git merge --ff-only feature/paid-order-count
git branch -d feature/paid-order-count
git log --oneline -4
cd ../../..

--ff-only разрешает простой перенос main вперёд, если основная ветка не получила других коммитов. Расходящиеся ветки и конфликты требуют отдельного решения и не входят в этот урок.

Сломайте это

Создайте ветку с незавершёнными изменениями

Незакоммиченные изменения принадлежат рабочей папке, а не ветке. Если изменить файл на main, а затем выполнить git switch -c feature/..., правка обычно перейдёт вместе с рабочей папкой. Это может быть полезно, но для новичка часто смешивает задачи.

Безопасный порядок перед ответвлением:

bash
git switch main
git status
git switch -c feature/task-name

Если git status не чист, сначала объясните каждую запись и решите, к какой задаче она относится.

Примите ветку за отдельную копию данных

Переключение ветки меняет tracked-файлы. Оно не создаёт второй .env, вторую выгрузку или отдельное виртуальное окружение. Не используйте ветки как механизм разделения чувствительных данных или локальной конфигурации.

Решите, что push уже создал PR

После git push ветка появилась в remote, но обсуждение ещё не создано. Проверьте GitHub: PR должен иметь правильные base и head, описание, reviewer и видимый diff.

Quiz · 4 вопроса

Проверьте себя

Ответьте после практики. Объяснение появится сразу после выбора.

01

Что делает команда git switch -c feature/paid-order-count, запущенная из чистой main?

Команда создаёт именованную ветку от текущего коммита и переключает рабочую папку на неё; отдельной копии проекта не возникает.

02

Как соотносятся git push и pull request на GitHub?

Push публикует Git-объекты и ссылку ветки. PR — отдельный объект платформы, связывающий base, head, описание, проверки и обсуждение.

03

После JOIN таблицы orders с одной строкой на заказ и order_items с одной строкой на товар SUM(orders.amount) выросла со 195 до 315. Какое ревью корректно?

Это блокирующий дефект корректности: связь один-ко-многим меняет grain и завышает выручку. Замечание должно назвать механизм, риск, исправление и проверку.

04

Автор исправил замечание в той же feature-ветке, сделал новый коммит и push. Что произойдёт с открытым PR?

PR следит за head-веткой, поэтому новые опубликованные коммиты обновляют предложение. После исправления reviewer может перепроверить риск и изменить решение.

def load_packet_builder():
    spec = importlib.util.spec_from_file_location("pr_review_packet_test", PACKET_PATH)
    if spec is None or spec.loader is None:
        raise RuntimeError("Cannot load pull request packet builder")

def git(root: Path, *arguments: str) -> subprocess.CompletedProcess[str]:
    return subprocess.run(
        ["git", "-C", str(root), *arguments],
        check=True,
        capture_output=True,
        text=True,
    )


def write_file(root: Path, relative: str, content: str) -> None:
    path = root / relative
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(content, encoding="utf-8")


def commit(root: Path, message: str, *paths: str) -> None:
    git(root, "add", "--", *paths)
    git(root, "commit", "-q", "-m", message)


def initialize_repository(root: Path) -> None:
    git(root, "init", "-q")
    git(root, "config", "user.name", "Course Student")
    git(root, "config", "user.email", "student@example.com")
    write_file(root, "README.md", "# Revenue project\n")
    commit(root, "Initialize revenue project", "README.md")
    git(root, "branch", "-M", "main")


def write_documents(root: Path) -> tuple[Path, Path]:
    body = root.parent / f"{root.name}-pull-request.md"
    body.write_text(
        (
            "## Задача\n\nДобавить число заказов в дневной отчёт по выручке.\n\n"
            "## Что изменено\n\nВ запрос и определение метрики добавлен счётчик заказов.\n\n"
            "## Как проверено\n\nПроверены полный diff ветки и ручной контрольный пример.\n\n"
            "## Ограничения\n\nРасчёт предполагает одну строку на заказ в источнике.\n"
        ),
        encoding="utf-8",
    )
    review = root.parent / f"{root.name}-review.md"
    review.write_text(
        (
            "## Решение\n\nRequest changes\n\n"
            "## Файл и строки\n\nreview_case.sql, SUM после соединения с товарами.\n\n"
            "## Наблюдение\n\nОдин заказ повторяется для каждой товарной позиции.\n\n"
            "## Риск\n\nВыручка завышается и становится зависимой от числа товаров.\n\n"
            "## Что исправить\n\nАгрегировать товары до одной строки на заказ до соединения.\n\n"
            "## Как проверить\n\nСверить выручку с независимой суммой таблицы заказов.\n"
        ),
        encoding="utf-8",
    )
    return body, review

                base="main",
                body_path=body,
                review_path=review,

        self.assertEqual(report["head"], "feature/paid-order-count")
        self.assertEqual(report["review_decision"], "Request changes")

from __future__ import annotations

import argparse
import json
import re
import subprocess
from pathlib import Path
from typing import Any


PR_REQUIRED_SECTIONS = (
    "Задача",
    "Что изменено",
    "Как проверено",
    "Ограничения",
)
REVIEW_REQUIRED_SECTIONS = (
    "Решение",
    "Файл и строки",
    "Наблюдение",
    "Риск",
    "Что исправить",
    "Как проверить",
)
ALLOWED_REVIEW_DECISIONS = {
    "comment": "Comment",
    "approve": "Approve",
    "request changes": "Request changes",
}


def run_git(
    root: Path,
    *arguments: str,
    check: bool = True,
) -> subprocess.CompletedProcess[str]:
    result = subprocess.run(
        ["git", "-C", str(root), "--no-pager", *arguments],
        capture_output=True,
        text=True,
        check=False,
    )
    if check and result.returncode != 0:
        message = result.stderr.strip() or result.stdout.strip() or "unknown Git error"
        raise RuntimeError(message)
    return result


def resolve_repository(path: Path) -> Path:
    candidate = path.expanduser()
    if not candidate.exists():
        raise ValueError(f"path does not exist: {candidate}")
    result = run_git(candidate, "rev-parse", "--show-toplevel", check=False)
    if result.returncode != 0:
        raise ValueError(f"not a Git working tree: {candidate}")
    return Path(result.stdout.strip()).resolve()


def verify_ref(root: Path, ref: str) -> str:
    result = run_git(root, "rev-parse", "--verify", f"{ref}^{{commit}}", check=False)
    if result.returncode != 0:
        raise ValueError(f"unknown commit or branch: {ref}")
    return result.stdout.strip()


def split_nul(value: str) -> list[str]:
    return [item for item in value.split("\0") if item]


def current_branch(root: Path) -> str | None:
    result = run_git(root, "symbolic-ref", "--quiet", "--short", "HEAD", check=False)
    return result.stdout.strip() if result.returncode == 0 else None


def parse_sections(markdown: str) -> dict[str, str]:
    matches = list(re.finditer(r"^##\s+(.+?)\s*$", markdown, flags=re.MULTILINE))
    sections: dict[str, str] = {}
    for index, match in enumerate(matches):
        start = match.end()
        end = matches[index + 1].start() if index + 1 < len(matches) else len(markdown)
        sections[match.group(1).strip()] = markdown[start:end].strip()
    return sections


def without_comments(value: str) -> str:
    return re.sub(r"<!--.*?-->", "", value, flags=re.DOTALL).strip()


def meaningful_section(value: str) -> bool:
    cleaned = without_comments(value)
    meaningful_lines = []
    for line in cleaned.splitlines():
        stripped = line.strip()
        if not stripped or stripped.startswith("- ["):
            continue
        if stripped in {"```", "```text", "```bash", "```shell"}:
            continue
        meaningful_lines.append(stripped)
    return len(" ".join(meaningful_lines)) >= 20


def load_markdown(path: Path | None, label: str) -> tuple[str, str]:
    if path is None:
        return "", f"{label} path is required."
    candidate = path.expanduser().resolve()
    if not candidate.is_file():
        return "", f"{label} does not exist: {candidate}"
    return candidate.read_text(encoding="utf-8"), ""


def missing_meaningful_sections(
    sections: dict[str, str],
    required: tuple[str, ...],
    *,
    short_sections: set[str] | None = None,
) -> list[str]:
    short_sections = short_sections or set()
    missing = []
    for section in required:
        value = sections.get(section, "")
        is_filled = bool(without_comments(value)) if section in short_sections else meaningful_section(value)
        if not is_filled:
            missing.append(section)
    return missing


def review_decision(sections: dict[str, str]) -> str:
    value = without_comments(sections.get("Решение", ""))
    first_line = next((line.strip() for line in value.splitlines() if line.strip()), "")
    normalized = first_line.strip("`*_ .").casefold()
    return ALLOWED_REVIEW_DECISIONS.get(normalized, "")


def commit_list(root: Path, base: str, head: str) -> list[dict[str, str]]:
    output = run_git(
        root,
        "log",
        "--reverse",
        "--format=%h%x1f%s%x1e",
        f"{base}..{head}",
    ).stdout
    commits: list[dict[str, str]] = []
    for record in output.split("\x1e"):
        record = record.strip()
        if not record:
            continue
        commit_hash, subject = record.split("\x1f", 1)
        commits.append({"hash": commit_hash, "subject": subject})
    return commits


def evaluate_pull_request(
    path: Path,
    base: str,
    head: str = "HEAD",
    body_path: Path | None = None,
    review_path: Path | None = None,
) -> dict[str, Any]:
    root = resolve_repository(path)
    base_hash = verify_ref(root, base)
    head_hash = verify_ref(root, head)
    branch = current_branch(root) if head == "HEAD" else head
    merge_base_result = run_git(root, "merge-base", base, head, check=False)
    merge_base = (
        merge_base_result.stdout.strip() if merge_base_result.returncode == 0 else ""
    )

    commits = commit_list(root, base, head) if merge_base else []
    files = (
        split_nul(
            run_git(root, "diff", "--name-only", "-z", f"{base}...{head}").stdout
        )
        if merge_base
        else []
    )
    status = split_nul(
        run_git(
            root,
            "status",
            "--porcelain=v1",
            "-z",
            "--untracked-files=all",
        ).stdout
    )

    body_text, body_error = load_markdown(body_path, "PR description")
    body_sections = parse_sections(body_text)
    missing_body_sections = missing_meaningful_sections(
        body_sections,
        PR_REQUIRED_SECTIONS,
    )

    review_text, review_error = load_markdown(review_path, "Review")
    review_sections = parse_sections(review_text)
    missing_review_sections = missing_meaningful_sections(
        review_sections,
        REVIEW_REQUIRED_SECTIONS,
        short_sections={"Решение"},
    )
    decision = review_decision(review_sections)

    diff_stat = (
        run_git(root, "diff", "--shortstat", f"{base}...{head}").stdout.strip()
        if merge_base
        else ""
    )
    behind = (
        int(run_git(root, "rev-list", "--count", f"{head}..{base}").stdout.strip())
        if merge_base
        else 0
    )

    separate_head = base_hash != head_hash and branch not in {None, base}
    body_ok = not body_error and not missing_body_sections
    review_ok = not review_error and not missing_review_sections and bool(decision)
    checks = [
        {
            "id": "separate-head",
            "passed": separate_head,
            "message": (
                f"Head `{branch}` отличается от base `{base}`."
                if separate_head
                else "Head должен быть отдельной именованной веткой с изменениями."
            ),
        },
        {
            "id": "merge-base",
            "passed": bool(merge_base),
            "message": (
                f"Merge base: {merge_base[:12]}."
                if merge_base
                else "У base и head не найден общий предок."
            ),
        },
        {
            "id": "commits",
            "passed": bool(commits),
            "message": f"Коммитов в предложении: {len(commits)}.",
        },
        {
            "id": "changed-files",
            "passed": bool(files),
            "message": f"Изменённых файлов: {len(files)}.",
        },
        {
            "id": "clean-tree",
            "passed": not status,
            "message": (
                "Рабочее дерево чистое."
                if not status
                else f"Незакоммиченных записей: {len(status)}."
            ),
        },
        {
            "id": "pr-description",
            "passed": body_ok,
            "message": (
                "Описание PR связывает задачу, изменение, проверку и ограничения."
                if body_ok
                else body_error
                or "Нужно заполнить разделы: " + ", ".join(missing_body_sections)
            ),
        },
        {
            "id": "review",
            "passed": review_ok,
            "message": (
                f"Ревью заполнено; решение: {decision}."
                if review_ok
                else review_error
                or "Нужно заполнить разделы: " + ", ".join(missing_review_sections)
                if missing_review_sections
                else "Решение должно быть Comment, Approve или Request changes."
            ),
        },
    ]

    return {
        "repository": str(root),
        "base": base,
        "head": branch or head,
        "merge_base": merge_base,
        "ahead": len(commits),
        "behind": behind,
        "diff_stat": diff_stat,
        "ready": all(check["passed"] for check in checks),
        "checks": checks,
        "commits": commits,
        "files": files,
        "working_tree": status,
        "pr_sections": list(body_sections),
        "review_sections": list(review_sections),
        "review_decision": decision,
    }


def markdown_escape(value: str) -> str:
    return value.replace("\\", "\\\\").replace("|", "\\|").replace("`", "'")


def render_markdown(report: dict[str, Any]) -> str:
    lines = [
        "# Pull request review packet",
        "",
        f"- Repository: `{markdown_escape(report['repository'])}`",
        f"- Range: `{report['base']}...{report['head']}`",
        f"- Commits ahead: {report['ahead']}",
        f"- Base-only commits: {report['behind']}",
        f"- Diff: {report['diff_stat'] or 'empty'}",
        f"- Review decision: **{report['review_decision'] or 'not set'}**",
        f"- Result: **{'ready for review' if report['ready'] else 'needs work'}**",
        "",
        "## Checks",
        "",
        "| Check | Result | Details |",
        "|---|---|---|",
    ]
    for check in report["checks"]:
        result = "PASS" if check["passed"] else "FAIL"
        lines.append(
            f"| `{check['id']}` | {result} | {markdown_escape(check['message'])} |"
        )

    lines.extend(["", "## Commits", ""])
    if report["commits"]:
        for commit in report["commits"]:
            lines.append(f"- `{commit['hash']}` {markdown_escape(commit['subject'])}")
    else:
        lines.append("- _No proposed commits_")

    lines.extend(["", "## Changed files", ""])
    if report["files"]:
        lines.extend(f"- `{markdown_escape(path)}`" for path in report["files"])
    else:
        lines.append("- _No changed files_")
    return "\n".join(lines) + "\n"


def main() -> None:
    parser = argparse.ArgumentParser(
        description="Validate a branch, PR description, and analytical review"
    )
    parser.add_argument("repository", nargs="?", default=".")
    parser.add_argument("--base", default="main")
    parser.add_argument("--head", default="HEAD")
    parser.add_argument("--body", type=Path, required=True)
    parser.add_argument("--review", type=Path, required=True)
    parser.add_argument("--format", choices=("markdown", "json"), default="markdown")
    parser.add_argument("--output", type=Path)
    args = parser.parse_args()

    try:
        report = evaluate_pull_request(
            Path(args.repository),
            base=args.base,
            head=args.head,
            body_path=args.body,
            review_path=args.review,
        )
    except (RuntimeError, ValueError) as error:
        parser.exit(2, f"pr-review-packet: {error}\n")

    rendered = (
        json.dumps(report, ensure_ascii=False, indent=2) + "\n"
        if args.format == "json"
        else render_markdown(report)
    )
    if args.output:
        args.output.write_text(rendered, encoding="utf-8")
    else:
        print(rendered, end="")
    raise SystemExit(0 if report["ready"] else 1)


if __name__ == "__main__":
    main()

До изменения дневная выручка на этих заказах равна `195`. Проверьте, сохраняет ли
