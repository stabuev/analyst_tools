02

Как правильно прочитать версию Python 3.12.7?

Python записывает выпуск как major.minor.patch: поколение 3, линия 12 и исправляющий выпуск 7.

Версии Python и совместимость

Версия Python — это свойство конкретного запущенного процесса, а не всей машины. Сначала выясните, какой интерпретатор работает, и только потом исправляйте окружение.

Тип: Learn Треки: Core Пререквизиты: 00/05 — Секреты и безопасная работа с данными Время: ~60 минут Результат: определяет версию и путь фактически запущенного Python, различает требование проекта и локальный выбор интерпретатора и документирует обоснованный контракт версии

Цели обучения

  • Объяснять простыми словами, что делает интерпретатор Python.
  • Читать номер версии как major.minor.patch и понимать роль каждой части.
  • Узнавать версию и путь Python внутри того процесса, где выполняется код.
  • Различать requires-python, .python-version и фактически запущенный интерпретатор.
  • Выбирать минимальную версию по требованиям проекта, а не по привычке автора.
  • Не путать декларацию совместимости с доказательством запуска и прохождения тестов.
  • Сохранять решение о версии в том же Git-процессе, который собран в предыдущей фазе.

Проблема

В фазе 00 вы создали revenue-project, провели изменение через ветку и ревью, а затем отделили публикуемые файлы от секретов и рабочих выгрузок. Граница проекта уже понятна. Теперь в нём должен появиться первый Python-код для обработки синтетического sample.

Автор запускает код и пишет коллеге:

text
Всё работает. Нужен Python 3.

Коллега отвечает:

text
У меня тоже Python 3, но import tomllib завершается ошибкой.

Оба сообщения могут быть правдивыми. У автора запущен Python 3.12, а у коллеги — 3.10. Модуль tomllib входит в стандартную библиотеку начиная с Python 3.11, поэтому общего слова «Python 3» недостаточно.

Есть и менее очевидная ситуация. Терминал показывает Python 3.12, а открытый ранее notebook продолжает работать на Python 3.10. Установка нового Python на компьютер не переключает уже запущенный процесс автоматически.

Чтобы проект можно было воспроизвести, нужно ответить на три разных вопроса:

  1. Какие версии Python допускает проект?
  2. Какую линию Python должен выбрать локальный инструмент?
  3. Какой интерпретатор действительно выполняет код прямо сейчас?

В этом уроке не нужно писать собственный менеджер версий или парсер сложных требований. Наша задача — увидеть фактический процесс, зафиксировать минимальный понятный контракт и подготовить проект к созданию окружения с uv в следующем уроке.

Концепция

Что такое интерпретатор

Файл с расширением .py — это текст с инструкциями. Чтобы инструкции выполнились, их читает программа Python. Эту программу называют интерпретатором.

Упрощённая цепочка выглядит так:

text
команда в терминале
        ↓
командная оболочка (shell) находит исполняемый файл Python
        ↓
Python создаёт процесс
        ↓
процесс читает и выполняет код

На одной машине может быть несколько интерпретаторов:

text
/usr/bin/python3
/opt/homebrew/bin/python3
<project>/.venv/bin/python
Python внутри другого окружения notebook

Они могут иметь разные версии и разные наборы установленных библиотек. Поэтому фраза «на компьютере установлен Python 3.12» ещё не говорит, какой из этих файлов выполнил конкретный скрипт.

Как читать номер версии

Рассмотрим Python 3.12.7:

text
3 . 12 . 7
│   │    └─ patch: исправления внутри линии 3.12
│   └────── minor: линия возможностей и совместимости
└────────── major: крупное поколение языка
  • Major меняется редко и обозначает крупное поколение языка. Сейчас рабочие проекты курса используют Python 3.
  • Minor появляется с очередным выпуском языка: 3.11, 3.12, 3.13. В новой линии могут появиться новый синтаксис, модули стандартной библиотеки и изменения API.
  • Patch содержит исправления ошибок и безопасности внутри minor-линии: например, 3.12.6 и 3.12.7 относятся к одной линии 3.12.

Для начального выбора совместимости чаще всего важна minor-линия. Жёстко требовать ровно 3.12.7 обычно не нужно: это может запретить более свежий исправляющий выпуск 3.12.8 без содержательной причины.

Python не обещает, что любой код автоматически работает на любой более новой версии. Номер помогает сформулировать ожидание, но реальную совместимость подтверждает запуск проекта и его проверок.

Как узнать фактически запущенный Python

Внутри Python доступны два особенно полезных наблюдения:

python
import sys

print(sys.version_info[:3])
print(sys.executable)

sys.version_info[:3] показывает major, minor и patch текущего процесса. sys.executable показывает путь к интерпретатору, который этот процесс запустил.

Из терминала ту же проверку можно выполнить без создания файла:

bash
python3 -c 'import sys; print(sys.version_info[:3]); print(sys.executable)'

Ключевое выражение — текущего процесса. Если ошибка возникла в notebook, проверку нужно выполнить в notebook. Если ошибка возникла в системе автоматических проверок (CI), нужны сведения из этого запуска. Вывод из вашего локального терминала не описывает автоматически чужой процесс.

Команда shell и интерпретатор процесса

Команда:

bash
command -v python3

показывает, какой файл shell сейчас найдёт для имени python3. Команда:

bash
python3 -c 'import sys; print(sys.executable)'

запускает найденный Python и спрашивает путь уже у процесса. В обычном терминале эти пути часто совпадают. Сравнение всё равно полезно: оно связывает имя команды с конкретным исполняемым файлом.

В уже открытом notebook ситуация может быть другой:

text
shell:    /opt/homebrew/bin/python3       → Python 3.12
notebook: <home>/old-project/bin/python  → Python 3.10

Это не «две версии одного процесса». Это два разных процесса, запущенных разными интерпретаторами. Как выбирать ядро notebook, или Jupyter kernel, мы подробно разберём в уроке 01/04.

Интерпретатор и окружение — не одно и то же

Интерпретатор выполняет Python-код. Окружение связывает выбранный интерпретатор с установленными для проекта библиотеками и командами.

text
окружение проекта
├── конкретный Python
├── установленные библиотеки
└── команды, доступные внутри окружения

Два процесса на Python 3.12 могут видеть разные библиотеки. И наоборот, одинаковый список библиотек нельзя без проверки переносить между Python 3.10 и 3.12. В этом уроке мы только выбираем и наблюдаем интерпретатор; окружение создадим в 01/02.

Три уровня версии в проекте

У проекта есть три связанных, но не взаимозаменяемых утверждения.

Уровень Пример На какой вопрос отвечает
Требование проекта requires-python = ">=3.11" Какая минимальная версия допускается проектом?
Локальный выбор .python-version содержит 3.12 Какую линию предложить совместимому инструменту?
Наблюдение процесса sys.version_info показывает 3.12.7 Какой Python действительно выполняет код сейчас?

Они могут согласоваться:

text
проект требует 3.11 или новее
        ↓
локально выбрана линия 3.12
        ↓
процесс запущен на 3.12.7

А могут расходиться:

text
проект требует 3.11 или новее
        ↓
в .python-version записано 3.12
        ↓
процесс всё ещё запущен на 3.10.14

Последний случай означает, что файл с выбором ещё не был применён инструментом либо код запущен в другом окружении.

requires-python — требование проекта

Минимальное требование хранится в pyproject.toml:

toml
[project]
name = "revenue-project"
version = "0.1.0"
requires-python = ">=3.11"

Запись >=3.11 читается как «Python 3.11 или новее». В этом уроке достаточно понимать одну нижнюю границу. Остальную структуру pyproject.toml, зависимости и более сложные ограничения мы разберём позже.

Минимум выбирают по самой ранней версии, на которой работают необходимые части проекта:

  • синтаксис Python;
  • модули стандартной библиотеки;
  • внешние библиотеки;
  • тесты и рабочие инструменты.

В курсе используется минимум Python 3.11. Это общее требование учебного проекта, а не случайная версия на компьютере автора.

Поле requires-python является декларацией. Оно сообщает инструментам о требовании, но не устанавливает Python и не доказывает, что код уже проверен на каждой будущей версии.

.python-version — локальный выбор для инструмента

Файл .python-version может содержать:

text
3.12

Совместимые инструменты, включая uv, используют это значение при выборе Python. Обычный текстовый файл сам по себе ничего не переключает: после его создания уже запущенный notebook не изменится, а shell без менеджера версий может продолжить находить прежний бинарник.

Полезная формула:

text
requires-python описывает допустимый минимум
.python-version предлагает локальную линию
sys.executable показывает факт

Совместимость зависимостей

У внешних библиотек тоже есть требования к Python. Возможная версия проекта находится на пересечении:

text
требования нашего кода
∩ требование requires-python
∩ поддержка Python библиотеками
∩ доступность сборок для операционной системы

Пока у revenue-project нет внешних Python-зависимостей, мы не будем изображать их проверку вручную. В следующем уроке uv создаст окружение, добавит библиотеку и покажет, можно ли разрешить требования вместе.

Начните из корня курса и проверьте проект готовым инструментом из урока о Git:

Шаг 2. Наблюдайте Python, а не угадывайте

Сначала посмотрите, что сообщает shell:

Теперь спросите сам запущенный процесс:

Ответьте своими словами:

  1. Каковы major, minor и patch?

  4. Совпали ли два пути?

Если путь содержит имя пользователя, не копируйте его бездумно в публикуемый документ. В дальнейшем используйте нейтральную запись вроде <home>/miniconda3/bin/python.

Шаг 3. Проверьте минимальное требование курса

Курс и его инструменты требуют Python 3.11 или новее. Проверьте текущий процесс:

bash
python3 -c \
  'import sys; print("OK: Python 3.11+" if sys.version_info >= (3, 11) else "STOP: нужен Python 3.11+")'

Если вы получили STOP, не меняйте требование проекта на 3.10 только ради продолжения. Пока сохраните наблюдение и установите поддерживаемый Python по docs/getting-started.md; в следующем уроке эту задачу возьмёт на себя uv.

Шаг 4. Создайте минимальный контракт проекта

Убедитесь, что pyproject.toml ещё не существует:

bash
git status --short
test ! -e pyproject.toml && echo 'pyproject.toml is new'

Создайте только минимальные метаданные:

bash
printf '%s\n' \
  '[project]' \
  'name = "revenue-project"' \
  'version = "0.1.0"' \
  'requires-python = ">=3.11"' \
  > pyproject.toml

Прочитайте файл:

bash
sed -n '1,20p' pyproject.toml

Не добавляйте зависимости вручную: это задача следующего урока. Сейчас важно одно поле — минимальная версия Python.

Шаг 5. Зафиксируйте выбранную локальную линию

Запишите текущую minor-линию без patch:

bash
python3 -c \
  'import sys; print(f"{sys.version_info.major}.{sys.version_info.minor}")' \
  > .python-version
cat .python-version

Если текущий Python равен 3.12.7, файл будет содержать 3.12. Это не попытка зафиксировать единственную допустимую patch-версию. Любой подходящий исправляющий выпуск линии 3.12 может стать фактическим интерпретатором.

Сразу повторите наблюдение:

bash
python3 --version
python3 -c 'import sys; print(sys.executable)'

Если ничего не изменилось, всё работает ожидаемо: .python-version не переключает уже запущенный процесс без инструмента, который умеет читать этот файл.

Шаг 6. Заполните карточку решения

Скопируйте самостоятельный шаблон в документацию проекта:

bash
mkdir -p docs
cp \
  ../../../phases/01-reproducible-project/01-python-versions/outputs/python-version-record-template.md \
  docs/python-version.md

Заполните наблюдения. Достаточный фрагмент выглядит так:

Что проверили Наблюдение Интерпретация

Путь процесса <home>/miniconda3/bin/python3 запущен Python из пользовательской установки
requires-python >=3.11 проект требует Python 3.11 или новее

Не копируйте пример как готовый ответ: версия и путь должны происходить из вашего процесса. В разделе ограничений честно напишите, что пока код не запускался на нескольких minor-версиях и зависимости ещё не зафиксированы.

Посмотрите рабочую копию:

Разберите сообщение коллеги

Коллега прислал:

text
Команда python3 --version в терминале: Python 3.12.7
В notebook sys.version_info: 3.10.14
В notebook sys.executable: /usr/bin/python3
Ошибка: ModuleNotFoundError: No module named 'tomllib'

Из этих фактов следует:

  1. Терминал и notebook используют разные интерпретаторы.
  2. Ошибка произошла внутри Python 3.10, где tomllib ещё не входит в стандартную библиотеку.
  3. Установка Python 3.12 уже произошла, но notebook на него не переключён.
  4. Проверять и менять нужно ядро notebook, а не требование проекта и не случайный пакет.

Так выглядит полезная диагностика: она называет процесс, версию, путь и связь с ошибкой.

Отличайте требование от доказанной поддержки

Представьте, что в проекте записано:

toml
requires-python = ">=3.11"

Это означает, что проект не должен запускаться на Python 3.10. Но строка сама по себе не доказывает работу на 3.11, 3.12, 3.13 и всех будущих версиях.

Уровни доказательства различаются:

text
записали требование
        ↓
создали окружение на выбранном Python
        ↓
установили зафиксированные зависимости
        ↓
запустили код и тесты
        ↓
повторили на каждой заявленной линии

В этом уроке выполнен первый шаг и собраны локальные наблюдения. Остальные появятся по мере прохождения фазы.

Меняйте минимальную версию как контракт

Минимум можно поднять, если:

  • проект начинает использовать новый синтаксис;
  • нужен модуль из более новой стандартной библиотеки;
  • важная зависимость перестала поддерживать старую линию;
  • команда больше не может надёжно тестировать старую версию.

В pull request полезно написать:

text
Минимальная версия поднята с 3.10 до 3.11, потому что проект использует tomllib из
стандартной библиотеки. После обновления окружения тесты пройдут на 3.11 и 3.12.

Фраза «у меня установлен 3.12» описывает машину автора, а не причину изменить контракт для всех пользователей.

Сценарии разбираются по наблюдаемому поведению. Не устанавливайте случайные версии Python и не ломайте системный интерпретатор ради упражнения.

Ожидайте, что .python-version переключит всё сама

Вы записали 3.12, но python3 --version всё ещё показывает 3.11.

Это не ошибка текстового файла. .python-version — вход для совместимого менеджера, а не исполняемая команда. В следующем уроке uv прочитает выбор и создаст окружение.

Понизьте требование до версии коллеги

Коллега запускает Python 3.10, а проект требует >=3.11. Самое быстрое на вид решение — изменить контракт на >=3.10.

Так делать можно только после проверки кода и зависимостей на 3.10. Если проект использует возможности 3.11, правильное действие — выбрать поддерживаемый интерпретатор, а не переписать требование под случайную машину.

Зафиксируйте точную patch-версию без причины

text
.python-version = 3.12.7

Точный patch может быть оправдан регламентом поставки. Для обычного учебного проекта он часто создаёт лишнее ограничение и мешает использовать исправления безопасности внутри линии 3.12. В этом уроке достаточно выбора minor-линии.

Проверьте только python3 --version

Терминал показывает 3.12, поэтому автор делает вывод, что notebook тоже работает на 3.12.

Проверка описывает только новый процесс из shell. В notebook нужно отдельно выполнить sys.version_info и sys.executable.

Считайте декларацию тестом

Автор записал requires-python = ">=3.11" и объявил поддержку 3.13, хотя проект на ней не запускался.

Quiz · 4 вопроса

Проверьте себя

Ответьте после практики. Объяснение появится сразу после выбора.

01

Терминал показывает Python 3.12, но sys.version_info внутри notebook показывает 3.10. Какой вывод обоснован?

Версия является свойством процесса. Вывод внутри notebook описывает его kernel, а команда в shell — отдельно запущенный терминальный процесс.

02

В .python-version записали 3.12, но команда python3 по-прежнему запускает 3.11. Что это означает?

.python-version хранит желаемый локальный выбор. Реальное переключение выполняет менеджер Python или окружений, например uv в следующем уроке.

03

Проект требует Python >=3.11, а ошибка возникла в процессе на Python 3.10. Какое первое решение корректнее?

Наблюдаемый процесс не удовлетворяет контракту. Сначала выбирают поддерживаемый Python; расширять поддержку старой версии можно только после содержательной проверки.

04

Автор предлагает указать ровно Python 3.12.7, потому что эта версия стоит на его ноутбуке. Что следует ответить?

Patch-pin иногда нужен способу поставки, но не должен возникать из случайной локальной установки. Контракт версии отражает требования проекта и доказанную политику поддержки.

# Проверка решения о версии Python

Рубрика проверяет не оформление документа, а связь наблюдений с решением. Номер версии без
пути к интерпретатору и объяснения контракта считается недостаточным результатом.

## Результат готов, если

- [ ] Версия и путь определены внутри фактически проверяемого Python-процесса через
  `sys.version_info` и `sys.executable`.
- [ ] Отдельно записано, какую команду `python3` нашёл shell, и объяснено, почему эта
  проверка не заменяет наблюдение внутри notebook, IDE или другого процесса.
- [ ] Номер версии корректно разобран на major, minor и patch без утверждения, что patch
  создаёт новую несовместимую линию.
- [ ] `requires-python` описан как требование проекта, а `.python-version` — как подсказка
  совместимому инструменту для локального выбора.
- [ ] Выбранная линия `.python-version` не противоречит `requires-python`.
- [ ] Минимальная версия имеет причину, связанную с кодом, стандартной библиотекой,
  зависимостями или общей политикой курса, а не только с машиной автора.
- [ ] Документ не выдаёт декларацию совместимости за доказательство запуска: перечислены
  версии, которые действительно проверялись.
- [ ] В публикуемом документе нет секретов, абсолютного домашнего пути с именем
  пользователя и несвязанных локальных данных.
- [ ] `pyproject.toml`, `.python-version` и карточка решения просмотрены в staged и полном
  diff перед merge.

## Сильный результат дополнительно показывает

- [ ] Для расхождения shell и notebook назван конкретный процесс, внутри которого нужно
  повторить диагностику.
- [ ] Автор различает выбор интерпретатора и окружение с пакетами и не пытается исправить
  версию случайной установкой зависимостей.
- [ ] Изменение минимальной версии оформлено как изменение контракта с причиной и планом
  повторных проверок.
- [ ] Reviewer понимает, что следующим шагом будут создание окружения, фиксация
  зависимостей и воспроизводимый запуск, а не копирование локального `.venv`.

## Итог проверки

- **Готово:** все обязательные критерии выполнены, а три уровня версии не смешаны.
- **Нужна правка:** наблюдения есть, но не объяснены либо требование проекта и локальный
  выбор названы синонимами.
- **Нельзя принимать:** версия угадана по чужому выводу, контракт изменён только ради
  зелёной проверки либо в Git подготовлены локальные или чувствительные данные.
