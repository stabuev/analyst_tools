Ответьте до чтения: результат поможет заметить исходные предположения.

01

Скрипт с import numpy работает у автора, но после клонирования коллега получает ModuleNotFoundError. Какого факта пока точно недостаточно для воспроизводимости?

Локальный успешный импорт описывает только машину автора. Проекту нужны объявленная зависимость, зафиксированное разрешение и повторяемая команда создания окружения.

02

Команда python3 и команда из проектной .venv используют одну версию Python. Можно ли из этого заключить, что они видят одинаковые библиотеки?

Версия интерпретатора и установленный набор библиотек — разные состояния. Два окружения на Python 3.12 могут содержать разные версии NumPy или не содержать её вовсе.

Окружения и зависимости с uv

В Git хранят не готовую .venv, а достаточный рецепт, по которому uv заново создаёт окружение и запускает проект.

Тип: Build Треки: Core Пререквизиты: 00/04 — Секреты и безопасная работа с данными Время: ~90 минут Результат: фиксирует поддерживаемую версию Python, добавляет зависимость через uv и восстанавливает проверенное рабочее состояние из lock-файла

Цели обучения

  • Объяснять простыми словами, зачем проекту отдельное окружение.
  • Различать требование requires-python, локальный выбор .python-version и фактически запущенный sys.executable.
  • Отличать версию интерпретатора от набора библиотек в окружении.
  • Различать требования в pyproject.toml, зафиксированное разрешение в uv.lock и установленные пакеты в .venv.
  • Добавлять зависимость командой uv add, а не только устанавливать её на своей машине.
  • Запускать код через uv run без обязательной ручной активации окружения.
  • Использовать --locked, когда lock-файл не должен незаметно измениться.
  • Удалять .venv и восстанавливать работоспособный проект через uv sync --locked.
  • Объяснять, почему uv.lock хранят в Git, а .venv — нет.

Проблема

После фазы 00 у нас есть безопасный Git-репозиторий revenue-project: в нём отделены публикуемые файлы, sample и локальные данные. Но репозиторий пока не отвечает на два вопроса:

  1. Какой Python подходит проекту и какой процесс действительно запускает код?
  2. Какие внешние библиотеки нужны и как коллеге установить тот же проверенный набор?

Фраза «у меня установлен Python и NumPy» не отвечает ни на один из них. На одной машине может быть несколько Python, а один и тот же Python может участвовать в нескольких окружениях с разными пакетами.

Представьте, что аналитик добавил в расчёт:

python
import numpy as np

На его компьютере скрипт работает: numpy когда-то установили глобально для другого notebook. Коллега клонирует репозиторий и получает:

text
ModuleNotFoundError: No module named 'numpy'

Совет pip install numpy устраняет ошибку только на одной машине и только сейчас. Из него непонятно:

  • является ли numpy обязательной зависимостью проекта;
  • какой диапазон версий допустим;
  • какую конкретную версию выбрал автор;
  • с каким Python она была разрешена;
  • как повторить установку позже;
  • не запустился ли код случайно из другого окружения.

Копировать готовый каталог .venv тоже неправильно. Внутри него находятся пути и файлы, связанные с конкретной операционной системой, архитектурой и установкой Python. Это расходный локальный каталог, а не переносимый архив проекта.

Нужен воспроизводимый процесс:

text
требование Python + зависимости проекта + lock-файл + код и sample
                              ↓
                         uv sync
                              ↓
                    локальная .venv
                              ↓
                         uv run
                              ↓
                   проверенный результат

Концепция

Три утверждения о Python решают разные задачи

Для небольшого аналитического проекта достаточно различать три уровня:

Уровень Пример На какой вопрос отвечает
Требование проекта requires-python = ">=3.11" Какая минимальная версия допустима?
Локальный выбор .python-version содержит 3.12 Какую совместимую линию предложить uv?
Факт запуска sys.version_info и sys.executable Какой Python выполняет код сейчас?

requires-python — декларация совместимости, а не команда установки. .python-version — подсказка инструменту, а не доказательство переключения уже запущенного процесса. Доказательство даёт сам процесс:

bash
python3 -c 'import sys; print(sys.version_info[:3]); print(sys.executable)'

Для курса нужен Python 3.11 или новее. Обычно нет причины фиксировать конкретный patch, например только 3.12.7: исправляющие выпуски внутри линии 3.12 должны оставаться доступны. Реальную совместимость всё равно подтверждают запуск и проверки проекта.

Версия Python и окружение — не одно и то же. Интерпретатор выполняет код, а окружение связывает выбранный интерпретатор с библиотеками и командами проекта. Два окружения на Python 3.12 могут содержать разные версии NumPy или не содержать NumPy вовсе.

Окружение отвечает на вопрос «что видит этот проект»

На одном компьютере могут одновременно существовать несколько проектов:

text
sales-report  → Python 3.12 + numpy 2.x
old-dashboard → Python 3.10 + numpy 1.x

Если устанавливать всё в один глобальный Python, требования проектов начинают влиять друг на друга. Обновление библиотеки ради одного проекта может сломать другой. Виртуальное окружение даёт проекту отдельное место для интерпретатора, библиотек и команд.

Для uv проектное окружение по умолчанию находится в .venv:

text
revenue-project/
├── .venv/
│   ├── bin/       # Scripts/ в Windows
│   └── lib/
├── pyproject.toml
└── uv.lock

Изоляция не означает виртуальную машину или отдельную операционную систему. .venv изолирует прежде всего Python-пакеты и команды проекта. Системные библиотеки, ОС и аппаратная архитектура остаются внешними условиями.

Три состояния, которые нельзя смешивать

1. Декларация: pyproject.toml

В [project].dependencies записано, что разрешено установить:

toml
[project]
name = "revenue-project"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "numpy>=2,<3",
]

Диапазон >=2,<3 означает: проекту подходит версия NumPy не ниже 2.0, но ниже 3.0. Это требование, а не утверждение о фактически установленной версии.

2. Разрешение: uv.lock

Несколько зависимостей могут предъявлять требования друг к другу. Разрешить зависимости — значит найти совместимый набор конкретных версий и источников. Результат uv записывает в uv.lock.

Lock-файл:

  • создаётся и обновляется инструментом;
  • содержит точные выбранные версии полного графа;
  • учитывает ограничения Python и платформенные условия;
  • хранится в Git;
  • не редактируется вручную.

uv.lock является кроссплатформенным: он может описывать варианты для нескольких платформ. Это не обещание, что двоичные файлы на macOS и Linux совпадут побайтно. Это единое зафиксированное разрешение, из которого uv выбирает подходящие артефакты для конкретной поддерживаемой машины.

3. Установленное состояние: .venv

.venv — то, что сейчас действительно установлено локально. Каталог можно удалить и получить заново. Он:

  • зависит от машины и выбранного интерпретатора;
  • может отстать от pyproject.toml и uv.lock;
  • не является источником правды;
  • не хранится в Git.

Короткая модель:

Уровень Вопрос Пример
pyproject.toml Что проект допускает? numpy>=2,<3
uv.lock Что именно разрешил uv? конкретная версия NumPy и источник
.venv Что установлено на этой машине? файлы выбранной сборки NumPy

Прямая и транзитивная зависимость

Если код проекта импортирует numpy, NumPy — прямая зависимость: проект объявляет её сам. Если NumPy или другая прямая библиотека требует ещё один пакет, тот становится транзитивной зависимостью.

В pyproject.toml обычно перечисляют прямые требования проекта. uv.lock фиксирует весь получившийся граф. Не нужно вручную переносить каждую транзитивную библиотеку в dependencies: её владелец уже описал связь в metadata своего пакета.

uv add, uv sync и uv run решают разные задачи

Команда:

bash
uv add 'numpy>=2,<3'

добавляет требование проекта, обновляет lock-файл и синхронизирует окружение. Это команда для осознанного изменения контракта.

Команда:

bash
uv sync

приводит .venv к состоянию проекта. Если окружения нет, создаёт его. Если lock-файла ещё нет, обычный sync может создать или обновить его.

Команда:

bash
uv run python src/revenue_summary.py data/sample/orders.csv

запускает процесс в проектном окружении. Перед запуском обычный uv run проверяет и при необходимости обновляет lock-файл и среду.

Для строгого воспроизведения используйте:

bash
uv run --locked python src/revenue_summary.py data/sample/orders.csv

--locked означает: lock-файл должен существовать и соответствовать metadata проекта. Если он устарел, команда завершится ошибкой вместо скрытого обновления.

Активация полезна, но не обязательна

Окружение можно активировать:

bash
source .venv/bin/activate

В PowerShell команда выглядит иначе:

powershell
.venv\Scripts\Activate.ps1

После активации обычное имя python указывает на Python окружения, пока открыт этот shell. Но состояние shell легко забыть или потерять при открытии нового терминала.

Воспроизводимая инструкция может обходиться без активации:

bash
uv run python -c 'import sys; print(sys.executable)'

Поэтому в курсе основной путь — явный uv run. Активация остаётся удобной для редактора или продолжительной интерактивной работы.

Воспроизводимость шире одного lock-файла

uv.lock фиксирует Python-зависимости, но не хранит:

  • исходный код;
  • входные данные;
  • переменные окружения и секреты;
  • системные библиотеки;
  • внешние сервисы;
  • доказательство корректности расчёта.

Поэтому мы проверим не только успешную установку NumPy, но и запуск контрольного расчёта на синтетическом sample. Совпадение установки без совпадения результата ещё не доказывает работоспособность проекта.

Соберите это

Практика продолжает тот же work/00-02-git/revenue-project. Мы не создаём отдельную лабораторию: к безопасному репозиторию из фазы 00 добавляется минимальный Python-контракт.

Шаг 1. Вернитесь к чистому проекту и наблюдайте Python

Из корня курса:

bash
cd work/00-02-git/revenue-project
git switch main
git status --short

Пустой вывод git status --short означает, что новая работа не смешивается с незавершённым изменением. Посмотрите, какой Python сейчас найдёт shell и что сообщает сам процесс:

bash
python3 --version
command -v python3
python3 -c 'import sys; print(sys.version_info[:3]); print(sys.executable)'

Этот Python ещё не обязан стать Python проекта. Наблюдение нужно, чтобы не путать привычное имя команды с конкретным процессом. Затем создайте ветку:

bash
git switch -c feature/uv-environment

Шаг 2. Запишите минимальный контракт Python

Создайте pyproject.toml только с теми полями, которые уже нужны uv:

toml
[project]
name = "revenue-project"
version = "0.1.0"
requires-python = ">=3.11"

Затем выберите поддерживаемую minor-линию для локальной работы:

bash
uv python pin 3.12
cat .python-version
sed -n '1,20p' pyproject.toml

Здесь >=3.11 описывает допустимый минимум проекта, а 3.12 — конкретный локальный выбор. Это не противоречие. Если для вашей платформы нужна другая поддерживаемая линия, укажите её явно, но не ослабляйте требование только ради случайно установленного старого Python.

Шаг 3. Исключите производное окружение из Git

Проверьте, есть ли правило:

bash
grep -n '^\.venv/$' .gitignore

Если вывода нет, добавьте строку:

gitignore
.venv/

На этом шаге .venv может ещё не существовать. Важно сначала задать границу Git, а уже потом создавать большой локальный каталог.

Шаг 4. Создайте пустое проектное окружение

bash
uv sync

Команда прочитает requires-python и .python-version, выберет совместимый Python, создаст .venv и первый uv.lock.

Сравните два процесса:

bash
python3 -c 'import sys; print(sys.version_info[:3]); print(sys.executable)'
uv run python -c 'import sys; print(sys.version_info[:3]); print(sys.executable)'

Первый путь зависит от вашего shell. Второй должен вести в .venv текущего проекта. Теперь проверьте Git:

bash
git check-ignore -v .venv/
git status --short

В статусе появится uv.lock, но не содержимое .venv.

Шаг 5. Добавьте маленькую реальную задачу

Скопируйте готовый контрольный скрипт урока:

bash
mkdir -p src
cp \
  ../../../phases/01-reproducible-project/01-uv-environments/outputs/revenue_summary.py \
  src/revenue_summary.py

Скрипт читает уже знакомый синтетический data/sample/orders.csv, оставляет оплаченные заказы и считает их количество, выручку и среднее значение. Расчёт намеренно мал: здесь мы изучаем не NumPy и не новую метрику, а связь кода с окружением.

Попробуйте строгий запуск до объявления NumPy:

bash
uv run --locked python src/revenue_summary.py data/sample/orders.csv

Ожидается ModuleNotFoundError: чистое проектное окружение честно показывает, что зависимость не объявлена. Обычный python3 на вашей машине может как упасть, так и случайно найти глобальный NumPy. Именно поэтому его успех не является доказательством.

Шаг 6. Добавьте прямую зависимость через uv

bash
uv add 'numpy>=2,<3'

Посмотрите не только сообщение команды, но и изменившиеся состояния:

bash
git diff -- pyproject.toml uv.lock
uv tree
uv run python -c 'import numpy; print(numpy.__version__)'

В pyproject.toml появится допустимый диапазон. В uv.lock будет записана конкретная разрешённая версия. В .venv появятся установленные файлы этой версии.

Не переписывайте число из numpy.__version__ в pyproject.toml как единственно верное. Диапазон описывает совместимость проекта, а точный текущий выбор уже хранится в lock-файле.

Шаг 7. Запустите контрольный расчёт

bash
uv run --locked python src/revenue_summary.py data/sample/orders.csv

Ожидаемый результат для sample из фазы 00:

json
{"average_paid_order": 97.5, "paid_order_count": 2, "revenue": 195.0}

Проверьте смысл результата, а не только код возврата:

  • оплаченных строк две: TEST-001 и TEST-003;
  • выручка равна 120 + 75 = 195;
  • среднее по двум оплаченным заказам равно 97.5;
  • строка pending на 80 не входит в расчёт.

Шаг 8. Докажите восстановление, удалив только .venv

Сначала сохраните наблюдаемую версию NumPy:

bash
uv run --locked python -c 'import numpy; print(numpy.__version__)' \
  > /tmp/numpy-version-before.txt
pwd

pwd должен заканчиваться на work/00-02-git/revenue-project. После этого удалите только локальное окружение:

bash
rm -rf .venv
test ! -d .venv && echo '.venv удалена'

Восстановите его из зафиксированного контракта:

bash
uv sync --locked
uv run --locked python -c 'import numpy; print(numpy.__version__)' \
  > /tmp/numpy-version-after.txt
diff /tmp/numpy-version-before.txt /tmp/numpy-version-after.txt
uv run --locked python src/revenue_summary.py data/sample/orders.csv

Пустой вывод diff подтверждает ту же разрешённую версию прямой зависимости. Повторный JSON подтверждает, что восстановленное окружение действительно запускает проект.

Шаг 9. Проведите изменение через Git

Проверьте границу публикации:

bash
git status --short
git check-ignore -v .venv/
git diff -- .gitignore pyproject.toml uv.lock src/revenue_summary.py

Подготовьте только воспроизводимый рецепт и код:

bash
git add .gitignore pyproject.toml uv.lock src/revenue_summary.py
git diff --cached --name-only
git diff --cached

В staged-наборе не должно быть .venv. Завершите изменение:

bash
git commit -m "Add reproducible uv environment"
git status --short

Используйте это

Минимальная инструкция для коллеги

После клонирования проекта коллеге не нужно угадывать отдельные команды установки:

bash
uv sync --locked
uv run --locked python src/revenue_summary.py data/sample/orders.csv

Первая команда создаёт локальную .venv из существующего lock-файла. Вторая запускает реальную задачу и запрещает неявно менять разрешение.

Если установка проходит, а расчёт падает, проблема уже не сводится к «пакет не установлен». Нужно читать ошибку кода, проверять входные данные и внешние условия.

Повседневная работа и строгая проверка

Во время разработки удобно:

bash
uv run python src/revenue_summary.py data/sample/orders.csv

uv сам поддержит lock-файл и окружение в актуальном состоянии. В проверяемой инструкции, CI или воспроизведении чужого результата полезнее строгий режим:

bash
uv lock --check
uv sync --check --locked
uv run --locked python src/revenue_summary.py data/sample/orders.csv

Здесь команды не должны «чинить по дороге» незакоммиченные изменения контракта.

Когда добавлять зависимость

Если код проекта импортирует библиотеку при обычном запуске, используйте:

bash
uv add package-name

Не добавляйте библиотеку только потому, что она открыта в соседнем notebook или может когда-нибудь пригодиться. Каждая зависимость увеличивает граф разрешения, поверхность обновлений и число возможных конфликтов.

Настройки инструментов и packaging-поля будут появляться в pyproject.toml позже, ровно в тех уроках, где они понадобятся. Здесь достаточно минимальных metadata и одной прямой runtime-зависимости.

Сломайте это

Установите пакет только в .venv

Представьте, что вместо uv add выполнено:

bash
uv pip install pandas

Импорт может заработать локально, но pandas не обязательно появится в [project].dependencies и согласованном uv.lock. После удаления .venv это состояние пропадёт. Для зависимости проекта используйте uv add pandas.

Измените manifest без обновления lock-файла

Добавьте требование вручную в pyproject.toml, но не запускайте uv lock. Затем:

bash
uv run --locked python src/revenue_summary.py data/sample/orders.csv

Строгий запуск должен завершиться ошибкой stale lockfile. Это полезный отказ: uv не выдаёт старое разрешение за соответствующее новой декларации.

Исправление — осознанно обновить контракт:

bash
uv lock
uv sync

или выполнять исходное изменение через uv add.

Не сохраните uv.lock

Если коллега получает только pyproject.toml, uv может заново подобрать версии из доступных на момент установки. Они будут удовлетворять диапазонам, но не обязательно совпадут с проверенным набором автора.

Проверьте:

bash
git ls-files --error-unmatch uv.lock

Запустите код мимо проекта

bash
python3 src/revenue_summary.py data/sample/orders.csv

Команда использует Python из текущего PATH. Если NumPy там есть, ошибка маскируется; если нет — появляется на машине автора. Сравните:

bash
python3 -c 'import sys; print(sys.executable)'
uv run python -c 'import sys; print(sys.executable)'

Попробуйте закоммитить .venv

bash
git status --short .venv
git check-ignore -v .venv/

Первый вывод должен быть пустым, второй — назвать правило .gitignore. Если .venv уже попала под tracking, одного нового правила недостаточно: Git продолжит следить за ранее добавленными файлами.

Quiz · 5 вопроса

Проверьте себя

Ответьте после практики. Объяснение появится сразу после выбора.

01

Аналитик хочет сделать NumPy обязательной зависимостью проекта. Какое действие лучше всего сохраняет намерение в проектном контракте?

uv add объявляет прямую зависимость, обновляет разрешение и синхронизирует окружение. Обычная установка только в .venv не делает пакет частью контракта проекта.

02

После удаления .venv нужно восстановить именно проверенный набор зависимостей и не разрешать uv незаметно менять lock-файл. Какая команда подходит?

uv sync --locked создаёт или синхронизирует окружение из существующего актуального uv.lock и отказывается работать, если lock-файл не соответствует metadata проекта.

03

В pyproject.toml вручную добавили pandas, но uv.lock не обновили. Что должен сделать uv run --locked?

--locked запрещает неявное обновление lock-файла. Рассинхронизация становится явной ошибкой и требует осознанно обновить контракт.

04

Коллега выполнил uv sync --locked без ошибок. Какой дополнительный шаг лучше всего проверит, что восстановленное окружение действительно годится для проекта?

Успешная установка подтверждает состояние зависимостей, но не корректность запуска проекта. Контрольная реальная задача проверяет импорт, код и ожидаемый результат вместе.

05

В pyproject.toml указано requires-python = ">=3.11", в .python-version записано 3.12, а uv run сообщает Python 3.12.8. Как это интерпретировать?

Требование проекта, локальный выбор minor-линии и факт текущего процесса отвечают на разные вопросы. Эти три значения согласуются и не обязаны быть записаны одинаково.

from __future__ import annotations

import argparse
import csv
import json
from pathlib import Path

import numpy as np


def read_paid_amounts(path: Path) -> np.ndarray:
    """Read amounts of paid orders from the course sample CSV."""
    with path.open(encoding="utf-8", newline="") as source:
        reader = csv.DictReader(source)
        fields = set(reader.fieldnames or [])
        missing = {"status", "amount"} - fields
        if missing:
            names = ", ".join(sorted(missing))
            raise ValueError(f"missing required columns: {names}")

        amounts: list[float] = []
        for row_number, row in enumerate(reader, start=2):
            if row["status"] != "paid":
                continue
            try:
                amounts.append(float(row["amount"]))
            except (TypeError, ValueError) as error:
                raise ValueError(
                    f"row {row_number}: amount must be a number"
                ) from error

    if not amounts:
        raise ValueError("the file contains no paid orders")
    return np.asarray(amounts, dtype=float)


def summarize_paid_orders(path: Path) -> dict[str, float | int]:
    amounts = read_paid_amounts(path)
    return {
        "paid_order_count": int(amounts.size),
        "revenue": round(float(amounts.sum()), 2),
        "average_paid_order": round(float(amounts.mean()), 2),
    }


def build_parser() -> argparse.ArgumentParser:
    parser = argparse.ArgumentParser(
        description="Summarize paid orders from a synthetic CSV sample"
    )
    parser.add_argument("orders_csv", type=Path)
    return parser


def main() -> int:
    parser = build_parser()
    args = parser.parse_args()
    try:
        result = summarize_paid_orders(args.orders_csv)
    except (OSError, ValueError) as error:
        parser.exit(2, f"revenue-summary: {error}\n")
    print(json.dumps(result, ensure_ascii=False, sort_keys=True))
    return 0


if __name__ == "__main__":
    raise SystemExit(main())
