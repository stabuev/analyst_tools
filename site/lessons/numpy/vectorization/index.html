Проверка перед уроком

Ответьте до чтения: результат поможет заметить исходные предположения.

01

Аналитик заменил поэлементный Python-цикл выражением над ndarray. Что в этом уроке называется векторизацией?

Векторизация переносит поэлементную работу из интерпретируемого Python-цикла в операции над массивами. Она не гарантирует параллельность и не сводится к одному преобразованию типа.

02

Перед сравнением времени циклического и векторного расчета выручки что нужно сделать в первую очередь?

Сначала фиксируют предметный контракт, подтверждают эквивалентность наиболее детального результата и определяют scope. Иначе можно точно измерить разные расчеты.

Векторизация и производительность

Векторизация переносит повторяющуюся операцию из Python-цикла в операции над массивами, но ускорение имеет смысл обсуждать только после проверки результата, границ измерения и расхода памяти.

Результат: переносит поэлементный расчет из Python-цикла в операции NumPy, проверяет построчную и итоговую эквивалентность, честно сравнивает повторные времена и объясняет компромисс между скоростью, памятью и понятностью.

Цели обучения

После урока вы сможете:

  • назвать единицу наблюдения и предметный смысл оси в векторизуемом расчете;
  • построить прозрачный Python-цикл как контрольную реализацию формулы;
  • выразить ту же формулу через поэлементные операции и агрегацию NumPy;
  • проверить наиболее детальный результат до сравнения итоговой суммы;
  • отличить расчетное время от end-to-end времени вместе с подготовкой данных;
  • выполнить предварительный вызов, несколько замеров и сравнить медианы;
  • правильно прочитать отношение времен, не превращая его в универсальное обещание;
  • оценить память входных и временных массивов;
  • объяснить, когда обычный цикл или обработка блоками разумнее полной векторизации.

Проблема

В таблице продаж одна строка соответствует не заказу целиком, а одной позиции заказа. Для каждой позиции нужно рассчитать выручку после скидки:

text
line_revenue = price * quantity * (1 - discount)

Затем выручка всех позиций складывается:

text
net_revenue = sum(line_revenue)

В исследовательском ноутбуке аналитик написал понятный цикл. На нескольких строках он работает мгновенно, а на сотнях тысяч позиций начинает занимать заметное время. Возникает естественная идея: заменить цикл выражением NumPy.

Но более короткая запись сама по себе не отвечает на важные вопросы:

  • совпадает ли формула для каждой позиции;
  • одинаковые ли данные получают две реализации;
  • входят ли преобразования списков в массивы в измеряемое время;
  • достаточно ли велик набор данных, чтобы сравнение имело рабочий смысл;
  • сколько временных массивов создает выражение;
  • можно ли переносить измеренное ускорение на другую машину и другой pipeline.

Если сразу запустить таймер, легко очень точно измерить две разные задачи. В этом уроке мы сначала зафиксируем вычислительный контракт, затем докажем эквивалентность и только после этого сравним время.

Концепция

Один расчет связывает все предыдущие уроки NumPy

Векторизация не является отдельным трюком, который отменяет модель массива. Она собирает вместе уже изученные идеи:

Предыдущая тема Роль в расчете выручки
ndarray хранит однородные цены, количества и скидки
Shape и axes все три массива относятся к одной оси line_item
dtype определяет диапазон, точность и тип результата умножения
Индексация и маски позволяют проверить или отобрать отдельные позиции
Агрегации sum() убирает ось позиций и возвращает общий итог
Broadcasting объясняет, как скаляр 1.0 применяется ко всем скидкам
Случайность локальный Generator создает воспроизводимые входы benchmark

Если форма, тип или смысл оси не определены, векторизация лишь быстрее выполнит неопределенный расчет.

Сначала предметный контракт

В учебном примере есть три одномерных массива:

Массив Смысл элемента Ожидаемая форма Рабочий dtype Допустимые значения
prices цена единицы товара (line_item,) float64 конечное число >= 0
quantities число единиц (line_item,) int64 целое число >= 0
discounts доля скидки (line_item,) float64 конечное число в [0, 1]

Одинаковая длина здесь означает не просто техническую совместимость. Элементы с одним индексом должны описывать одну и ту же позицию заказа:

text
prices[i], quantities[i], discounts[i] -> line_item[i]

Если массив скидок переставлен относительно массива цен, формы останутся одинаковыми и NumPy выполнит расчет без ошибки. Поэтому shape-контракт необходим, но не доказывает согласованность строк исходных данных.

Контрольный пример на трех позициях

Возьмем значения, которые можно проверить без компьютера:

Позиция Цена Количество Скидка Выручка после скидки
0 100 2 0% 100 * 2 * 1.00 = 200
1 80 1 25% 80 * 1 * 0.75 = 60
2 50 4 10% 50 * 4 * 0.90 = 180

Наиболее детальный ожидаемый результат:

text
line_revenue = [200, 60, 180]

И только затем итог:

text
net_revenue = 200 + 60 + 180 = 440

Это разделение принципиально. Одинаковая итоговая сумма может получиться у разных наборов построчных значений. Например, [200, 60, 180] и [180, 60, 200] оба дают 440, но приписывают выручку не тем позициям.

Что именно означает векторизация

Python-цикл выполняет тело цикла отдельно для каждого элемента:

text
получить три Python-объекта
-> выполнить Python-умножение и вычитание
-> обновить Python-переменную
-> перейти к следующей позиции

Векторное выражение передает NumPy целые массивы и описание операции:

text
умножить соответствующие элементы массивов
-> применить скидку
-> агрегировать результат

Циклы не исчезают. Они выполняются внутри библиотеки в скомпилированном коде над однородными буферами памяти. Python организует несколько крупных операций вместо сотен тысяч мелких итераций.

В этом курсе под векторизацией понимается именно выражение расчета через array API вместо Python-цикла по отдельным элементам. Это слово само по себе не обещает:

  • использование GPU;
  • параллельное выполнение на нескольких ядрах;
  • применение SIMD в каждом конкретном вызове;
  • отсутствие внутренних циклов;
  • отсутствие временных массивов;
  • автоматическое ускорение любого алгоритма.

Конкретная операция NumPy может использовать аппаратные оптимизации, но пользовательский контракт остается тем же: массивы, формы, типы и операция над ними.

Поэлементные операции и редукция — разные этапы

Выражение:

python
line_revenue = prices * quantities * (1.0 - discounts)

сохраняет ось line_item:

text
prices:       (line_item=n,)
quantities:   (line_item=n,)
discounts:    (line_item=n,)
line_revenue: (line_item=n,)

Агрегация:

python
net_revenue = line_revenue.sum(dtype=np.float64)

убирает единственную ось и возвращает скаляр. Это два разных наблюдаемых результата:

  1. выручка каждой позиции;
  2. общая выручка набора позиций.

Проверять только второй результат недостаточно: ошибки отдельных строк могут компенсироваться при сложении.

Как ведут себя типы

При контракте урока:

text
prices.dtype       = float64
quantities.dtype   = int64
discounts.dtype    = float64

выражение 1.0 - discounts возвращает float64, смешанное умножение с ценами также дает float64, а сумма явно накапливается в float64.

Порядок операций имеет значение. Если и цены, и количества хранятся в узком целом типе, то промежуточное prices * quantities может переполниться до встречи с вещественной скидкой. Поэтому готовый артефакт сначала проверяет данные и приводит цены к float64, а количества — к int64. Это применение урока о dtype, а не косметическое форматирование.

Почему NumPy часто быстрее

Обе реализации выполняют линейное число операций: при удвоении числа позиций объем работы примерно удваивается. Разница обычно возникает в постоянных накладных расходах:

  • Python-цикл интерпретирует каждую итерацию;
  • NumPy получает одну крупную операцию над однородным массивом;
  • данные ndarray расположены компактнее Python-объектов;
  • библиотека может использовать оптимизированные машинные инструкции.

Из этого не следует, что NumPy быстрее на любом размере. Для трех элементов подготовка и вызов операции могут стоить больше самого цикла. Для ветвистого или последовательного алгоритма естественной векторной операции может вообще не быть.

Три независимых контракта

Корректное сравнение удобно разделить на три вопроса.

1. Контракт результата

  • одна и та же предметная формула;
  • одинаковые позиции и значения входов;
  • согласованные shape и dtype;
  • построчный результат совпадает с заданной точностью;
  • итоговая сумма совпадает с заданной точностью.

2. Контракт измерения

  • явно названа работа внутри таймера;
  • обе реализации получают заранее подготовленные данные;
  • до замеров выполнен предварительный вызов;
  • сделано несколько повторов;
  • сохранены все времена и выбран способ сводки;
  • размер входа и среда записаны вместе с результатом.

3. Контракт вывода

  • ускорение относится только к измеренному входу и scope;
  • результат не объявляется свойством всех машин и всех задач;
  • стоимость памяти рассматривается вместе со временем;
  • быстрый неправильный расчет не считается улучшением.

Calculation-only и end-to-end — разные вопросы

Граница измерения называется scope. В этом уроке основной артефакт отвечает на узкий вопрос:

Сколько занимает сама формула и суммирование, если обе реализации уже получили удобное для них подготовленное представление данных?

Это calculation_only:

text
Python branch: готовые списки -> цикл и сумма
NumPy branch:  готовые ndarray -> выражение и np.sum

За пределами таймера остаются генерация, проверка, преобразование ndarray в списки, сверка результатов и запись JSON.

Другой рабочий вопрос может звучать так:

Что быстрее от исходных Python-списков до итогового числа?

Тогда NumPy-ветка обязана включать np.asarray. Это уже end-to-end scope:

text
Python branch: исходные списки -> цикл и сумма
NumPy branch:  исходные списки -> np.asarray -> выражение и np.sum

Оба сравнения допустимы, но отвечают на разные вопросы. Нельзя исключить преобразование из подписи отчета, а потом сделать вывод о полной стоимости pipeline.

Зачем нужны предварительный вызов и повторы

Для измерения используется time.perf_counter_ns(): монотонный таймер с высоким разрешением для измерения прошедшего времени. Он измеряет wall time, поэтому в результат могут попасть переключения процессов и фоновая нагрузка.

Один предварительный вызов помогает не смешивать основное измерение с частью разовых эффектов первого запуска. Это не «очищает» машину и не гарантирует стабильность.

Несколько повторов показывают разброс. В уроке сводкой служит медиана — центральное значение упорядоченных времен. Отдельный долгий запуск меньше влияет на медиану, чем на среднее. При этом медиана семи повторов не превращает локальный микробенчмарк в универсальный закон.

Как читать speedup

В артефакте отношение вычисляется так:

text
speedup = median_loop_seconds / median_vectorized_seconds

Примеры интерпретации:

Результат Что он означает в зафиксированном benchmark
20.0 медиана цикла в 20 раз больше медианы NumPy-ветки
1.0 медианы примерно равны
0.7 в этом измерении цикл оказался быстрее

Фраза «NumPy всегда быстрее в 20 раз» не следует ни из одного такого числа. Корректная формулировка называет размер, scope, реализации, число повторов и среду.

Векторизация тратит память на промежуточные результаты

Для n позиций входные ndarray урока занимают:

text
prices:       n * 8 bytes
quantities:   n * 8 bytes
discounts:    n * 8 bytes
--------------------------------
inputs:       n * 24 bytes

Прямое выражение можно мысленно разложить так:

python
gross = prices * quantities
multiplier = 1.0 - discounts
line_revenue = gross * multiplier

Каждый из трех результатов имеет float64 и занимает еще n * 8 байт. Во время последнего умножения эти буферы могут одновременно существовать, поэтому учебная верхняя оценка временных ndarray равна 3 * n * 8 байт.

Для миллиона позиций это примерно:

text
входные буферы ndarray:       24 MB
три временных float64-вектора: 24 MB

Это не оценка всего процесса: в ней нет Python-списков, Python-объектов, служебных данных аллокатора и памяти интерпретатора. Артефакт явно сохраняет эту границу оценки.

Если память является ограничением, один рабочий буфер можно переиспользовать:

python
workspace = np.empty_like(prices, dtype=np.float64)
np.subtract(1.0, discounts, out=workspace)
np.multiply(workspace, quantities, out=workspace)
np.multiply(workspace, prices, out=workspace)
net_revenue = workspace.sum(dtype=np.float64)

Такой вариант сложнее читать и легче случайно испортить изменением не того массива. Оптимизировать память стоит после оценки, а не ради самого использования out=.

Когда цикл остается разумным

Обычный цикл не является ошибкой сам по себе. Он может быть хорошим решением, если:

  • данных мало и время не влияет на рабочий процесс;
  • следующий результат зависит от предыдущего состояния;
  • логика содержит сложные ветвления, для которых нет ясной операции массива;
  • векторная запись создает слишком большие временные массивы;
  • обработка блоками проще укладывается в ограничение памяти;
  • основное время занимает сеть, диск или база данных, а не Python-цикл;
  • векторная версия настолько сложна, что ее трудно проверить и сопровождать.

Цель не в том, чтобы удалить все циклы. Цель — выбрать понятную и проверенную реализацию с приемлемой стоимостью.

Соберите это

Шаг 1. Зафиксируйте маленький контрольный набор

Начните со списков, а не со случайного миллиона строк:

python
prices = [100.0, 80.0, 50.0]
quantities = [2, 1, 4]
discounts = [0.0, 0.25, 0.10]

expected_line_revenue = [200.0, 60.0, 180.0]
expected_total = 440.0

Назовите ось списков line_item. Элементы с одним индексом относятся к одной позиции.

Шаг 2. Проверьте минимальный предметный контракт

Прозрачная проверка для учебных списков:

python
import math


def validate_line_items(prices, quantities, discounts):
    if not (len(prices) == len(quantities) == len(discounts)):
        raise ValueError("inputs must have equal lengths")
    if not prices:
        raise ValueError("inputs must not be empty")

    if any(not math.isfinite(price) or price < 0 for price in prices):
        raise ValueError("prices must be finite and non-negative")
    if any(
        isinstance(quantity, bool)
        or not isinstance(quantity, int)
        or quantity < 0
        for quantity in quantities
    ):
        raise ValueError("quantities must be non-negative integers")
    if any(
        not math.isfinite(discount) or not 0 <= discount <= 1
        for discount in discounts
    ):
        raise ValueError("discounts must be finite and in [0, 1]")

Проверка выполняется до расчета и до таймера. Benchmark не должен измерять поведение на данных, которые не соответствуют смыслу задачи.

Шаг 3. Постройте построчный Python-baseline

python
def python_line_revenue(prices, quantities, discounts):
    return [
        price * quantity * (1.0 - discount)
        for price, quantity, discount in zip(
            prices,
            quantities,
            discounts,
            strict=True,
        )
    ]

zip(..., strict=True) не позволяет тихо отбросить хвост более длинного списка. Это последняя техническая защита, но она не заменяет понятное сообщение валидатора.

Проверьте подробный результат:

python
validate_line_items(prices, quantities, discounts)
loop_rows = python_line_revenue(prices, quantities, discounts)

assert loop_rows == expected_line_revenue
assert sum(loop_rows) == expected_total

Шаг 4. Постройте baseline итогового расчета

Для измерения не обязательно сохранять отдельный список выручки:

python
def python_net_revenue(prices, quantities, discounts):
    total = 0.0
    for price, quantity, discount in zip(
        prices,
        quantities,
        discounts,
        strict=True,
    ):
        total += price * quantity * (1.0 - discount)
    return total

Построчная функция нужна для диагностики эквивалентности. Итоговая функция отражает фактический расчет, время которого мы хотим сравнить.

text

Используйте это

Подготовьте массивы с явными типами

Выразите построчную формулу через массивы

vector_rows = (

Измерьте одну реализацию несколько раз

python
import statistics
import time


def measure_seconds(function, repeat):
    function()  # один предварительный вызов вне измерений
    durations = []

    for _ in range(repeat):
        started = time.perf_counter_ns()
        function()
        elapsed_ns = time.perf_counter_ns() - started
        durations.append(elapsed_ns / 1_000_000_000)

    return statistics.median(durations), durations

Замерьте две заранее подготовленные ветки:

vector_median, vector_runs = measure_seconds(

            dtype=np.float64,
        )
    ),
    repeat=7,

def numpy_from_lists(prices, quantities, discounts):

Итог совпал, но позиции перепутаны. Сначала сравнивайте наиболее детальный результат.

Сравнить разные формулы

python
# Python: скидка является долей
price * quantity * (1.0 - discount)

# NumPy: скидку ошибочно вычитают как денежную сумму
prices * quantities - discounts

Такой benchmark сравнивает не реализации одного расчета, а разные бизнес-правила.

Тихо усечь входы

python
prices = [100.0, 80.0, 50.0]
quantities = [2, 1]
discounts = [0.0, 0.25, 0.1]

Обычный zip остановится на двух элементах. strict=True и предварительная проверка длин превращают потерю строки в явную ошибку.

Применить скидки не к тем позициям

python
discounts = discounts[::-1]

Форма осталась (n,), поэтому NumPy не знает, что строки потеряли согласованность. Совместимость формы не проверяет ключи исходной таблицы.

Пропустить предметную валидацию

python
discounts = np.array([0.1, -0.2, 1.5])

Выражение выполнится быстро и воспроизводимо, но отрицательная скидка увеличит цену, а скидка выше единицы даст отрицательную выручку. Обе реализации могут согласованно посчитать бессмысленный результат.

Получить переполнение до вещественной операции

python
prices = np.array([30_000], dtype=np.int16)
quantities = np.array([3], dtype=np.int16)

result = prices * quantities * (1.0 - discounts)

Первое умножение выполняется в узком целом типе и может переполниться. Явно готовьте рабочие типы до расчета.

Объявить проигрыш NumPy ошибкой

Для десяти элементов NumPy-ветка может оказаться медленнее. Это наблюдение о выбранном размере и scope, а не поломка библиотеки. Урок намеренно не требует speedup > 1.

Создать слишком много временных массивов

Длинная цепочка поэлементных выражений может выделить несколько буферов размером со весь набор. Ускорение CPU не компенсирует нехватку памяти. Оцените nbytes, рассмотрите переиспользование буфера или обработку крупными блоками.

Использовать out= и испортить вход

python
np.subtract(1.0, discounts, out=discounts)

Теперь исходные скидки заменены множителями после скидки. Переиспользуйте отдельный рабочий буфер и явно проверяйте, какие массивы разрешено изменять.

Векторизовать последовательный процесс как независимые строки

Если состояние на позиции i зависит от результата позиции i - 1, простая поэлементная формула меняет алгоритм. Иногда нужен специализированный накопительный метод, а иногда понятный цикл остается правильным решением.

Оптимизировать не то место

Если 95% времени pipeline занимает SQL-запрос или чтение файла, ускорение короткой формулы в 20 раз почти не изменит end-to-end время. Поиск реального bottleneck относится к профилированию и подробно рассматривается в фазе 12.

Принять векторизацию за параллельность

  • NumPy: CPU/SIMD optimizations — углубление после основной части: runtime dispatch и аппаратно-зависимые SIMD-оптимизации; материал помогает не смешивать пользовательскую векторизацию через array API с конкретными машинными инструкциями.

Проверьте себя

Ответьте после практики. Объяснение появится сразу после выбора.

01

Python-ветка вернула построчные значения [200, 60, 180], NumPy-ветка — [180, 60, 200]. Общая сумма обеих равна 440. Как поступить?

Компенсирующие или переставленные значения могут сохранить агрегат. Эквивалентность сначала проверяют на наиболее детальном наблюдаемом результате.

02

Benchmark объявлен calculation-only: цикл получает готовые списки, NumPy — готовые ndarray. Что должно остаться вне таймера?

Узкий calculation-only scope измеряет уже подготовленные вычислительные ветки. Все исключения должны быть явно перечислены в отчете.

03

Рабочий pipeline получает Python-списки, а решение требуется оценить от этого входа до итоговой суммы. Как честно измерить NumPy-ветку?

Если исходное представление — часть поставленного вопроса, стоимость перехода к ndarray входит в end-to-end ветку. Это отдельный benchmark, а не замена подписи calculation-only.

04

На десяти позициях медиана NumPy-ветки больше медианы простого цикла. Какой вывод обоснован?

Вызов операций массива тоже имеет стоимость. Локальный результат на малом n допустим, но не описывает другие размеры и среды.

05

Выражение prices * quantities * (1 - discounts) стало быстрее цикла, но на полном наборе процесс упирается в память. Что проверить следующим шагом?

Цепочка операций может одновременно держать несколько буферов размером с данные. Скорость и память являются разными частями решения.

06

Почему unit test артефакта не требует speedup > 10?

Тесты проверяют формулу, валидацию, структуру отчета и положительные времена. Жесткий performance-порог без контролируемой среды создает нестабильный тест.

07

Остаток после каждой операции зависит от остатка после предыдущей. Простая поэлементная запись считает все строки независимо. Какое решение разумнее?

Векторная реализация должна сохранять алгоритм. Для состояния, зависящего от предыдущего шага, нужен подходящий накопительный механизм либо прозрачный цикл.

08

Как корректно передать результат локального benchmark коллеге?

Число без условий невоспроизводимо и легко переинтерпретировать. Отчет должен связывать измерение с конкретными данными, реализациями, средой и допустимым утверждением.

from __future__ import annotations

import argparse
import json
import platform
import statistics
import time
from collections.abc import Callable, Sequence
from pathlib import Path
from typing import Any

import numpy as np

MAX_SIZE = 2_000_000
MIN_REPEAT = 3
MAX_REPEAT = 101
MAX_SEED = 2**64 - 1
ROW_RTOL = 1e-12
ROW_ATOL = 1e-12
TOTAL_RTOL = 1e-12
TOTAL_ATOL = 1e-8


class BenchmarkError(ValueError):
    """Raised when benchmark inputs or results violate the contract."""


def _validate_size(size: int) -> None:
    if isinstance(size, bool) or not isinstance(size, (int, np.integer)):
        raise BenchmarkError("size must be an integer")
    if size <= 0 or size > MAX_SIZE:
        raise BenchmarkError(f"size must be between 1 and {MAX_SIZE:,}")


def _validate_repeat(repeat: int) -> None:
    if isinstance(repeat, bool) or not isinstance(repeat, (int, np.integer)):
        raise BenchmarkError("repeat must be an integer")
    if repeat < MIN_REPEAT or repeat > MAX_REPEAT:
        raise BenchmarkError(f"repeat must be between {MIN_REPEAT} and {MAX_REPEAT}")


def _validate_seed(seed: int) -> None:
    if isinstance(seed, bool) or not isinstance(seed, (int, np.integer)):
        raise BenchmarkError("seed must be an integer")
    if seed < 0 or seed > MAX_SEED:
        raise BenchmarkError(f"seed must be between 0 and {MAX_SEED}")


def prepare_inputs(
    prices: Any,
    quantities: Any,
    discounts: Any,
) -> tuple[np.ndarray, np.ndarray, np.ndarray]:
    """Validate the line-item contract and return contiguous calculation arrays."""

    raw_prices = np.asarray(prices)
    raw_quantities = np.asarray(quantities)
    raw_discounts = np.asarray(discounts)
    arrays = {
        "prices": raw_prices,
        "quantities": raw_quantities,
        "discounts": raw_discounts,
    }

    for name, values in arrays.items():
        if values.ndim != 1:
            raise BenchmarkError(f"{name} must be one-dimensional")
        if np.issubdtype(values.dtype, np.bool_):
            raise BenchmarkError(f"{name} must not use boolean dtype")
        is_real_number = np.issubdtype(values.dtype, np.integer) or np.issubdtype(
            values.dtype, np.floating
        )
        if values.dtype == np.dtype("O") or not is_real_number:
            raise BenchmarkError(f"{name} must contain real numeric values")

    if not (raw_prices.shape == raw_quantities.shape == raw_discounts.shape):
        raise BenchmarkError("input arrays must have equal shapes")
    if raw_prices.size == 0:
        raise BenchmarkError("input arrays must not be empty")
    if not np.issubdtype(raw_quantities.dtype, np.integer):
        raise BenchmarkError("quantities must use an integer dtype")

    normalized_prices = np.ascontiguousarray(raw_prices, dtype=np.float64)
    normalized_quantities = np.ascontiguousarray(raw_quantities, dtype=np.int64)
    normalized_discounts = np.ascontiguousarray(raw_discounts, dtype=np.float64)

    if not np.all(np.isfinite(normalized_prices)):
        raise BenchmarkError("prices must contain only finite values")
    if not np.all(np.isfinite(normalized_discounts)):
        raise BenchmarkError("discounts must contain only finite values")
    if np.any(normalized_prices < 0.0):
        raise BenchmarkError("prices must be non-negative")
    if np.any(normalized_quantities < 0):
        raise BenchmarkError("quantities must be non-negative")
    if np.any((normalized_discounts < 0.0) | (normalized_discounts > 1.0)):
        raise BenchmarkError("discounts must be between 0 and 1")

    return normalized_prices, normalized_quantities, normalized_discounts


def python_line_revenue(
    prices: Sequence[float],
    quantities: Sequence[int],
    discounts: Sequence[float],
) -> list[float]:
    """Transparent per-line baseline for already validated Python sequences."""

    if not (len(prices) == len(quantities) == len(discounts)):
        raise BenchmarkError("input sequences must have equal lengths")
    return [
        price * quantity * (1.0 - discount)
        for price, quantity, discount in zip(
            prices,
            quantities,
            discounts,
            strict=True,
        )
    ]


def numpy_line_revenue(
    prices: np.ndarray,
    quantities: np.ndarray,
    discounts: np.ndarray,
) -> np.ndarray:
    """Vectorized per-line calculation for prepared one-dimensional arrays."""

    if not (prices.shape == quantities.shape == discounts.shape):
        raise BenchmarkError("input arrays must have equal shapes")
    if prices.ndim != 1:
        raise BenchmarkError("input arrays must be one-dimensional")
    return prices * quantities * (1.0 - discounts)


def python_net_revenue(
    prices: Sequence[float],
    quantities: Sequence[int],
    discounts: Sequence[float],
) -> float:
    """Loop and reduction used in the timed Python baseline."""

    if not (len(prices) == len(quantities) == len(discounts)):
        raise BenchmarkError("input sequences must have equal lengths")
    total = 0.0
    for price, quantity, discount in zip(
        prices,
        quantities,
        discounts,
        strict=True,
    ):
        total += price * quantity * (1.0 - discount)
    return total


def numpy_net_revenue(
    prices: np.ndarray,
    quantities: np.ndarray,
    discounts: np.ndarray,
) -> float:
    """Vectorized expression and reduction used in the timed NumPy branch."""

    line_revenue = numpy_line_revenue(prices, quantities, discounts)
    return float(np.sum(line_revenue, dtype=np.float64))


def generate_inputs(
    size: int,
    seed: int,
) -> tuple[np.ndarray, np.ndarray, np.ndarray]:
    """Create reproducible line-item arrays with the declared business domain."""

    _validate_size(size)
    _validate_seed(seed)
    rng = np.random.default_rng(seed)
    prices = rng.uniform(1.0, 500.0, size=size)
    quantities = rng.integers(1, 6, size=size, dtype=np.int64)
    discounts = rng.choice(np.array([0.0, 0.05, 0.1, 0.2]), size=size)
    return prices, quantities, discounts


def compare_results(
    price_list: Sequence[float],
    quantity_list: Sequence[int],
    discount_list: Sequence[float],
    prices: np.ndarray,
    quantities: np.ndarray,
    discounts: np.ndarray,
) -> dict[str, Any]:
    """Check the most detailed output before comparing aggregate totals."""

    loop_rows = np.asarray(
        python_line_revenue(price_list, quantity_list, discount_list),
        dtype=np.float64,
    )
    vector_rows = numpy_line_revenue(prices, quantities, discounts)
    try:
        np.testing.assert_allclose(
            loop_rows,
            vector_rows,
            rtol=ROW_RTOL,
            atol=ROW_ATOL,
        )
    except AssertionError as error:
        raise BenchmarkError("line-level implementations disagree") from error

    loop_total = python_net_revenue(price_list, quantity_list, discount_list)
    vector_total = numpy_net_revenue(prices, quantities, discounts)
    if not np.isclose(
        loop_total,
        vector_total,
        rtol=TOTAL_RTOL,
        atol=TOTAL_ATOL,
    ):
        raise BenchmarkError(
            f"aggregate implementations disagree: loop={loop_total}, vectorized={vector_total}"
        )

    return {
        "line_values_close": True,
        "line_shape": list(vector_rows.shape),
        "max_line_absolute_difference": float(np.max(np.abs(loop_rows - vector_rows), initial=0.0)),
        "totals_close": True,
        "loop_total": loop_total,
        "vectorized_total": vector_total,
        "absolute_total_difference": abs(loop_total - vector_total),
        "tolerances": {
            "line_rtol": ROW_RTOL,
            "line_atol": ROW_ATOL,
            "total_rtol": TOTAL_RTOL,
            "total_atol": TOTAL_ATOL,
        },
    }


def measure_seconds(
    function: Callable[[], float],
    repeat: int,
) -> tuple[float, list[float]]:
    """Warm up once, then return the median and every measured duration."""

    _validate_repeat(repeat)
    function()
    durations: list[float] = []
    for _ in range(repeat):
        started = time.perf_counter_ns()
        function()
        elapsed_ns = max(time.perf_counter_ns() - started, 1)
        durations.append(elapsed_ns / 1_000_000_000)
    return statistics.median(durations), durations


def estimate_array_memory(size: int) -> dict[str, Any]:
    """Estimate known ndarray storage, not allocator or Python-object overhead."""

    _validate_size(size)
    float64_vector_bytes = size * np.dtype(np.float64).itemsize
    int64_vector_bytes = size * np.dtype(np.int64).itemsize
    input_array_bytes = 2 * float64_vector_bytes + int64_vector_bytes
    return {
        "input_array_bytes": input_array_bytes,
        "one_float64_vector_bytes": float64_vector_bytes,
        "straight_expression_peak_temporary_array_count": 3,
        "straight_expression_estimated_peak_temporary_bytes": (3 * float64_vector_bytes),
        "estimate_boundary": (
            "ndarray data buffers only; excludes allocator, Python lists, "

def environment_report() -> dict[str, str]:
    return {
        "python_version": platform.python_version(),
        "numpy_version": np.__version__,
        "system": platform.system(),
        "release": platform.release(),
        "machine": platform.machine(),
    }


def benchmark(size: int, repeat: int, seed: int) -> dict[str, Any]:
    """Run a calculation-only comparison with explicit correctness gates."""

    _validate_size(size)
    _validate_repeat(repeat)
    _validate_seed(seed)

    raw_prices, raw_quantities, raw_discounts = generate_inputs(size, seed)
    prices, quantities, discounts = prepare_inputs(
        raw_prices,
        raw_quantities,
        raw_discounts,
    )
    price_list = prices.tolist()
    quantity_list = quantities.tolist()
    discount_list = discounts.tolist()

    correctness = compare_results(
        price_list,
        quantity_list,
        discount_list,
        prices,
        quantities,
        discounts,
    )

    loop_median, loop_runs = measure_seconds(
        lambda: python_net_revenue(price_list, quantity_list, discount_list),
        repeat,
    )
    vector_median, vector_runs = measure_seconds(
        lambda: numpy_net_revenue(prices, quantities, discounts),
        repeat,
    )

    return {
        "contract_version": 1,
        "experiment": "line-item net revenue: Python loop vs NumPy expression",
        "input_generation": {
            "size": size,
            "seed": seed,
            "generator": "numpy.random.Generator",
            "bit_generator": type(np.random.default_rng(seed).bit_generator).__name__,
        },
        "input_contract": {
            "axis_names": ["line_item"],
            "shape": [size],
            "dtypes": {
                "prices": str(prices.dtype),
                "quantities": str(quantities.dtype),
                "discounts": str(discounts.dtype),
            },
            "domain": {
                "prices": "finite and >= 0",
                "quantities": "integer and >= 0",
                "discounts": "finite and in [0, 1]",
            },
        },
        "correctness": correctness,
        "scope": {
            "name": "calculation_only",
            "included": [
                "line-item formula",
                "reduction to total revenue",
            ],
            "excluded": [
                "input generation",
                "input validation",
                "ndarray-to-list conversion",
                "correctness checks",
                "JSON serialization",
            ],
            "representations": {
                "loop": "prepared Python lists",
                "vectorized": "prepared contiguous NumPy arrays",
            },
        },
        "timing": {
            "timer": "time.perf_counter_ns",
            "warmup_calls_per_implementation": 1,
            "repeat": repeat,
            "summary_statistic": "median",
            "loop_seconds": {
                "median": loop_median,
                "runs": loop_runs,
            },
            "vectorized_seconds": {
                "median": vector_median,
                "runs": vector_runs,
            },
            "speedup_loop_over_vectorized": loop_median / vector_median,
        },
        "memory": estimate_array_memory(size),
        "environment": environment_report(),
        "claim_boundary": (
            "The speedup describes this input, scope, implementation and environment; "
            "it is not a universal NumPy guarantee."
        ),
    }


def main() -> int:
    parser = argparse.ArgumentParser(
        description="Compare a Python loop and a NumPy line-item calculation"
    )
    parser.add_argument("--size", type=int, default=100_000)
    parser.add_argument("--repeat", type=int, default=7)
    parser.add_argument("--seed", type=int, default=42)
    parser.add_argument("--output", type=Path)
    args = parser.parse_args()

    try:
        report = benchmark(args.size, args.repeat, args.seed)
        text = json.dumps(report, ensure_ascii=False, indent=2) + "\n"
        if args.output:
            args.output.write_text(text, encoding="utf-8")
        else:
            print(text, end="")
    except (BenchmarkError, OSError) as error:
        parser.exit(2, f"vectorization-benchmark: {error}\n")
    return 0


if __name__ == "__main__":
    raise SystemExit(main())
