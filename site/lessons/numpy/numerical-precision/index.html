Ответьте до чтения: результат поможет заметить исходные предположения.

После векторизации рассчитанная выручка отличается от результата прозрачного цикла на несколько последних знаков. Что нужно решить до выбора функции сравнения?

Сначала формулируют контракт: точное или приближенное совпадение, единицы результата и допустимую предметную ошибку. Только после этого выбирают инструмент и параметры.

02

Какие результаты обычно следует сравнивать точно, а не с floating-point допуском?

Идентификаторы, счетчики и маски по смыслу дискретны: приближенное совпадение может скрыть настоящую ошибку.

Численная точность и сравнение результатов

Численный результат надежен не тогда, когда тест «зеленый», а когда заранее определено, что именно должно совпасть, какая ошибка допустима в единицах задачи и какие значения считаются поломкой.

Тип: Case

Треки: Core

Пререквизиты: 02-numpy/03-dtypes, 02-numpy/05-aggregations, 02-numpy/06-broadcasting, 02-numpy/08-vectorization

Время: ~60 минут

Результат: формулирует численный контракт результата: выбирает точное или приближенное сравнение, обосновывает rtol и atol из единиц и масштаба, контролирует shape, dtype, диапазон и конечность и диагностирует переполнение, underflow и ошибку накопления.

Цели обучения

После урока вы сможете:

  • отличить точное значение от результата приближенных floating-point вычислений;
  • выбрать точное или приближенное сравнение по смыслу результата;
  • объяснить формулу atol + rtol * abs(expected) и роль каждой части;
  • вывести допуски из единиц измерения и допустимой предметной ошибки;
  • проверить shape до сравнения значений и решить, входит ли dtype в контракт;
  • задать явную политику для NaN, inf и деления на ноль;
  • обнаружить integer overflow до выполнения операции в узком типе;
  • распознать floating overflow, underflow и потерю точности при накоплении;
  • получить короткий диагностический отчет вместо одного результата True или False.

Проблема

В предыдущем уроке мы заменили Python-цикл векторным расчетом выручки. Сначала сравнили выручку каждой позиции, затем итоговую сумму. Представим, что новая реализация вернула:

text
expected: [200.0, 60.0, 180.0]
actual:   [200.0, 59.99999999999999, 180.0]

Точное сравнение объявит вторую позицию ошибочной. Аналитик заменяет его на np.allclose(actual, expected) и тест проходит. Но это еще не доказательство корректности:

  • стандартный допуск мог оказаться слишком широким для этой задачи;
  • массивы разных форм могли неявно совместиться через broadcasting;
  • одинаковые inf могли появиться из одной и той же ошибки деления;
  • итоговая сумма могла совпасть при неправильных значениях отдельных строк;
  • промежуточное целое произведение могло переполниться до преобразования во float64.

Есть и противоположная ошибка: требовать побитового совпадения от двух корректных floating-point реализаций, которые выполняют операции в разном порядке. Тогда тест проверяет случайную деталь вычисления и ломается без изменения бизнес-результата.

Нам нужен не «очень строгий» и не «очень мягкий» тест. Нужен численный контракт: описание ожидаемой структуры, представления, допустимой ошибки и поведения на опасных значениях.

Концепция

Численный контракт состоит из нескольких слоев

Проверяйте результат по порядку. Каждый следующий слой имеет смысл только после предыдущего.

Слой Вопрос Пример нарушения
Смысл Одинаковую ли величину и в одинаковых ли единицах мы сравниваем? рубли сравнили с копейками
Shape Совпадают ли предметные оси и форма? (orders,) сравнили с (1, orders)
Dtype Важно ли именно представление результата? счетчик неожиданно стал float64
Конечность Допустимы ли NaN и бесконечности? деление на ноль породило inf
Вид сравнения Результат должен совпасть точно или приближенно? вычисленный float сравнили через ==
Допуск Какая ошибка допустима в единицах задачи? для копеек разрешили отклонение в рубль
Диапазон операции Могла ли ошибка возникнуть до сравнения? int8 переполнился при сложении
Диагностика Где и почему контракт нарушен? тест вернул только False

Этот порядок связывает всю фазу NumPy. Shape пришел из уроков об осях и broadcasting, dtype и диапазон — из урока о типах, накопление — из агрегатов, а необходимость сравнить детальный результат до итога — из векторизации.

Сначала решите: точное или приближенное сравнение

Не всякое значение в NumPy нужно сравнивать с допуском.

Результат Ожидаемый контракт Подходящий инструмент
идентификаторы, счетчики, индексы точное совпадение np.array_equal или обычный assert
булева маска точное совпадение формы и каждого значения np.testing.assert_array_equal
целочисленная сумма без переполнения точное совпадение np.testing.assert_array_equal
результат вычислений с float предметно допустимое приближение np.testing.assert_allclose с явными допусками
деньги с требованием точности до копейки точный контракт в минимальных денежных единицах целые копейки; иногда Decimal вне NumPy

Простой вопрос для выбора:

Может ли корректная реализация получить соседнее представимое число из-за округления или другого порядка операций?

Если нет, требуется точное сравнение. Если да, нужен обоснованный численный допуск. Приближенное сравнение не должно маскировать ошибки там, где результат по смыслу точный.

Почему float хранит приближение

Число с плавающей точкой хранит конечное число двоичных разрядов. Большинство привычных десятичных дробей, включая 0.1, нельзя записать конечной двоичной дробью. В памяти хранится ближайшее доступное значение:

python
0.1 + 0.2
# 0.30000000000000004

Это не случайность и не ошибка Python. Между соседними представимыми числами есть расстояние — шаг численной сетки. Он зависит от:

  • dtype: сетка float32 грубее сетки float64;
  • масштаба: около больших чисел соседние значения дальше друг от друга;
  • последовательности операций: каждое округление может повлиять на следующее;
  • способа накопления: сумма многих чисел может зависеть от порядка сложения.

np.finfo(dtype).eps показывает расстояние от 1.0 до следующего представимого числа данного типа. Это полезная характеристика формата, но не универсальный допуск для аналитического теста. Результат в миллиардах, доля около нуля и длинная сумма имеют разные масштабы и разные предметные требования.

Как устроено приближенное сравнение

NumPy использует условие:

text
abs(actual - expected) <= atol + rtol * abs(expected)

Где:

  • actual — проверяемый результат;
  • expected — эталон, относительно которого задан масштаб;
  • atol — абсолютный допуск в тех же единицах, что и результат;
  • rtol — безразмерная допустимая доля от модуля эталона.

Формула не полностью симметрична: в относительной части используется именно expected. Поэтому заранее назначьте эталон и не меняйте аргументы местами случайно.

Почему около нуля нужен atol

Если expected = 0, относительная часть равна нулю:

text
rtol * abs(0) = 0

Остается только atol. Например, для остаточного дисбаланса в рублях можно решить, что значения до одной копейки считаются практически нулевыми:

text
abs(actual - 0) <= 0.01

Это предметное решение, а не свойство NumPy.

Почему rtol может быть опасен на больших значениях

Пусть ожидаемая выручка равна 1_000_000_000 рублей. Даже небольшой на вид rtol=1e-9 разрешает примерно один рубль относительной ошибки:

text
1e-9 * 1_000_000_000 = 1

Если контракт требует расхождения не более одной копейки, уместнее:

python
rtol = 0.0
atol = 0.01

Для другой задачи — например, приближенного результата итерационного алгоритма — может быть оправдана и относительная часть. Допуск нельзя переносить между величинами только потому, что у них одинаковый dtype.

Как выбрать rtol и atol

Используйте короткую последовательность вопросов.

  1. Назовите единицы. Рубли, секунды, доля, количество пользователей, байты?
  2. Назовите типичный диапазон. Значения около нуля, сотен или миллиардов?
  3. Определите допустимую абсолютную ошибку. Что еще не меняет решение в этих единицах?
  4. Решите, должна ли ошибка расти вместе с масштабом. Если да, добавьте rtol.
  5. Проверьте края. Что формула разрешает при нуле, типичном и максимальном значении?
  6. Запишите обоснование рядом с тестом. Не оставляйте только два числа без смысла.

Примеры рассуждения:

Величина Возможный контракт Почему
число заказов точное равенство целый счетчик не является приближением
выручка в рублях, рассчитанная из копеек rtol=0, atol=0.01 ошибка ограничена одной копейкой независимо от масштаба
доля конверсии например, atol=1e-10, малый rtol есть значения около нуля и относительное округление

Числа в таблице — примеры формы решения, а не готовые стандарты для всех проектов. Стандартные значения np.allclose удобны для интерактивной проверки, но не заменяют обоснование production-контракта. Округление результата перед сравнением тоже не заменяет допуск: оно меняет данные и создает отдельные пограничные эффекты.

Shape проверяется до значений

Broadcasting полезен в вычислениях, но опасен в тесте, если должен совпадать каждый элемент одной и той же предметной оси:

python
actual = np.array([1.0, 2.0])       # shape (2,)
expected = np.array([[1.0, 2.0]])  # shape (1, 2)

Эти массивы совместимы для broadcasting, но описывают разные контракты формы. Поэтому сначала требуется:

python
assert actual.shape == expected.shape

И только затем — сравнение значений. Совместимость вычисления не равна совпадению предметной структуры.

Dtype может быть метаданными или обязательной частью контракта

Массивы int64([1, 2]) и float64([1.0, 2.0]) содержат численно равные значения. Иногда этого достаточно. Но если результат должен оставаться счетчиком, экономить память или сохранять определенный диапазон, неожиданный dtype — самостоятельная ошибка.

Разделяйте два вопроса:

  1. совпали ли значения;
  2. совпало ли требуемое представление.

Не делайте проверку dtype обязательной по привычке. Включайте ее, когда тип является частью интерфейса или влияет на корректность следующей операции.

NaN и inf требуют разных политик

NaN обычно означает отсутствующий или неопределенный результат. По умолчанию даже два NaN не считаются равными:

python
np.nan == np.nan
# False

equal_nan=True уместен только при явном контракте: например, обе реализации должны оставить пропуск в одних и тех же позициях. Он не означает, что любые пропуски допустимы.

inf и -inf чаще сигнализируют о переполнении или делении на ноль. Две реализации могут вернуть одинаковую бесконечность из-за одной общей ошибки. Поэтому артефакт урока считает даже совпадающие бесконечности нарушением контракта конечного аналитического результата.

Если в конкретном математическом объекте бесконечность допустима, это уже другой контракт, который следует описать и проверить отдельно.

Integer overflow нужно предотвращать до операции

Python integers могут расширяться, а NumPy integers имеют фиксированный диапазон:

python
np.iinfo(np.int8)
# min=-128, max=127

Сложение 120 + 20 не помещается в int8. Если выполнить его в узком dtype, неверное значение уже появится. Последующее преобразование в int64 не восстановит потерянный результат.

Безопасный порядок:

  1. оценить будущий диапазон;
  2. расширить dtype до операции или вычислить контрольный результат как Python integer;
  3. проверить результат по np.iinfo(target_dtype);
  4. только затем преобразовать в целевой dtype.

np.errstate управляет floating-point категориями ошибок и не является универсальной защитой от integer overflow. Для целых чисел нужен явный range-контракт.

У float есть верхняя и нижняя границы

np.finfo(dtype) помогает увидеть четыре разные характеристики:

Характеристика Смысл
eps шаг сетки около 1.0
max максимальное конечное положительное значение
tiny минимальное положительное нормальное значение
smallest_subnormal минимальное положительное представимое значение

Между tiny и smallest_subnormal находятся subnormal-значения. Они позволяют постепенно приближаться к нулю, но имеют меньше значащих разрядов. Еще меньший результат округляется до нуля — это underflow.

python
info = np.finfo(np.float32)

with np.errstate(over="raise"):
    too_large = np.float32(info.max) * np.float32(2)

with np.errstate(under="raise"):
    too_small = np.float32(info.smallest_subnormal) * np.float32(0.5)

Overflow обычно заметнее: конечный результат превращается в inf. Underflow опаснее тем, что очень малое ненулевое значение может стать обычным нулем. Если масштаб является частью смысла задачи, ноль нельзя автоматически интерпретировать как «события не было».

Для некоторых алгоритмов решение лежит не в более широком допуске, а в изменении вычисления: масштабировании данных, другой формуле или работе в логарифмах. Подробный численный анализ выходит за рамки урока; здесь важно распознать границу представления и не принять ее за факт предметной области.

Деление на ноль — это решение, а не предупреждение

Для отношения numerator / denominator заранее выберите политику:

  • завершить расчет ошибкой при невалидной позиции;
  • оставить NaN, если отношение там по смыслу не определено;
  • записать другое значение только при предметном обосновании.

Ноль не является автоматически нейтральной заменой: коэффициент 0 означает определенное значение, а не отсутствие возможности посчитать коэффициент.

Рабочий шаблон использует where и заранее подготовленный out:

python
left, right = np.broadcast_arrays(numerator, denominator)
valid = np.isfinite(left) & np.isfinite(right) & (right != 0)
result = np.full(left.shape, np.nan, dtype=np.float64)

with np.errstate(divide="raise", invalid="raise", over="raise"):
    np.divide(left, right, out=result, where=valid)

После вычисления проверьте, что расположение не конечных значений соответствует выбранной политике. np.errstate делает ошибку наблюдаемой, но не решает за аналитика, что означает проблемная строка.

Ошибка может накапливаться

Рассмотрим сумму:

python
values = np.array([1e8, 1.0, -1e8], dtype=np.float32)

Точное математическое значение равно 1. Но число 1 слишком мало относительно 1e8 для сетки float32: при промежуточном сложении вклад может потеряться. Это пример потери значимости и чувствительности к порядку операций.

Более широкий аккумулятор часто помогает:

python
sum_float32 = np.sum(values, dtype=np.float32)
sum_float64 = np.sum(values, dtype=np.float64)

Но есть важная граница: float64-аккумулятор получает уже сохраненные значения float32. Он может уменьшить новые ошибки сложения, но не восстанавливает информацию, потерянную при первоначальном преобразовании входов.

Еще один источник проблем — вычитание двух близких больших чисел. Поэтому для дисперсии лучше использовать np.var, а не наивную формулу:

python
np.mean(values * values) - np.mean(values) ** 2

Специализированная функция обычно яснее передает намерение и может использовать более устойчивый алгоритм. Однако ее результат все равно нужно проверять по предметному контракту.

Соберите это

Сначала соберем минимальную прозрачную проверку без NumPy API сравнения. Ее задача — показать, откуда берется решение для одного числа.

python
def is_close(actual, expected, *, rtol, atol):
    absolute_error = abs(actual - expected)
    allowed_error = atol + rtol * abs(expected)
    return {
        "passed": absolute_error <= allowed_error,
        "absolute_error": absolute_error,
        "allowed_error": allowed_error,
    }

Проверим обычное floating-point округление:

python
report = is_close(
    0.1 + 0.2,
    0.3,
    rtol=1e-9,
    atol=1e-12,
)

assert report["passed"]
assert report["absolute_error"] <= report["allowed_error"]

Проверим значение около нуля:

python
strict = is_close(1e-9, 0.0, rtol=1e-5, atol=0.0)
with_absolute_tolerance = is_close(1e-9, 0.0, rtol=1e-5, atol=1e-8)

assert not strict["passed"]
assert with_absolute_tolerance["passed"]

В отчете важны оба числа. Факт «не прошло» становится объяснимым: фактическая ошибка оказалась больше разрешенной.

Для точного результата допуск не нужен:

python
actual_counts = np.array([12, 7, 0])
expected_counts = np.array([12, 7, 0])

np.testing.assert_array_equal(actual_counts, expected_counts)

Перед integer-операцией можно сначала посчитать в расширяемых Python integers:

python
target = np.dtype("int8")
info = np.iinfo(target)
result = int(120) + int(20)

if not info.min <= result <= info.max:
    raise OverflowError(f"{result} is outside {target} range")

А для деления сначала запишите политику человеческим языком:

text

Использовать eps как универсальный допуск

eps относится к шагу сетки около единицы. Он ничего не говорит о допустимой ошибке выручки в миллиардах или вероятности около нуля.

Он уменьшает ошибку последующих сложений, но не восстанавливает разряды, потерянные при хранении каждого элемента.

01

Ожидаемая выручка равна 1 000 000 000 рублей, а контракт разрешает отклонение не более одной копейки. Какая проверка лучше выражает это требование?

02

Эталонный остаточный дисбаланс равен 0, а значение 1e-9 по правилам задачи считается практически нулевым. Какую часть допуска нужно определить в первую очередь?

В формуле NumPy rtol умножается на abs(expected). При нулевом эталоне допустимую ненулевую ошибку задает atol.

03

actual имеет shape (2,), expected — (1, 2), а значения совпадают после broadcasting. Как должен поступить quality gate для построчного результата?

Broadcasting подтверждает техническую совместимость операции, но не совпадение предметных осей. Shape-контракт проверяют первым.

04

Две реализации коэффициента вернули inf в одной позиции, и np.isclose считает значения совпавшими. Что делать, если контракт требует конечный аналитический результат?

Совпадающие бесконечности могут быть общей ошибкой двух реализаций. При finite-result contract любая бесконечность является нарушением.

05

Когда допустимо включить equal_nan=True при сравнении двух результатов?

equal_nan=True выражает конкретную missingness policy: обе реализации должны оставить допустимый пропуск в одной позиции. Это не общее разрешение на NaN.

06

Нужно сложить 120 и 20 и сохранить результат в int8. Какой порядок предотвращает тихое переполнение?

Проверка диапазона и расширение должны произойти до узкой операции. После переполнения преобразование в широкий dtype уже не восстановит правильное значение 140.

07

Что гарантирует суммирование массива float32 с accumulator dtype=float64?

Более широкий аккумулятор работает с уже сохраненными float32. Он может точнее складывать их, но не знает потерянных исходных разрядов.

08

Очень малое положительное значение после умножения в float32 стало нулем. Отличие от нуля важно для вывода. Какой вывод корректен?

Underflow может округлить ненулевой результат до subnormal или нуля. Если это меняет смысл, требуется пересмотреть представление или алгоритм, а не объявлять ноль фактом.

from __future__ import annotations

import argparse
import json
import math
from pathlib import Path
from typing import Any

import numpy as np

DEFAULT_MAX_MISMATCHES = 5
MAX_MISMATCHES_LIMIT = 100


class NumericalQualityError(ValueError):
    """Raised when a numerical result violates an explicit quality contract."""


def _validate_tolerance(name: str, value: object) -> float:
    if isinstance(value, (bool, np.bool_)) or not isinstance(
        value,
        (int, float, np.integer, np.floating),
    ):
        raise NumericalQualityError(f"{name} must be a real number")
    numeric_value = float(value)
    if not math.isfinite(numeric_value) or numeric_value < 0.0:
        raise NumericalQualityError(f"{name} must be finite and non-negative")
    return numeric_value


def _validate_max_mismatches(value: int) -> None:
    if isinstance(value, bool) or not isinstance(value, (int, np.integer)):
        raise NumericalQualityError("max_mismatches must be an integer")
    if value <= 0 or value > MAX_MISMATCHES_LIMIT:
        raise NumericalQualityError(f"max_mismatches must be between 1 and {MAX_MISMATCHES_LIMIT}")


def _as_real_numeric_array(values: object, *, name: str) -> np.ndarray:
    array = np.asarray(values)
    if array.size == 0:
        raise NumericalQualityError(f"{name} must not be empty")
    if array.dtype.kind not in "iuf":
        raise NumericalQualityError(f"{name} must contain real numeric values")
    return array


def _json_scalar(value: object) -> int | float | str:
    scalar = np.asarray(value).item()
    if isinstance(scalar, (int, np.integer)):
        return int(scalar)
    numeric_value = float(scalar)
    if math.isnan(numeric_value):
        return "nan"
    if math.isinf(numeric_value):
        return "inf" if numeric_value > 0 else "-inf"
    return numeric_value


def tolerance_report(
    actual: object,
    expected: object,
    *,
    rtol: float,
    atol: float,
    equal_nan: bool = False,
    require_same_dtype: bool = False,
    max_mismatches: int = DEFAULT_MAX_MISMATCHES,
) -> dict[str, Any]:
    """Compare equally shaped real arrays under an explicit numerical contract."""

    checked_rtol = _validate_tolerance("rtol", rtol)
    checked_atol = _validate_tolerance("atol", atol)
    _validate_max_mismatches(max_mismatches)
    if not isinstance(equal_nan, (bool, np.bool_)):
        raise NumericalQualityError("equal_nan must be boolean")
    if not isinstance(require_same_dtype, (bool, np.bool_)):
        raise NumericalQualityError("require_same_dtype must be boolean")

    actual_array = _as_real_numeric_array(actual, name="actual")
    expected_array = _as_real_numeric_array(expected, name="expected")
    if actual_array.shape != expected_array.shape:
        raise NumericalQualityError(
            f"shape mismatch: actual {actual_array.shape}, expected {expected_array.shape}"
        )

    actual_float = actual_array.astype(np.float64, copy=False)
    expected_float = expected_array.astype(np.float64, copy=False)
    actual_nan = np.isnan(actual_float)
    expected_nan = np.isnan(expected_float)
    actual_inf = np.isinf(actual_float)
    expected_inf = np.isinf(expected_float)
    infinity_mask = actual_inf | expected_inf
    finite_pair = np.isfinite(actual_float) & np.isfinite(expected_float)

    integer_pair = np.issubdtype(actual_array.dtype, np.integer) and np.issubdtype(
        expected_array.dtype,
        np.integer,
    )
    with np.errstate(invalid="ignore", over="ignore"):
        allowed_error = checked_atol + checked_rtol * np.abs(expected_float)
        if integer_pair:
            # np.isclose converts large integers to floating point and can make adjacent
            # int64 values above 2**53 look equal. Python integers preserve the exact
            # difference while still allowing an explicitly requested tolerance.
            exact_difference = np.abs(actual_array.astype(object) - expected_array.astype(object))
            absolute_error = np.asarray(exact_difference, dtype=np.float64)
            close = np.fromiter(
                (
                    difference <= allowed
                    for difference, allowed in zip(
                        exact_difference.flat,
                        allowed_error.flat,
                        strict=True,
                    )
                ),
                dtype=bool,
                count=actual_array.size,
            ).reshape(actual_array.shape)
        else:
            close = np.isclose(
                actual_array,
                expected_array,
                rtol=checked_rtol,
                atol=checked_atol,
                equal_nan=bool(equal_nan),
            )
            absolute_error = np.full(actual_array.shape, np.nan, dtype=np.float64)
            difference = np.empty(actual_array.shape, dtype=np.float64)
            np.subtract(
                actual_float,
                expected_float,
                out=difference,
                where=finite_pair,
            )
            np.abs(difference, out=absolute_error, where=finite_pair)

    # A matching infinity is still a failed finite-result contract. Matching NaN values
    # may pass only after the caller explicitly enables equal_nan.
    close = np.asarray(close, dtype=bool)
    close[infinity_mask] = False

    mismatch_mask = ~close
    mismatch_count = int(np.count_nonzero(mismatch_mask))
    mismatch_indices = np.argwhere(mismatch_mask)
    mismatch_examples: list[dict[str, Any]] = []
    for raw_index in mismatch_indices[:max_mismatches]:
        index = tuple(int(component) for component in raw_index)
        actual_value = actual_array[index]
        expected_value = expected_array[index]
        if np.isinf(actual_float[index]) or np.isinf(expected_float[index]):
            reason = "infinity_not_allowed"
        elif np.isnan(actual_float[index]) or np.isnan(expected_float[index]):
            reason = "nan_positions_do_not_match_or_equal_nan_is_false"
        else:
            reason = "outside_tolerance"
        mismatch_examples.append(
            {
                "index": list(index),
                "actual": _json_scalar(actual_value),
                "expected": _json_scalar(expected_value),
                "absolute_error": (
                    float(absolute_error[index]) if np.isfinite(absolute_error[index]) else None
                ),
                "allowed_error": (
                    float(allowed_error[index]) if np.isfinite(allowed_error[index]) else None
                ),
                "reason": reason,
            }
        )

    finite_absolute_errors = absolute_error[finite_pair]
    finite_allowed_errors = allowed_error[finite_pair]
    dtype_matches = actual_array.dtype == expected_array.dtype
    value_comparison_passed = mismatch_count == 0
    dtype_contract_passed = bool(dtype_matches or not require_same_dtype)
    passed = value_comparison_passed and dtype_contract_passed

    return {
        "status": "passed" if passed else "failed",
        "passed": passed,
        "shape": list(actual_array.shape),
        "element_count": int(actual_array.size),
        "actual_dtype": str(actual_array.dtype),
        "expected_dtype": str(expected_array.dtype),
        "dtype_matches": bool(dtype_matches),
        "require_same_dtype": bool(require_same_dtype),
        "dtype_contract_passed": dtype_contract_passed,
        "comparison": {
            "formula": "abs(actual - expected) <= atol + rtol * abs(expected)",
            "reference": "expected",
            "rtol": checked_rtol,
            "atol": checked_atol,
            "equal_nan": bool(equal_nan),
            "infinity_policy": "always fail",
        },
        "value_comparison_passed": value_comparison_passed,
        "mismatch_count": mismatch_count,
        "mismatch_examples": mismatch_examples,
        "mismatch_examples_truncated": mismatch_count > len(mismatch_examples),
        "max_absolute_error_on_finite_pairs": (
            float(finite_absolute_errors.max()) if finite_absolute_errors.size else None
        ),
        "max_allowed_error_on_finite_pairs": (
            float(finite_allowed_errors.max()) if finite_allowed_errors.size else None
        ),
        "non_finite": {
            "actual_nan_count": int(np.count_nonzero(actual_nan)),
            "expected_nan_count": int(np.count_nonzero(expected_nan)),
            "actual_infinity_count": int(np.count_nonzero(actual_inf)),
            "expected_infinity_count": int(np.count_nonzero(expected_inf)),
        },
    }


def assert_numerically_close(
    actual: object,
    expected: object,
    *,
    rtol: float,
    atol: float,
    equal_nan: bool = False,
    require_same_dtype: bool = False,
) -> None:
    report = tolerance_report(
        actual,
        expected,
        rtol=rtol,
        atol=atol,
        equal_nan=equal_nan,
        require_same_dtype=require_same_dtype,
    )
    if report["passed"]:
        return

    reasons: list[str] = []
    if not report["dtype_contract_passed"]:
        reasons.append(
            f"dtype mismatch: actual {report['actual_dtype']}, expected {report['expected_dtype']}"
        )
    if report["mismatch_examples"]:
        first = report["mismatch_examples"][0]
        reasons.append(
            f"first mismatch at index {first['index']}: "
            f"actual={first['actual']}, expected={first['expected']}, "
            f"reason={first['reason']}"
        )
    raise NumericalQualityError(
        f"numerical contract failed with {report['mismatch_count']} value mismatches; "
        + "; ".join(reasons)
    )


def safe_divide(
    numerator: object,
    denominator: object,
    *,
    fill_value: float | None = None,
) -> np.ndarray:
    """Divide real arrays under an explicit invalid-position policy."""

    left = _as_real_numeric_array(numerator, name="numerator").astype(
        np.float64,
        copy=False,
    )
    right = _as_real_numeric_array(denominator, name="denominator").astype(
        np.float64,
        copy=False,
    )
    left, right = np.broadcast_arrays(left, right)

    checked_fill_value: float | None = None
    if fill_value is not None:
        checked_fill_value = _validate_fill_value(fill_value)

    valid = np.isfinite(left) & np.isfinite(right) & (right != 0.0)
    invalid_count = int(valid.size - np.count_nonzero(valid))
    if fill_value is None and invalid_count:
        raise NumericalQualityError(
            f"division has {invalid_count} invalid positions; choose error or fill policy"
        )

    result = np.full(
        left.shape,
        checked_fill_value if fill_value is not None else np.nan,
        dtype=np.float64,
    )
    with np.errstate(divide="raise", invalid="raise", over="raise", under="raise"):
        try:
            np.divide(left, right, out=result, where=valid)
        except FloatingPointError as error:
            raise NumericalQualityError(f"floating-point division failed: {error}") from error
    return result


def _validate_fill_value(value: object) -> float:
    if isinstance(value, (bool, np.bool_)) or not isinstance(
        value,
        (int, float, np.integer, np.floating),
    ):
        raise NumericalQualityError("fill_value must be a real scalar")
    numeric_value = float(value)
    if math.isinf(numeric_value):
        raise NumericalQualityError("fill_value must be finite or NaN, not infinity")
    return numeric_value


def _as_integer_object_array(values: object, *, name: str) -> np.ndarray:
    array = np.asarray(values, dtype=object)
    if array.size == 0:
        raise NumericalQualityError(f"{name} must not be empty")
    for value in array.ravel():
        if isinstance(value, (bool, np.bool_)) or not isinstance(
            value,
            (int, np.integer),
        ):
            raise NumericalQualityError(f"{name} must contain integer values")
    return array


def checked_integer_add(left: object, right: object, *, dtype: str) -> np.ndarray:
    """Add with Python integers, then verify the requested fixed-width range."""

    try:
        requested = np.dtype(dtype)
    except TypeError as error:
        raise NumericalQualityError(f"unknown dtype: {dtype}") from error
    if not np.issubdtype(requested, np.integer):
        raise NumericalQualityError("checked_integer_add requires an integer dtype")

    left_array = _as_integer_object_array(left, name="left")
    right_array = _as_integer_object_array(right, name="right")
    left_array, right_array = np.broadcast_arrays(left_array, right_array)
    result_object = left_array + right_array
    info = np.iinfo(requested)
    for value in result_object.ravel():
        if value < info.min or value > info.max:
            raise NumericalQualityError(
                f"result {value} is outside {requested.name} range [{info.min}, {info.max}]"
            )
    return np.asarray(result_object.tolist(), dtype=requested)


def floating_range_report(dtype: str) -> dict[str, float | int | str]:
    try:
        requested = np.dtype(dtype)
    except TypeError as error:
        raise NumericalQualityError(f"unknown dtype: {dtype}") from error
    if requested.name not in {"float16", "float32", "float64"}:
        raise NumericalQualityError("floating_range_report supports float16, float32 and float64")
    info = np.finfo(requested)
    return {
        "dtype": requested.name,
        "bits": info.bits,
        "eps_near_one": float(info.eps),
        "max_finite": float(info.max),
        "smallest_positive_normal": float(info.tiny),
        "smallest_positive_subnormal": float(info.smallest_subnormal),
    }


def summation_report(
    values: object,
    *,
    storage_dtype: str = "float32",
    accumulator_dtype: str = "float64",
) -> dict[str, Any]:
    raw = _as_real_numeric_array(values, name="summation values")
    try:
        storage = np.dtype(storage_dtype)
        accumulator = np.dtype(accumulator_dtype)
    except TypeError as error:
        raise NumericalQualityError("unknown summation dtype") from error
    if not np.issubdtype(storage, np.floating) or not np.issubdtype(
        accumulator,
        np.floating,
    ):
        raise NumericalQualityError("summation dtypes must be floating")
    if accumulator.itemsize < storage.itemsize:
        raise NumericalQualityError("accumulator dtype must be at least as wide as storage dtype")

    with np.errstate(over="ignore", invalid="ignore"):
        stored = raw.astype(storage)
    if not np.isfinite(stored).all():
        raise NumericalQualityError("summation values must remain finite after storage conversion")

    reference = math.fsum(float(value) for value in stored.ravel())
    storage_sum = float(np.sum(stored, dtype=storage))
    accumulator_sum = float(np.sum(stored, dtype=accumulator))
    return {
        "input_shape": list(stored.shape),
        "element_count": int(stored.size),
        "storage_dtype": storage.name,
        "accumulator_dtype": accumulator.name,
        "python_fsum_reference_over_stored_values": reference,
        "storage_accumulator_sum": storage_sum,
        "chosen_accumulator_sum": accumulator_sum,
        "storage_accumulator_absolute_error": abs(storage_sum - reference),
        "chosen_accumulator_absolute_error": abs(accumulator_sum - reference),
        "reference_boundary": (
            "math.fsum is applied after conversion to storage_dtype; information "
            "lost during storage conversion cannot be restored"
        ),
    }


def parse_json(raw: str, *, name: str) -> Any:
    try:
        return json.loads(raw)
    except json.JSONDecodeError as error:
        raise NumericalQualityError(f"{name} must be valid JSON") from error


def main() -> int:
    parser = argparse.ArgumentParser(description="Run a NumPy numerical quality gate")
    parser.add_argument("--actual", default="[0.30000000000000004, 10.0]")
    parser.add_argument("--expected", default="[0.3, 10.0]")
    parser.add_argument("--rtol", type=float, default=1e-9)
    parser.add_argument("--atol", type=float, default=1e-12)
    parser.add_argument("--equal-nan", action="store_true")
    parser.add_argument("--require-same-dtype", action="store_true")
    parser.add_argument("--max-mismatches", type=int, default=DEFAULT_MAX_MISMATCHES)
    parser.add_argument("--output", type=Path)
    args = parser.parse_args()

    try:
        comparison = tolerance_report(
            parse_json(args.actual, name="actual"),
            parse_json(args.expected, name="expected"),
            rtol=args.rtol,
            atol=args.atol,
            equal_nan=args.equal_nan,
            require_same_dtype=args.require_same_dtype,
            max_mismatches=args.max_mismatches,
        )
        report = {
            "contract_version": 1,
            "gate": "numpy numerical comparison",
            "comparison": comparison,
            "decision": {
                "status": comparison["status"],
                "exit_code": 0 if comparison["passed"] else 1,
            },
        }
        text = json.dumps(report, ensure_ascii=False, indent=2, allow_nan=False) + "\n"
        if args.output:
            args.output.write_text(text, encoding="utf-8")
        else:
            print(text, end="")
    except (NumericalQualityError, OSError) as error:
        parser.exit(2, f"numerical-quality: {error}\n")
    return 0 if comparison["passed"] else 1


if __name__ == "__main__":
    raise SystemExit(main())
