Вопрос раньше графика

Сначала зафиксируйте, какое решение должен поддержать анализ, и только затем выбирайте визуальное представление.

Тип: Learn
Треки: Core
Пререквизиты: 05/11
Время: ~75 минут
Результат: преобразует рабочий вопрос в контракт визуального доказательства с единицей анализа, сравнением, метрикой, ожидаемым паттерном и критерием остановки.

  • Связывать визуальный вопрос с конкретным решением, а не с абстрактным интересом к данным.
  • Фиксировать population, grain, метрику и знаменатель до построения графика.
  • Выбирать семейство представлений по аналитической операции: trend, comparison, distribution, relationship или composition.
  • Отделять ожидаемый паттерн от результата и причинное объяснение от наблюдаемой связи.
  • Задавать stop rule, который запрещает интерпретацию неподготовленных данных.

Проблема

После мартовского релиза команда увидела снижение семидневной активации и попросила:

Построй график активации и найди, почему она упала.

Формулировка кажется понятной, но не задает аналитический контракт:

  • какую продуктовую или техническую развилку должен поддержать результат;
  • кто входит в population;
  • что является одной единицей наблюдения;
  • как именно рассчитывается активация и кто входит в знаменатель;
  • какие периоды или сегменты сравниваются;
  • какие данные нельзя интерпретировать;
  • где заканчивается наблюдение и начинается причинная гипотеза.

Если сразу открыть plotting-библиотеку, эти решения все равно придется принять. Только они окажутся спрятаны в фильтрах, groupby, настройках осей и выбранном типе графика. Две правдоподобные картинки могут отвечать на разные вопросы:

  • доля активации по календарным дням регистрации;
  • доля по недельным когортам;
  • число активированных пользователей;
  • доля среди всех регистраций, включая неполные окна;
  • доля только среди пользователей с полными семью днями наблюдения.

График не исправляет неоднозначный вопрос. Он делает одно из возможных толкований визуально убедительным.

Концепция

Visual question brief - это машинно-проверяемый договор между рабочим вопросом, данными, сравнением и будущим представлением.

text
решение
  -> вопрос
  -> population и grain
  -> метрика и знаменатель
  -> сравнение
  -> ожидаемый паттерн
  -> stop rule
  -> chart plan
  -> проверяемое наблюдение

Решение раньше любопытства

Вопрос «что происходит с активацией?» допускает бесконечное исследование. Решение ограничивает поиск:

text
Определить сегмент для следующей технической и продуктовой диагностики.

Этот контракт не обещает доказать причину релиза. Он требует найти проверяемое место для следующего шага: платформу, версию приложения или канал.

Population и grain

Population отвечает, о ком разрешено делать вывод. Grain отвечает, что означает одна строка или одна независимая единица сравнения.

Для user_journeys:

text
Population: новые пользователи подписочного сервиса.
Grain: один пользователь с полным семидневным окном после регистрации.

Исходная таблица намеренно нарушает этот идеальный вход:

  • один user_id доставлен повторно;
  • у последних регистраций observed_days < 7;
  • outcome-поля неполных окон отсутствуют.

Поэтому grain в вопросе одновременно становится требованием к будущему аудиту данных.

Метрика является формулой

Название activation_7d недостаточно. Для rate-метрики нужны событие, population policy и знаменатель:

text
Доля пользователей с activated_7d=true
среди observed_days=7 после удаления дубликатов.

Если включить неполные окна в знаменатель как неактивированных пользователей, последние когорты будут выглядеть хуже даже без реального изменения поведения.

Сравнение является частью вопроса

Фраза «после релиза» требует явных уровней:

  • baseline: когорты до 2026-03-02;
  • comparison: когорты начиная с 2026-03-02;
  • time grain: cohort_week;
  • сегменты для drill-down: platform и acquisition_channel.

Без baseline график показывает последовательность чисел, но не определяет контрольную точку решения.

Тип вопроса раньше типа mark

Brief использует пять аналитических операций:

Тип вопроса Что сравнивается Базовый план
trend значение по упорядоченному времени линия с видимыми наблюдениями
comparison категории на общей шкале сортированный dot plot
distribution форма и хвосты числовой величины ECDF и документированный histogram
relationship совместное изменение двух величин scatter с контролем overplotting
composition доли частей общего целого выровненные доли вместе с counts

Это не автоматический выбор единственно правильного графика. Это rubric: набор предпочтительных представлений, необходимого контекста и опасных приемов для проверки.

Expected pattern не является ответом

Ожидаемый паттерн делает исходное рассуждение наблюдаемым:

text
Общий спад может сочетать изменение channel mix
и отдельное ухудшение Android 2.4.

Он нужен, чтобы заранее назвать альтернативы и искать опровержение. После просмотра графика нельзя незаметно переписать ожидание так, будто найденный паттерн предполагался с начала.

Stop rule защищает интерпретацию

Stop rule для этого вопроса:

text
Не интерпретировать тренд, пока неполные окна,
дубликаты и знаменатель не проверены.

Такой критерий не равен «исследовать, пока не станет понятно». Он задает наблюдаемое условие допуска к выводу. В других задачах stop rule может ограничивать число проверяемых сегментов или объявлять, какой минимальный размер группы нужен для сравнения.

Соберите это

Сначала соберите уменьшенную версию brief без plotting-библиотек.

Шаг 1. Объявите обязательные поля

python
required = (
    "question",
    "decision",
    "grain",
    "metric",
    "comparison",
    "expected_pattern",
    "stop_rule",
)

Отсутствие любого поля означает не «неполное описание картинки», а неполный договор интерпретации.

Шаг 2. Проверьте заполненность

python
missing = [
    field
    for field in required
    if not str(brief[field]).strip()
]
ready = not missing

Эта проверка еще не доказывает содержательную корректность, но делает пропуск поля машинно наблюдаемым.

Шаг 3. Сформулируйте brief до чтения графика

python
brief = {
    "question": "Изменилась ли семидневная активация после релиза?",
    "decision": "Решить, какой сегмент требует отдельной диагностики.",
    "grain": "Один пользователь с полным семидневным окном.",
    "metric": "Доля activated_7d среди observed_days=7.",
    "comparison": "Недели до и после 2026-03-02, затем platform и channel.",
    "expected_pattern": "Возможны channel mix и отдельная Android regression.",
    "stop_rule": "Не интерпретировать результат до аудита окон и дубликатов.",
}

Запустите прозрачный пример:

bash
uv run --locked python code/main.py

Результат содержит ready, сам brief и список missing. В нем еще нет выбора mark: ручной механизм сначала проверяет постановку задачи.

Используйте это

Самостоятельный CLI усиливает минимальный механизм:

  • валидирует структуру JSON без неявных дополнительных полей;
  • требует знаменатель для rate-метрики;
  • проверяет baseline и уровни сравнения;
  • требует time_column для trend и x/y для relationship;
  • формирует chart plan и rubric;
  • добавляет контрольные вопросы для grain, population, metric, comparison и stop rule;
  • предупреждает о причинной формулировке, но не запрещает исследовать ассоциацию.

Запустите встроенный пример:

bash
uv run --locked python outputs/visual_question_brief.py --example

Отчет содержит четыре независимых слоя:

text
brief
chart_plan
required_checks
interpretation_contract

chart_plan для trend связывает cohort_week с осью x, activation_7d с y и предлагает сегменты как facet или color. Одновременно он требует:

  • одинаковый time grain;
  • полные observation windows;
  • размер выборки или знаменатель для каждого периода;
  • uncertainty, если значения являются оценками.

Для собственного вопроса передайте JSON через --spec:

json
{
  "id": "onboarding-by-platform",
  "question": "Как различается длительность onboarding между платформами?",
  "decision": "Выбрать платформу для следующего разбора длинного onboarding.",
  "question_type": "distribution",
  "population": "Пользователи с завершенным первым onboarding flow.",
  "grain": "Один пользователь и его первый завершенный onboarding flow.",
  "metric": {
    "name": "onboarding_seconds",
    "kind": "duration",
    "definition": "Длительность первого завершенного onboarding flow в секундах."
  },
  "comparison": {
    "dimension": "platform",
    "levels": ["web", "ios", "android"],
    "baseline": "web"
  },
  "expected_pattern": "Android может иметь более длинный правый хвост после релиза.",
  "stop_rule": "Не сравнивать платформы до проверки отрицательных значений и пропусков."
}
bash
uv run --locked python outputs/visual_question_brief.py \
  --spec question.json \
  --output brief.json

Коды возврата:

  • 0: спецификация корректна, brief сформирован;
  • 2: JSON нельзя прочитать или контракт вопроса нарушен.

CLI не читает user_journeys.csv и не подтверждает expected pattern. Его граница ответственности - проверить постановку и подготовить план доказательства для следующих шагов.

Сломайте это

Проверьте, как brief останавливает распространенные ошибки.

  1. Удалите metric.denominator у метрики с kind="rate".
  2. Укажите baseline, которого нет в comparison.levels.
  3. Повторите один уровень сравнения дважды.
  4. Для trend удалите comparison.time_column.
  5. Для relationship не задавайте x или y.
  6. Добавьте неизвестное поле chart: "line".
  7. Сформулируйте вопрос как «каково влияние релиза на активацию».

Первые шесть случаев должны завершиться структурированной ошибкой и кодом 2. Седьмой случай остается допустимым, но отчет добавляет предупреждение: визуализация может показать ассоциацию, а причинное утверждение требует эксперимента или явного causal design.

Особенно важен неизвестный chart. Пользователь не должен заранее проталкивать тип картинки мимо вопроса и данных. Конкретный mark появится после аудита и контрольной таблицы.

Проверьте это

Контрольные факты:

  • встроенный пример имеет статус ready;
  • trend использует cohort_week по x и activation_7d по y;
  • rate без знаменателя отклоняется;
  • distribution rubric требует binning, missing-value и outlier policy;
  • relationship rubric требует явные x, y, контроль overplotting и стратификацию;
  • дублирующиеся уровни сравнения отклоняются;
  • причинная формулировка дает warning, а не причинный вывод;
  • CLI пишет одинаковый корректный JSON в stdout и --output.

Поставьте результат

outputs/visual_question_brief.py - самостоятельный CLI-конструктор visual question brief. Он принимает вопрос в JSON, возвращает нормализованный контракт, chart plan, проверки и ограничения интерпретации.

Артефакт можно использовать:

  • перед EDA-ноутбуком как входной договор;
  • в issue или аналитическом тикете перед началом расчета;
  • как machine-readable приложение к отчету;
  • в ревью, чтобы проверить соответствие графика исходному вопросу;
  • как вход для аудитора данных и генераторов графиков следующих уроков.

Brief не гарантирует хороший анализ сам по себе. Он делает ключевые решения видимыми и проверяемыми до того, как оформление графика начнет убеждать читателя.

Упражнения

  1. Составьте comparison brief для сравнения activation между тремя тарифами и обоснуйте baseline.
  2. Преобразуйте вопрос о распределении first_order_amount_rub так, чтобы пропуски без заказа не трактовались как нулевые суммы.
  3. Добавьте в артефакт новый тип вопроса ranking с rubric, тестами и явным правилом обработки равных значений.

from __future__ import annotations

import importlib.util
import json
import subprocess
import sys
import unittest
from pathlib import Path
from tempfile import TemporaryDirectory

ROOT = Path(__file__).resolve().parents[1]
ARTIFACT = ROOT / "outputs" / "visual_question_brief.py"
SPEC = importlib.util.spec_from_file_location("visual_question_brief", ARTIFACT)
if SPEC is None or SPEC.loader is None:
    raise RuntimeError(f"cannot load {ARTIFACT}")
BRIEF = importlib.util.module_from_spec(SPEC)
SPEC.loader.exec_module(BRIEF)


def example() -> dict:
    return json.loads(json.dumps(BRIEF.EXAMPLE_SPEC))


class VisualQuestionBriefTest(unittest.TestCase):
    def test_example_is_ready_and_preserves_decision_contract(self) -> None:
        report = BRIEF.build_brief(example())
        self.assertEqual(report["status"], "ready")
        self.assertEqual(report["brief"]["id"], "activation-after-release")
        self.assertIn("следующей", report["interpretation_contract"]["decision"])
        self.assertEqual(len(report["required_checks"]), 5)

    def test_trend_plan_uses_time_and_requires_complete_windows(self) -> None:
        report = BRIEF.build_brief(example())
        plan = report["chart_plan"]
        self.assertEqual(plan["encodings"]["x"], "cohort_week")
        self.assertEqual(plan["encodings"]["y"], "activation_7d")
        self.assertIn("complete observation windows", plan["required_context"])

    def test_distribution_plan_exposes_bins_and_outlier_policy(self) -> None:
        spec = example()
        spec["question_type"] = "distribution"
        spec["comparison"] = {
            "dimension": "platform",
            "levels": ["all complete windows"],
            "baseline": "all complete windows",
        }
        spec["metric"] = {
            "name": "onboarding_seconds",
            "kind": "duration",
            "definition": "Duration of the first completed onboarding flow in seconds.",
        }
        report = BRIEF.build_brief(spec)
        self.assertIn("ECDF", report["chart_plan"]["primary_view"])
        self.assertIn("outlier policy and scale choice", report["chart_plan"]["required_context"])

    def test_relationship_plan_requires_explicit_x_and_y(self) -> None:
        spec = example()
        spec["question_type"] = "relationship"
        spec["comparison"] = {
            "dimension": "platform",
            "levels": ["all complete windows"],
            "baseline": "all complete windows",
            "x": "sessions_7d",
            "y": "onboarding_seconds",
        }
        report = BRIEF.build_brief(spec)
        self.assertEqual(
            report["chart_plan"]["encodings"],
            {"x": "sessions_7d", "y": "onboarding_seconds"},
        )
        self.assertIn("overplotting control", report["chart_plan"]["required_context"])

    def test_rate_without_denominator_is_rejected(self) -> None:
        spec = example()
        del spec["metric"]["denominator"]
        with self.assertRaisesRegex(BRIEF.BriefError, "metric.denominator"):
            BRIEF.build_brief(spec)

    def test_duplicate_comparison_levels_are_rejected(self) -> None:
        spec = example()
        spec["comparison"]["levels"] = ["до релиза", "до релиза"]
        with self.assertRaisesRegex(BRIEF.BriefError, "duplicate"):
            BRIEF.build_brief(spec)

    def test_causal_wording_produces_warning_not_causal_claim(self) -> None:
        spec = example()
        spec["question"] = "Каково влияние релиза на семидневную активацию новых пользователей?"
        report = BRIEF.build_brief(spec)
        self.assertEqual(len(report["warnings"]), 1)

    def test_cli_writes_same_valid_json_to_stdout_and_output(self) -> None:

            self.assertEqual(json.loads(result.stdout), json.loads(output.read_text()))
            self.assertEqual(json.loads(result.stdout)["status"], "ready")

from __future__ import annotations

import argparse
import json
import re
import sys
from copy import deepcopy
from pathlib import Path
from typing import Any


class BriefError(ValueError):
    """Raised when a visual question specification is incomplete or inconsistent."""


QUESTION_PLANS = {
    "trend": {
        "primary_view": "line with visible observations",
        "alternative_view": "small-multiple dot plots",
        "required_context": [
            "consistent time grain",
            "complete observation windows",
            "sample size or denominator for every period",
            "uncertainty when values are estimates",
        ],
        "avoid": [
            "connecting missing periods as observed values",
            "dual axes for unrelated metrics",
            "interpreting the last incomplete period",
        ],
    },
    "comparison": {
        "primary_view": "sorted dot plot on a common scale",
        "alternative_view": "aligned bars with an honest baseline",
        "required_context": [
            "explicit baseline group",
            "group sample sizes",
            "same metric definition for every group",
            "uncertainty when values are estimates",
        ],
        "avoid": [
            "truncated bar baseline",
            "alphabetical ordering when magnitude is the question",
            "using area or volume for one-dimensional magnitude",
        ],
    },
    "distribution": {
        "primary_view": "ECDF with a documented histogram",
        "alternative_view": "box plot with visible raw or binned observations",
        "required_context": [
            "binning rule when a histogram is used",
            "sample size",
            "missing-value policy",
            "outlier policy and scale choice",
        ],
        "avoid": [
            "deleting outliers only to improve appearance",
            "comparing histograms with different bin edges",
            "reporting only the mean for a skewed distribution",
        ],
    },
    "relationship": {
        "primary_view": "scatter with transparency and grouped summaries",
        "alternative_view": "hexbin or small multiples for dense data",
        "required_context": [
            "units and ranges of both variables",
            "overplotting control",
            "stratified control summaries",
            "warning that association is not causation",
        ],
        "avoid": [
            "fitting a trend without showing observations",
            "hiding segment reversals in one aggregate line",
            "describing association as a causal effect",
        ],
    },
    "composition": {
        "primary_view": "aligned share bars with absolute counts",
        "alternative_view": "small multiples of group shares",
        "required_context": [
            "common denominator",
            "absolute count beside every share",
            "stable category definitions",
            "explicit treatment of missing categories",
        ],
        "avoid": [
            "shares without denominators",
            "too many categories in a pie chart",
            "comparing areas or angles when precise differences matter",
        ],
    },
}

METRIC_KINDS = {"rate", "count", "amount", "duration", "continuous"}
IDENTIFIER = re.compile(r"^[a-z][a-z0-9-]*$")
CAUSAL_TERMS = (
    "cause",
    "causal",
    "effect of",
    "impact of",
    "причин",
    "эффект",
    "влияни",
)

EXAMPLE_SPEC = {
    "id": "activation-after-release",
    "question": "Как изменилась семидневная активация после релиза 2026-03-02?",
    "decision": "Определить сегмент для следующей технической и продуктовой диагностики.",
    "question_type": "trend",
    "population": "Новые пользователи подписочного сервиса.",
    "grain": "Один пользователь с полным семидневным окном после регистрации.",
    "metric": {
        "name": "activation_7d",
        "kind": "rate",
        "definition": "Доля пользователей с activated_7d=true среди полных окон.",
        "denominator": "Пользователи с observed_days=7 после удаления дубликатов.",
    },
    "comparison": {
        "dimension": "cohort_week",
        "levels": ["до релиза", "после релиза"],
        "baseline": "до релиза",
        "time_column": "cohort_week",
        "segments": ["platform", "acquisition_channel"],
    },
    "expected_pattern": (
        "Общий спад может сочетать изменение channel mix и отдельное ухудшение Android 2.4."
    ),
    "stop_rule": (
        "Не интерпретировать тренд, пока неполные окна, дубликаты и знаменатель не проверены."
    ),
}


def require_text(value: Any, location: str, *, minimum: int = 1) -> str:
    if not isinstance(value, str) or len(value.strip()) < minimum:
        raise BriefError(f"{location} must be text with at least {minimum} characters")
    return value.strip()


def require_mapping(value: Any, location: str) -> dict[str, Any]:
    if not isinstance(value, dict):
        raise BriefError(f"{location} must be an object")
    return value


def require_text_list(value: Any, location: str, *, minimum: int = 1) -> list[str]:
    if not isinstance(value, list) or len(value) < minimum:
        raise BriefError(f"{location} must contain at least {minimum} values")
    values = [require_text(item, f"{location}[{index}]") for index, item in enumerate(value)]
    if len(values) != len(set(values)):
        raise BriefError(f"{location} contains duplicate values")
    return values


def normalize_metric(value: Any) -> dict[str, str]:
    metric = require_mapping(value, "metric")
    required = {"name", "kind", "definition"}
    allowed = required | {"denominator"}
    missing = sorted(required - set(metric))
    unexpected = sorted(set(metric) - allowed)
    if missing or unexpected:
        raise BriefError(f"metric fields: missing={missing}, unexpected={unexpected}")
    kind = require_text(metric["kind"], "metric.kind")
    if kind not in METRIC_KINDS:
        raise BriefError(f"metric.kind must be one of {sorted(METRIC_KINDS)}")
    normalized = {
        "name": require_text(metric["name"], "metric.name"),
        "kind": kind,
        "definition": require_text(metric["definition"], "metric.definition", minimum=15),
    }
    if kind == "rate":
        normalized["denominator"] = require_text(
            metric.get("denominator"),
            "metric.denominator",
            minimum=10,
        )
    elif "denominator" in metric:
        normalized["denominator"] = require_text(
            metric["denominator"],
            "metric.denominator",
        )
    return normalized


def normalize_comparison(value: Any, question_type: str) -> dict[str, Any]:
    comparison = require_mapping(value, "comparison")
    required = {"dimension", "levels", "baseline"}
    allowed = required | {"time_column", "segments", "x", "y"}
    missing = sorted(required - set(comparison))
    unexpected = sorted(set(comparison) - allowed)
    if missing or unexpected:
        raise BriefError(f"comparison fields: missing={missing}, unexpected={unexpected}")
    levels = require_text_list(
        comparison["levels"],
        "comparison.levels",
        minimum=2 if question_type in {"trend", "comparison", "composition"} else 1,
    )
    baseline = require_text(comparison["baseline"], "comparison.baseline")
    if baseline not in levels:
        raise BriefError("comparison.baseline must be present in comparison.levels")
    normalized: dict[str, Any] = {
        "dimension": require_text(comparison["dimension"], "comparison.dimension"),
        "levels": levels,
        "baseline": baseline,
    }
    if "segments" in comparison:
        normalized["segments"] = require_text_list(
            comparison["segments"],
            "comparison.segments",
        )
    if question_type == "trend":
        normalized["time_column"] = require_text(
            comparison.get("time_column"),
            "comparison.time_column",
        )
    if question_type == "relationship":
        normalized["x"] = require_text(comparison.get("x"), "comparison.x")
        normalized["y"] = require_text(comparison.get("y"), "comparison.y")
    return normalized


def normalize_spec(spec: dict[str, Any]) -> dict[str, Any]:
    required = {
        "id",
        "question",
        "decision",
        "question_type",
        "population",
        "grain",
        "metric",
        "comparison",
        "expected_pattern",
        "stop_rule",
    }
    missing = sorted(required - set(spec))
    unexpected = sorted(set(spec) - required)
    if missing or unexpected:
        raise BriefError(f"top-level fields: missing={missing}, unexpected={unexpected}")
    identifier = require_text(spec["id"], "id")
    if not IDENTIFIER.fullmatch(identifier):
        raise BriefError("id must use lowercase kebab-case")
    question_type = require_text(spec["question_type"], "question_type")
    if question_type not in QUESTION_PLANS:
        raise BriefError(f"question_type must be one of {sorted(QUESTION_PLANS)}")
    return {
        "id": identifier,
        "question": require_text(spec["question"], "question", minimum=20),
        "decision": require_text(spec["decision"], "decision", minimum=20),
        "question_type": question_type,
        "population": require_text(spec["population"], "population", minimum=10),
        "grain": require_text(spec["grain"], "grain", minimum=10),
        "metric": normalize_metric(spec["metric"]),
        "comparison": normalize_comparison(spec["comparison"], question_type),
        "expected_pattern": require_text(
            spec["expected_pattern"],
            "expected_pattern",
            minimum=20,
        ),
        "stop_rule": require_text(spec["stop_rule"], "stop_rule", minimum=20),
    }


def causal_warning(spec: dict[str, Any]) -> list[str]:
    text = " ".join([spec["question"], spec["decision"], spec["expected_pattern"]]).casefold()
    if any(term in text for term in CAUSAL_TERMS):
        return [
            "The wording suggests causality. A visualization can show association, "
            "but identification requires an experiment or an explicit causal design."
        ]
    return []


def build_brief(spec: dict[str, Any]) -> dict[str, Any]:
    normalized = normalize_spec(spec)
    plan = deepcopy(QUESTION_PLANS[normalized["question_type"]])
    comparison = normalized["comparison"]
    encodings: dict[str, str] = {"y": normalized["metric"]["name"]}
    if normalized["question_type"] == "trend":
        encodings["x"] = comparison["time_column"]
    elif normalized["question_type"] == "relationship":
        encodings = {"x": comparison["x"], "y": comparison["y"]}
    else:
        encodings["x"] = comparison["dimension"]
    if comparison.get("segments"):
        encodings["facet_or_color"] = ", ".join(comparison["segments"])
    plan["encodings"] = encodings
    checks = [
        {
            "id": "grain",
            "question": f"Does every source row match this grain: {normalized['grain']}?",
        },
        {
            "id": "population",
            "question": f"Does the filtered data represent: {normalized['population']}?",
        },
        {
            "id": "metric",
            "question": (
                "Can the plotted values be reproduced from a control table using: "
                f"{normalized['metric']['definition']}?"
            ),
        },
        {
            "id": "comparison",
            "question": (
                f"Are all levels of {comparison['dimension']} comparable to baseline "
                f"{comparison['baseline']}?"
            ),
        },
        {
            "id": "stop-rule",
            "question": f"Has this stop rule been satisfied: {normalized['stop_rule']}?",
        },
    ]
    return {
        "version": "1.0.0",
        "status": "ready",
        "brief": normalized,
        "chart_plan": plan,
        "required_checks": checks,
        "interpretation_contract": {
            "observation": "State only what the checked values and encodings show.",
            "explanation": "Label explanations as hypotheses unless independently identified.",
            "decision": normalized["decision"],
            "stop_rule": normalized["stop_rule"],
        },
        "warnings": causal_warning(normalized),
    }


def load_spec(path: Path) -> dict[str, Any]:
    try:
        value = json.loads(path.read_text(encoding="utf-8"))
    except OSError as error:
        raise BriefError(f"cannot read spec: {error}") from error
    except json.JSONDecodeError as error:
        raise BriefError(f"invalid JSON spec: {error.msg}") from error
    if not isinstance(value, dict):
        raise BriefError("spec must contain a JSON object")
    return value


def render_json(value: Any) -> str:
    return json.dumps(value, ensure_ascii=False, indent=2, sort_keys=True) + "\n"


def main(argv: list[str] | None = None) -> int:
    parser = argparse.ArgumentParser(
        description="Validate a visual question and produce a chart plan"
    )
    source = parser.add_mutually_exclusive_group(required=True)
    source.add_argument("--spec", type=Path, help="Path to a visual question JSON spec")
    source.add_argument("--example", action="store_true", help="Use the built-in phase example")
    parser.add_argument("--output", type=Path, help="Optional path for the JSON brief")
    args = parser.parse_args(argv)
    try:
        spec = deepcopy(EXAMPLE_SPEC) if args.example else load_spec(args.spec)
        report = build_brief(spec)
    except BriefError as error:
        sys.stdout.write(render_json({"error": str(error)}))
        return 2
    content = render_json(report)
    if args.output:
        args.output.parent.mkdir(parents=True, exist_ok=True)
        args.output.write_text(content, encoding="utf-8")
    sys.stdout.write(content)
    return 0


if __name__ == "__main__":
    raise SystemExit(main())
