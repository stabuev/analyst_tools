# Вопрос раньше графика

> Сначала зафиксируйте, какое решение должен поддержать анализ, и только затем выбирайте визуальное представление.

**Тип:** Learn  
**Треки:** Core  
**Пререквизиты:** 05/11  
**Время:** ~75 минут  
**Результат:** преобразует рабочий вопрос в контракт визуального доказательства с
единицей анализа, сравнением, метрикой, ожидаемым паттерном и критерием остановки.

## Цели обучения

- Связывать визуальный вопрос с конкретным решением, а не с абстрактным интересом к данным.
- Фиксировать population, grain, метрику и знаменатель до построения графика.
- Выбирать семейство представлений по аналитической операции: trend, comparison,
  distribution, relationship или composition.
- Отделять ожидаемый паттерн от результата и причинное объяснение от наблюдаемой связи.
- Задавать stop rule, который запрещает интерпретацию неподготовленных данных.

## Проблема

После мартовского релиза команда увидела снижение семидневной активации и попросила:

> Построй график активации и найди, почему она упала.

Формулировка кажется понятной, но не задает аналитический контракт:

- какую продуктовую или техническую развилку должен поддержать результат;
- кто входит в population;
- что является одной единицей наблюдения;
- как именно рассчитывается активация и кто входит в знаменатель;
- какие периоды или сегменты сравниваются;
- какие данные нельзя интерпретировать;
- где заканчивается наблюдение и начинается причинная гипотеза.

Если сразу открыть plotting-библиотеку, эти решения все равно придется принять. Только
они окажутся спрятаны в фильтрах, `groupby`, настройках осей и выбранном типе графика.
Две правдоподобные картинки могут отвечать на разные вопросы:

- доля активации по календарным дням регистрации;
- доля по недельным когортам;
- число активированных пользователей;
- доля среди всех регистраций, включая неполные окна;
- доля только среди пользователей с полными семью днями наблюдения.

График не исправляет неоднозначный вопрос. Он делает одно из возможных толкований
визуально убедительным.

## Концепция

Visual question brief - это машинно-проверяемый договор между рабочим вопросом, данными,
сравнением и будущим представлением.

```text
решение
  -> вопрос
  -> population и grain
  -> метрика и знаменатель
  -> сравнение
  -> ожидаемый паттерн
  -> stop rule
  -> chart plan
  -> проверяемое наблюдение
```

### Решение раньше любопытства

Вопрос «что происходит с активацией?» допускает бесконечное исследование. Решение
ограничивает поиск:

```text
Определить сегмент для следующей технической и продуктовой диагностики.
```

Этот контракт не обещает доказать причину релиза. Он требует найти проверяемое место для
следующего шага: платформу, версию приложения или канал.

### Population и grain

**Population** отвечает, о ком разрешено делать вывод. **Grain** отвечает, что означает
одна строка или одна независимая единица сравнения.

Для `user_journeys`:

```text
Population: новые пользователи подписочного сервиса.
Grain: один пользователь с полным семидневным окном после регистрации.
```

Исходная таблица намеренно нарушает этот идеальный вход:

- один `user_id` доставлен повторно;
- у последних регистраций `observed_days < 7`;
- outcome-поля неполных окон отсутствуют.

Поэтому grain в вопросе одновременно становится требованием к будущему аудиту данных.

### Метрика является формулой

Название `activation_7d` недостаточно. Для rate-метрики нужны событие, population policy
и знаменатель:

```text
Доля пользователей с activated_7d=true
среди observed_days=7 после удаления дубликатов.
```

Если включить неполные окна в знаменатель как неактивированных пользователей, последние
когорты будут выглядеть хуже даже без реального изменения поведения.

### Сравнение является частью вопроса

Фраза «после релиза» требует явных уровней:

- baseline: когорты до `2026-03-02`;
- comparison: когорты начиная с `2026-03-02`;
- time grain: `cohort_week`;
- сегменты для drill-down: `platform` и `acquisition_channel`.

Без baseline график показывает последовательность чисел, но не определяет контрольную
точку решения.

### Тип вопроса раньше типа mark

Brief использует пять аналитических операций:

| Тип вопроса | Что сравнивается | Базовый план |
|---|---|---|
| `trend` | значение по упорядоченному времени | линия с видимыми наблюдениями |
| `comparison` | категории на общей шкале | сортированный dot plot |
| `distribution` | форма и хвосты числовой величины | ECDF и документированный histogram |
| `relationship` | совместное изменение двух величин | scatter с контролем overplotting |
| `composition` | доли частей общего целого | выровненные доли вместе с counts |

Это не автоматический выбор единственно правильного графика. Это rubric: набор
предпочтительных представлений, необходимого контекста и опасных приемов для проверки.

### Expected pattern не является ответом

Ожидаемый паттерн делает исходное рассуждение наблюдаемым:

```text
Общий спад может сочетать изменение channel mix
и отдельное ухудшение Android 2.4.
```

Он нужен, чтобы заранее назвать альтернативы и искать опровержение. После просмотра
графика нельзя незаметно переписать ожидание так, будто найденный паттерн предполагался с
начала.

### Stop rule защищает интерпретацию

Stop rule для этого вопроса:

```text
Не интерпретировать тренд, пока неполные окна,
дубликаты и знаменатель не проверены.
```

Такой критерий не равен «исследовать, пока не станет понятно». Он задает наблюдаемое
условие допуска к выводу. В других задачах stop rule может ограничивать число проверяемых
сегментов или объявлять, какой минимальный размер группы нужен для сравнения.

## Соберите это

Сначала соберите уменьшенную версию brief без plotting-библиотек.

### Шаг 1. Объявите обязательные поля

```python
required = (
    "question",
    "decision",
    "grain",
    "metric",
    "comparison",
    "expected_pattern",
    "stop_rule",
)
```

Отсутствие любого поля означает не «неполное описание картинки», а неполный договор
интерпретации.

### Шаг 2. Проверьте заполненность

```python
missing = [
    field
    for field in required
    if not str(brief[field]).strip()
]
ready = not missing
```

Эта проверка еще не доказывает содержательную корректность, но делает пропуск поля
машинно наблюдаемым.

### Шаг 3. Сформулируйте brief до чтения графика

```python
brief = {
    "question": "Изменилась ли семидневная активация после релиза?",
    "decision": "Решить, какой сегмент требует отдельной диагностики.",
    "grain": "Один пользователь с полным семидневным окном.",
    "metric": "Доля activated_7d среди observed_days=7.",
    "comparison": "Недели до и после 2026-03-02, затем platform и channel.",
    "expected_pattern": "Возможны channel mix и отдельная Android regression.",
    "stop_rule": "Не интерпретировать результат до аудита окон и дубликатов.",
}
```

Запустите прозрачный пример:

```bash
uv run --locked python code/main.py
```

Результат содержит `ready`, сам brief и список `missing`. В нем еще нет выбора mark:
ручной механизм сначала проверяет постановку задачи.

## Используйте это

Самостоятельный CLI усиливает минимальный механизм:

- валидирует структуру JSON без неявных дополнительных полей;
- требует знаменатель для rate-метрики;
- проверяет baseline и уровни сравнения;
- требует `time_column` для trend и `x`/`y` для relationship;
- формирует chart plan и rubric;
- добавляет контрольные вопросы для grain, population, metric, comparison и stop rule;
- предупреждает о причинной формулировке, но не запрещает исследовать ассоциацию.

Запустите встроенный пример:

```bash
uv run --locked python outputs/visual_question_brief.py --example
```

Отчет содержит четыре независимых слоя:

```text
brief
chart_plan
required_checks
interpretation_contract
```

`chart_plan` для trend связывает `cohort_week` с осью `x`, `activation_7d` с `y` и
предлагает сегменты как facet или color. Одновременно он требует:

- одинаковый time grain;
- полные observation windows;
- размер выборки или знаменатель для каждого периода;
- uncertainty, если значения являются оценками.

Для собственного вопроса передайте JSON через `--spec`:

```json
{
  "id": "onboarding-by-platform",
  "question": "Как различается длительность onboarding между платформами?",
  "decision": "Выбрать платформу для следующего разбора длинного onboarding.",
  "question_type": "distribution",
  "population": "Пользователи с завершенным первым onboarding flow.",
  "grain": "Один пользователь и его первый завершенный onboarding flow.",
  "metric": {
    "name": "onboarding_seconds",
    "kind": "duration",
    "definition": "Длительность первого завершенного onboarding flow в секундах."
  },
  "comparison": {
    "dimension": "platform",
    "levels": ["web", "ios", "android"],
    "baseline": "web"
  },
  "expected_pattern": "Android может иметь более длинный правый хвост после релиза.",
  "stop_rule": "Не сравнивать платформы до проверки отрицательных значений и пропусков."
}
```

```bash
uv run --locked python outputs/visual_question_brief.py \
  --spec question.json \
  --output brief.json
```

Коды возврата:

- `0`: спецификация корректна, brief сформирован;
- `2`: JSON нельзя прочитать или контракт вопроса нарушен.

CLI не читает `user_journeys.csv` и не подтверждает expected pattern. Его граница
ответственности - проверить постановку и подготовить план доказательства для следующих
шагов.

## Сломайте это

Проверьте, как brief останавливает распространенные ошибки.

1. Удалите `metric.denominator` у метрики с `kind="rate"`.
2. Укажите baseline, которого нет в `comparison.levels`.
3. Повторите один уровень сравнения дважды.
4. Для `trend` удалите `comparison.time_column`.
5. Для `relationship` не задавайте `x` или `y`.
6. Добавьте неизвестное поле `chart: "line"`.
7. Сформулируйте вопрос как «каково влияние релиза на активацию».

Первые шесть случаев должны завершиться структурированной ошибкой и кодом `2`. Седьмой
случай остается допустимым, но отчет добавляет предупреждение: визуализация может показать
ассоциацию, а причинное утверждение требует эксперимента или явного causal design.

Особенно важен неизвестный `chart`. Пользователь не должен заранее проталкивать тип
картинки мимо вопроса и данных. Конкретный mark появится после аудита и контрольной
таблицы.

## Проверьте это

Контрольные факты:

- встроенный пример имеет статус `ready`;
- trend использует `cohort_week` по `x` и `activation_7d` по `y`;
- rate без знаменателя отклоняется;
- distribution rubric требует binning, missing-value и outlier policy;
- relationship rubric требует явные `x`, `y`, контроль overplotting и стратификацию;
- дублирующиеся уровни сравнения отклоняются;
- причинная формулировка дает warning, а не причинный вывод;
- CLI пишет одинаковый корректный JSON в stdout и `--output`.

Запустите behavioral tests:

```bash
uv run --locked python -m unittest discover -s tests
```

Проверьте общий набор данных фазы:

```bash
uv run --locked python ../data/generate_data.py --check
```

Committed `tiny` содержит 25 строк и 24 уникальных пользователя. Такое расхождение не
нужно исправлять внутри brief: оно должно остановить интерпретацию и перейти в аудит
данных следующего урока.

## Поставьте результат

`outputs/visual_question_brief.py` - самостоятельный CLI-конструктор visual question
brief. Он принимает вопрос в JSON, возвращает нормализованный контракт, chart plan,
проверки и ограничения интерпретации.

Артефакт можно использовать:

- перед EDA-ноутбуком как входной договор;
- в issue или аналитическом тикете перед началом расчета;
- как machine-readable приложение к отчету;
- в ревью, чтобы проверить соответствие графика исходному вопросу;
- как вход для аудитора данных и генераторов графиков следующих уроков.

Brief не гарантирует хороший анализ сам по себе. Он делает ключевые решения видимыми и
проверяемыми до того, как оформление графика начнет убеждать читателя.

## Упражнения

1. Составьте `comparison` brief для сравнения activation между тремя тарифами и
   обоснуйте baseline.
2. Преобразуйте вопрос о распределении `first_order_amount_rub` так, чтобы пропуски без
   заказа не трактовались как нулевые суммы.
3. Добавьте в артефакт новый тип вопроса `ranking` с rubric, тестами и явным правилом
   обработки равных значений.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Visual question | Название графика | Проверяемый вопрос, связанный с решением и данными |
| Population | Все строки файла | Множество сущностей, к которым относится вывод |
| Grain | Шаг делений оси | Смысл одной строки или независимой единицы анализа |
| Denominator | Техническая деталь rate | Явное множество наблюдений, относительно которого считается доля |
| Baseline | Первая категория | Обоснованная контрольная точка сравнения |
| Expected pattern | Заранее известный ответ | Проверяемое ожидание, которое разрешено опровергнуть |
| Stop rule | Момент закрытия ноутбука | Условие допуска к интерпретации или завершения поиска |
| Encoding | Украшение mark | Отображение поля данных в позицию, цвет, размер или другой визуальный канал |
| Association | Причинный эффект | Наблюдаемая совместная изменчивость без доказанного механизма причины |

## Дополнительное чтение

- [A Nested Model for Visualization Design and Validation](https://www.cs.ubc.ca/labs/imager/tr/2009/NestedModel/NestedModel.pdf) - изучите четыре уровня проектирования и угрозы валидности при слишком раннем переходе к visual encoding.
- [Vega-Lite: A Grammar of Interactive Graphics](https://idl.cs.washington.edu/papers/vega-lite/) - посмотрите, как декларативная спецификация отделяет данные, encodings и interaction от ручного рисования.
- [Vega-Lite Encoding](https://vega.github.io/vega-lite/docs/encoding.html) - сопоставьте поля и semantic types с каналами `x`, `y`, `color`, `size`, `facet` и `tooltip`.
- [Government Analysis Function: Data visualisation charts](https://analysisfunction.civilservice.gov.uk/policy-store/data-visualisation-charts/) - разберите требования к сообщению графика, источнику, доступному описанию и данным для воспроизведения.
