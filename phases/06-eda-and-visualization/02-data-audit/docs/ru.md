# Аудит набора данных

> До первого графика докажите, что строки, ключи, окна наблюдения и пропуски имеют понятный смысл.

**Тип:** Case  
**Треки:** Core  
**Пререквизиты:** 06/01  
**Время:** ~90 минут  
**Результат:** проверяет grain, ключи, типы, пропуски, диапазоны, дубликаты и полноту
окна наблюдения до построения графиков и отделяет дефект данных от сигнала.

## Цели обучения

- Сверять фактический grain с объявленным primary key.
- Разделять структурные, случайные и вызванные окном наблюдения пропуски.
- Отличать невозможное значение от допустимого экстремума.
- Формировать decision log, который определяет фильтры последующего EDA.

## Проблема

Таблица `user_journeys` выглядит готовой для графика активации: одна строка похожа на
одного пользователя, есть платформа, когорта и `activated_7d`. Но в `tiny` лежат 25
строк и только 24 уникальных `user_id`, две последние регистрации наблюдались меньше
семи дней, одна длительность onboarding равна `-1`, а пустой `app_version` для web
является нормой.

Если применить одно правило `dropna()` или считать все пустые outcomes как `false`,
аудит сам создаст ложный сигнал.

## Концепция

Аудит EDA-входа отвечает на четыре разных вопроса:

| Слой | Проверка | Решение |
|---|---|---|
| Structure | колонки, типы, категории | можно ли читать значения |
| Grain | ключ и дубликаты | что означает одна строка |
| Observation | полнота семидневного окна | известен ли outcome |
| Domain | диапазоны и null policy | возможно ли значение |

Пропуск классифицируется по причине. `app_version=""` для web структурен.
`activated_7d=""` при `observed_days<7` означает еще не наблюдавшийся outcome.
`country=""` допустим, но его долю нужно показать. Пустой `user_id` был бы дефектом.

Невозможный `onboarding_seconds=-1` нельзя смешивать с допустимым heavy-tail значением
`3600`: первое нарушает домен, второе требует устойчивой статистики и видимости на
графике.

## Соберите это

Прозрачный пример читает CSV стандартной библиотекой, считает `user_id` и отдельно
находит неполные окна:

```bash
uv run --locked python code/main.py
```

Ключевой принцип:

```python
incomplete = observed_days < 7
```

Эта маска не должна превращаться в `activated_7d=False`. Она исключает строку только из
метрики, требующей полного семидневного окна.

## Используйте это

CLI применяет `contract.json`, проверяет типы, категории, диапазоны, primary key,
observation policy и классифицирует missingness:

```bash
uv run --locked python outputs/eda_audit.py \
  --input ../data/tiny/user_journeys.csv \
  --contract ../data/contract.json \
  --allow-failures
```

Отчет содержит `checks`, `failure_ids`, `activation_blockers`, `missingness` и
`decision_log`. Без `--allow-failures` известные дефекты дают код `1`, поэтому CLI можно
использовать как quality gate. Код `2` означает, что файл или контракт нельзя прочитать.

`ready_for_activation` уже, чем общий `valid`: например, экстремальный onboarding важен
для распределения, но сам по себе не меняет корректность знаменателя activation.

## Сломайте это

1. Удалите `user_id` из CSV: аудит должен остановиться до domain checks.
2. Заполните outcome у строки с `observed_days=2`: observation policy должна упасть.
3. Оставьте пустой `app_version` у Android: структурное правило нарушено.
4. Запишите `eight` в `sessions_7d`: ошибка должна содержать номер строки.
5. Добавьте неизвестную платформу: категория должна появиться в отчете.

Не исправляйте данные внутри аудитора. Артефакт локализует решение и сохраняет исходные
bytes проверяемыми через SHA-256.

## Проверьте это

Для committed `tiny` должны выполняться контрольные факты:

- 25 строк и 24 уникальных пользователя;
- дубликат `J018`;
- два неполных окна без outcomes;
- один отрицательный и один экстремальный onboarding;
- один пропуск `country`;
- пустой `app_version` для web соответствует policy.

```bash
uv run --locked python -m unittest discover -s tests
uv run --locked python ../data/generate_data.py --check
```

После удаления повторной доставки, неполных окон и невозможного отрицательного времени
очищенный срез становится готов к activation-анализу.

## Поставьте результат

`outputs/eda_audit.py` - самостоятельный аудитор EDA-входа. Его JSON можно передать
следующему шагу, приложить к отчету или использовать в CI. Decision log фиксирует, какие
строки исключены из конкретной метрики и почему.

## Упражнения

1. Добавьте проверку, что `cohort_week` всегда приходится на понедельник.
2. Введите severity `error`, `warning`, `info` вместо бинарного статуса.
3. Добавьте сравнение фактического диапазона дат с `time_range` контракта.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Data audit | Автоматическая очистка | Наблюдаемая проверка пригодности входа |
| Structural missingness | Случайно потерянное значение | Поле неприменимо по модели данных |
| Observation window | Диапазон оси | Период, нужный для полного наблюдения outcome |
| Invalid value | Любой выброс | Значение вне допустимого домена |
| Decision log | Технический лог | Обоснование фильтров и трактовок анализа |
