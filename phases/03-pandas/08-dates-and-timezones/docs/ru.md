# Даты, интервалы и часовые пояса

> Момент храните на общей шкале UTC, календарную дату вычисляйте в явно выбранной зоне.

**Тип:** Case
**Треки:** Core
**Пререквизиты:** 03/07
**Время:** ~105 минут
**Результат:** нормализует timestamps и рассчитывает бизнес-календарь без сдвига дня.

## Цели обучения

- Различать момент времени и локальное календарное представление.
- Разбирать смешанные offsets через `format="mixed", utc=True`.
- Использовать `tz_convert`, а не удаление зоны.
- Проверять интервалы и календарные границы.

## Проблема

Заказ в `23:30Z` относится к следующему календарному дню в Москве. Если обрезать строку
до первых десяти символов или удалить timezone, дневная метрика попадёт не в тот день.
Смешанные offsets нельзя надёжно сравнивать как строки.

## Концепция

UTC timestamp — точка на временной шкале. Timezone задаёт правила отображения точки в
локальные дату и время, включая исторические переходы. `tz_localize` назначает зону naive
времени, `tz_convert` отображает уже timezone-aware момент в другой зоне.

Календарная неделя и длительность — разные операции: неделя зависит от бизнес-зоны,
длительность считается между UTC-моментами.

## Соберите это

Переведите вручную:

```text
2026-02-01 23:30 UTC -> 2026-02-02 02:30 Europe/Moscow
```

Затем вычислите два момента с offsets `+03:00` и `Z`, которые обозначают одну точку, и
проверьте их равенство после нормализации.

## Используйте это

```python
utc = pd.to_datetime(
    orders["ordered_at"],
    format="mixed",
    errors="coerce",
    utc=True,
)
moscow = utc.dt.tz_convert("Europe/Moscow")
local_date = moscow.dt.date
```

```bash
uv run --locked python outputs/time_normalizer.py \
  ../data/tiny/orders.csv --column ordered_at --timezone Europe/Moscow
```

## Сломайте это

Передайте несуществующую зону, неразбираемую дату, пустое значение и end раньше start.
Отдельно сравните результат удаления timezone через `tz_localize(None)` с правильным
`tz_convert`: первое действие теряет контекст, второе сохраняет момент.

## Проверьте это

- Все валидные моменты стали `datetime64[..., UTC]`.
- Непустая неверная дата вызывает ошибку.
- Пустая дата остаётся пропуском.
- Локальная дата рассчитана после `tz_convert`.
- Отрицательная длительность выявляется.

## Поставьте результат

`outputs/time_normalizer.py` добавляет UTC и локальные timestamp, дату, час и начало недели.
Timezone является обязательным параметром, поэтому календарная семантика видима в запуске.

## Упражнения

1. Сравните локальные даты заказа в Москве и Нью-Йорке.
2. Рассчитайте часы от регистрации пользователя до заказа.
3. Добавьте проверку допустимого временного диапазона набора.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| UTC | Ещё одна локальная зона | Общая временная шкала |
| Offset | Полный timezone | Смещение в конкретный момент без всех правил зоны |
| `tz_convert` | Назначает зону naive-времени | Переводит aware-момент в другое представление |
| Calendar window | То же, что длительность | Граница, зависящая от локального календаря |
