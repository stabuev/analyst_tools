# GroupBy и единица анализа

> `groupby` не просто считает несколько чисел. Он превращает исходные строки в таблицу
> с другой единицей наблюдения. Поэтому до кода нужно назвать grain входа, популяцию,
> смысл агрегатов и grain результата.

**Тип:** Case

**Треки:** Core

**Пререквизиты:** 03/04

**Время:** ~105 минут

**Результат:** на типизированной таблице товарных позиций объявляет входной и выходной
grain, выбирает агрегаты по смыслу метрик, сохраняет неизвестность неполной суммы и
сверяет результат с ручным контрольным расчётом.

## Цели обучения

После урока вы сможете:

- объяснить агрегацию как переход от одного grain к другому;
- до `groupby` назвать входной ключ, популяцию, ключи результата и смысл каждой метрики;
- вручную воспроизвести механизм `split -> apply -> combine` на нескольких строках;
- различать `size`, `count`, `nunique`, `sum` и `mean` по вопросу, на который они отвечают;
- использовать именованные агрегации и получать ключ группы обычным столбцом;
- отличать пропуск в ключе группы от пропуска в измеряемом значении;
- не превращать полностью или частично неизвестную сумму в доказанный ноль;
- проверять новый grain, полноту, dtype, число строк и контрольные totals;
- объяснить, почему `GroupBy.apply` не нужен для обычных агрегатов.

## Где мы находимся в маршруте

Первые четыре урока pandas последовательно добавили четыре контракта:

| Урок | Вопрос | Что уже умеет студент |
|---|---|---|
| `03/01` | Что означает строка? | Отличать индекс от бизнес-ключа, объявлять и проверять grain |
| `03/02` | Можно ли доверять значениям? | Проверять dtype, missing и invalid до вычислений |
| `03/03` | Какие строки входят в расчёт? | Строить согласованную nullable-маску и явно разрешать `NA` |
| `03/04` | Что вычислить для каждой строки? | Добавлять производные `Series`, сохраняя строки, индекс и grain |

Теперь появляется новый вопрос:

```text
03/01: что представляет строка?      -> объявляем grain и ключ
03/02: можно ли вычислять?           -> проверяем dtype и пропуски
03/03: какие строки входят?          -> определяем популяцию
03/04: что вычислить для каждой?     -> сохраняем grain
03/05: какой итог нужен для группы?  -> изменяем grain
03/06: как соединить таблицы?        -> проверяем cardinality
```

В `03/04` таблица `order_items` осталась на grain товарной позиции. Функция добавила
`line_total`, но сознательно не стала считать сумму заказа. Именно этот результат теперь
станет входом агрегации.

## Проблема

После предыдущего урока у нас есть уже типизированные товарные позиции:

| Метка | `order_id` | `product_id` | `line_total` |
|---|---|---|---:|
| `item-a` | `O1001` | `P01` | 800.0 |
| `item-b` | `O1001` | `P02` | 800.0 |
| `item-c` | `O1002` | `P03` | пропуск |
| `item-d` | `O1003` | `P04` | 75.5 |
| `item-e` | `O1004` | `P05` | 0.0 |

Входной grain:

```text
одна строка -> одна товарная позиция заказа
кандидатный ключ -> order_id + product_id
```

Для следующего объединения нужна таблица на уровне заказа:

```text
одна строка -> один заказ
кандидатный ключ -> order_id
```

Значит, две строки `O1001` должны превратиться в одну. Повтор `order_id` во входе здесь
не является дублем: он ожидается на grain товарной позиции. Дублем будет повтор всей
пары `(order_id, product_id)`.

До вызова `groupby` нужно принять четыре решения:

1. **Входной grain:** что означает каждая строка и какой ключ его поддерживает?
2. **Популяция:** все ли проверенные строки участвуют в расчёте?
3. **Метрики:** что именно считаем внутри каждой группы и как трактуем пропуски?
4. **Выходной grain:** что означает строка результата и какой ключ должен стать уникальным?

Если одно из решений не принято, pandas всё равно вернёт таблицу. Именно поэтому ошибка
агрегации часто выглядит правдоподобным числом, а не исключением.

## Концепция

### Агрегация меняет grain

Преобразование из `03/04` добавляло значение каждой исходной строке:

```text
5 товарных позиций -> 5 товарных позиций
```

Агрегация сворачивает несколько строк одной группы в одну:

```text
5 товарных позиций -> 4 заказа
```

`order_id` повторяется во входе и становится уникальным в результате. Это не побочный
эффект `groupby`, а целевое изменение единицы наблюдения.

Число строк после агрегации определяется не названием функции `sum` или `mean`, а числом
уникальных комбинаций ключей группировки в выбранной популяции.

### Ключи группировки являются обещанием о результате

Сравните два вызова:

```python
items.groupby("order_id")
items.groupby(["order_id", "product_id"])
```

Первый создаёт группу для каждого заказа и действительно уменьшает детализацию. Второй
использует исходный составной ключ, поэтому при корректном входном grain почти каждая
строка останется отдельной группой. Код выполнится, но требуемого перехода к заказам не
произойдёт.

Поэтому список ключей читается как человеческое утверждение:

> Одна строка результата будет описывать одну уникальную комбинацию этих значений.

### Популяция определяется до группировки

Фильтрация и агрегация решают разные задачи:

| Операция | Вопрос | Что меняется |
|---|---|---|
| Фильтрация | Какие наблюдения участвуют? | Состав строк |
| Преобразование | Что вычислить для каждой строки? | Столбцы, но не grain |
| Агрегация | Какой итог нужен для каждой группы? | Grain результата |

Если бизнес-вопрос относится только к выбранной популяции, сначала примените проверенную
маску из `03/03`, сохраните отчёт о `True`, `False` и `NA`, а уже затем группируйте
разрешённые строки. В этом уроке входная таблица уже является нужной популяцией, поэтому
мы не прячем дополнительный фильтр внутрь функции агрегации.

### `split -> apply -> combine` — модель механизма

`GroupBy` удобно понимать как три шага.

**Split — разделить.** Значение `order_id` направляет каждую строку в группу:

```text
O1001 -> item-a, item-b
O1002 -> item-c
O1003 -> item-d
O1004 -> item-e
```

**Apply — применить агрегаты.** Внутри каждой группы отдельно считаются число строк,
число известных сумм и сумма известных значений.

**Combine — собрать.** Результаты групп соединяются в новую таблицу, где одна строка
соответствует одному `order_id`.

Слово `apply` в названии модели не требует метода `GroupBy.apply`. Для обычных сумм,
счётчиков, минимумов и средних pandas предоставляет специализированные агрегаты. Они
яснее показывают намерение и не заставляют самостоятельно определять форму результата
произвольной callback-функции.

### Один универсальный `count` не существует

Разные агрегаты отвечают на разные вопросы:

| Агрегат | Что считает | Как относится к пропуску |
|---|---|---|
| `size` | все строки группы | включает строки с пропущенным значением |
| `count` | известные значения выбранного столбца | исключает пропуски этого столбца |
| `nunique` | разные известные значения | по умолчанию исключает пропуск |
| `sum` | сумму известных значений | обычно пропускает `NA`; полностью пустая сумма может стать нулём |
| `mean` | среднее известных значений | исключает `NA` и меняет знаменатель |

На группе из трёх строк со значениями `P01`, `P01`, `NA` получатся разные ответы:

```text
size                 = 3  # строки
count(product_id)    = 2  # известные записи
nunique(product_id)  = 1  # разные известные продукты
```

Ни один ответ не «правильнее» остальных без вопроса. Имя выходной метрики должно
фиксировать смысл: `line_count`, `known_amount_lines`, `distinct_products`, а не
безымянный `count`.

### Пропуск в ключе и пропуск в значении — разные проблемы

Пропуск в обязательном `order_id` означает, что строку нельзя связать с заказом. Для
нашего входного контракта это ошибка до `groupby`; параметр `dropna=False` не превращает
неизвестный идентификатор в корректный заказ.

У необязательного аналитического разреза политика может быть другой. Например, при
группировке по необязательному `segment`:

- `dropna=True` исключит строки с неизвестным сегментом из групп;
- `dropna=False` сохранит отдельную группу неизвестного сегмента.

Это решение о популяции отчёта. Его нельзя автоматически переносить на обязательные
бизнес-ключи.

Пропуск в `line_total` имеет другой смысл: заказ известен, но одна из его частей пока не
оценена. Такой заказ должен остаться в результате вместе с evidence о неполноте.

### Сумма известных частей не всегда является суммой целого

Для `O1002` единственная `line_total` неизвестна. Обычный `sum` может вернуть `0.0`, хотя
данные не доказывают нулевую стоимость. При частично известном заказе проблема ещё
опаснее:

```text
известная позиция:     100.0
неизвестная позиция:   NA
обычная сумма:         100.0
полная сумма заказа:   неизвестна
```

Поэтому артефакт хранит два разных результата:

- `known_amount_total` — сумма известных позиций; `NA`, если не известно ни одной;
- `order_amount` — строгая сумма заказа; `NA`, если неизвестна хотя бы одна позиция.

Рядом сохраняются:

- `line_count` — все позиции;
- `known_amount_lines` — позиции с известной суммой;
- `missing_amount_lines` — позиции с неизвестной суммой.

Так частичная сумма не получает слишком сильное название, а пропуск не превращается в
ноль без предметного основания.

### Именованная агрегация связывает столбец, функцию и смысл

В pandas рабочий контракт удобно записать именованными агрегациями:

```python
grouped = items.groupby("order_id", as_index=False, sort=False).agg(
    line_count=("product_id", "size"),
    known_amount_lines=("line_total", "count"),
    known_amount_total=("line_total", "sum"),
)
```

Каждая строка внутри `.agg(...)` отвечает на три вопроса:

```text
как назвать результат?
из какого столбца взять значения?
какой агрегат применить?
```

`as_index=False` оставляет `order_id` обычным столбцом. Это удобно для проверки
бизнес-ключа и для `merge` в следующем уроке. Технический индекс результата не становится
бизнес-ключом только потому, что pandas умеет помещать group keys в индекс.

`sort=False` сохраняет порядок первого появления групп в учебном примере. На смысл
метрик порядок не влияет; если публикации нужна определённая сортировка, её задают
отдельным явным шагом.

### Среднее требует правильной единицы анализа

Средняя стоимость товарной позиции и средняя сумма заказа — разные метрики:

```text
mean(line_total) до агрегации       -> средняя стоимость позиции
mean(order_amount) после агрегации  -> средняя сумма полного заказа
```

Нельзя назвать первое значение `average_order_value`: заказы с большим числом позиций
получат больший вес.

Опасно и простое среднее групповых средних. Если одна группа содержит два значения
`10, 20`, а другая одно значение `100`, то:

```text
среднее первой группы       = 15
среднее второй группы       = 100
среднее двух средних        = 57.5
среднее всех трёх значений  = 43.33...
```

Для повторной агрегации среднего нужны как минимум исходная сумма и знаменатель. В этом
уроке достаточно распознать ловушку; полноценные продуктовые метрики появятся позже.

### Граница урока

Здесь мы изучаем только уменьшение детализации через встроенные агрегаты.

- `GroupBy.transform` сохраняет число строк и будет нужен для групповых признаков позже.
- `GroupBy.filter` отбирает целые группы и не заменяет построчную маску `03/03`.
- `GroupBy.apply` нужен для нестандартных задач, но не является стартовой точкой.
- `observed` относится к категориальным ключам; категории появятся в `03/09`.
- временные группировки появятся после дат и часовых поясов в `03/08`.
- объединение order-level результата с `orders` относится к следующему уроку `03/06`.

## Соберите это

Сначала воспроизведите агрегацию без pandas. Используйте те же пять строк, но представьте
их списком словарей:

```python
rows = [
    {"order_id": "O1001", "line_total": 800.0},
    {"order_id": "O1001", "line_total": 800.0},
    {"order_id": "O1002", "line_total": None},
    {"order_id": "O1003", "line_total": 75.5},
    {"order_id": "O1004", "line_total": 0.0},
]
```

До кода предскажите:

1. сколько групп получится;
2. сколько строк будет у `O1001`;
3. чем должны различаться `O1002` и `O1004`;
4. какой столбец станет ключом результата.

### Шаг 1. Разделите строки по ключу

```python
groups = {}

for row in rows:
    order_id = row["order_id"]
    groups.setdefault(order_id, []).append(row["line_total"])
```

Теперь структура выглядит так:

```text
O1001 -> [800.0, 800.0]
O1002 -> [None]
O1003 -> [75.5]
O1004 -> [0.0]
```

Этот словарь делает наблюдаемым шаг `split`. pandas построит группы эффективнее, но их
предметный смысл останется тем же.

### Шаг 2. Посчитайте полноту и известную сумму

```python
control = []

for order_id, values in groups.items():
    known = [value for value in values if value is not None]
    line_count = len(values)
    known_count = len(known)
    missing_count = line_count - known_count
    known_total = sum(known) if known else None
    order_total = known_total if missing_count == 0 else None

    control.append(
        {
            "order_id": order_id,
            "line_count": line_count,
            "known_amount_lines": known_count,
            "missing_amount_lines": missing_count,
            "known_amount_total": known_total,
            "order_amount": order_total,
        }
    )
```

Получится четыре записи:

```text
O1001 -> lines=2, known=2, missing=0, known_total=1600.0, order=1600.0
O1002 -> lines=1, known=0, missing=1, known_total=None,   order=None
O1003 -> lines=1, known=1, missing=0, known_total=75.5,  order=75.5
O1004 -> lines=1, known=1, missing=0, known_total=0.0,   order=0.0
```

Ручная версия медленнее `groupby`, но показывает все решения: ключ, состав групп,
знаменатели и правило неизвестной суммы. Готовая функция `manual_order_rollup` находится
в артефакте и используется как независимый контроль pandas-результата.

## Используйте это

### Шаг 1. Создайте уже типизированный вход

Не читайте и не очищайте CSV повторно. В реальном pipeline это результат `03/02` и
`03/04`:

```python
import pandas as pd

items = pd.DataFrame(
    {
        "order_id": ["O1001", "O1001", "O1002", "O1003", "O1004"],
        "product_id": ["P01", "P02", "P03", "P04", "P05"],
        "line_total": [800.0, 800.0, pd.NA, 75.5, 0.0],
    },
    index=["item-a", "item-b", "item-c", "item-d", "item-e"],
).astype(
    {
        "order_id": "string",
        "product_id": "string",
        "line_total": "Float64",
    }
)
```

Проверьте входные обещания:

```python
assert not items[["order_id", "product_id"]].isna().any().any()
assert not items.duplicated(["order_id", "product_id"]).any()
assert str(items["order_id"].dtype) == "string"
assert str(items["product_id"].dtype) == "string"
assert str(items["line_total"].dtype) == "Float64"
```

Отдельный `order_id` повторяется и не должен проходить проверку уникальности на этом
grain:

```python
assert items["order_id"].duplicated().any()
```

### Шаг 2. Выполните именованные агрегации

```python
result = (
    items.groupby("order_id", as_index=False, sort=False)
    .agg(
        line_count=("product_id", "size"),
        known_amount_lines=("line_total", "count"),
        known_amount_total=("line_total", "sum"),
    )
    .astype(
        {
            "order_id": "string",
            "line_count": "Int64",
            "known_amount_lines": "Int64",
            "known_amount_total": "Float64",
        }
    )
)
```

На этом шаге pandas уже создал одну строку на заказ, но политика неизвестной суммы ещё
не завершена.

### Шаг 3. Сделайте неполноту наблюдаемой

```python
result["missing_amount_lines"] = (
    result["line_count"] - result["known_amount_lines"]
).astype("Int64")
```

Эта разность связывает `size` и `count`:

```text
все строки - строки с известной суммой = строки с неизвестной суммой
```

### Шаг 4. Не выдавайте пустую или частичную сумму за полный итог

Сначала исправьте полностью пустую сумму:

```python
result["known_amount_total"] = result["known_amount_total"].mask(
    result["known_amount_lines"].eq(0),
    pd.NA,
)
```

Затем опубликуйте строгую сумму заказа только для полных групп:

```python
result["order_amount"] = result["known_amount_total"].mask(
    result["missing_amount_lines"].gt(0),
    pd.NA,
).astype("Float64")
```

Здесь `mask` из `03/04` используется уже после агрегации: он не меняет группы, а явно
задаёт предметную политику выходного значения.

### Шаг 5. Проверьте новый grain и reconciliation

```python
assert result["order_id"].is_unique
assert len(result) == items["order_id"].nunique()
assert result["line_count"].sum() == len(items)
assert result["known_amount_lines"].sum() == items["line_total"].notna().sum()
assert result["missing_amount_lines"].sum() == items["line_total"].isna().sum()
```

Ожидаемый результат:

```text
order_id  line_count  known_amount_lines  missing_amount_lines  known_amount_total  order_amount
O1001              2                   2                     0              1600.0        1600.0
O1002              1                   0                     1                <NA>          <NA>
O1003              1                   1                     0                75.5          75.5
O1004              1                   1                     0                 0.0           0.0
```

### Шаг 6. Запустите готовый артефакт

## Сломайте это

### Поломка 1. Запретить любой повтор `order_id`

```python
assert items["order_id"].is_unique
```

Проверка упадёт на корректном `O1001`, потому что применяет ключ выходного grain к
входной таблице товарных позиций. До агрегации проверяйте составную пару
`(order_id, product_id)`; после неё — отдельный `order_id`.

### Поломка 2. Пропустить настоящий дубль позиции

Если скопировать `item-a`, сумма `O1001` станет `2400.0`. `groupby` не знает, что пара
`O1001, P01` должна быть уникальна, и честно сложит обе строки. Входной ключ нужно
проверить до агрегата.

### Поломка 3. Повторно разобрать `line_total`

```python
line_total = pd.to_numeric(items["line_total"], errors="coerce")
```

Такой код скрывает нарушение контракта `03/02`: непустое `oops` станет неотличимо от
допустимого пропуска. Агрегация принимает `Float64` и отклоняет другой dtype вместо
повторной очистки.

### Поломка 4. Заменить неизвестную позицию нулём

```python
items["line_total"] = items["line_total"].fillna(0)
```

После этого `O1002` будет выглядеть полным заказом с суммой `0.0`. Ноль — известное
значение; `NA` — отсутствие знания. Их нельзя объединять без предметного правила.

### Поломка 5. Довериться нулю полностью пустой группы

```python
items.groupby("order_id")["line_total"].sum()
```

Технический результат для `O1002` может быть нулём. Без `count` невозможно понять,
получен ли ноль из реальных нулевых позиций или из отсутствующих значений. Поэтому сумма
публикуется вместе со знаменателями и явной completeness-политикой.

### Поломка 6. Перепутать `size`, `count` и `nunique`

```python
line_count = group["line_total"].count()
```

Такой `line_count` потеряет строки с неизвестной суммой. Для числа позиций используйте
`size`; для числа известных сумм — `count(line_total)`; для разных сущностей —
`nunique` соответствующего идентификатора.

### Поломка 7. Незаметно изменить популяцию через ключ с пропусками

```python
orders.groupby("segment").agg(...)
```

При стандартной политике строки с неизвестным `segment` не попадут ни в одну группу.
Если они должны оставаться в отчёте, используйте `dropna=False` и покажите отдельную
unknown-группу. Для обязательного `order_id` правильнее остановиться раньше с ошибкой.

### Поломка 8. Сохранить исходный grain лишним ключом

```python
items.groupby(["order_id", "product_id"]).agg(...)
```

Результат почти не уменьшит число строк. Формально агрегация выполнена, но таблица
по-прежнему описывает товарные позиции, а не заказы.

### Поломка 9. Назвать среднее позиции средним заказом

```python
average_order_value = items["line_total"].mean()
```

Это среднее по позициям. Заказы с несколькими товарами получают больший вес. Сначала
нужно получить одну полную сумму на заказ и только затем считать среднее по заказам.

### Поломка 10. Начать с `GroupBy.apply`

```python
items.groupby("order_id").apply(custom_summary)
```

Произвольная функция скрывает форму и dtype результата и заставляет вручную повторять
встроенные операции. Для `size`, `count` и `sum` используйте именованные агрегации.
`apply` оставьте задаче, которую действительно нельзя выразить готовыми агрегатами.

## Проверьте это

### Контрольный пример

Для пяти учебных строк подтвердите четыре группы и значения:

```text
O1001 -> 2 строки, 2 известны, 0 пропущено, полный итог 1600.0
O1002 -> 1 строка, 0 известны, 1 пропущена, полный итог неизвестен
O1003 -> 1 строка, 1 известна, 0 пропущено, полный итог 75.5
O1004 -> 1 строка, 1 известна, 0 пропущено, полный итог 0.0
```

Ручной `manual_order_rollup` и pandas-функция должны совпасть по каждой группе, а не
только по общей сумме.

### Входной контракт

- объект является `DataFrame`;
- столбцы `order_id`, `product_id`, `line_total` существуют ровно по одному разу;
- идентификаторы имеют dtype `string`, не пропущены и не пусты;
- `line_total` уже имеет dtype `Float64`;
- пара `(order_id, product_id)` уникальна;
- повтор отдельного `order_id` разрешён;
- входной объект не изменяется.

### Выходной контракт

- одна строка соответствует одному `order_id`;
- `order_id` уникален и остаётся обычным столбцом;
- счётчики имеют dtype `Int64`;
- суммы имеют dtype `Float64`;
- нулевая сумма остаётся `0.0`;
- полностью или частично неизвестный итог заказа остаётся `pd.NA`;
- пустой типизированный вход возвращает пустой типизированный результат без фиктивной
  группы.

### Reconciliation-инварианты

```text
sum(line_count) == число входных строк
sum(known_amount_lines) == число известных line_total
sum(missing_amount_lines) == число пропущенных line_total
line_count == known_amount_lines + missing_amount_lines для каждой группы
число строк результата == число уникальных order_id
```

Для полных заказов дополнительно:

```text
known_amount_total == order_amount
```

Для неполных заказов:

```text
missing_amount_lines > 0
order_amount is NA
```

## Поставьте результат

Готовый артефакт находится в
`phases/03-pandas/05-groupby/outputs/order_item_rollup.py`.

Его публичный контракт:

```python
aggregate_order_items(frame: pd.DataFrame) -> pd.DataFrame
```

Функция:

- принимает уже типизированный результат построчных преобразований `03/04`;
- проверяет входной grain `(order_id, product_id)`;
- разрешает законный повтор `order_id`;
- сворачивает позиции до одной строки на заказ;
- сохраняет число всех, известных и неизвестных позиций;
- разделяет сумму известных частей и строгую полную сумму заказа;
- возвращает явные nullable-dtype;
- проверяет reconciliation строк;
- не изменяет входной объект;
- не читает CSV, не очищает данные, не фильтрует популяцию и не использует
  `GroupBy.apply`.

Такой order-level результат можно безопасно передать следующему уроку. В `03/06` он
будет объединён с `orders`, а cardinality `one-to-one` станет проверяемым обещанием, а не
предположением.

## Ключевые термины

| Термин | Распространённое заблуждение | Точное значение |
|---|---|---|
| Group key | Любой удобный столбец | Поле или набор полей, задающий группы и grain результата |
| Aggregation | Любая операция после `groupby` | Операция, уменьшающая каждую группу до одного или нескольких итогов |
| `size` | Число известных значений | Число строк группы независимо от пропусков измеряемого столбца |
| `count` | Число строк | Число непропущенных значений выбранного столбца |
| `nunique` | То же, что `count` | Число разных известных значений |
| Named aggregation | Косметическое переименование | Контракт «имя результата -> исходный столбец -> агрегат» |
| Reconciliation | Повторить ту же формулу | Независимо сверить строки, знаменатели и totals до и после изменения grain |
| Complete total | Сумма известных частей | Итог, для которого известны все обязательные составляющие |
