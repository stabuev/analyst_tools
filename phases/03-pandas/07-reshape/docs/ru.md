# Pivot, melt и tidy data

> Reshape меняет расположение фактов в таблице, но не должен менять их смысл,
> терять идентификаторы или прятать агрегацию.

**Тип:** Build

**Треки:** Core

**Пререквизиты:** 03/06

**Время:** ~90 минут

**Результат:** объявляет grain wide- и long-таблиц, переносит уровни одной переменной
между названиями столбцов и строками, проверяет уникальность ключа pivot-ячейки и
отделяет изменение формы от агрегации.

## Цели обучения

После урока вы сможете:

- объяснить wide- и long-форму как два расположения одних и тех же фактов;
- до преобразования назвать grain входа, grain результата и полный идентификатор строки;
- определить, какие столбцы являются уровнями одной переменной и действительно могут
  быть собраны в общий столбец значений;
- вручную предсказать строки после `melt`, включая строки с пропущенным измерением;
- использовать `melt` с явными `id_vars`, `value_vars`, `var_name` и `value_name`;
- сформулировать составной ключ будущей pivot-ячейки и доказать его уникальность;
- применять `pivot` только для изменения формы без агрегации;
- отличать `pivot` от `pivot_table` и выбирать агрегат по смыслу наблюдений;
- не подменять отсутствующую комбинацию нулём без предметного основания;
- сверять количество строк, идентификаторы, значения и пропуски после reshape.

## Проблема

Есть небольшая таблица количества заказов по статусам:

| `user_id` | `currency` | `paid` | `refunded` | `pending` |
|---|---|---:|---:|---:|
| `U1` | `RUB` | 2 | 0 | 1 |
| `U2` | `RUB` | 1 | пропуск | 0 |

Её grain:

```text
одна строка -> один пользователь в одной валюте
кандидатный ключ -> user_id + currency
```

Значения `paid`, `refunded` и `pending` отвечают на один вопрос — сколько заказов было в
конкретном статусе. Различается только сам статус, но сейчас он зашит в названия
столбцов.

Для графика с цветом по статусу удобнее такая форма:

| `user_id` | `currency` | `status` | `order_count` |
|---|---|---|---:|
| `U1` | `RUB` | `paid` | 2 |
| `U1` | `RUB` | `refunded` | 0 |
| `U1` | `RUB` | `pending` | 1 |
| `U2` | `RUB` | `paid` | 1 |
| `U2` | `RUB` | `refunded` | пропуск |
| `U2` | `RUB` | `pending` | 0 |

Новый grain:

```text
одна строка -> один статус пользователя в одной валюте
кандидатный ключ -> user_id + currency + status
```

Строк стало больше, но заказов не прибавилось. Значение `2` не пересчитано: оно лишь
переместилось из ячейки столбца `paid` в строку со `status="paid"`.

Опасность начинается, если воспринимать reshape как косметическую перестановку. Можно:

- забыть `currency` и потерять часть идентичности исходной строки;
- собрать в один `value` выручку в рублях и длительность в минутах;
- удалить пропущенное значение и изменить наблюдаемую популяцию;
- получить несколько значений для одной будущей ячейки;
- скрыть конфликт через случайную агрегацию `pivot_table`;
- заменить отсутствующую комбинацию нулём без доказательства.

Все эти варианты могут дать аккуратную таблицу без синтаксической ошибки.

## Концепция

### Wide и long описывают расположение данных

**Wide-форма**, или широкая форма, хранит уровни одной переменной в нескольких названиях
столбцов. В примере `paid`, `refunded` и `pending` являются уровнями переменной `status`.

**Long-форма**, или длинная форма, хранит эти уровни в строках одного столбца `status`, а
соответствующие измерения — в `order_count`.

Переход можно представить как два встречных движения:

```text
melt:  название столбца -> значение в строке
pivot: значение в строке -> название столбца
```

При чистом reshape значение не пересчитывается. Меняются оси таблицы и grain её строки.

### Не всякая wide-таблица плохая и не всякая long-таблица tidy

Tidy data — это не команда «сделать таблицу как можно длиннее». У модели есть три
ориентира:

1. каждая переменная хранится в своём столбце;
2. каждое наблюдение хранится в своей строке;
3. каждый тип наблюдаемой сущности хранится в своей таблице.

Столбцы `revenue` и `order_count` являются разными переменными с разными единицами
измерения. Их нормально оставить рядом:

| `user_id` | `revenue` | `order_count` |
|---|---:|---:|
| `U1` | 4200.0 | 3 |

Если механически собрать их в `metric` и `value`, столбец `value` начнёт смешивать рубли
и штуки. Форма станет длиннее, но смысл — слабее.

Обратно, `paid`, `refunded` и `pending` — не три разные метрики. Это три уровня одной
переменной `status`, измеряемые одной величиной `order_count`. Их можно безопасно
собрать, если они имеют общий смысл, единицу измерения и согласованный dtype.

Перед `melt` задайте вопрос:

> Я объединяю разные уровни одной переменной или просто разные числовые столбцы?

Название метода не может ответить на этот предметный вопрос вместо аналитика.

### Идентификаторы должны полностью сохранять исходную строку

`id_vars` — столбцы, которые связывают каждую новую long-строку с исходным наблюдением.
В нашем примере это `user_id` и `currency` вместе.

Если один пользователь может иметь строки в нескольких валютах, одного `user_id`
недостаточно. После удаления `currency` разные wide-строки будут выглядеть одинаково.
Это та же ошибка неполного составного ключа, которую мы разбирали для `merge` в `03/06`.

Технический `RangeIndex` не заменяет идентификаторы. `melt` по умолчанию создаёт новый
индекс. Если метка несёт бизнес-смысл, сначала сделайте её обычным проверяемым столбцом;
если не несёт — не пытайтесь сохранять её как идентичность наблюдения.

### Число строк после `melt` можно предсказать

Пусть:

- `R` — число wide-строк;
- `M` — число явно выбранных `value_vars`.

Тогда:

```text
число long-строк = R * M
```

Для двух пользователей и трёх статусов получится `2 * 3 = 6` строк.

Пропущенное измерение тоже занимает ячейку исходной таблицы. Поэтому `melt` сохраняет
его как long-строку с `NA`; оно не уменьшает ожидаемое число строк. Если затем вызвать
`dropna`, это уже отдельное решение о популяции, а не часть reshape.

Явный `value_vars` важен ещё по одной причине. Если его не указать, pandas возьмёт все
столбцы, кроме `id_vars`. После появления нового служебного столбца он тоже может
незаметно попасть в long-таблицу.

### Ключ pivot-ячейки продолжает модель составного ключа

При обратном преобразовании значения `status` станут заголовками. Чтобы найти значение
конкретной ячейки, нужны:

```text
ключ строки будущей wide-таблицы + будущий столбец
```

Для примера:

```text
(user_id, currency) + status
= (user_id, currency, status)
```

Эта комбинация должна соответствовать не более чем одной строке long-таблицы. Если для
`(U1, RUB, paid)` есть значения `2` и `3`, pandas не может узнать, какое из них поместить
в ячейку. `pivot` правильно остановится: операция обещала изменить форму, а не выбрать
правило объединения фактов.

### `pivot` и `pivot_table` отвечают на разные вопросы

| Операция | Что делает | Требование к ключу ячейки |
|---|---|---|
| `pivot` | Только меняет форму | Ключ уже уникален |
| `pivot_table` | Сначала агрегирует, затем меняет форму | Повторы допустимы, но нужен осмысленный `aggfunc` |

`pivot_table` не является «более терпеливым pivot». Это агрегация из `03/05`, совмещённая
с изменением формы. У неё по умолчанию используется среднее. Среднее может быть
математически вычислимо и одновременно не отвечать бизнес-вопросу.

Надёжная модель:

```text
повтор ключа ячейки — ошибка данных
-> исправить источник или полный ключ

повтор ключа ячейки — несколько ожидаемых наблюдений
-> выбрать агрегат по смыслу
-> явно агрегировать
-> проверить уникальный новый grain
-> выполнить pivot
```

Раздельные шаги особенно полезны во время обучения и отладки: виден результат
агрегации, а `pivot` остаётся проверкой её grain.

### Пропуск в wide-ячейке не доказывает ноль

После `pivot` пустая ячейка может означать как минимум две ситуации:

- такой комбинации ключей вообще не было в long-таблице;
- комбинация была, но её измеренное значение неизвестно.

В wide-представлении обе ситуации могут выглядеть как `NA`. Значение `0` можно подставить
только когда предметный контракт доказывает: отсутствие строки действительно означает
ноль. Например, в полном реестре заказов отсутствие заказов определённого статуса может
означать нулевое количество. Для выручки из неполной выгрузки то же решение уже опасно.

У этого различия есть следствие: обратимость зависит от структуры входа. Прямоугольную
wide-таблицу можно перевести в long и вернуть обратно без потери ячеек. Но если исходная
long-таблица была разреженной и не содержала некоторых комбинаций, `pivot -> melt` может
создать для них явные строки с `NA`. Поэтому сверяйте не только значения, но и ожидаемое
множество ключей.

### Граница урока

Здесь нужны только две базовые операции:

- `melt` для wide → long;
- `pivot` для проверенного long → wide.

Мы не изучаем сейчас `stack`, `unstack`, устройство `MultiIndex`, `wide_to_long`,
`explode`, `crosstab` и pivot по нескольким столбцам значений. Эти механизмы полезны в
отдельных задачах, но не улучшают центральную модель и легко превращают первый урок о
reshape в справочник API.

## Соберите это

Сначала выполните преобразование без pandas.

Исходная строка:

```python
wide_row = {
    "user_id": "U1",
    "currency": "RUB",
    "paid": 2,
    "refunded": 0,
    "pending": 1,
}
statuses = ["paid", "refunded", "pending"]
```

Одна wide-строка должна породить три long-строки:

```python
manual_long = [
    {
        "user_id": wide_row["user_id"],
        "currency": wide_row["currency"],
        "status": status,
        "order_count": wide_row[status],
    }
    for status in statuses
]
```

В этой реализации виден весь механизм:

1. идентификаторы копируются в каждую новую строку;
2. название исходного measured-столбца становится значением `status`;
3. содержимое ячейки становится `order_count`;
4. одна wide-строка создаёт столько long-строк, сколько выбрано статусов.

Теперь проделайте то же со второй исходной строкой. Не удаляйте `refunded=<NA>`: это
одна из шести ожидаемых ячеек. После ручной сборки проверьте:

```text
grain -> user_id + currency + status
строк -> 2 * 3 = 6
ключ  -> (user_id, currency, status) уникален
NA    -> одно неизвестное измерение сохранено
```

## Используйте это

### Wide → long через `melt`

Соберём тот же пример как уже типизированный `DataFrame`:

```python
import pandas as pd

wide = pd.DataFrame(
    {
        "user_id": ["U1", "U2"],
        "currency": ["RUB", "RUB"],
        "paid": [2, 1],
        "refunded": [0, pd.NA],
        "pending": [1, 0],
    }
).astype(
    {
        "user_id": "string",
        "currency": "string",
        "paid": "Int64",
        "refunded": "Int64",
        "pending": "Int64",
    }
)
```

Прямой вызов pandas:

```python
long = wide.melt(
    id_vars=["user_id", "currency"],
    value_vars=["paid", "refunded", "pending"],
    var_name="status",
    value_name="order_count",
    ignore_index=True,
)
```

Каждый параметр фиксирует часть контракта:

| Параметр | Обещание |
|---|---|
| `id_vars` | Эти столбцы полностью идентифицируют wide-строку |
| `value_vars` | Эти столбцы являются уровнями одной переменной |
| `var_name` | Так называется переменная, чьи уровни были в заголовках |
| `value_name` | Так называется единое измеряемое значение |
| `ignore_index=True` | Старый технический индекс не является идентичностью результата |

Учебный артефакт усиливает этот вызов проверками:

```python
from outputs.reshape_contract import to_long

long = to_long(
    wide,
    id_vars=["user_id", "currency"],
    value_vars=["paid", "refunded", "pending"],
    variable_name="status",
    value_name="order_count",
)
```

Он отклонит неполный или повторяющийся идентификатор, пересечение ролей столбцов,
несовместимые dtype измерений, конфликт выходных имён и дублирующиеся названия столбцов.
После `melt` он сверит ожидаемое число строк и количество пропусков.

Строгая проверка одинакового dtype не доказывает общий смысл `paid` и `refunded`; это
обязан доказать аналитик. Она лишь не позволяет незаметно сложить, например, `Int64` и
`Float64` в один столбец после уже объявленного dtype-контракта.

### Long → wide через проверенный `pivot`

Сначала назовите ключ будущей ячейки:

```python
cell_key = ["user_id", "currency", "status"]
assert not long.duplicated(cell_key).any()
```

Затем выполните чистое изменение формы:

```python
from outputs.reshape_contract import pivot_unique

wide_again = pivot_unique(
    long,
    index=["user_id", "currency"],
    columns="status",
    values="order_count",
)
```

`pivot_unique` не принимает `aggfunc` и не заполняет пропуски. Если ключ ячейки
повторяется, функция показывает примеры конфликтных ключей и останавливается.

Порядок новых wide-столбцов не является бизнес-смыслом преобразования и может не
совпасть с исходным. Для отчёта задайте его явно:

```python
expected_columns = [
    "user_id",
    "currency",
    "paid",
    "refunded",
    "pending",
]
wide_again = wide_again.loc[:, expected_columns]
```

Теперь можно сверить round trip:

```python
pd.testing.assert_frame_equal(
    wide_again,
    wide.reset_index(drop=True),
    check_names=False,
)
```

### Когда сначала действительно нужна агрегация

Рассмотрим не готовые количества, а строки заказов:

| `order_id` | `user_id` | `status` |
|---|---|---|
| `O1` | `U1` | `paid` |
| `O2` | `U1` | `paid` |
| `O3` | `U1` | `refunded` |

Здесь grain — один заказ, поэтому повтор `(U1, paid)` ожидаем. Будущая ячейка должна
содержать количество разных заказов. Сначала принимаем решение об агрегации:

```python
status_counts = (
    orders.groupby(
        ["user_id", "status"],
        as_index=False,
        sort=False,
    )
    .agg(order_count=("order_id", "nunique"))
)
```

Теперь grain результата — одна пара `(user_id, status)`, её ключ уникален, и можно
использовать `pivot_unique`:

```python
status_matrix = pivot_unique(
    status_counts,
    index=["user_id"],
    columns="status",
    values="order_count",
)
```

То же можно записать короче через `pivot_table`, но агрегат всё равно должен быть явным:

```python
status_matrix = orders.pivot_table(
    index=["user_id"],
    columns="status",
    values="order_id",
    aggfunc="nunique",
    observed=True,
)
```

Раздельный вариант `groupby -> pivot` лучше показывает два разных решения: сначала что
считать, затем как расположить результат. Для количества заказов отсутствующую
комбинацию можно заменить нулём только если `orders` является полным реестром выбранной
популяции и каждый заказ обязан иметь один известный статус.

## Сломайте это

### 1. Удалите часть идентификатора

Добавьте для `U1` две wide-строки в разных валютах и передайте только
`id_vars=["user_id"]`. Строгий артефакт обнаружит, что `user_id` не идентифицирует одну
wide-строку. Исправление — вернуть `currency`, а не удалить «дубликат».

### 2. Объедините разные переменные

Попробуйте собрать `revenue` в рублях и `delivery_minutes` в общий `value`. Даже если
привести оба столбца к одному dtype, общий столбец не получит общей единицы измерения.
Проверка dtype ловит часть ошибок, но семантический контракт остаётся вашей задачей.

### 3. Не указывайте `value_vars`

Добавьте служебный столбец `data_quality_flag` и вызовите обычный `melt` только с
`id_vars`. pandas включит все остальные столбцы. Артефакт требует явный непустой список
измерений и не допускает такого расширения по умолчанию.

### 4. Спрячьте идентичность в индексе

Назначьте строкам красивые метки, не создавая бизнес-столбец, и выполните `melt` с
`ignore_index=True`. Метки исчезнут. Это не потеря бизнес-данных, если индекс был
техническим; если он был значимым, входной контракт был описан неверно.

### 5. Удалите `NA` после `melt`

`long.dropna(subset=["order_count"])` сократит шесть ожидаемых строк до пяти. Иногда это
нужная популяция для графика, но тогда решение должно называться фильтрацией и иметь
отдельную сверку, а не маскироваться под reshape.

### 6. Продублируйте ключ ячейки

Добавьте второе значение для `(U1, RUB, paid)`. `pivot_unique` остановится. Не заменяйте
его автоматически на `pivot_table`: сначала выясните, является ли строка дублем данных,
пропущенной частью ключа или ожидаемым отдельным наблюдением.

### 7. Положитесь на `pivot_table` без `aggfunc`

По умолчанию будет выбрано среднее. Для суммы выручки, количества заказов или последнего
статуса это разные и обычно неверные решения. Явно назовите агрегат и объясните, какую
единицу анализа он создаёт.

### 8. Заполните все пропуски нулями

Сравните две причины `NA`: отсутствующая комбинация и существующее, но неизвестное
измерение. Универсальный `fillna(0)` делает их неразличимыми и превращает неизвестность
в доказанный факт.

## Проверьте это

Перед `melt`:

- назван grain wide-таблицы;
- `id_vars` являются полным уникальным ключом wide-строки;
- обязательные идентификаторы не пропущены и не пусты;
- `value_vars` перечислены явно;
- все `value_vars` описывают уровни одной переменной, одну единицу измерения и один
  объявленный dtype;
- имена будущих `variable` и `value` столбцов не конфликтуют со входом.

После `melt`:

- grain дополнен новой переменной;
- строк ровно `len(wide) * len(value_vars)`;
- каждый исходный идентификатор повторён `len(value_vars)` раз;
- число измеренных ячеек и пропусков согласуется со входом;
- технический индекс не используется как бизнес-ключ;
- входной `DataFrame` не изменён.

Перед `pivot`:

- названы ключ строки wide-результата и столбец, чьи значения станут заголовками;
- полный ключ ячейки `index + columns` не содержит пропусков и уникален;
- повторы не скрываются неявной агрегацией;
- отдельно решено, означает ли отсутствие комбинации ноль или неизвестность.

После `pivot`:

- число строк равно числу уникальных комбинаций `index`;
- ключ wide-строки уникален;
- значения и пропуски не пересчитаны;
- порядок столбцов задан явно, если он является контрактом публикации;
- для прямоугольного входа round trip возвращает исходные факты.

Запустите поведенческие тесты урока:

Тесты проверяют не только успешный пример, но и неполный идентификатор, разные dtype,

типизированный пустой вход, round trip и отсутствие мутации.

## Поставьте результат

Артефакт `outputs/reshape_contract.py` содержит две чистые функции:

- `to_long` проверяет полный идентификатор wide-строки, роли и dtype столбцов, выполняет
  `melt` и сверяет строки с пропусками;
- `pivot_unique` проверяет ключ будущей ячейки, выполняет `pivot` без агрегации и
  сохраняет пропуски как пропуски.

Артефакт не читает файлы, не очищает категории, не агрегирует сырые заказы и не содержит
CLI. Он принимает уже подготовленный `DataFrame`, поэтому его можно встроить в
аналитический pipeline именно на шаге изменения формы.

Минимальный запуск примера:

```bash
uv run --locked python phases/03-pandas/07-reshape/code/main.py
```

## Ключевые термины

| Термин | Распространённое заблуждение | Точное значение |
|---|---|---|
| Wide-форма | Любая таблица с большим числом столбцов | Представление, где уровни переменной могут находиться в названиях столбцов |
| Long-форма | Автоматически правильная tidy-таблица | Представление, где уровни переменной хранятся в строках |
| Tidy data | Все метрики нужно собрать в `metric/value` | Переменные, наблюдения и типы сущностей имеют согласованное место |
| Identifier variable | Любой столбец, который не расплавили | Полный набор полей, сохраняющий идентичность исходного наблюдения |
| Measured variable | Любой числовой столбец | Одна измеряемая величина с общим смыслом, единицей и dtype |
| Ключ pivot-ячейки | Только `index` или только `columns` | Комбинация всех `index` и значения `columns`, адресующая одну ячейку |
| `pivot` | Умеет сам решить судьбу дублей | Чистое изменение формы, требующее уникального ключа ячейки |
| `pivot_table` | Более удобный вариант `pivot` | Агрегация с последующим изменением формы |
