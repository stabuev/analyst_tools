# Pivot, melt и tidy data

> Reshape меняет расположение значений, но не должен прятать неоднозначность идентификаторов.

**Тип:** Build
**Треки:** Core
**Пререквизиты:** 03/06
**Время:** ~90 минут
**Результат:** переводит wide и long формы и отклоняет неоднозначный pivot.

## Цели обучения

- Отличать identifier variables от measured variables.
- Предсказывать число строк после `melt`.
- Проверять уникальность pivot-ячеек.
- Выбирать long или wide форму под задачу.

## Проблема

В wide-таблице `paid`, `refunded`, `pending` находятся в именах столбцов. Для графика или
универсальной агрегации удобнее две колонки `status` и `orders`. Обратный pivot возможен
только тогда, когда каждой паре `user_id,status` соответствует одно значение. Иначе
`pivot_table` молча агрегирует конфликт, если функция не выбрана осознанно.

## Концепция

Tidy-форма хранит одну переменную в одном столбце и одно наблюдение в одной строке.
`melt` повторяет идентификаторы для каждого измеряемого столбца. `pivot` переносит
значения категории в заголовки, но требует уникального ключа ячейки.

Если дубли требуют суммы или среднего, это отдельная агрегация с отдельным решением, а не
техническая деталь reshape.

## Соберите это

Для двух пользователей и трёх метрик вручную постройте шесть long-строк. Затем составьте
ключ будущей ячейки `(user_id, metric)` и убедитесь, что он уникален перед обратным
поворотом.

## Используйте это

```python
long = wide.melt(
    id_vars=["user_id"],
    value_vars=["paid", "refunded"],
    var_name="status",
    value_name="orders",
)
wide_again = long.pivot(index="user_id", columns="status", values="orders")
```

Артефакт сначала проверяет ключи, затем вызывает `pivot`.

## Сломайте это

Продублируйте пару `user_id,status` с другим значением. `pivot_unique` должен остановиться.
Удалите identifier, добавьте пропуск в нём и сравните число строк до и после `melt`.

## Проверьте это

- Все identifier-столбцы сохранены.
- Число long-строк равно `rows * metrics`.
- Ключ ячейки уникален до pivot.
- После round trip значения не изменились.
- Скрытая агрегация не используется.

`outputs/reshape_contract.py` содержит безопасные `to_long`, `pivot_unique` и пример

## Упражнения

1. Переведите сегментные метрики из wide в long.
2. Добавьте второй identifier `currency`.
3. Реализуйте осознанный `pivot_table` с суммой и отдельным тестом дублей.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Tidy data | Только длинная таблица | Переменные в столбцах, наблюдения в строках |
| Identifier | Любой неизменённый столбец | Поле, сохраняющее личность наблюдения |
| Pivot cell | Одна исходная строка | Комбинация index и columns будущей таблицы |
| Hidden aggregation | Удобное исправление дублей | Неявное изменение смысла нескольких наблюдений |
