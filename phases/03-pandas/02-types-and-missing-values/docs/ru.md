# Типы данных и пропуски

> Dtype — часть контракта данных: молчаливое приведение меняет допустимые операции и смысл пропуска.

**Тип:** Build
**Треки:** Core
**Пререквизиты:** 03/01
**Время:** ~90 минут
**Результат:** приводит столбцы к nullable dtype и обнаруживает значения, которые потерялись бы при приведении.

## Цели обучения

- Отличать `str`, `string`, `object` и категориальные данные.
- Использовать `Int64`, `Float64` и `boolean` с `pd.NA`.
- Нормализовать timestamps с разными offsets в UTC.
- Не превращать ошибочные значения в пропуски незаметно.

## Проблема

CSV не хранит dtype. Значения `1200.00`, пустая строка и `oops` сначала могут оказаться
строками. Вызов `to_numeric(errors="coerce")` превратит и допустимый пропуск, и ошибку
`oops` в одно значение `NaN`. После этого невозможно понять, что было отсутствием факта,
а что нарушением контракта.

## Концепция

Тип определяет множество допустимых значений и операций. Nullable-типы pandas отделяют
пропуск от необходимости переходить к `float64`: целые значения остаются `Int64`,
логические — `boolean`. В pandas 3 строки по умолчанию используют новый `str` dtype, но
для явного контракта удобно задавать `string`.

Пропуск не имеет универсального решения. В `amount` пустота допустима для pending-заказа,
но `oops` — ошибка. В ключе `order_id` пропуск недопустим. Решение принимается по
семантике столбца, а не одной командой `fillna`.

## Соберите это

Для каждого столбца выпишите:

1. семантический тип;
2. допустим ли пропуск;
3. какие строки считаются валидными;
4. что делать с неразбираемым значением.

Ручной конвертер сначала разбирает значение, затем сравнивает неразобранные позиции с
исходными непустыми строками. Только истинно пустые значения могут стать `pd.NA` без
ошибки.

## Используйте это

```python
amount = pd.to_numeric(orders["amount"], errors="coerce").astype("Float64")
ordered_at = pd.to_datetime(
    orders["ordered_at"],
    errors="coerce",
    format="mixed",
    utc=True,
)
```

Артефакт делает проверку строгой:

```bash
uv run --locked python outputs/dtype_audit.py \
  ../data/tiny/orders.csv \
  --schema '{"order_id":"string","amount":"Float64","ordered_at":"datetime_utc"}'
```

## Сломайте это

Замените сумму на `unknown`, целое количество на `1.5`, boolean на `perhaps`, а дату на
`yesterday`. Все четыре значения должны завершать проверку ошибкой, а не новым пропуском.
Отдельно проверьте пустую строку: если контракт разрешает пропуск, она должна сохраниться
как `pd.NA`.

## Проверьте это

- dtype после конвертации совпадает с контрактом.
- Непустые неразбираемые строки вызывают ошибку.
- Дробное значение не обрезается при переводе в `Int64`.
- Смешанные UTC offsets становятся одной временной шкалой.
- Число пропусков до и после объяснимо.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Nullable dtype | Обычный NumPy dtype | pandas-тип, который хранит `pd.NA` |
| Coercion | Безопасная очистка | Приведение, способное скрыть ошибку как пропуск |
| UTC | Часовой пояс пользователя | Единая временная шкала для сравнения моментов |
| Missingness | Одно техническое значение | Семантическое отсутствие факта с правилами по столбцу |
