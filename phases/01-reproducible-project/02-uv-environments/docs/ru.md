# Окружения и зависимости с uv

> В Git хранят не готовую `.venv`, а достаточный рецепт, по которому `uv` заново
> создаёт окружение и запускает проект.

**Тип:** Build
**Треки:** Core
**Пререквизиты:** 01/01 — Версии Python и совместимость
**Время:** ~75 минут
**Результат:** добавляет зависимость через `uv`, запускает проект в его окружении и
восстанавливает то же рабочее состояние из lock-файла

## Цели обучения

- Объяснять простыми словами, зачем проекту отдельное окружение.
- Различать требования в `pyproject.toml`, зафиксированное разрешение в `uv.lock` и
  установленные пакеты в `.venv`.
- Добавлять зависимость командой `uv add`, а не только устанавливать её на своей машине.
- Запускать код через `uv run` без обязательной ручной активации окружения.
- Использовать `--locked`, когда lock-файл не должен незаметно измениться.
- Удалять `.venv` и восстанавливать работоспособный проект через `uv sync --locked`.
- Объяснять, почему `uv.lock` хранят в Git, а `.venv` — нет.

## Проблема

В предыдущем уроке в `revenue-project` появились:

- `requires-python = ">=3.11"` — минимальное требование проекта;
- `.python-version` — выбранная для локальной работы minor-линия Python;
- наблюдение фактически запущенного интерпретатора.

Но проект пока не отвечает на следующий вопрос: **какие внешние библиотеки нужны коду и
как коллеге установить тот же набор?**

Представьте, что аналитик добавил в расчёт:

```python
import numpy as np
```

На его компьютере скрипт работает: `numpy` когда-то установили глобально для другого
notebook. Коллега клонирует репозиторий и получает:

```text
ModuleNotFoundError: No module named 'numpy'
```

Совет `pip install numpy` устраняет ошибку только на одной машине и только сейчас. Из него
непонятно:

- является ли `numpy` обязательной зависимостью проекта;
- какой диапазон версий допустим;
- какую конкретную версию выбрал автор;
- с каким Python она была разрешена;
- как повторить установку позже;
- не запустился ли код случайно из другого окружения.

Копировать готовый каталог `.venv` тоже неправильно. Внутри него находятся пути и файлы,
связанные с конкретной операционной системой, архитектурой и установкой Python. Это
расходный локальный каталог, а не переносимый архив проекта.

Нужен воспроизводимый процесс:

```text
требование Python + зависимости проекта + lock-файл + код и sample
                              ↓
                         uv sync
                              ↓
                    локальная .venv
                              ↓
                         uv run
                              ↓
                   проверенный результат
```

## Концепция

### Окружение отвечает на вопрос «что видит этот проект»

На одном компьютере могут одновременно существовать несколько проектов:

```text
sales-report  → Python 3.12 + numpy 2.x
old-dashboard → Python 3.10 + numpy 1.x
```

Если устанавливать всё в один глобальный Python, требования проектов начинают влиять друг
на друга. Обновление библиотеки ради одного проекта может сломать другой. Виртуальное
окружение даёт проекту отдельное место для интерпретатора, библиотек и команд.

Для `uv` проектное окружение по умолчанию находится в `.venv`:

```text
revenue-project/
├── .venv/
│   ├── bin/       # Scripts/ в Windows
│   └── lib/
├── pyproject.toml
└── uv.lock
```

Изоляция не означает виртуальную машину или отдельную операционную систему. `.venv`
изолирует прежде всего Python-пакеты и команды проекта. Системные библиотеки, ОС и
аппаратная архитектура остаются внешними условиями.

### Три состояния, которые нельзя смешивать

#### 1. Декларация: `pyproject.toml`

В `[project].dependencies` записано, что разрешено установить:

```toml
[project]
name = "revenue-project"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "numpy>=2,<3",
]
```

Диапазон `>=2,<3` означает: проекту подходит версия NumPy не ниже 2.0, но ниже 3.0. Это
требование, а не утверждение о фактически установленной версии.

#### 2. Разрешение: `uv.lock`

Несколько зависимостей могут предъявлять требования друг к другу. **Разрешить
зависимости** — значит найти совместимый набор конкретных версий и источников. Результат
`uv` записывает в `uv.lock`.

Lock-файл:

- создаётся и обновляется инструментом;
- содержит точные выбранные версии полного графа;
- учитывает ограничения Python и платформенные условия;
- хранится в Git;
- не редактируется вручную.

`uv.lock` является кроссплатформенным: он может описывать варианты для нескольких
платформ. Это не обещание, что двоичные файлы на macOS и Linux совпадут побайтно. Это
единое зафиксированное разрешение, из которого `uv` выбирает подходящие артефакты для
конкретной поддерживаемой машины.

#### 3. Установленное состояние: `.venv`

`.venv` — то, что сейчас действительно установлено локально. Каталог можно удалить и
получить заново. Он:

- зависит от машины и выбранного интерпретатора;
- может отстать от `pyproject.toml` и `uv.lock`;
- не является источником правды;
- не хранится в Git.

Короткая модель:

| Уровень | Вопрос | Пример |
|---|---|---|
| `pyproject.toml` | Что проект допускает? | `numpy>=2,<3` |
| `uv.lock` | Что именно разрешил `uv`? | конкретная версия NumPy и источник |
| `.venv` | Что установлено на этой машине? | файлы выбранной сборки NumPy |

### Прямая и транзитивная зависимость

Если код проекта импортирует `numpy`, NumPy — **прямая зависимость**: проект объявляет её
сам. Если NumPy или другая прямая библиотека требует ещё один пакет, тот становится
**транзитивной зависимостью**.

В `pyproject.toml` обычно перечисляют прямые требования проекта. `uv.lock` фиксирует весь
получившийся граф. Не нужно вручную переносить каждую транзитивную библиотеку в
`dependencies`: её владелец уже описал связь в metadata своего пакета.

### `uv add`, `uv sync` и `uv run` решают разные задачи

Команда:

```bash
uv add 'numpy>=2,<3'
```

добавляет требование проекта, обновляет lock-файл и синхронизирует окружение. Это команда
для осознанного изменения контракта.

Команда:

```bash
uv sync
```

приводит `.venv` к состоянию проекта. Если окружения нет, создаёт его. Если lock-файла
ещё нет, обычный `sync` может создать или обновить его.

Команда:

```bash
uv run python src/revenue_summary.py data/sample/orders.csv
```

запускает процесс в проектном окружении. Перед запуском обычный `uv run` проверяет и при
необходимости обновляет lock-файл и среду.

Для строгого воспроизведения используйте:

```bash
uv run --locked python src/revenue_summary.py data/sample/orders.csv
```

`--locked` означает: lock-файл должен существовать и соответствовать metadata проекта.
Если он устарел, команда завершится ошибкой вместо скрытого обновления.

### Активация полезна, но не обязательна

Окружение можно активировать:

```bash
source .venv/bin/activate
```

В PowerShell команда выглядит иначе:

```powershell
.venv\Scripts\Activate.ps1
```

После активации обычное имя `python` указывает на Python окружения, пока открыт этот
shell. Но состояние shell легко забыть или потерять при открытии нового терминала.

Воспроизводимая инструкция может обходиться без активации:

```bash
uv run python -c 'import sys; print(sys.executable)'
```

Поэтому в курсе основной путь — явный `uv run`. Активация остаётся удобной для редактора
или продолжительной интерактивной работы.

### Воспроизводимость шире одного lock-файла

`uv.lock` фиксирует Python-зависимости, но не хранит:

- исходный код;
- входные данные;
- переменные окружения и секреты;
- системные библиотеки;
- внешние сервисы;
- доказательство корректности расчёта.

Поэтому мы проверим не только успешную установку NumPy, но и запуск контрольного расчёта
на синтетическом sample. Совпадение установки без совпадения результата ещё не доказывает
работоспособность проекта.


Первый путь зависит от вашего shell. Второй должен вести в `.venv` текущего проекта.
Теперь проверьте Git:


В статусе появится `uv.lock`, но не содержимое `.venv`.

Скопируйте готовый контрольный скрипт урока:

Попробуйте строгий запуск **до** объявления NumPy:

Ожидается `ModuleNotFoundError`: чистое проектное окружение честно показывает, что

### Шаг 5. Добавьте прямую зависимость через `uv`

```bash
uv add 'numpy>=2,<3'
```

Посмотрите не только сообщение команды, но и изменившиеся состояния:

```bash
git diff -- pyproject.toml uv.lock
uv tree
uv run python -c 'import numpy; print(numpy.__version__)'
```

В `pyproject.toml` появится допустимый диапазон. В `uv.lock` будет записана конкретная
разрешённая версия. В `.venv` появятся установленные файлы этой версии.

Не переписывайте число из `numpy.__version__` в `pyproject.toml` как единственно верное.
Диапазон описывает совместимость проекта, а точный текущий выбор уже хранится в lock-файле.

### Шаг 6. Запустите контрольный расчёт

```bash
uv run --locked python src/revenue_summary.py data/sample/orders.csv
```

Ожидаемый результат для sample из фазы 00:

```json
{"average_paid_order": 97.5, "paid_order_count": 2, "revenue": 195.0}
```

Проверьте смысл результата, а не только код возврата:

- оплаченных строк две: `TEST-001` и `TEST-003`;
- выручка равна `120 + 75 = 195`;
- среднее по двум оплаченным заказам равно `97.5`;
- строка `pending` на 80 не входит в расчёт.

### Шаг 7. Докажите восстановление, удалив только `.venv`

Сначала сохраните наблюдаемую версию NumPy:

```bash
uv run --locked python -c 'import numpy; print(numpy.__version__)' \
  > /tmp/numpy-version-before.txt
pwd
```

`pwd` должен заканчиваться на `work/00-03-git/revenue-project`. После этого удалите только
локальное окружение:

```bash
rm -rf .venv
test ! -d .venv && echo '.venv удалена'
```

Восстановите его из зафиксированного контракта:

```bash
uv sync --locked
uv run --locked python -c 'import numpy; print(numpy.__version__)' \
  > /tmp/numpy-version-after.txt
diff /tmp/numpy-version-before.txt /tmp/numpy-version-after.txt
uv run --locked python src/revenue_summary.py data/sample/orders.csv
```

Пустой вывод `diff` подтверждает ту же разрешённую версию прямой зависимости. Повторный
JSON подтверждает, что восстановленное окружение действительно запускает проект.

## Используйте это

### Минимальная инструкция для коллеги

После клонирования проекта коллеге не нужно угадывать отдельные команды установки:

```bash
uv sync --locked
uv run --locked python src/revenue_summary.py data/sample/orders.csv
```

Первая команда создаёт локальную `.venv` из существующего lock-файла. Вторая запускает
реальную задачу и запрещает неявно менять разрешение.

Если установка проходит, а расчёт падает, проблема уже не сводится к «пакет не
установлен». Нужно читать ошибку кода, проверять входные данные и внешние условия.

### Повседневная работа и строгая проверка

Во время разработки удобно:

```bash
uv run python src/revenue_summary.py data/sample/orders.csv
```

`uv` сам поддержит lock-файл и окружение в актуальном состоянии. В проверяемой инструкции,
CI или воспроизведении чужого результата полезнее строгий режим:

```bash
uv lock --check
uv sync --check --locked
uv run --locked python src/revenue_summary.py data/sample/orders.csv
```

Здесь команды не должны «чинить по дороге» незакоммиченные изменения контракта.

### Когда добавлять зависимость

Если код проекта импортирует библиотеку при обычном запуске, используйте:

```bash
uv add package-name
```

Не добавляйте библиотеку только потому, что она открыта в соседнем notebook или может
когда-нибудь пригодиться. Каждая зависимость увеличивает граф разрешения, поверхность
обновлений и число возможных конфликтов.

Инструменты разработки, группы и расширенную структуру `pyproject.toml` мы разберём в
следующем уроке. Здесь достаточно одной прямой runtime-зависимости.

## Сломайте это

### Установите пакет только в `.venv`

Представьте, что вместо `uv add` выполнено:

```bash
uv pip install pandas
```

Импорт может заработать локально, но `pandas` не обязательно появится в
`[project].dependencies` и согласованном `uv.lock`. После удаления `.venv` это состояние
пропадёт. Для зависимости проекта используйте `uv add pandas`.

### Измените manifest без обновления lock-файла

Добавьте требование вручную в `pyproject.toml`, но не запускайте `uv lock`. Затем:

```bash
uv run --locked python src/revenue_summary.py data/sample/orders.csv
```

Строгий запуск должен завершиться ошибкой stale lockfile. Это полезный отказ: `uv` не
выдаёт старое разрешение за соответствующее новой декларации.

Исправление — осознанно обновить контракт:

```bash
uv lock
uv sync
```

или выполнять исходное изменение через `uv add`.

### Не сохраните `uv.lock`

Если коллега получает только `pyproject.toml`, `uv` может заново подобрать версии из
доступных на момент установки. Они будут удовлетворять диапазонам, но не обязательно
совпадут с проверенным набором автора.

Проверьте:

```bash
git ls-files --error-unmatch uv.lock
```

### Запустите код мимо проекта

```bash
python3 src/revenue_summary.py data/sample/orders.csv
```

Команда использует Python из текущего `PATH`. Если NumPy там есть, ошибка маскируется;
если нет — появляется на машине автора. Сравните:

```bash
python3 -c 'import sys; print(sys.executable)'
uv run python -c 'import sys; print(sys.executable)'
```

### Попробуйте закоммитить `.venv`

```bash
git status --short .venv
git check-ignore -v .venv/
```

Первый вывод должен быть пустым, второй — назвать правило `.gitignore`. Если `.venv` уже
попала под tracking, одного нового правила недостаточно: Git продолжит следить за ранее
добавленными файлами.

## Проверьте это

### Проверки собственного проекта

Из `revenue-project` выполните:

```bash
uv lock --check
uv sync --check --locked
uv run --locked python -c \
  'import sys, numpy; print(sys.executable); print(numpy.__version__)'
uv run --locked python src/revenue_summary.py data/sample/orders.csv
git check-ignore -v .venv/
git ls-files --error-unmatch pyproject.toml uv.lock .python-version
git status --short
```

Результат готов, если:

1. lock-файл соответствует `pyproject.toml`;
2. `.venv` соответствует lock-файлу;
3. путь Python находится внутри проектной `.venv`;
4. NumPy импортируется из строгого запуска;
5. контрольный расчёт снова даёт выручку `195.0`;
6. `uv.lock` tracked, `.venv` ignored;
7. рабочее дерево чистое после коммита.

### Проверки артефакта урока

Из корня курса:

```bash
uv run --locked python -m unittest discover \
  -s phases/01-reproducible-project/02-uv-environments/tests \
  -p "test_*.py" -v
uv run --locked python \
  phases/01-reproducible-project/02-uv-environments/code/main.py
```

Пять behavioral tests проверяют контрольный расчёт, обязательные столбцы, нечисловую
сумму, отсутствие оплаченных заказов и JSON-интерфейс CLI. Они проверяют код, который
служит smoke-задачей. Состояние окружения проверяют сами команды `uv lock`, `uv sync` и
`uv run` — второй самописный менеджер зависимостей для этого не нужен.

## Поставьте результат

После урока прежний проект выглядит так:

```text
revenue-project/
├── .gitignore
├── .python-version
├── .venv/                      # локально, не в Git
├── data/
│   └── sample/orders.csv
├── docs/
├── pyproject.toml              # допустимые требования
├── queries/
├── src/
│   └── revenue_summary.py      # контрольная задача с NumPy
└── uv.lock                     # точное разрешение, в Git
```

Именованный артефакт урока — `outputs/revenue_summary.py`. Его можно скопировать в
проект с колонками `status` и `amount` и запускать:

```bash
uv run --locked python outputs/revenue_summary.py data/sample/orders.csv
```

Главный поставленный результат шире одного скрипта: репозиторий содержит зависимость,
lock-файл, код и безопасный sample, а `.venv` восстанавливается одной командой.

В следующем уроке мы подробно прочитаем `pyproject.toml`: отделим metadata проекта от
runtime-зависимостей, групп разработки и настроек инструментов. Здесь мы использовали
только минимальную часть manifest, необходимую для рабочего `uv`-цикла.

## Упражнения

1. Выпишите для текущего проекта по одному факту из `pyproject.toml`, `uv.lock` и
   `.venv`. Объясните, почему ни один из трёх уровней не заменяет остальные.
2. Удалите `.venv`, восстановите её через `uv sync --locked` и сравните версию NumPy и
   JSON расчёта до и после восстановления.
3. Создайте копию sample с одним дополнительным оплаченным заказом. До запуска вручную
   рассчитайте ожидаемые count, revenue и average, затем сравните с CLI.
4. В отдельной учебной ветке измените диапазон NumPy вручную. Покажите отказ строгого
   запуска, восстановите согласованность и объясните diff `pyproject.toml` и `uv.lock`.

## Ключевые термины

| Термин | Распространённое заблуждение | Точное значение |
|---|---|---|
| Окружение | Полная виртуальная машина | Изолированный Python и установленные для проекта пакеты и команды |
| `.venv` | Переносимый источник воспроизводимости | Локальное производное установленное состояние |
| Requirement | Точная установленная версия | Допустимое требование к версии или источнику зависимости |
| Разрешение зависимостей | Скачивание одного пакета | Выбор совместимых конкретных версий всего графа требований |
| `uv.lock` | Копия списка из `pyproject.toml` | Зафиксированное разрешение прямых и транзитивных зависимостей |
| Прямая зависимость | Любой пакет внутри `.venv` | Библиотека, которую проект объявляет и использует непосредственно |
| Транзитивная зависимость | Пакет, который нужно всегда объявить вручную | Зависимость другой зависимости проекта |
| `uv add` | Установка пакета только локально | Изменение декларации, lock-файла и проектного окружения |
| `uv sync` | Запуск аналитического кода | Приведение `.venv` к состоянию проекта |
| `uv run` | Другое имя для глобального `python` | Запуск команды в найденном проектном окружении |
| `--locked` | Команда обновить все пакеты | Требование не менять lock-файл неявно и отказ при рассинхронизации |
| Smoke-задача | Полная проверка корректности проекта | Небольшой реальный запуск, подтверждающий базовую работоспособность окружения |
