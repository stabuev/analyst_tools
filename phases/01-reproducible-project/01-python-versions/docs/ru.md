# Версии Python и совместимость

> «У меня запускается» описывает один интерпретатор; воспроизводимый проект описывает
> поддерживаемый диапазон и проверяет, какой Python выполняет код прямо сейчас.

**Тип:** Build
**Треки:** Core
**Пререквизиты:** 00/06 — Секреты и безопасная работа с данными
**Время:** ~45 минут
**Результат:** объясняет связь интерпретатора, окружения и совместимости зависимостей

## Цели обучения

- Отличать команду `python` в shell от интерпретатора текущего процесса.
- Читать `sys.version_info` и `sys.executable`.
- Задавать диапазон совместимости через `[project].requires-python`.
- Объяснять различие между `requires-python` и `.python-version`.
- Проверять нижнюю и верхнюю границы на матрице кандидатных версий.
- Понимать, почему смена Python требует пересоздания окружения и повторной проверки
  зависимостей.

## Проблема

Аналитик запускает notebook, скрипт и тесты командами, которые выглядят одинаково:

```bash
python analysis.py
python -m pytest
jupyter lab
```

Но за ними могут стоять разные интерпретаторы:

```text
/usr/bin/python3
/opt/homebrew/bin/python3
project/.venv/bin/python
kernel из старого окружения
```

На одной машине код использует Python 3.12, на другой — 3.10. В первом есть нужная
возможность стандартной библиотеки, во втором импорт завершается ошибкой. Или сам код
запускается, но выбранная версия зависимости больше не поддерживает старый Python.

Фраза «проект написан на Python 3» почти ничего не гарантирует. Нужны ответы на три
разных вопроса:

1. Какой Python совместим с проектом?
2. Какой Python должен выбрать локальный инструмент?
3. Какой бинарник выполняет этот конкретный процесс?

Если ответы не зафиксированы и не проверяются, окружение нельзя воспроизвести
предсказуемо.

## Концепция

### Интерпретатор, окружение и зависимости

**Интерпретатор** — исполняемый файл Python. Внутри процесса его можно определить через:

```python
import sys

print(sys.version_info)
print(sys.executable)
```

`sys.version_info` сообщает версию запущенного Python. `sys.executable` обычно содержит
абсолютный путь к бинарнику, который выполняет код.

**Окружение** связывает конкретный интерпретатор с каталогами установленных пакетов и
командами. Virtual environment не превращает Python 3.11 в Python 3.12. Оно создаётся на
основе определённого интерпретатора.

**Зависимости** имеют собственные ограничения: поддерживаемые версии Python, доступные
wheels, platform tags и зависимости от других пакетов. Поэтому совместимость проекта
выглядит как пересечение:

```text
возможности кода
∩ requires-python проекта
∩ поддержка Python зависимостями
∩ доступные сборки для платформы
```

После смены базового Python старое окружение нельзя считать доказательством
воспроизводимости. Его нужно пересоздать и снова установить и проверить зависимости.

### `requires-python` — диапазон поддержки

В `pyproject.toml` поле `requires-python` является стандартной metadata проекта:

```toml
[project]
name = "analytics-project"
version = "0.1.0"
requires-python = ">=3.11,<3.13"
```

Здесь clauses разделены запятыми и применяются одновременно:

- `>=3.11` — Python 3.11 включён;
- `<3.13` — Python 3.13 исключён;
- Python 3.12 находится внутри диапазона.

Это не утверждение, что каждая версия внутри диапазона уже установлена. Это обещание
проекта: такие интерпретаторы считаются поддерживаемыми и должны участвовать в проверках.

Нижняя граница определяется самой ранней версией, на которой работают:

- синтаксис проекта;
- используемые модули стандартной библиотеки;
- зависимости;
- тесты и инструменты.

Верхняя граница нужна, если будущая или уже выпущенная версия ещё не проверена либо
несовместима с критичной зависимостью. Не добавляйте её автоматически из страха, но и не
обещайте `>=3.11` навсегда, если проект никогда не проверялся на следующей major/minor
версии.

### `.python-version` — выбор для инструмента

Файл:

```text
3.11
```

сообщает совместимым инструментам предпочтительную локальную версию. Например, `uv`
учитывает `.python-version`, а при работе с проектом также уважает `requires-python`.

Это два разных утверждения:

```text
requires-python = ">=3.11,<3.13"  # поддерживаются 3.11 и 3.12
.python-version = 3.11            # локально сначала выбираем 3.11
```

`.python-version` является соглашением инструментов, а не заменой packaging metadata.
Если оставить только selector, установщик пакета не узнает диапазон поддержки. Если
оставить только диапазон, инструменту может понадобиться самому выбрать одну из нескольких
доступных версий.

### Версия в shell и версия процесса

Команда:

```bash
python3 --version
```

показывает бинарник, который shell нашёл для имени `python3`. Она не доказывает, что уже
запущенный Jupyter kernel использует тот же путь.

Диагностику выполняйте внутри проблемного процесса:

```python
import sys

print("version:", ".".join(map(str, sys.version_info[:3])))
print("executable:", sys.executable)
```

Сравнивайте абсолютный путь, а не только номер версии. Два Python 3.12 могут иметь разные
пакеты и разные окружения.

## Соберите это

Создайте учебный каталог:

```bash
mkdir version-lab
```

Из корня репозитория курса запустите артефакт:

```bash
python3 phases/01-reproducible-project/01-python-versions/outputs/python_version_check.py \
  init version-lab \
  --name version-lab \
  --requires '>=3.11,<3.13' \
  --select 3.11
```

Команда создаёт минимальный контракт:

```text
version-lab/
├── .python-version
└── pyproject.toml
```

CLI намеренно не редактирует существующие файлы автоматически. Если контракт уже есть,
его изменение влияет на пользователей, CI и resolver, поэтому должно проходить через
обычный diff и review.

### Шаг 1: прочитайте диапазон

```bash
cat version-lab/pyproject.toml
```

Результат:

```toml
[project]
name = "version-lab"
version = "0.1.0"
requires-python = ">=3.11,<3.13"
```

В этом уроке `pyproject.toml` используется только как стандартное место для
`requires-python`. Остальные роли файла, таблицы и конфигурация инструментов разбираются в
уроке `01/03`.

Проверьте несколько версий вручную:

| Кандидат | Результат | Причина |
|---|---|---|
| 3.10.14 | Нет | Ниже 3.11 |
| 3.11.9 | Да | Внутри диапазона |
| 3.12.4 | Да | Внутри диапазона |
| 3.13.0 | Нет | Верхняя граница исключающая |

### Шаг 2: прочитайте selector

```bash
cat version-lab/.python-version
```

Значение `3.11` выбирает любую подходящую patch-версию линии 3.11. Фиксация patch до
`3.11.9` даёт более точный выбор, но может усложнить запуск на машине, где доступна другая
security patch. Уровень точности должен соответствовать способу поставки проекта.

Для учебного курса достаточно minor selector и диапазона metadata. Управление установкой
Python и окружением появится в следующем уроке про `uv`.

### Шаг 3: определите текущий процесс

```bash
python3 -c \
  'import sys; print(sys.version_info[:3]); print(sys.executable)'
```

Запишите:

- major, minor и patch;
- абсолютный путь;
- совпадает ли major/minor с `.python-version`;
- входит ли версия в `requires-python`.

Не исправляйте несовпадение alias-командой вслепую. Сначала поймите, кто запустил процесс:
shell, IDE, notebook kernel, CI runner или scheduler.

### Шаг 4: проверьте контракт CLI

Из каталога урока:

```bash
python3 outputs/python_version_check.py check path/to/version-lab \
  --candidate 3.10.14 \
  --candidate 3.11.9 \
  --candidate 3.12.4 \
  --candidate 3.13.0
```

Из корня курса:

```bash
python3 phases/01-reproducible-project/01-python-versions/outputs/python_version_check.py \
  check version-lab \
  --candidate 3.10.14 \
  --candidate 3.11.9 \
  --candidate 3.12.4 \
  --candidate 3.13.0
```

Проверка завершится успешно только если:

- `requires-python` существует и понятен CLI;
- `.python-version` находится внутри диапазона;
- текущий Python входит в диапазон;
- текущий major/minor совпадает с selector;
- кандидатные версии записаны корректно.

Если текущая машина использует Python 3.12, а selector равен 3.11, инструмент честно
покажет несовпадение. Для продолжения практики выберите установленную и поддерживаемую
версию, но не расширяйте диапазон только ради зелёного отчёта.

## Используйте это

### Проверяйте процесс, а не предположение

Перед расследованием «пакет установлен, но import не работает» выполните:

```bash
python3 -c 'import sys; print(sys.executable)'
python3 -m pip --version
```

Обе команды должны указывать на ожидаемое окружение. Вызов `python3 -m pip` связывает pip
с выбранным Python яснее, чем отдельная команда `pip`.

В notebook:

```python
import sys

{
    "version": tuple(sys.version_info[:3]),
    "executable": sys.executable,
}
```

Если путь kernel отличается от shell, установка пакета в shell не меняет уже выбранный
kernel.

### Стройте матрицу до изменения диапазона

Сформируйте отчёт:

```bash
python3 outputs/python_version_check.py check . \
  --candidate 3.11 \
  --candidate 3.12 \
  --candidate 3.13 \
  --format json \
  --output /tmp/python-compatibility.json
```

Матрица отвечает только на вопрос, допускает ли версия текущий metadata-контракт. Она не
запускает код на неустановленных интерпретаторах. Настоящая поддержка требует CI или
локального запуска тестов на каждой заявленной линии.

Это важное различие:

```text
specifier допускает версию ≠ проект доказанно работает на версии
```

### Меняйте нижнюю границу обоснованно

Причины поднять минимум:

- код использует новый синтаксис;
- нужен модуль, появившийся в новой стандартной библиотеке;
- критичная зависимость прекратила поддержку старой версии;
- команда больше не может надёжно тестировать старую линию.

В pull request укажите:

```text
Минимум поднят с 3.10 до 3.11, потому что проект использует tomllib из стандартной
библиотеки. CI проверяет 3.11 и 3.12.
```

Фраза «у меня установлен 3.11» не является причиной изменить публичный контракт.

### Не обещайте совместимость по одной машине

Если автор работает на 3.12, из этого не следует ни `==3.12`, ни `>=3.12`.

- `==3.12.7` может быть слишком узким и запрещать совместимые security patches.
- `>=3.12` без верхней границы обещает поддержку ещё не проверенных будущих версий.
- `>=3.11,<3.13` разумно только при тестах 3.11 и 3.12 и осознанном исключении 3.13.

Контракт описывает политику поддержки проекта, а не снимок ноутбука автора.

## Сломайте это

### Запустите другой Python

Измените `.python-version` на minor, отличающийся от текущего:

```bash
printf '3.11\n' > .python-version
python3 path/to/python_version_check.py check .
```

Если процесс работает на 3.12, `selection-match` завершится ошибкой. Это не означает, что
3.12 несовместим с проектом: selector и диапазон отвечают на разные вопросы.

### Выберите версию вне диапазона

```toml
requires-python = ">=3.11,<3.13"
```

```text
# .python-version
3.13
```

Проверка `selector` остановит контракт. Инструмент не должен выбирать Python, который сам
проект объявил неподдерживаемым.

### Запишите требование человеческой фразой

```toml
requires-python = "Python 3.11 or newer"
```

Это не version specifier. Используйте формальный синтаксис:

```toml
requires-python = ">=3.11"
```

CLI урока поддерживает числовые clauses `>=`, `>`, `<=`, `<`, `==`, `!=`, `~=` и
wildcard для `==`/`!=`. Pre-release, local versions и сложные варианты намеренно не
реализованы: для полного синтаксиса источником истины остаётся спецификация PyPA и
production packaging tools.

### Уберите верхнюю границу без проверки

```toml
requires-python = ">=3.11"
```

Синтаксически это корректно. Но теперь metadata допускает любую будущую версию. Если
проект публикуется как пакет или разворачивается автоматически, resolver может выбрать
комбинацию, которую команда никогда не тестировала.

Решение не в обязательном `<3.13` для всех проектов. Решение — согласовать metadata,
матрицу CI и реальную политику поддержки.

### Поменяйте Python внутри существующего окружения

Представьте `.venv`, созданный на 3.11, после чего selector изменили на 3.12. Старый
каталог может продолжать ссылаться на прежний интерпретатор и содержать несовместимые
binary wheels.

Не «ремонтируйте» такое окружение последовательностью случайных установок. Пересоздайте
его из контракта и lock-файла. Эту процедуру мы соберём в уроке `01/02`.

## Проверьте это

Запустите тесты урока:

```bash

  -s phases/01-reproducible-project/01-python-versions/tests \
  -p "test_*.py" -v
```

Шесть тестов проверяют:

- контракт, совместимый с текущим интерпретатором;
- включающие и исключающие границы диапазона;
- runtime ниже минимальной версии;
- selector за верхней границей;
- `~=`, wildcard и исключение версии;
- отсутствующий selector и некорректный specifier.

Запустите демонстрацию:

```bash
python3 phases/01-reproducible-project/01-python-versions/code/main.py
```

Перед review соберите диагностический минимум:

```bash
python3 --version
python3 -c 'import sys; print(sys.executable)'
cat .python-version
python3 path/to/python_version_check.py check .
```

В CI проверка должна запускаться отдельно для каждой действительно поддерживаемой minor
версии. Матрица текста не заменяет матрицу процессов.

## Поставьте результат

Результат урока — version contract из двух файлов:

```toml
# pyproject.toml
[project]
name = "analytics-project"
version = "0.1.0"
requires-python = ">=3.11,<3.13"
```

```text
# .python-version
3.11
```

и воспроизводимая проверка:

```bash
python3 outputs/python_version_check.py check .
```

Manifest находится в `outputs/artifact.json`. CLI использует `tomllib` и другие модули
стандартной библиотеки Python 3.11+, не устанавливает интерпретаторы и не разрешает
зависимости.

В следующем уроке `uv` превратит этот декларативный контракт в выбранный Python,
виртуальное окружение и воспроизводимую установку. Здесь важно сохранить границу:
сначала определяем, что проект поддерживает, затем выбираем инструмент доставки.

## Ключевые термины

| Термин | Распространённое заблуждение | Точное значение |
|---|---|---|
| Интерпретатор | Любая команда `python` | Конкретный бинарник, выполняющий процесс |
| `sys.executable` | Путь к исходному коду | Путь к текущему Python executable |
| Environment | Версия Python | Интерпретатор плюс установленные пакеты и пути |
| `requires-python` | Версия автора | Стандартный диапазон совместимости проекта |
| `.python-version` | Полный контракт поддержки | Selector предпочтительного интерпретатора для инструментов |
| Minor version | Любая patch-сборка | Линия вроде 3.11 внутри major 3 |
| Patch version | Новая несовместимая линия | Исправляющий выпуск вроде 3.11.9 |
| Version specifier | Человеческое описание | Формальные clauses сравнения версий |
| Compatibility matrix | Установленные Python | Набор кандидатов и ожидаемый статус поддержки |

## Дополнительное чтение

- [PyPA: pyproject.toml specification](https://packaging.python.org/en/latest/specifications/pyproject-toml/#requires-python) — найдите нормативное определение поля `requires-python` и его связь с core metadata.
- [PyPA: Version specifiers](https://packaging.python.org/en/latest/specifications/version-specifiers/) — разберите `~=`, включающие и исключающие сравнения, wildcard и объединение clauses.
- [Python Docs: sys.version_info](https://docs.python.org/3/library/sys.html#sys.version_info) — изучите структурированное представление версии запущенного интерпретатора.
- [Python Docs: sys.executable](https://docs.python.org/3/library/sys.html#sys.executable) — проверьте точную семантику пути и случаи, когда он может быть пустым.
- [uv Docs: Python versions](https://docs.astral.sh/uv/concepts/python-versions/) — посмотрите, как `uv` согласует `requires-python`, `.python-version` и явно запрошенный Python.
