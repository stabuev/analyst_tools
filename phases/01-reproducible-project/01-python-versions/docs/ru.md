## Концепция

### Что такое интерпретатор

Файл с расширением `.py` — это текст с инструкциями. Чтобы инструкции выполнились, их
читает программа Python. Эту программу называют **интерпретатором**.

Упрощённая цепочка выглядит так:

```text
команда в терминале
        ↓
командная оболочка (shell) находит исполняемый файл Python
        ↓
Python создаёт процесс
        ↓
процесс читает и выполняет код
```

На одной машине может быть несколько интерпретаторов:

```text
/usr/bin/python3
/opt/homebrew/bin/python3
<project>/.venv/bin/python
Python внутри другого окружения notebook
```

Они могут иметь разные версии и разные наборы установленных библиотек. Поэтому фраза «на
компьютере установлен Python 3.12» ещё не говорит, какой из этих файлов выполнил конкретный
скрипт.

### Как читать номер версии

Рассмотрим Python `3.12.7`:

```text
3 . 12 . 7
│   │    └─ patch: исправления внутри линии 3.12
│   └────── minor: линия возможностей и совместимости
└────────── major: крупное поколение языка
```

- **Major** меняется редко и обозначает крупное поколение языка. Сейчас рабочие проекты
  курса используют Python 3.
- **Minor** появляется с очередным выпуском языка: 3.11, 3.12, 3.13. В новой линии могут
  появиться новый синтаксис, модули стандартной библиотеки и изменения API.
- **Patch** содержит исправления ошибок и безопасности внутри minor-линии: например,
  3.12.6 и 3.12.7 относятся к одной линии 3.12.

Для начального выбора совместимости чаще всего важна minor-линия. Жёстко требовать ровно
`3.12.7` обычно не нужно: это может запретить более свежий исправляющий выпуск `3.12.8`
без содержательной причины.

Python не обещает, что любой код автоматически работает на любой более новой версии.
Номер помогает сформулировать ожидание, но реальную совместимость подтверждает запуск
проекта и его проверок.

### Как узнать фактически запущенный Python

Внутри Python доступны два особенно полезных наблюдения:

```python
import sys

print(sys.version_info[:3])
print(sys.executable)
```

`sys.version_info[:3]` показывает major, minor и patch текущего процесса.
`sys.executable` показывает путь к интерпретатору, который этот процесс запустил.

Из терминала ту же проверку можно выполнить без создания файла:

```bash
python3 -c 'import sys; print(sys.version_info[:3]); print(sys.executable)'
```

Ключевое выражение — **текущего процесса**. Если ошибка возникла в notebook, проверку
нужно выполнить в notebook. Если ошибка возникла в системе автоматических проверок (CI),
нужны сведения из этого запуска. Вывод из вашего локального терминала не описывает
автоматически чужой процесс.

### Команда shell и интерпретатор процесса

Команда:

```bash
command -v python3
```

показывает, какой файл shell сейчас найдёт для имени `python3`. Команда:

```bash
python3 -c 'import sys; print(sys.executable)'
```

запускает найденный Python и спрашивает путь уже у процесса. В обычном терминале эти пути
часто совпадают. Сравнение всё равно полезно: оно связывает имя команды с конкретным
исполняемым файлом.

В уже открытом notebook ситуация может быть другой:

```text
shell:    /opt/homebrew/bin/python3       → Python 3.12
notebook: <home>/old-project/bin/python  → Python 3.10
```

Это не «две версии одного процесса». Это два разных процесса, запущенных разными
интерпретаторами. Как выбирать ядро notebook, или Jupyter kernel, мы подробно разберём в
уроке `01/04`.

### Интерпретатор и окружение — не одно и то же

**Интерпретатор** выполняет Python-код. **Окружение** связывает выбранный интерпретатор с
установленными для проекта библиотеками и командами.

```text
окружение проекта
├── конкретный Python
├── установленные библиотеки
└── команды, доступные внутри окружения
```

Два процесса на Python 3.12 могут видеть разные библиотеки. И наоборот, одинаковый список
библиотек нельзя без проверки переносить между Python 3.10 и 3.12. В этом уроке мы только
выбираем и наблюдаем интерпретатор; окружение создадим в `01/02`.

### Три уровня версии в проекте

У проекта есть три связанных, но не взаимозаменяемых утверждения.

| Уровень | Пример | На какой вопрос отвечает |
|---|---|---|
| Требование проекта | `requires-python = ">=3.11"` | Какая минимальная версия допускается проектом? |
| Локальный выбор | `.python-version` содержит `3.12` | Какую линию предложить совместимому инструменту? |
| Наблюдение процесса | `sys.version_info` показывает `3.12.7` | Какой Python действительно выполняет код сейчас? |

Они могут согласоваться:

```text
проект требует 3.11 или новее
        ↓
локально выбрана линия 3.12
        ↓
процесс запущен на 3.12.7
```

А могут расходиться:

```text
проект требует 3.11 или новее
        ↓
в .python-version записано 3.12
        ↓
процесс всё ещё запущен на 3.10.14
```

Последний случай означает, что файл с выбором ещё не был применён инструментом либо код
запущен в другом окружении.

### `requires-python` — требование проекта

Минимальное требование хранится в `pyproject.toml`:

```toml
[project]
name = "revenue-project"
version = "0.1.0"
requires-python = ">=3.11"
```

Запись `>=3.11` читается как «Python 3.11 или новее». В этом уроке достаточно понимать
одну нижнюю границу. Остальную структуру `pyproject.toml`, зависимости и более сложные
ограничения мы разберём позже.

Минимум выбирают по самой ранней версии, на которой работают необходимые части проекта:

- синтаксис Python;
- модули стандартной библиотеки;
- внешние библиотеки;
- тесты и рабочие инструменты.

В курсе используется минимум Python 3.11. Это общее требование учебного проекта, а не
случайная версия на компьютере автора.

Поле `requires-python` является декларацией. Оно сообщает инструментам о требовании, но не
устанавливает Python и не доказывает, что код уже проверен на каждой будущей версии.

### `.python-version` — локальный выбор для инструмента

Файл `.python-version` может содержать:

```text
3.12
```

Совместимые инструменты, включая `uv`, используют это значение при выборе Python.
Обычный текстовый файл сам по себе ничего не переключает: после его создания уже
запущенный notebook не изменится, а shell без менеджера версий может продолжить находить
прежний бинарник.

Полезная формула:

```text
requires-python описывает допустимый минимум
.python-version предлагает локальную линию
sys.executable показывает факт
```

### Совместимость зависимостей

У внешних библиотек тоже есть требования к Python. Возможная версия проекта находится на
пересечении:

```text
требования нашего кода
∩ требование requires-python
∩ поддержка Python библиотеками
∩ доступность сборок для операционной системы
```

Пока у `revenue-project` нет внешних Python-зависимостей, мы не будем изображать их
проверку вручную. В следующем уроке `uv` создаст окружение, добавит библиотеку и покажет,
можно ли разрешить требования вместе.

### Шаг 5. Зафиксируйте выбранную локальную линию

Запишите текущую minor-линию без patch:

```bash

  'import sys; print(f"{sys.version_info.major}.{sys.version_info.minor}")' \

cat .python-version
```

Если текущий Python равен `3.12.7`, файл будет содержать `3.12`. Это не попытка
зафиксировать единственную допустимую patch-версию. Любой подходящий исправляющий выпуск
линии 3.12 может стать фактическим интерпретатором.

Сразу повторите наблюдение:

```bash

```

запущенный процесс без инструмента, который умеет читать этот файл.

## Используйте это

### Разберите сообщение коллеги

Коллега прислал:

```text
Команда python3 --version в терминале: Python 3.12.7
В notebook sys.version_info: 3.10.14
В notebook sys.executable: /usr/bin/python3
Ошибка: ModuleNotFoundError: No module named 'tomllib'
```

Из этих фактов следует:

1. Терминал и notebook используют разные интерпретаторы.
2. Ошибка произошла внутри Python 3.10, где `tomllib` ещё не входит в стандартную
   библиотеку.
3. Установка Python 3.12 уже произошла, но notebook на него не переключён.
4. Проверять и менять нужно ядро notebook, а не требование проекта и не случайный пакет.

Так выглядит полезная диагностика: она называет процесс, версию, путь и связь с ошибкой.

### Отличайте требование от доказанной поддержки

Представьте, что в проекте записано:

```toml
requires-python = ">=3.11"
```

Это означает, что проект не должен запускаться на Python 3.10. Но строка сама по себе не
доказывает работу на 3.11, 3.12, 3.13 и всех будущих версиях.

Уровни доказательства различаются:

```text
записали требование
        ↓
создали окружение на выбранном Python
        ↓
установили зафиксированные зависимости
        ↓
запустили код и тесты
        ↓
повторили на каждой заявленной линии
```

В этом уроке выполнен первый шаг и собраны локальные наблюдения. Остальные появятся по
мере прохождения фазы.

### Меняйте минимальную версию как контракт

Минимум можно поднять, если:

- проект начинает использовать новый синтаксис;
- нужен модуль из более новой стандартной библиотеки;
- важная зависимость перестала поддерживать старую линию;
- команда больше не может надёжно тестировать старую версию.

В pull request полезно написать:

```text
Минимальная версия поднята с 3.10 до 3.11, потому что проект использует tomllib из
стандартной библиотеки. После обновления окружения тесты пройдут на 3.11 и 3.12.
```

Фраза «у меня установлен 3.12» описывает машину автора, а не причину изменить контракт
для всех пользователей.

### Завершите ветку

Проверьте карточку по рубрике. Если вы работаете через GitHub, отправьте ветку и проведите
обычное ревью из урока `00/04`.

Для полностью локальной практики:

```bash
git switch main
git merge --ff-only feature/python-version-contract
git branch -d feature/python-version-contract
git status --short
git log --oneline -5
```

Теперь тот же `revenue-project` содержит первый контракт воспроизводимости и готов к
созданию окружения.

### Ожидайте, что `.python-version` переключит всё сама

Вы записали `3.12`, но `python3 --version` всё ещё показывает 3.11.

Это не ошибка текстового файла. `.python-version` — вход для совместимого менеджера, а не
исполняемая команда. В следующем уроке `uv` прочитает выбор и создаст окружение.

### Понизьте требование до версии коллеги

Коллега запускает Python 3.10, а проект требует `>=3.11`. Самое быстрое на вид решение —
изменить контракт на `>=3.10`.

Так делать можно только после проверки кода и зависимостей на 3.10. Если проект использует
возможности 3.11, правильное действие — выбрать поддерживаемый интерпретатор, а не
переписать требование под случайную машину.

### Зафиксируйте точную patch-версию без причины

```text
.python-version = 3.12.7
```

Точный patch может быть оправдан регламентом поставки. Для обычного учебного проекта он
часто создаёт лишнее ограничение и мешает использовать исправления безопасности внутри
линии 3.12. В этом уроке достаточно выбора minor-линии.

### Проверьте только `python3 --version`

Терминал показывает 3.12, поэтому автор делает вывод, что notebook тоже работает на 3.12.

Проверка описывает только новый процесс из shell. В notebook нужно отдельно выполнить
`sys.version_info` и `sys.executable`.

### Считайте декларацию тестом

Автор записал `requires-python = ">=3.11"` и объявил поддержку 3.13, хотя проект на ней не
запускался.

Metadata сообщает намерение инструментам. Доказательство совместимости требует реального
окружения, установки зависимостей и проверок на заявленной версии.

## Проверьте это

Проверьте `docs/python-version.md` по отдельной рубрике:

```bash
sed -n '1,240p' \
  ../../../phases/01-reproducible-project/01-python-versions/outputs/python-version-record-rubric.md
```

Затем повторите фактические проверки из `revenue-project`:

Результат готов, если вы можете без подсказки ответить:

2. Какую линию предлагает `.python-version`?

Пустой `git status --short` подтверждает завершённый Git-цикл, но не заменяет проверку

## Ключевые термины

| Термин | Распространённое заблуждение | Точное значение |
|---|---|---|
| Интерпретатор | Любая команда `python` | Конкретная программа, выполняющая Python-код |
| Процесс | Весь проект на диске | Один запущенный экземпляр программы со своим интерпретатором и состоянием |
| `sys.version_info` | Версия Python на всей машине | Версия интерпретатора текущего процесса |
| `sys.executable` | Путь к анализируемому скрипту | Путь к Python, который запустил текущий процесс |
| Major | Первая цифра любого пакета | Крупное поколение самого Python, например 3 |
| Minor | Незначительное исправление | Линия Python с новыми возможностями, например 3.12 |
| Patch | Новая несовместимая линия | Исправляющий выпуск внутри minor-линии, например 3.12.7 |
| Окружение | Только версия Python | Интерпретатор и доступный проекту набор библиотек и команд |
| `requires-python` | Версия на машине автора | Декларация минимальной версии, допускаемой проектом |
| `.python-version` | Команда переключения Python | Подсказка совместимому инструменту для выбора локальной линии |
| Совместимость | Наличие строки в метаданных | Подтверждённая способность кода и зависимостей работать на выбранной версии |
