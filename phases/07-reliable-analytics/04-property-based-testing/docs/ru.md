# Property-based testing с Hypothesis

> Генератор полезен только после того, как сформулировано свойство и независимый oracle.

**Тип:** Build  
**Треки:** core  
**Пререквизиты:** 07/03  
**Время:** ~90 минут

## Цели обучения

- формулировать свойства денежных агрегатов и дедупликации;
- ограничивать стратегии реальным доменом входов;
- читать shrunk counterexample как новый regression case.

## Проблема

Пять вручную выбранных примеров редко покрывают пустой batch, перестановку строк,
повторное применение дедупликации и копеечные значения. Случайный smoke test тоже слаб:
он не объясняет, что именно должно оставаться истинным.

## Концепция

Property-based test генерирует множество входов для свойства. Полезные свойства:
partition sum совпадает с paid revenue, перестановка строк не меняет агрегат,
дедупликация идемпотентна, а выбранная версия имеет максимальную sequence.
При сбое Hypothesis уменьшает вход до минимального контрпримера.

## Соберите это

Изучите стратегии и свойства в `outputs/property_suite.py`. Suite детерминирован:
каждое свойство получает 100 примеров, внешняя example database отключена.

```bash
uv run --locked python phases/07-reliable-analytics/04-property-based-testing/outputs/property_suite.py
```

Артефакт также демонстрирует ошибку «округлить каждую сумму до рублей перед сложением»
и находит для нее минимальный копеечный вход.

## Используйте это

Применяйте property tests к чистым функциям и устойчивым инвариантам. Не генерируйте
произвольные строки для поля status: стратегия должна выражать настоящий домен, иначе
большая часть примеров проверяет только parser.

## Сломайте это

Измените paid revenue так, чтобы он включал `pending`, либо сделайте дедупликацию
зависимой от порядка входа. Посмотрите, какое свойство падает и до какого входа
Hypothesis выполняет shrinking.

## Проверьте это

```bash
uv run --locked python -m unittest discover \
  -s phases/07-reliable-analytics/04-property-based-testing/tests
```

## Поставьте результат

Результат: `outputs/property_suite.py` с четырьмя исполняемыми свойствами и
машинным отчетом о shrunk counterexample.

## Упражнения

1. Добавьте свойство `0 <= paid_revenue <= total_amount`.
2. Сгенерируйте события с одинаковой sequence и зафиксируйте tie-break policy.
3. Сохраните найденный новый defect как обычный regression test.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Property | Ожидаемый output одного fixture | Утверждение, истинное для целого домена входов |
| Strategy | Генератор любых значений | Описание допустимого и граничного пространства входов |
| Shrinking | Уменьшение числа запусков | Поиск более простого входа, который сохраняет сбой |
