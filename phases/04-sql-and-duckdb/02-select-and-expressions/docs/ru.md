# SELECT и выражения

> Хороший SELECT описывает не только вычисление, но и контракт результирующей таблицы.

**Тип:** Build
**Треки:** Core
**Пререквизиты:** 04/01
**Время:** ~75 минут
**Результат:** строит проекцию, вычисляемые поля и фильтры и сверяет их с ручной таблицей.

## Цели обучения

- Разделять источник строк, фильтр и проекцию результата.
- Давать вычисляемым выражениям стабильные имена и типы.
- Передавать значения параметрами, а не собирать SQL строковой конкатенацией.
- Проверять grain, схему и порядок результата.

## Проблема

Заказчику нужны оплаченные RUB-заказы дороже 100 рублей с комиссией 5%. Запрос может
вернуть правильные числа сегодня, но остаться плохим интерфейсом: `SELECT *` незаметно
изменит схему, неявный cast сломается на новом значении, а `LIMIT` без `ORDER BY` даст
нестабильную выборку.

## Концепция

Запрос формирует новую таблицу. Ее контракт состоит как минимум из:

- grain: одна строка на `order_id`;
- списка и порядка колонок;
- типов вычисляемых значений;
- правил фильтрации;
- детерминированного порядка, если результат сравнивают или показывают человеку.

Логически `FROM` выбирает источник, `WHERE` оставляет строки, `SELECT` строит выражения, а
`ORDER BY` упорядочивает результат. Текст `SELECT` пишется первым, но это не первый
логический шаг.

## Соберите это

Для трех ручных строк примените правила последовательно:

| order_id | status | currency | amount | остается? |
|---|---|---|---:|---|
| O1001 | paid | RUB | 1200 | да |
| O1002 | refunded | RUB | 800 | нет |
| O1004 | pending | USD | NULL | нет |

Для `O1001` вычисление `1200 * 1.05` дает `1260`. Сначала получите ожидаемую таблицу
руками, затем сравнивайте SQL именно с ней, а не только с количеством строк.

Запустите пример:

```bash
uv run --locked python code/main.py
```

## Используйте это

Артефакт выполняет параметризованный запрос:

```sql
SELECT
    order_id,
    user_id,
    upper(trim(currency)) AS currency,
    amount::DECIMAL(18, 2) AS amount,
    round(amount::DECIMAL(18, 2) * (1 + ?), 2) AS amount_with_fee
FROM read_csv(?, header = true, all_varchar = true, nullstr = '')
WHERE lower(trim(status)) = 'paid'
  AND upper(trim(currency)) = upper(?)
  AND amount::DECIMAL(18, 2) >= ?
ORDER BY amount::DECIMAL(18, 2) DESC, order_id;
```

Знаки `?` получают значения отдельно от SQL. Путь, валюта и пороги не становятся частью
синтаксиса запроса.

## Сломайте это

Проверьте четыре ошибки:

1. замените список колонок на `*` и добавьте колонку в CSV;
2. уберите `ORDER BY` и сравните порядок после изменения источника;
3. подставьте валюту через f-string и передайте строку `RUB' OR TRUE --`;
4. удалите явный cast `amount` и добавьте нечисловое значение.

Артефакт использует параметры и отклоняет отрицательные пороги, поэтому эти ошибки
становятся наблюдаемыми.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Projection | Выбор строк | Выбор и вычисление колонок результата |
| Predicate | Любое выражение | Условие, результат которого используется как логическое значение |
| Alias | Косметическое имя | Стабильное имя выражения в контракте |
| Parameter | Фрагмент SQL | Значение, переданное отдельно от синтаксиса |
| Deterministic order | Порядок файла | Порядок, явно заданный `ORDER BY` |
