# Grain, ключи и связи

> До запроса нужно доказать, что вы понимаете единицу строки и допустимые связи таблиц.

**Тип:** Learn
**Треки:** Core
**Пререквизиты:** 03/11
**Время:** ~75 минут
**Результат:** объявляет grain таблиц, проверяет ключи и находит нарушения связей.

## Цели обучения

- Отличать смысл строки, primary key и foreign key.
- Проверять простой и составной ключ на `NULL` и дубликаты.
- Находить orphan-ключи без потери проблемных строк.
- Превращать контракт данных в воспроизводимый DuckDB-аудит.

## Проблема

Вам передали четыре CSV и запрос «посчитать выручку по пользователям». Названия выглядят
знакомо: `users`, `orders`, `order_items`, `events`. Но до первого `JOIN` неизвестно:

- одна ли строка в `orders` действительно означает один заказ;
- уникален ли `event_id` после повторной доставки событий;
- существует ли каждый `orders.user_id` в `users`;
- является ли одна строка `order_items` заказом или товарной позицией.

Если принять неверный grain, запрос может синтаксически выполниться и вернуть правдоподобное
число. Повторный `event_id` завысит активность, неизвестный пользователь исчезнет при
`INNER JOIN`, а две позиции заказа размножат `orders.amount`.

## Концепция

**Grain** отвечает на вопрос: «Что означает одна строка?» Он задается словами до выбора
ключа и до вычислений.

| Таблица | Grain | Primary key |
|---|---|---|
| `users` | один пользователь | `user_id` |
| `orders` | один заказ | `order_id` |
| `order_items` | одна товарная позиция заказа | `order_id, product_id` |
| `events` | одна доставленная строка события | `event_id` |

**Primary key** должен однозначно идентифицировать строку заявленного grain:

1. ни одна часть ключа не равна `NULL`;
2. комбинация значений не повторяется.

Составной ключ проверяется как единая комбинация. Повтор `order_id` в `order_items`
нормален: один заказ содержит несколько товаров. Повтор пары `order_id, product_id`
нарушает контракт.

**Foreign key** описывает допустимую связь. `orders.user_id -> users.user_id` означает:
каждый непустой пользователь заказа должен существовать в `users`. Значение без
соответствия называется orphan key. Внешний ключ не доказывает уникальность дочерней
таблицы и не определяет cardinality всего `JOIN`.

Файлы CSV не хранят ограничения базы данных. Поэтому контракт и фактические проверки
нужны отдельно: описание сообщает намерение, SQL проверяет наблюдаемые данные.

## Соберите это

Сначала проверьте механизм без DuckDB.

### Шаг 1. Найдите дубликат ключа

Для последовательности событий:

```text
E0001, E0002, E0005, E0005
```

постройте словарь частот:

```text
E0001 -> 1
E0002 -> 1
E0005 -> 2
```

Группа `E0005` нарушает уникальность. Здесь одна duplicate group и две строки,
участвующие в дубликате.

### Шаг 2. Проверьте внешний ключ как разность множеств

```text
orders.user_id = {U001, U002, U999}
users.user_id  = {U001, U002}
```

Разность дочерних и родительских ключей равна `{U999}`. Это orphan key. Не удаляйте его
фильтром: задача аудита состоит в том, чтобы сделать нарушение видимым.

Исполняемый пример повторяет ручную проверку через `Counter`, затем сравнивает результат
с DuckDB:

```bash
uv run --locked python code/main.py
```

## Используйте это

DuckDB читает CSV напрямую. Для проверки дубликатов нужен тот же механизм частот:

```sql
SELECT event_id, count(*) AS row_count
FROM read_csv('../data/tiny/events.csv', header = true, all_varchar = true)
WHERE event_id IS NOT NULL
GROUP BY event_id
HAVING count(*) > 1;
```

Проверка orphan-ключей сохраняет дочернюю сторону и ищет отсутствие родителя:

```sql
WITH
orders AS (
    SELECT *
    FROM read_csv('../data/tiny/orders.csv', header = true, all_varchar = true)
),
users AS (
    SELECT *
    FROM read_csv('../data/tiny/users.csv', header = true, all_varchar = true)
)
SELECT orders.user_id, count(*) AS order_count
FROM orders
WHERE orders.user_id IS NOT NULL
  AND NOT EXISTS (
      SELECT 1
      FROM users
      WHERE users.user_id = orders.user_id
  )
GROUP BY orders.user_id;
```

Артефакт выполняет эти проверки для всех таблиц из `contract.json`:

```bash
uv run --locked python outputs/grain_key_audit.py \
  --data-dir ../data/tiny \
  --contract ../data/contract.json \
  --allow-failures
```

Отчет показывает:

- фактические и ожидаемые колонки;
- число строк;
- `NULL` и duplicate groups в primary key;
- примеры повторившихся комбинаций;
- orphan keys и число затронутых строк;
- общий `valid` и количество проваленных проверок.

Пути передаются в DuckDB параметрами. Имена таблиц и колонок берутся из контракта только
после проверки формата SQL-идентификатора.

## Сломайте это

Tiny-набор намеренно содержит два нарушения:

- `events.event_id = E0005` доставлен дважды;
- `orders.user_id = U999` отсутствует в `users`.

Без `--allow-failures` аудитор является quality gate и завершается с кодом `1`:

```bash
uv run --locked python outputs/grain_key_audit.py \
  --data-dir ../data/tiny \
  --contract ../data/contract.json
```

Попробуйте также:

1. удалить колонку `product_id` из копии `order_items.csv`;
2. сделать пустым один `users.user_id`;
3. продублировать только `order_id`, но оставить разные `product_id`;
4. добавить позицию для несуществующего заказа.

В третьем случае составной ключ остается допустимым. Остальные изменения должны сделать
соответствующую проверку невалидной.

## Проверьте это

Контрольные факты для committed tiny-набора:

- `users`: 8 строк, ключ `user_id` валиден;
- `orders`: 12 строк, ключ `order_id` валиден;
- `order_items`: 14 строк, составной ключ валиден;
- `events`: 16 строк, одна duplicate group и две duplicate rows;
- связь `orders -> users`: один orphan key и одна orphan row;
- связь `order_items -> orders` валидна;
- итоговый отчет содержит ровно две проваленные проверки.

Запустите behavioral tests:

```bash
uv run --locked python -m unittest discover -s tests
```

Тесты создают дополнительные временные данные с `NULL`-ключом и отсутствующей колонкой,
поэтому проверяют поведение, а не только запоминают committed CSV.

## Поставьте результат

`outputs/grain_key_audit.py` — самостоятельный CLI. Ему нужны каталог CSV и машинный
контракт. Артефакт не зависит от текста урока и подходит как локальный preflight перед
аналитическим запросом или как шаг CI.

Код возврата:

- `0`: все проверки прошли;
- `1`: данные прочитаны, но grain, ключ или связь нарушены;
- `2`: контракт или входные файлы нельзя проверить;
- `0` с `--allow-failures`: учебный режим для просмотра известных дефектов.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Grain | Количество строк | Смысл одной строки таблицы |
| Primary key | Любая колонка с `id` | Непустая уникальная комбинация, идентифицирующая grain |
| Composite key | Несколько независимых ключей | Один ключ из комбинации нескольких колонок |
| Foreign key | Гарантия `one-to-one` | Требование существования ключа в родительской таблице |
| Orphan key | Строка, которую надо удалить | Дочерний ключ без соответствующего родителя |
| Cardinality | Число колонок | Допустимое число соответствий между строками таблиц |
