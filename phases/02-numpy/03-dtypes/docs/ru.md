# Dtype, память и диапазоны

> Выбирайте `dtype` не по значениям маленького примера, а по предметному контракту: будущему диапазону, пропускам, допустимой ошибке и операциям, которые будут выполнены над массивом.

**Тип:** Build
**Треки:** Core
**Пререквизит:** `02-numpy/02-shape-and-axes`
**Время:** ~75 минут
**Результат:** оценивает память массива, выбирает `dtype` по допустимому диапазону, пропускам и точности, а затем проверяет, что преобразование не меняет значения недопустимым образом.

## Цели обучения

После урока вы сможете:

- связать знакомые `shape` и `size` с `itemsize` и `nbytes`;
- объяснить человеческим языком различие знаковых и беззнаковых целых типов;
- различать наблюдаемый, ожидаемый и возникающий после вычислений диапазоны;
- выбрать политику хранения пропусков до преобразования массива;
- объяснить практическую разницу между `float32` и `float64` через допустимую ошибку;
- обнаружить потерю дробной части, выход за диапазон и переполнение узкого integer;
- использовать `astype`, `np.can_cast` и `np.result_type`, понимая границы каждой проверки;
- проверить кандидат на соответствие dtype-контракту и оценить память планируемого массива.

## Проблема

В первом уроке мы выяснили, что у обычного `ndarray` один общий `dtype`. Во втором —
научились записывать форму вместе с предметными именами осей и предсказывать форму
результата.

Теперь объединим эти две идеи. Аналитическая система должна хранить почасовое количество
событий для магазинов:

```text
(store=500, day=365, hour=24)
```

Из контракта формы уже можно получить число элементов:

```text
size = 500 * 365 * 24 = 4 380 000
```

Tiny-выборка выглядит так:

```python
sample = [0, 12, 200]
```

Python и NumPy на обычной 64-битной платформе, скорее всего, создадут из нее `int64`.
Буфер планируемого массива тогда потребует:

```text
4 380 000 элементов * 8 байт = 35 040 000 байт
```

Если посмотреть только на текущие значения `0..200`, возникает соблазн выбрать `uint8`:

```text
4 380 000 элементов * 1 байт = 4 380 000 байт
```

Памяти потребуется в восемь раз меньше. Но описание источника разрешает рекламные пики до
`20 000` событий в час. `uint8` хранит только `0..255`, поэтому экономный тип нарушает
контракт будущих данных. Для диапазона `0..20 000` нужен как минимум `uint16`:

```text
4 380 000 элементов * 2 байта = 8 760 000 байт
```

`uint16` все еще экономит 75% буфера относительно `int64`, но не отбрасывает допустимые
значения.

Даже это решение еще не окончательное. Перед преобразованием нужно выяснить:

1. Могут ли после корректировок появиться отрицательные значения?
2. Возможны ли пропуски и как они должны храниться?
3. Будут ли значения складываться, умножаться или усредняться?
4. Какой диапазон появится после этих операций?
5. Допустима ли потеря точности, если тип вещественный?

Именно поэтому задача урока — не найти самый маленький `dtype`. Нужно найти достаточно
компактный тип, который сохраняет смысл данных и будущих вычислений.

## Концепция

### `dtype` — общий контракт элементов массива

Python-список может одновременно хранить целое число, строку и `None`. Обычный массив
NumPy интерпретирует все элементы по одному правилу:

```python
import numpy as np

values = np.array([1, 2, 3], dtype=np.int16)
```

`int16` сообщает NumPy как минимум три вещи:

- каждый элемент является целым числом;
- на каждый элемент отводится 16 бит, то есть 2 байта;
- допустимый диапазон ограничен возможностями знакового 16-битного представления.

У отдельного Python-числа нет обязанности занимать тот же фиксированный объём, что и
элемент `ndarray`. Поэтому Python `int` и NumPy `int16` нельзя считать взаимозаменяемыми
контрактами только потому, что оба сейчас показывают число `12`.

Один общий `dtype` также напоминает о границе модели. В прошлом уроке массив имел ось
`metric`, где рядом стояли количество заказов и выручка. Если показатели требуют разных
типов, правил пропусков или единиц хранения, удобнее держать их в отдельных однородных
массивах либо перейти к именованной табличной структуре. Узкий `dtype` не должен быть
причиной смешивать разные сущности.

### Как форма превращается в оценку памяти

Мы уже знаем:

```text
size = произведение длин shape
```

Теперь добавляется размер одного элемента:

```text
nbytes = size * itemsize
```

NumPy показывает оба значения непосредственно:

```python
values = np.zeros((2, 3), dtype=np.uint16)

assert values.size == 6
assert values.itemsize == 2
assert values.nbytes == 12
```

| Атрибут | Что означает | Что не означает |
|---|---|---|
| `size` | число элементов | число байтов |
| `itemsize` | байты одного элемента | диапазон значений сам по себе |
| `nbytes` | размер буфера элементов | вся память процесса Python |

`nbytes` не учитывает объект массива, служебные структуры, временные результаты и другие
объекты программы. Но для больших плотных массивов это полезная нижняя оценка и удобный
способ сравнить кандидатов до выделения памяти.

Планируемую память можно посчитать без создания огромного массива:

```python
from math import prod

planned_shape = (500, 365, 24)
planned_size = prod(planned_shape)
uint16_bytes = planned_size * np.dtype("uint16").itemsize

assert planned_size == 4_380_000
assert uint16_bytes == 8_760_000
```

### Почему восемь бит дают диапазон `0..255`

Один бит принимает два состояния. `b` бит дают `2**b` комбинаций. Для восьми бит:

```text
2**8 = 256 комбинаций
```

У беззнакового типа все комбинации отданы неотрицательным числам. Отсчет начинается с
нуля, поэтому последняя комбинация соответствует не `256`, а `255`:

```text
uint8: 0..255
```

Общая формула:

```text
unsigned minimum = 0
unsigned maximum = 2**b - 1
```

Знаковый тип делит пространство между отрицательной и неотрицательной частями. Для
привычного представления целых в NumPy:

```text
signed minimum = -(2**(b - 1))
signed maximum = 2**(b - 1) - 1
```

Отсюда получаются границы:

| `dtype` | Байты | Минимум | Максимум |
|---|---:|---:|---:|
| `uint8` | 1 | 0 | 255 |
| `int8` | 1 | -128 | 127 |
| `uint16` | 2 | 0 | 65 535 |
| `int16` | 2 | -32 768 | 32 767 |
| `uint32` | 4 | 0 | 4 294 967 295 |
| `int32` | 4 | -2 147 483 648 | 2 147 483 647 |

Запоминать всю таблицу не нужно. Важно понимать устройство диапазона и проверять точные
границы через `np.iinfo`.

`unsigned` не является «более точным integer». Он просто не хранит отрицательные значения
и использует освободившиеся комбинации для более высокой верхней границы. Выбирайте его
только тогда, когда отрицательные значения запрещены предметным контрактом, а не потому,
что текущий файл случайно не содержит минусов.

### Три диапазона вместо одного

Безопасное решение учитывает три разных диапазона.

#### 1. Наблюдаемый диапазон

Это минимум и максимум конкретной выборки:

```text
sample: 0..200
```

Он нужен для диагностики текущих данных, но не описывает все будущие значения.

#### 2. Ожидаемый диапазон

Это границы из предметного контракта, спецификации источника или согласованной политики:

```text
allowed hourly events: 0..20 000
```

Именно этот диапазон должен помещаться в тип, если массив будет использоваться в
production. Если границы неизвестны, честный результат аудита — предупреждение, а не
уверенная рекомендация.

#### 3. Производный диапазон

Это значения, которые могут появиться после вычисления. Если сложить 365 дневных значений
до `20 000`, теоретический максимум равен:

```text
365 * 20 000 = 7 300 000
```

Исходное значение помещается в `uint16`, а годовой итог — уже нет. NumPy часто использует
более широкий аккумулятор для сумм узких integer, но полагаться на это без проверки
нельзя. Нужно предсказывать тип и диапазон результата, особенно перед сохранением обратно
в узкий массив или внешнюю схему.

Таким образом, «тип подходит исходному столбцу» и «тип подходит всему pipeline» — разные
утверждения.

### Идентификатор не становится измерением из-за integer dtype

Числовой идентификатор может выглядеть как обычное целое, но над ним обычно не нужно
складывать или вычислять среднее. При выборе хранения важно учитывать:

- максимальное значение всей системы, а не текущего файла;
- специальные или внешние диапазоны идентификаторов;
- возможную смену генератора ключей;
- необходимость сохранить ведущие нули, если это на самом деле код.

Поэтому правило «возьмем минимальный integer по `min` и `max`» особенно опасно для ID и
кодов. Сначала определите семантический тип значения, затем физическое представление.

### Пропуск — часть контракта, а не случайное значение

Обычные integer dtype NumPy не имеют отдельного представления `NaN`:

```python
np.array([1, np.nan, 3], dtype=np.int16)
```

Такое преобразование невозможно. Перед выбором типа нужно решить, что означает отсутствие
значения.

| Политика | Когда подходит | Риск |
|---|---|---|
| Остановить расчет | пропуски запрещены контрактом | pipeline требует исправления источника |
| Floating dtype с `NaN` | дробные значения естественны или потеря integer-семантики допустима | целые представляются как float |
| Отдельная маска валидности | нужен числовой буфер и явное состояние отсутствия | два массива нужно хранить согласованно |
| Sentinel, например `-1` | значение гарантированно вне предметного диапазона | будущий диапазон может пересечь sentinel |
| Nullable integer таблицы | нужны целые значения и пропуски в табличной модели | это уже модель pandas, а не обычный `ndarray` |

Нельзя автоматически заменить пропуск нулем: ноль часто является настоящим наблюдением.
Нельзя автоматически выбрать `-1`: в данных с корректировками отрицательное значение
может быть допустимым. Политика должна быть названа явно.

### `float32` и `float64`: вопрос допустимой ошибки

Вещественные типы хранят приближения. Большинство десятичных дробей не имеют точного
конечного двоичного представления, поэтому значения после вычислений могут немного
отличаться от математического результата.

Для практического выбора достаточно помнить порядок величин:

- `float32` занимает 4 байта и обычно сохраняет около 7 значащих десятичных цифр;
- `float64` занимает 8 байт и обычно сохраняет около 15–16 значащих цифр.

Число значащих цифр важнее количества знаков после запятой. Посмотрим на значение большого
масштаба:

```python
source = np.array([10_000_000.1], dtype=np.float64)
narrow = source.astype(np.float32)

print(source)
print(narrow)
```

В `float32` небольшая дробная часть рядом с десятью миллионами теряется. Ошибку можно
измерить:

```python
restored = narrow.astype(np.float64)
max_abs_error = np.max(np.abs(restored - source))

assert max_abs_error > 0.09
```

Если допустимая ошибка признака модели равна `0.2`, такое преобразование может пройти
контракт. Если финансовый итог должен совпадать до копейки, оно недопустимо.

`np.finfo(dtype).eps` показывает расстояние от `1.0` до следующего представимого числа
данного типа. Это свойство типа, а не универсальный допуск для любого массива. Ошибка
зависит от масштаба значений и последовательности операций. Устойчивые сравнения и
накопление ошибки подробно рассматриваются в уроке `02/09`.

Для денежных сумм часто безопаснее хранить целое число минимальных денежных единиц, если
такой контракт поддерживает предметную задачу. Но это не универсальное решение: валюты,
округление и внешние системы требуют отдельной спецификации.

### `astype` выполняет преобразование, но не утверждает его безопасность

```python
values = np.array([12.9, -2.1])
converted = values.astype(np.int16)

print(converted)
```

```text
[12 -2]
```

Дробная часть отброшена в сторону нуля. NumPy выполнил запрошенную операцию, но не знает,
допустима ли такая потеря для бизнеса.

Выход за диапазон при преобразовании уже созданного массива также может не остановить
расчет:

```python
values = np.array([256])
converted = values.astype(np.uint8)

assert converted[0] == 0
```

Поэтому безопасная последовательность выглядит так:

1. проверить наблюдаемый и ожидаемый диапазоны;
2. проверить пропуски и дробные значения;
3. выполнить преобразование;
4. измерить ошибку или точное совпадение после round-trip;
5. только затем заменить исходный тип в pipeline.

### Что проверяют `np.can_cast` и `np.result_type`

`np.can_cast` сравнивает типы, а не конкретные значения:

```python
np.can_cast(np.dtype("int64"), np.dtype("uint16"), casting="safe")
# False
```

Ответ консервативен и правильный на уровне типов: произвольный `int64` может быть
отрицательным или превышать `65 535`. Но конкретный предметный контракт `0..20 000`
помещается в `uint16`. Поэтому `np.can_cast` полезен как строгий барьер, но не заменяет
проверку диапазона конкретных данных и будущих границ.

`np.result_type` помогает предсказать тип результата совместной операции:

```python
result_dtype = np.result_type(
    np.array([1], dtype=np.uint8),
    np.array([1], dtype=np.int16),
)

assert result_dtype == np.dtype("int16")
```

Это полезнее догадки по одному операнду. Точные правила продвижения NumPy 2.x сложнее
одной таблицы; на этом этапе достаточно проверять тип результата для критичных операций,
а не запоминать все сочетания.

### Алгоритм выбора dtype

Перед изменением типа пройдите семь шагов.

1. **Назовите смысл значений.** Это количество, измерение, сумма, коэффициент, ID или код?
2. **Задайте допустимые будущие границы.** Не ограничивайтесь минимумом и максимумом tiny-выборки.
3. **Определите политику пропусков.** Ошибка, `NaN`, маска, sentinel или nullable-таблица?
4. **Учтите следующие операции.** Какой диапазон и тип появятся после суммы, разности или умножения?
5. **Задайте допустимую ошибку.** Для floating-преобразования она должна следовать из задачи.
6. **Сравните память кандидатов.** Используйте планируемый `shape`, а не только размер примера.
7. **Проверьте преобразование.** Диапазон, дробную часть, round-trip и тип результата операции.

Самый маленький тип рассматривается только на шестом шаге. Экономия памяти не может
компенсировать неверные данные.

## Соберите это

Построим прозрачную версию dtype-контракта до использования готовых справочников NumPy.

### Шаг 1. Оцените число элементов и память

```python
from math import prod

planned_shape = (500, 365, 24)
planned_size = prod(planned_shape)

bytes_per_int64 = 8
bytes_per_uint16 = 2

int64_nbytes = planned_size * bytes_per_int64
uint16_nbytes = planned_size * bytes_per_uint16

assert planned_size == 4_380_000
assert int64_nbytes == 35_040_000
assert uint16_nbytes == 8_760_000
```

Это продолжение shape-контракта прошлого урока: длины осей определяют `size`, а `dtype`
добавляет стоимость каждого элемента.

### Шаг 2. Выведите диапазон integer

```python
def integer_bounds(bits, signed):
    if signed:
        return -(2 ** (bits - 1)), 2 ** (bits - 1) - 1
    return 0, 2**bits - 1
```

```python
assert integer_bounds(8, signed=False) == (0, 255)
assert integer_bounds(8, signed=True) == (-128, 127)
assert integer_bounds(16, signed=False) == (0, 65_535)
assert integer_bounds(16, signed=True) == (-32_768, 32_767)
```

### Шаг 3. Разделите наблюдение и контракт

```python
sample_min = 0
sample_max = 200

expected_min = 0
expected_max = 20_000
```

Проверка `uint8` по выборке пройдет:

```python
uint8_min, uint8_max = integer_bounds(8, signed=False)

observed_fits_uint8 = (
    sample_min >= uint8_min
    and sample_max <= uint8_max
)

assert observed_fits_uint8
```

Проверка предметного контракта — нет:

```python
expected_fits_uint8 = (
    expected_min >= uint8_min
    and expected_max <= uint8_max
)

assert not expected_fits_uint8
```

Для `uint16` проходят обе проверки.

### Шаг 4. Зафиксируйте политику пропусков

Минимальная спецификация может выглядеть так:

```python
dtype_contract = {
    "semantic_type": "event_count",
    "expected_min": 0,
    "expected_max": 20_000,
    "allow_missing": False,
    "target_dtype": "uint16",
}
```

Если пропуск обнаружен при `allow_missing=False`, расчет должен остановиться. Если
пропуски разрешены, спецификация обязана назвать способ хранения; одного флага в
production-контракте недостаточно.

### Шаг 5. Проверьте преобразование float через round-trip

```python
source = np.array([10_000_000.1], dtype=np.float64)
converted = source.astype(np.float32)
restored = converted.astype(np.float64)

error = np.max(np.abs(restored - source))

assert error > 0.09
```

Теперь решение можно сравнить с бизнес-допуском:

```python
max_allowed_error = 0.01

assert error > max_allowed_error
```

Значит, `float32` должен быть отклонен для этого сценария, хотя само преобразование
синтаксически успешно.

### Шаг 6. Сформулируйте итог до кода

Для массива событий решение выглядит так:

```text
observed range: 0..200
expected range: 0..20 000
missing policy: missing values are rejected
planned shape: (500, 365, 24)
target dtype: uint16
projected buffer: 8 760 000 bytes
decision: approved for storage, derived aggregates require a separate dtype check
```

Это и есть прозрачный механизм, который автоматизирует артефакт урока.

## Используйте это

### Получите точные свойства типа

```python
uint16 = np.dtype("uint16")
info = np.iinfo(uint16)

assert uint16.itemsize == 2
assert info.min == 0
assert info.max == 65_535
assert info.bits == 16
```

Для вещественного типа:

```python
float32 = np.finfo(np.float32)

print(float32.bits)
print(float32.eps)
print(float32.min)
print(float32.max)
```

Эти значения описывают возможности типа. Они не определяют автоматически допустимую
ошибку вашего расчета.

### Проверьте integer-контракт аудитором

Из корня репозитория:

```bash
uv run --locked python phases/02-numpy/03-dtypes/outputs/dtype_audit.py \
  --values '[0, 12, 200]' \
  --target-dtype uint16 \
  --expected-min 0 \
  --expected-max 20000 \
  --planned-shape '[500, 365, 24]'
```

Главные поля отчета:

```text
status: approved
observed range: 0..200
expected range: 0..20 000
recommendation basis: expected_contract
smallest integer dtype: uint16
planned size: 4 380 000
target projected nbytes: 8 760 000
```

`--planned-shape` используется только для оценки памяти. Артефакт не утверждает, что
tiny-выборка уже имеет такую форму.

Теперь проверим привлекательный, но слишком узкий тип:

```bash
uv run --locked python phases/02-numpy/03-dtypes/outputs/dtype_audit.py \
  --values '[0, 12, 200]' \
  --target-dtype uint8 \
  --expected-min 0 \
  --expected-max 20000
```

Отчет вернет `status: rejected`: наблюдаемые значения помещаются, ожидаемый диапазон —
нет. Для отклоненного контракта CLI завершается с кодом `1`, но все равно печатает JSON с
причинами. Ошибка входных аргументов использует код `2` и понятное сообщение без traceback.

### Проверьте пропуски

```bash
uv run --locked python phases/02-numpy/03-dtypes/outputs/dtype_audit.py \
  --values '[12.5, null, 18.0]' \
  --target-dtype float32 \
  --expected-min 0 \
  --expected-max 100 \
  --allow-missing \
  --max-abs-error 0.001
```

Здесь `null` преобразуется в `NaN`, а floating target способен его представить. Флаг
`--allow-missing` означает, что пропуск разрешен этим учебным контрактом; в реальном
проекте рядом нужно описать его предметный смысл и обработку.

Если заменить target на `int16`, аудитор отклонит решение даже с `--allow-missing`: обычный
integer не умеет хранить `NaN`.

### Проверьте допустимую ошибку float

```bash
uv run --locked python phases/02-numpy/03-dtypes/outputs/dtype_audit.py \
  --values '[10000000.1]' \
  --target-dtype float32 \
  --expected-min 0 \
  --expected-max 20000000 \
  --max-abs-error 0.01
```

Преобразование будет отклонено: максимальная ошибка round-trip превышает `0.01`.

Если narrowing с `float64` до `float32` запрошен без `--max-abs-error`, аудитор возвращает
`warning`, а не выдумывает универсальный допуск.

### Прочитайте результат `np.can_cast` правильно

В approved-отчете переход `int64 -> uint16` может иметь:

```text

```

Это не противоречие. NumPy отвечает про все возможные `int64`, а dtype-контракт — про
явно ограниченный диапазон `0..20 000`. Отчет показывает оба уровня, чтобы пользователь
понимал основание решения.

## Сломайте это

### Ошибка 1. Выбрать тип только по tiny-выборке

```text
observed: 0..200
expected: 0..20 000
candidate: uint8
```

Сегодняшний файл пройдет, рекламный пик нарушит контракт. Наблюдаемый диапазон не заменяет
описание источника.

### Ошибка 2. Использовать unsigned до появления корректировки

```python
np.array([-1], dtype=np.uint16)
```

Если `-1` означает допустимую корректировку, unsigned выбран неверно. Если `-1` означает
пропуск, проблема находится в политике пропусков. В обоих случаях нельзя просто удалить
отрицательную строку.

### Ошибка 3. Довериться `astype` при выходе за диапазон

```python
value = np.array([256])
converted = value.astype(np.uint8)

assert converted[0] == 0
```

Запрошенное преобразование выполнилось с потерей значения. Диапазон нужно проверять до
`astype`.

### Ошибка 4. Незаметно отбросить дробную часть

```python
rates = np.array([12.9, -2.1])
integer_rates = rates.astype(np.int16)

assert np.array_equal(integer_rates, np.array([12, -2]))
```

Это не округление до ближайшего целого. Если дробная часть несет смысл, integer target
должен быть отклонен.

### Ошибка 5. Использовать sentinel, который стал допустимым значением

В первой версии контракта `-1` мог означать пропуск, потому что события были только
неотрицательными. После появления корректировок `-1` стал настоящим значением. Старые
пропуски теперь невозможно отличить от наблюдений без дополнительной информации.

### Ошибка 6. Потерять малое изменение рядом с большим float

```python
large = np.float32(10_000_000)
result = large + np.float32(0.1)

assert result == large
```

Операция выполнилась, но выбранный масштаб и точность `float32` не позволили сохранить
прибавление.

### Ошибка 7. Переполнить уже существующий узкий массив

```python
left = np.array([250], dtype=np.uint8)
right = np.array([10], dtype=np.uint8)
result = left + right

assert result[0] == 4
```

Математический результат `260` не помещается в `uint8`, и вычисление обернулось по
диапазону. Проверка при создании массива не защищает все будущие операции.

### Ошибка 8. Проверить хранение, но не тип агрегата

```python
daily = np.full(365, 20_000, dtype=np.uint16)
year_total = daily.sum()

print(year_total)
print(year_total.dtype)
```

NumPy обычно использует более широкий integer для такого суммирования. Это помогает, но
после расчета результат нельзя без проверки записывать обратно в `uint16`: значение
`7 300 000` там не помещается.

### Ошибка 9. Принять `nbytes` за всю память pipeline

Даже если исходный буфер занимает 9 МБ, операция может создать временный массив другого
типа, копию или несколько промежуточных результатов. `nbytes` — полезная оценка хранения,
но не профиль пикового потребления памяти.

## Поставьте результат

Именованный артефакт урока:

```text
phases/02-numpy/03-dtypes/outputs/dtype_audit.py
```

Аудитор принимает sample значений и отдельно заданный контракт:

- `--target-dtype` — проверяемый кандидат;
- `--expected-min`, `--expected-max` — будущие предметные границы;
- `--allow-missing` — явное разрешение пропусков;
- `--max-abs-error` — бюджет ошибки для преобразования;
- `--planned-shape` — форма для оценки памяти без выделения большого массива.

Отчет разделяет:

- фактический dtype, форму, память, пропуски и observed range;
- expected range и политику;
- возможности и планируемую память target dtype;
- type-level результат `np.can_cast`;
- максимальную round-trip ошибку;
- отдельные проверки и причины отказа;
- итоговый статус `approved`, `warning` или `rejected`.

Статус `approved` означает только соответствие переданному контракту. Если expected bounds
получены из догадки, одобрение технически воспроизводимо, но предметно необоснованно.
Сохраняйте источник границ рядом со схемой данных или решением о типе.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| `dtype` | «Python-тип внутри каждой ячейки» | Общее фиксированное представление элементов `ndarray` |
| `itemsize` | «Количество элементов» | Размер одного элемента в байтах |
| `nbytes` | «Вся память программы» | Объём буфера элементов массива |
| Signed integer | «Любой целый тип» | Integer, диапазон которого включает отрицательные значения |
| Unsigned integer | «Более точный integer» | Неотрицательный integer с большей верхней границей при той же разрядности |
| Наблюдаемый диапазон | «Гарантия будущих значений» | Минимум и максимум конкретной проверенной выборки |
| Ожидаемый диапазон | «Текущий `min` и `max`» | Допустимые будущие границы из предметного контракта |
| Производный диапазон | «Тот же диапазон, что у входа» | Возможные значения после вычисления |
| `NaN` | «Универсальный пропуск для любого dtype» | Специальное floating-point значение, которого нет в обычном integer dtype |
| Sentinel | «Безопасная замена `NaN`» | Обычное значение, зарезервированное политикой как маркер отсутствия |
| `astype` | «Безопасно меняет тип» | Выполняет запрошенное преобразование, которое может потерять данные |
| Round-trip | «Повторное сохранение файла» | Преобразование в target и обратно для измерения изменения значений |
| `np.can_cast` | «Проверяет текущие значения» | Консервативно проверяет совместимость типов по всем возможным значениям |
| `np.result_type` | «Показывает dtype первого аргумента» | Предсказывает общий dtype совместной операции переданных объектов |
| `eps` | «Универсальный допуск float» | Расстояние от `1.0` до следующего представимого числа данного floating dtype |

Особенно полезен разбор связи `dtype`, `itemsize`, `size` и `nbytes`: по нему можно самостоятельно оценить память до создания большого массива.

Обратите внимание на параметры `casting` и `copy`, а затем проверьте на своих данных, чем успешное выполнение операции отличается от сохранения значений.

- [`numpy.result_type` (EN)](https://numpy.org/doc/stable/reference/generated/numpy.result_type.html) — небольшой API-справочник для предсказания `dtype` результата до вычисления. Повторите примеры, а затем передайте типы из своего аудита и сравните прогноз с фактическим результатом операции.
- [NumPy: Data type promotion (EN)](https://numpy.org/doc/stable/reference/arrays.promotion.html) — понятное объяснение того, как NumPy выбирает общий тип для разных операндов. В первую очередь разберите иерархию `uint → int → float → complex` и отдельные правила для Python-скаляров в NumPy 2.x.
- [NEP 50: Promotion rules for Python scalars (EN)](https://numpy.org/neps/nep-0050-scalar-promotion.html) — первичный документ, объясняющий, почему современные правила продвижения типов были изменены. Это материал для углубления: читайте мотивацию и примеры поведения, а детали реализации можно пропустить.
