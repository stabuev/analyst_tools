# Векторизация и производительность

> Векторизация переносит повторяющуюся операцию из Python-цикла в операции над массивами,
> но ускорение имеет смысл обсуждать только после проверки результата, границ измерения и
> расхода памяти.

**Тип:** Case

**Треки:** Core

**Пререквизиты:** уроки `02-numpy/01-arrays`–`02-numpy/08-random-simulations`

**Время:** ~75 минут

**Результат:** переносит поэлементный расчет из Python-цикла в операции NumPy, проверяет
построчную и итоговую эквивалентность, честно сравнивает повторные времена и объясняет
компромисс между скоростью, памятью и понятностью.

## Цели обучения

После урока вы сможете:

- назвать единицу наблюдения и предметный смысл оси в векторизуемом расчете;
- построить прозрачный Python-цикл как контрольную реализацию формулы;
- выразить ту же формулу через поэлементные операции и агрегацию NumPy;
- отличить настоящий array API от `np.vectorize`, который лишь оборачивает Python-функцию;
- проверить наиболее детальный результат до сравнения итоговой суммы;
- отличить расчетное время от end-to-end времени вместе с подготовкой данных;
- выполнить предварительный вызов, несколько замеров и сравнить медианы;
- правильно прочитать отношение времен, не превращая его в универсальное обещание;
- оценить память входных и временных массивов;
- объяснить, когда обычный цикл или обработка блоками разумнее полной векторизации.

## Проблема

В таблице продаж одна строка соответствует не заказу целиком, а одной позиции заказа.
Для каждой позиции нужно рассчитать выручку после скидки:

```text
line_revenue = price * quantity * (1 - discount)
```

Затем выручка всех позиций складывается:

```text
net_revenue = sum(line_revenue)
```

В исследовательском ноутбуке аналитик написал понятный цикл. На нескольких строках он
работает мгновенно, а на сотнях тысяч позиций начинает занимать заметное время. Возникает
естественная идея: заменить цикл выражением NumPy.

Но более короткая запись сама по себе не отвечает на важные вопросы:

- совпадает ли формула для каждой позиции;
- одинаковые ли данные получают две реализации;
- входят ли преобразования списков в массивы в измеряемое время;
- достаточно ли велик набор данных, чтобы сравнение имело рабочий смысл;
- сколько временных массивов создает выражение;
- можно ли переносить измеренное ускорение на другую машину и другой pipeline.

Если сразу запустить таймер, легко очень точно измерить две разные задачи. В этом уроке
мы сначала зафиксируем вычислительный контракт, затем докажем эквивалентность и только
после этого сравним время.

## Концепция

### Один расчет связывает все предыдущие уроки NumPy

Векторизация не является отдельным трюком, который отменяет модель массива. Она собирает
вместе уже изученные идеи:

| Предыдущая тема | Роль в расчете выручки |
|---|---|
| `ndarray` | хранит однородные цены, количества и скидки |
| Shape и axes | все три массива относятся к одной оси `line_item` |
| `dtype` | определяет диапазон, точность и тип результата умножения |
| Индексация и маски | позволяют проверить или отобрать отдельные позиции |
| Агрегации | `sum()` убирает ось позиций и возвращает общий итог |
| Broadcasting | объясняет, как скаляр `1.0` применяется ко всем скидкам |
| Случайность | локальный `Generator` создает воспроизводимые входы benchmark |

Если форма, тип или смысл оси не определены, векторизация лишь быстрее выполнит
неопределенный расчет.

### Сначала предметный контракт

В учебном примере есть три одномерных массива:

| Массив | Смысл элемента | Ожидаемая форма | Рабочий `dtype` | Допустимые значения |
|---|---|---|---|---|
| `prices` | цена единицы товара | `(line_item,)` | `float64` | конечное число `>= 0` |
| `quantities` | число единиц | `(line_item,)` | `int64` | целое число `>= 0` |
| `discounts` | доля скидки | `(line_item,)` | `float64` | конечное число в `[0, 1]` |

Одинаковая длина здесь означает не просто техническую совместимость. Элементы с одним
индексом должны описывать одну и ту же позицию заказа:

```text
prices[i], quantities[i], discounts[i] -> line_item[i]
```

Если массив скидок переставлен относительно массива цен, формы останутся одинаковыми и
NumPy выполнит расчет без ошибки. Поэтому shape-контракт необходим, но не доказывает
согласованность строк исходных данных.

### Контрольный пример на трех позициях

Возьмем значения, которые можно проверить без компьютера:

| Позиция | Цена | Количество | Скидка | Выручка после скидки |
|---:|---:|---:|---:|---:|
| 0 | 100 | 2 | 0% | `100 * 2 * 1.00 = 200` |
| 1 | 80 | 1 | 25% | `80 * 1 * 0.75 = 60` |
| 2 | 50 | 4 | 10% | `50 * 4 * 0.90 = 180` |

Наиболее детальный ожидаемый результат:

```text
line_revenue = [200, 60, 180]
```

И только затем итог:

```text
net_revenue = 200 + 60 + 180 = 440
```

Это разделение принципиально. Одинаковая итоговая сумма может получиться у разных наборов
построчных значений. Например, `[200, 60, 180]` и `[180, 60, 200]` оба дают `440`, но
приписывают выручку не тем позициям.

### Что именно означает векторизация

Python-цикл выполняет тело цикла отдельно для каждого элемента:

```text
получить три Python-объекта
-> выполнить Python-умножение и вычитание
-> обновить Python-переменную
-> перейти к следующей позиции
```

Векторное выражение передает NumPy целые массивы и описание операции:

```text
умножить соответствующие элементы массивов
-> применить скидку
-> агрегировать результат
```

Циклы не исчезают. Они выполняются внутри библиотеки в скомпилированном коде над
однородными буферами памяти. Python организует несколько крупных операций вместо сотен
тысяч мелких итераций.

В этом курсе под векторизацией понимается именно выражение расчета через array API вместо
Python-цикла по отдельным элементам. Это слово само по себе не обещает:

- использование GPU;
- параллельное выполнение на нескольких ядрах;
- применение SIMD в каждом конкретном вызове;
- отсутствие внутренних циклов;
- отсутствие временных массивов;
- автоматическое ускорение любого алгоритма.

Конкретная операция NumPy может использовать аппаратные оптимизации, но пользовательский
контракт остается тем же: массивы, формы, типы и операция над ними.

### Название функции не доказывает векторизацию

У NumPy есть функция с многообещающим именем `np.vectorize`:

```python
def discounted_revenue(price: float, quantity: int, discount: float) -> float:
    return price * quantity * (1.0 - discount)

wrapped = np.vectorize(discounted_revenue)
line_revenue = wrapped(prices, quantities, discounts)
```

Такая запись принимает массивы и поддерживает broadcasting, но по существу вызывает
обычную Python-функцию для отдельных элементов. Это удобная адаптация интерфейса, а не
способ убрать Python-цикл ради производительности. Официальная документация прямо
описывает `np.vectorize` как средство удобства, а не ускорения.

Настоящее векторное выражение для этой задачи использует операции NumPy над целыми
массивами:

```python
line_revenue = prices * quantities * (1.0 - discounts)
```

Практическое правило:

- если array API или ufunc уже выражает операцию, используйте его;
- если передаете Python callback, выясните, не вызывается ли он отдельно для каждого
  элемента;
- не делайте вывод о механизме по имени функции или короткой записи;
- после проверки эквивалентности измерьте реальную рабочую нагрузку.

`np.vectorize` может быть полезен для удобного применения редкой скалярной функции, когда
производительность несущественна. Но заменить им цикл и объявить задачу
«векторизованной и ускоренной» нельзя.

### Поэлементные операции и редукция — разные этапы

Выражение:

```python
line_revenue = prices * quantities * (1.0 - discounts)
```

сохраняет ось `line_item`:

```text
prices:       (line_item=n,)
quantities:   (line_item=n,)
discounts:    (line_item=n,)
line_revenue: (line_item=n,)
```

Агрегация:

```python
net_revenue = line_revenue.sum(dtype=np.float64)
```

убирает единственную ось и возвращает скаляр. Это два разных наблюдаемых результата:

1. выручка каждой позиции;
2. общая выручка набора позиций.

Проверять только второй результат недостаточно: ошибки отдельных строк могут
компенсироваться при сложении.

### Как ведут себя типы

При контракте урока:

```text
prices.dtype       = float64
quantities.dtype   = int64
discounts.dtype    = float64
```

выражение `1.0 - discounts` возвращает `float64`, смешанное умножение с ценами также дает
`float64`, а сумма явно накапливается в `float64`.

Порядок операций имеет значение. Если и цены, и количества хранятся в узком целом типе,
то промежуточное `prices * quantities` может переполниться до встречи с вещественной
скидкой. Поэтому готовый артефакт сначала проверяет данные и приводит цены к `float64`, а
количества — к `int64`. Это применение урока о `dtype`, а не косметическое форматирование.

### Почему NumPy часто быстрее

Обе реализации выполняют линейное число операций: при удвоении числа позиций объем
работы примерно удваивается. Разница обычно возникает в постоянных накладных расходах:

- Python-цикл интерпретирует каждую итерацию;
- NumPy получает одну крупную операцию над однородным массивом;
- данные ndarray расположены компактнее Python-объектов;
- библиотека может использовать оптимизированные машинные инструкции.

Из этого не следует, что NumPy быстрее на любом размере. Для трех элементов подготовка и
вызов операции могут стоить больше самого цикла. Для ветвистого или последовательного
алгоритма естественной векторной операции может вообще не быть.

### Три независимых контракта

Корректное сравнение удобно разделить на три вопроса.

#### 1. Контракт результата

- одна и та же предметная формула;
- одинаковые позиции и значения входов;
- согласованные `shape` и `dtype`;
- построчный результат совпадает с заданной точностью;
- итоговая сумма совпадает с заданной точностью.

#### 2. Контракт измерения

- явно названа работа внутри таймера;
- обе реализации получают заранее подготовленные данные;
- до замеров выполнен предварительный вызов;
- сделано несколько повторов;
- сохранены все времена и выбран способ сводки;
- размер входа и среда записаны вместе с результатом.

#### 3. Контракт вывода

- ускорение относится только к измеренному входу и scope;
- результат не объявляется свойством всех машин и всех задач;
- стоимость памяти рассматривается вместе со временем;
- быстрый неправильный расчет не считается улучшением.

### Calculation-only и end-to-end — разные вопросы

Граница измерения называется scope. В этом уроке основной артефакт отвечает на узкий
вопрос:

> Сколько занимает сама формула и суммирование, если обе реализации уже получили удобное
> для них подготовленное представление данных?

Это `calculation_only`:

```text
Python branch: готовые списки -> цикл и сумма
NumPy branch:  готовые ndarray -> выражение и np.sum
```

За пределами таймера остаются генерация, проверка, преобразование ndarray в списки,
сверка результатов и запись JSON.

Другой рабочий вопрос может звучать так:

> Что быстрее от исходных Python-списков до итогового числа?

Тогда NumPy-ветка обязана включать `np.asarray`. Это уже end-to-end scope:

```text
Python branch: исходные списки -> цикл и сумма
NumPy branch:  исходные списки -> np.asarray -> выражение и np.sum
```

Оба сравнения допустимы, но отвечают на разные вопросы. Нельзя исключить преобразование
из подписи отчета, а потом сделать вывод о полной стоимости pipeline.

### Зачем нужны предварительный вызов и повторы

Для измерения используется `time.perf_counter_ns()`: монотонный таймер с высоким
разрешением для измерения прошедшего времени. Он измеряет wall time, поэтому в результат
могут попасть переключения процессов и фоновая нагрузка.

Один предварительный вызов помогает не смешивать основное измерение с частью разовых
эффектов первого запуска. Это не «очищает» машину и не гарантирует стабильность.

Несколько повторов показывают разброс. В уроке сводкой служит медиана — центральное
значение упорядоченных времен. Отдельный долгий запуск меньше влияет на медиану, чем на
среднее. При этом медиана семи повторов не превращает локальный микробенчмарк в
универсальный закон.

### Как читать speedup

В артефакте отношение вычисляется так:

```text
speedup = median_loop_seconds / median_vectorized_seconds
```

Примеры интерпретации:

| Результат | Что он означает в зафиксированном benchmark |
|---:|---|
| `20.0` | медиана цикла в 20 раз больше медианы NumPy-ветки |
| `1.0` | медианы примерно равны |
| `0.7` | в этом измерении цикл оказался быстрее |

Фраза «NumPy всегда быстрее в 20 раз» не следует ни из одного такого числа. Корректная
формулировка называет размер, scope, реализации, число повторов и среду.

### Векторизация тратит память на промежуточные результаты

Для `n` позиций входные ndarray урока занимают:

```text
prices:       n * 8 bytes
quantities:   n * 8 bytes
discounts:    n * 8 bytes
--------------------------------
inputs:       n * 24 bytes
```

Прямое выражение можно мысленно разложить так:

```python
gross = prices * quantities
multiplier = 1.0 - discounts
line_revenue = gross * multiplier
```

Каждый из трех результатов имеет `float64` и занимает еще `n * 8` байт. Во время
последнего умножения эти буферы могут одновременно существовать, поэтому учебная верхняя
оценка временных ndarray равна `3 * n * 8` байт.

Для миллиона позиций это примерно:

```text
входные буферы ndarray:       24 MB
три временных float64-вектора: 24 MB
```

Это не оценка всего процесса: в ней нет Python-списков, Python-объектов, служебных данных
аллокатора и памяти интерпретатора. Артефакт явно сохраняет эту границу оценки.

Если память является ограничением, один рабочий буфер можно переиспользовать:

```python
workspace = np.empty_like(prices, dtype=np.float64)
np.subtract(1.0, discounts, out=workspace)
np.multiply(workspace, quantities, out=workspace)
np.multiply(workspace, prices, out=workspace)
net_revenue = workspace.sum(dtype=np.float64)
```

Такой вариант сложнее читать и легче случайно испортить изменением не того массива.
Оптимизировать память стоит после оценки, а не ради самого использования `out=`.

### Когда цикл остается разумным

Обычный цикл не является ошибкой сам по себе. Он может быть хорошим решением, если:

- данных мало и время не влияет на рабочий процесс;
- следующий результат зависит от предыдущего состояния;
- логика содержит сложные ветвления, для которых нет ясной операции массива;
- векторная запись создает слишком большие временные массивы;
- обработка блоками проще укладывается в ограничение памяти;
- основное время занимает сеть, диск или база данных, а не Python-цикл;
- векторная версия настолько сложна, что ее трудно проверить и сопровождать.

Цель не в том, чтобы удалить все циклы. Цель — выбрать понятную и проверенную реализацию
с приемлемой стоимостью.

### Где заканчивается этот урок

Здесь нужен только минимальный навык: доказать эквивалентность, назвать scope, повторить
замер и не забыть о памяти. Полноценные правила benchmark, профилирование CPU и памяти,
cache policy и сравнение pandas, DuckDB и Polars рассматриваются в
[фазе 12 «Производительность аналитики»](../../../12-performance/README.md).

## Соберите это

### Шаг 1. Зафиксируйте маленький контрольный набор

Начните со списков, а не со случайного миллиона строк:

```python
prices = [100.0, 80.0, 50.0]
quantities = [2, 1, 4]
discounts = [0.0, 0.25, 0.10]

expected_line_revenue = [200.0, 60.0, 180.0]
expected_total = 440.0
```

Назовите ось списков `line_item`. Элементы с одним индексом относятся к одной позиции.

### Шаг 2. Проверьте минимальный предметный контракт

Прозрачная проверка для учебных списков:

```python
import math


def validate_line_items(prices, quantities, discounts):
    if not (len(prices) == len(quantities) == len(discounts)):
        raise ValueError("inputs must have equal lengths")
    if not prices:
        raise ValueError("inputs must not be empty")

    if any(not math.isfinite(price) or price < 0 for price in prices):
        raise ValueError("prices must be finite and non-negative")
    if any(
        isinstance(quantity, bool)
        or not isinstance(quantity, int)
        or quantity < 0
        for quantity in quantities
    ):
        raise ValueError("quantities must be non-negative integers")
    if any(
        not math.isfinite(discount) or not 0 <= discount <= 1
        for discount in discounts
    ):
        raise ValueError("discounts must be finite and in [0, 1]")
```

Проверка выполняется до расчета и до таймера. Benchmark не должен измерять поведение на
данных, которые не соответствуют смыслу задачи.

### Шаг 3. Постройте построчный Python-baseline

```python
def python_line_revenue(prices, quantities, discounts):
    return [
        price * quantity * (1.0 - discount)
        for price, quantity, discount in zip(
            prices,
            quantities,
            discounts,
            strict=True,
        )
    ]
```

`zip(..., strict=True)` не позволяет тихо отбросить хвост более длинного списка. Это
последняя техническая защита, но она не заменяет понятное сообщение валидатора.

Проверьте подробный результат:

```python
validate_line_items(prices, quantities, discounts)
loop_rows = python_line_revenue(prices, quantities, discounts)

assert loop_rows == expected_line_revenue
assert sum(loop_rows) == expected_total
```

### Шаг 4. Постройте baseline итогового расчета

Для измерения не обязательно сохранять отдельный список выручки:

```python
def python_net_revenue(prices, quantities, discounts):
    total = 0.0
    for price, quantity, discount in zip(
        prices,
        quantities,
        discounts,
        strict=True,
    ):
        total += price * quantity * (1.0 - discount)
    return total
```

Построчная функция нужна для диагностики эквивалентности. Итоговая функция отражает
фактический расчет, время которого мы хотим сравнить.

### Измерьте одну реализацию несколько раз

```python
import statistics
import time


def measure_seconds(function, repeat):
    function()  # один предварительный вызов вне измерений
    durations = []

    for _ in range(repeat):
        started = time.perf_counter_ns()
        function()
        elapsed_ns = time.perf_counter_ns() - started
        durations.append(elapsed_ns / 1_000_000_000)

    return statistics.median(durations), durations
```

Замерьте две заранее подготовленные ветки:

```python
loop_median, loop_runs = measure_seconds(
    lambda: python_net_revenue(
        prices_list,
        quantities_list,
        discounts_list,
    ),
    repeat=7,
)

vector_median, vector_runs = measure_seconds(
    lambda: float(
        np.sum(
            prices_array
            * quantities_array
            * (1.0 - discounts_array),
            dtype=np.float64,
        )
    ),
    repeat=7,
)

speedup = loop_median / vector_median
```

Не удаляйте отдельные времена после вычисления медианы. По ним видно, был ли один запуск
аномально долгим и насколько вообще стабилен локальный результат.

## Сломайте это

### Проверить только общий итог

```text
loop rows:       [200, 60, 180]
vectorized rows: [180, 60, 200]
sum:             440 в обеих ветках
```

Итог совпал, но позиции перепутаны. Сначала сравнивайте наиболее детальный результат.

### Сравнить разные формулы

```python
# Python: скидка является долей
price * quantity * (1.0 - discount)

# NumPy: скидку ошибочно вычитают как денежную сумму
prices * quantities - discounts
```

Такой benchmark сравнивает не реализации одного расчета, а разные бизнес-правила.

### Тихо усечь входы

```python
prices = [100.0, 80.0, 50.0]
quantities = [2, 1]
discounts = [0.0, 0.25, 0.1]
```

Обычный `zip` остановится на двух элементах. `strict=True` и предварительная проверка длин
превращают потерю строки в явную ошибку.

### Применить скидки не к тем позициям

```python
discounts = discounts[::-1]
```

Форма осталась `(n,)`, поэтому NumPy не знает, что строки потеряли согласованность.
Совместимость формы не проверяет ключи исходной таблицы.

### Пропустить предметную валидацию

```python
discounts = np.array([0.1, -0.2, 1.5])
```

Выражение выполнится быстро и воспроизводимо, но отрицательная скидка увеличит цену, а
скидка выше единицы даст отрицательную выручку. Обе реализации могут согласованно
посчитать бессмысленный результат.

### Получить переполнение до вещественной операции

```python
prices = np.array([30_000], dtype=np.int16)

discounts = np.array([0.1], dtype=np.float64)

result = prices * quantities * (1.0 - discounts)
```

Первое умножение выполняется в узком целом типе и может переполниться. Явно готовьте
рабочие типы до расчета.

Одно число не показывает разброс и может включить редкий шум среды.

Это может быть допустимым end-to-end сравнением, если исходное представление — список.

### Требовать конкретное ускорение в unit test

```python

```

Такой тест зависит от железа, нагрузки и размера данных. Behavioral tests должны
проверять формулу, контракт, структуру отчета и положительность измерений. Контроль
регрессии производительности требует отдельной стабильной среды и политики.

### Объявить проигрыш NumPy ошибкой

Для десяти элементов NumPy-ветка может оказаться медленнее. Это наблюдение о выбранном
размере и scope, а не поломка библиотеки. Урок намеренно не требует `speedup > 1`.

### Обернуть Python-функцию в `np.vectorize` и объявить ускорение

`np.vectorize(callback)` продолжает вызывать Python callback поэлементно. Короткая запись
и ndarray на выходе не доказывают, что вычислительный цикл перенесен в скомпилированную
операцию NumPy. Сравните механизм и измерения с выражением через ufunc или другой
подходящий array API.

### Создать слишком много временных массивов

Длинная цепочка поэлементных выражений может выделить несколько буферов размером со весь
набор. Ускорение CPU не компенсирует нехватку памяти. Оцените `nbytes`, рассмотрите
переиспользование буфера или обработку крупными блоками.

### Использовать `out=` и испортить вход

```python
np.subtract(1.0, discounts, out=discounts)
```

Теперь исходные скидки заменены множителями после скидки. Переиспользуйте отдельный
рабочий буфер и явно проверяйте, какие массивы разрешено изменять.

### Векторизовать последовательный процесс как независимые строки

Если состояние на позиции `i` зависит от результата позиции `i - 1`, простая
поэлементная формула меняет алгоритм. Иногда нужен специализированный накопительный метод,
а иногда понятный цикл остается правильным решением.

### Оптимизировать не то место

Если 95% времени pipeline занимает SQL-запрос или чтение файла, ускорение короткой
формулы в 20 раз почти не изменит end-to-end время. Поиск реального bottleneck относится
к профилированию и подробно рассматривается в фазе 12.

### Принять векторизацию за параллельность

Короткое выражение над массивами не доказывает использование нескольких ядер. Не
приписывайте механизму свойства, которых не измеряли и не проверяли.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Векторизация | «Циклов больше нет» | Повторяющаяся операция выражена через array API, а внутренний цикл выполняет библиотека |
| `np.vectorize` | «Автоматический ускоритель Python-функции» | Удобная обертка, которая применяет Python callback к элементам и не является заменой ufunc ради скорости |
| Поэлементная операция | «Всегда возвращает итог» | Применяет правило к согласованным элементам и обычно сохраняет предметные оси |
| Редукция | «Еще одно умножение массивов» | Убирает одну или несколько осей, сводя их к агрегату |
| Baseline | «Намеренно плохой код» | Прозрачная контрольная реализация той же предметной формулы |
| Эквивалентность | «Совпала общая сумма» | Согласованы наиболее детальные результаты и затем необходимые агрегаты |
| Benchmark | «Любой вызов таймера» | Эксперимент с зафиксированными входами, scope, повторами, средой и правилом сравнения |
| Scope | «Только размер массива» | Точная граница работы, включенной и исключенной из измерения |
| Warm-up | «Гарантия чистого замера» | Неизмеряемый предварительный вызов, отделяющий часть разовых эффектов |
| Медиана времени | «Среднее всех запусков» | Центральное значение упорядоченных повторных измерений |
| Speedup | «Свойство NumPy» | Отношение времен двух реализаций в конкретном benchmark |
| Временный массив | «Бесплатный результат выражения» | Дополнительный буфер, который существует во время вычисления и расходует память |
| End-to-end | «То же, что calculation-only» | Scope от исходного представления или входа до конечного результата |
| Bottleneck | «Самый длинный фрагмент кода» | Часть процесса, фактически ограничивающая его время или ресурс |

### На русском языке

- [С. В. Лемешевский: «Основы NumPy: массивы и векторные вычисления»](https://slemeshevsky.github.io/python-course/numpy/html/numpy-flatly.html) — начните с объяснения непрерывных буферов и сравнения массива со списком, затем откройте разделы об универсальных функциях и программировании массивами; опубликованные времена считайте иллюстрацией и повторите опыт в своей среде.
- [`time.perf_counter()` и `perf_counter_ns()`](https://docs-python.ru/standart-library/modul-time-python/funktsija-perf-counter-modulja-time/) — русскоязычное объяснение счетчика для коротких интервалов: обратите внимание, что смысл имеет разница двух показаний, а wall time включает ожидание и вмешательство среды.
- [Командная строка модуля `timeit`](https://docs-python.ru/standart-library/modul-timeit-python/interfejs-komandnoj-stroki-modulja-timeit/) — разберите параметры `--number`, `--repeat`, `--setup` и `--verbose`, а затем сопоставьте их с явными scope, предварительным вызовом и сохранением всех повторов в артефакте урока.

### На английском языке

- [NumPy: Universal functions basics](https://numpy.org/doc/stable/user/basics.ufuncs.html) — основной официальный материал о поэлементных ufunc: внутреннем цикле, broadcasting, выборе `dtype`, редукциях, буферах и аргументе `out`; это точный механизм под выражением урока.
- [`numpy.testing.assert_allclose`](https://numpy.org/doc/stable/reference/generated/numpy.testing.assert_allclose.html) — контракт проверки численной эквивалентности: изучите формулу `atol + rtol * abs(desired)`, асимметрию аргументов и особенно выбор `atol` около нуля.
- [NumPy: Copies and views](https://numpy.org/doc/stable/user/basics.copies.html) — официальный разбор представлений, копий и `base`; используйте его, чтобы отличать стоимость вычислительного выражения от скрытой стоимости преобразования или копирования входа.
- [`numpy.vectorize`](https://numpy.org/doc/stable/reference/generated/numpy.vectorize.html) — важная анти-ловушка: несмотря на название, функция предназначена прежде всего для удобства и по существу выполняет Python-цикл, поэтому не является заменой ufunc ради производительности.
- [Python: `timeit`](https://docs.python.org/3/library/timeit.html) — первичная документация о `setup`, `repeat`, `autorange`, накладных расходах и влиянии других процессов; сравните рекомендацию смотреть на весь вектор времен с медианой, выбранной в учебном отчете.
- [Harris et al.: “Array programming with NumPy”](https://www.nature.com/articles/s41586-020-2649-2) — первичная обзорная статья о модели array programming, устройстве NumPy и его роли в научной экосистеме; читайте после практики для более широкой архитектурной картины.
- [NumPy: CPU/SIMD optimizations](https://numpy.org/doc/stable/reference/simd/index.html) — углубление после основной части: runtime dispatch и аппаратно-зависимые SIMD-оптимизации; материал помогает не смешивать пользовательскую векторизацию через array API с конкретными машинными инструкциями.
