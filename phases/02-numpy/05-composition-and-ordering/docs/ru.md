# Сборка, сортировка и согласованность массивов

> Массивы мало уметь создать и отфильтровать. В аналитическом расчете приходится
> соединять порции данных, менять их порядок и считать категории так, чтобы значения,
> относящиеся к одному наблюдению, не разъехались.

**Тип:** Build

**Трек:** core

**Пререквизит:** `02-numpy/04-indexing-and-masks`

**Время:** ~90 минут

## Цели обучения

После урока вы сможете:

- отличить присоединение вдоль существующей оси от добавления новой оси;
- до запуска предсказать shape и предметные оси результата `concatenate` и `stack`;
- получить позиции сортировки через `argsort` и применить одну перестановку ко всем
  связанным массивам;
- объяснить, зачем аналитическому результату нужна устойчивая сортировка;
- найти уникальные значения, частоты и обратные индексы через `unique`;
- отличить значение минимума или максимума от позиции первого экстремума;
- проверить shape, dtype, порядок, политику `NaN` и сохранение соответствия наблюдений.

## Проблема

После предыдущих уроков мы умеем описывать массив через shape, оси и dtype, менять его
форму и выбирать наблюдения маской. Но реальные данные редко приходят одним готовым
массивом.

Представьте два пакета заказов:

```python
import numpy as np

amount_batch_1 = np.array([120.0, 80.0])
amount_batch_2 = np.array([120.0, 95.0])

order_id_batch_1 = np.array([101, 102])
order_id_batch_2 = np.array([103, 104])
```

Мы хотим:

1. собрать пакеты в одну последовательность;
2. расположить заказы по сумме;
3. сохранить связь суммы с `order_id`;
4. посчитать, сколько раз встречается каждая сумма;
5. найти заказ с минимальной и максимальной суммой.

Наивный код легко дает правдоподобный, но неверный результат:

```python
amounts = np.sort(np.concatenate([amount_batch_1, amount_batch_2]))
order_ids = np.concatenate([order_id_batch_1, order_id_batch_2])
```

Теперь первая сумма равна `80`, а первый идентификатор — `101`. В исходных данных сумма
`80` принадлежала заказу `102`. Оба массива по отдельности выглядят нормально, длины
совпадают, исключения нет — но аналитический смысл разрушен.

Главная мысль урока:

> Если несколько массивов описывают одни наблюдения, их общая позиция является частью
> данных. Любое изменение порядка нужно применять к ним одной и той же перестановкой.

## Концепция

### Наблюдение связывает параллельные массивы

В NumPy одно наблюдение часто хранится не в одной «строке», а в нескольких связанных
массивах:

```python
order_ids = np.array([101, 102, 103, 104])
amounts = np.array([120.0, 80.0, 120.0, 95.0])
segments = np.array(["A", "B", "C", "A"])
```

Позиция `i` означает один заказ:

```text
order_ids[i] <-> amounts[i] <-> segments[i]
```

Это и есть инвариант согласованности. Он не записан в dtype или shape. NumPy не знает,
что три массива связаны бизнес-смыслом, поэтому проверить и сохранить эту связь должен
автор расчета.

Перед структурной операцией задайте четыре вопроса:

1. Что является одним наблюдением?
2. Какая ось перечисляет наблюдения?
3. Создаем ли мы новую предметную ось или удлиняем существующую?
4. Какие массивы должны пройти через ту же операцию или перестановку?

### `concatenate`: удлинить существующую ось

`np.concatenate` соединяет массивы вдоль уже существующей оси. Число осей не меняется.

```python
first = np.array([[120.0, 2.0], [80.0, 1.0]])
second = np.array([[120.0, 4.0], [95.0, 2.0]])

features = np.concatenate([first, second], axis=0)

assert first.shape == (2, 2)       # (order, metric)
assert second.shape == (2, 2)      # (order, metric)
assert features.shape == (4, 2)    # (order, metric)
```

Мы не создали ось `batch`. Мы объявили, что после загрузки пакет больше не является
отдельной аналитической сущностью, и продолжили существующую ось `order`.

Правило формы для `concatenate`:

- длины по выбранной оси складываются;
- длины по всем остальным осям должны совпадать;
- `ndim` результата равен `ndim` входов.

Например, формы `(2, 3)` и `(4, 3)` можно соединить по `axis=0` в `(6, 3)`. По
`axis=1` их соединить нельзя: длина нетронутой оси `0` различается.

Соединение по оси признаков тоже возможно:

```python
amount = np.array([[120.0], [80.0], [95.0]])
items = np.array([[2.0], [1.0], [2.0]])

features = np.concatenate([amount, items], axis=1)
assert features.shape == (3, 2)  # (order, metric)
```

Но здесь особенно важно заранее доказать, что строка `i` в обоих массивах описывает один
и тот же заказ. Совпадение shape само по себе этого не доказывает.

### `stack`: создать новую ось

`np.stack` соединяет массивы вдоль новой оси. Все входные формы должны совпасть точно,
а `ndim` увеличивается на один.

Пусть две матрицы имеют оси `(store, day)`: план и факт продаж.

```python
plan = np.array([[10, 12, 11], [8, 9, 10]])
fact = np.array([[9, 13, 11], [7, 10, 10]])

comparison = np.stack([plan, fact], axis=0)

assert plan.shape == (2, 3)          # (store, day)
assert comparison.shape == (2, 2, 3) # (scenario, store, day)
```

Длина новой оси `scenario` равна количеству входных массивов. Позиция `0` означает
`plan`, позиция `1` — `fact`.

Сравним операции:

| Вопрос | `concatenate` | `stack` |
|---|---|---|
| Ось уже существовала? | Да | Нет |
| Требование к формам | Совпадают вне выбранной оси | Совпадают полностью |
| Меняется `ndim`? | Нет | Увеличивается на 1 |
| Типичный смысл | Еще наблюдения или признаки | Сценарий, модель, версия, повтор |

Выбор функции определяется не внешним видом результата, а предметным смыслом оси.
`vstack`, `hstack` и похожие сокращения удобны, но в начале обучения скрывают главный
вопрос: какая именно ось меняется. Поэтому в этом уроке мы используем явные
`concatenate(..., axis=...)` и `stack(..., axis=...)`.

### Сборка обычно создает новый буфер

В предыдущем уроке мы различали view и copy. `concatenate` и `stack` обычно выделяют
новый массив и копируют в него значения:

```python
left = np.array([1, 2])
right = np.array([3, 4])
result = np.concatenate([left, right])

assert not np.shares_memory(result, left)
assert not np.shares_memory(result, right)
```

Это важно для больших данных: однократная сборка нескольких частей нормальна, а
добавление по одному элементу в цикле многократно копирует уже собранное.

Плохой шаблон:

```python
result = np.array([], dtype=np.int64)
for value in range(10_000):
    result = np.concatenate([result, np.array([value])])
```

Если число элементов известно, создайте массив нужной формы заранее. Если части уже
существуют, соберите их одним вызовом.

### При сборке dtype может измениться

NumPy должен найти единый dtype результата:

```python
integers = np.array([1, 2], dtype=np.int32)
floats = np.array([3.5], dtype=np.float64)
combined = np.concatenate([integers, floats])

assert combined.dtype == np.float64
```

Такое продвижение типа может быть правильным, но не должно быть неожиданным. Опаснее
смешивание чисел и строк: числа могут превратиться в строки, после чего арифметика
перестанет иметь исходный смысл. Поэтому до и после сборки проверяйте dtype.

### `sort` возвращает значения, `argsort` — позиции

`np.sort(amounts)` отвечает на вопрос «каковы суммы по возрастанию?». Но для связанных
массивов этого недостаточно: нужны позиции исходных заказов в новом порядке.

```python
order_ids = np.array([101, 102, 103, 104])
amounts = np.array([120.0, 80.0, 120.0, 95.0])
segments = np.array(["A", "B", "C", "A"])

permutation = np.argsort(amounts, kind="stable")

sorted_amounts = amounts[permutation]
sorted_order_ids = order_ids[permutation]
sorted_segments = segments[permutation]

assert permutation.tolist() == [1, 3, 0, 2]
assert sorted_order_ids.tolist() == [102, 104, 101, 103]
```

`permutation` — это индексный массив из предыдущего урока. Он описывает одно изменение
порядка и применяется ко всем массивам с grain `order`.

Механизм можно увидеть без NumPy:

```python
permutation = sorted(range(len(amounts)), key=amounts.__getitem__)
```

Сначала сортируются не сами заказы, а их позиции. Затем позиции выбирают наблюдения в
нужном порядке.

### Почему сортировка должна быть устойчивой

Два заказа имеют сумму `120`. Устойчивая сортировка сохраняет их прежний взаимный
порядок: сначала заказ `101`, затем `103`.

Это полезно, когда:

- равные значения встречаются регулярно;
- исходный порядок уже имеет смысл, например время поступления;
- результат сравнивают между запусками;
- следующий шаг выбирает «первый» элемент среди равных.

Поэтому мы явно используем `kind="stable"`, а не полагаемся на алгоритм по умолчанию.

Для убывания есть тонкая ловушка:

```python
wrong = np.argsort(amounts, kind="stable")[::-1]
```

Срез `[::-1]` переворачивает весь результат, а вместе с ним — порядок внутри группы
равных значений. Сортировка перестает быть устойчивой в желаемом смысле. Учебный
артефакт ниже разворачивает группы различных значений, но сохраняет исходный порядок
внутри каждой группы.

### Обратная перестановка возвращает исходный порядок

Иногда после расчета в отсортированном порядке результат нужно вернуть к исходным
заказам. Для этого строят обратную перестановку:

```python
inverse = np.empty_like(permutation)
inverse[permutation] = np.arange(len(permutation))

assert np.array_equal(sorted_amounts[inverse], amounts)
```

Проверка восстановления — сильнее простой проверки длины. Она подтверждает, что ни одна
позиция не потеряна и не повторена.

### Пропуски в ключе сортировки требуют решения

`NaN` не означает обычное большое или маленькое число. Автоматическое расположение
пропусков в начале или конце может повлиять на выбор лучших и худших наблюдений.

До сортировки зафиксируйте политику:

- запретить пропуски и остановить расчет;
- расположить их в конце;
- отделить их в самостоятельную ветку;
- заполнить значением — только если такое заполнение имеет предметный смысл.

В артефакте политика по умолчанию — ошибка. Вариант `last` нужно выбрать явно.

### `unique`: не только список значений

`np.unique` по умолчанию возвращает отсортированные уникальные значения:

```python
segments = np.array(["B", "A", "B", "C", "A"])
unique = np.unique(segments)

assert unique.tolist() == ["A", "B", "C"]
```

Это не порядок первого появления. Для аналитики обычно нужны дополнительные результаты:

```python
unique, first_index, inverse, counts = np.unique(
    segments,
    return_index=True,
    return_inverse=True,
    return_counts=True,
)
```

Их смысл:

| Результат | Вопрос |
|---|---|
| `unique` | Какие различные значения есть? |
| `counts` | Сколько раз встретилось каждое значение из `unique`? |
| `first_index` | Где значение впервые встретилось в исходном массиве? |
| `inverse` | Как для каждой исходной позиции указать номер уникального значения? |

Обратные индексы позволяют восстановить исходный массив:

```python
assert np.array_equal(unique[inverse], segments)
```

А порядок первого появления можно получить через позиции:

```python
first_seen = unique[np.argsort(first_index, kind="stable")]
assert first_seen.tolist() == ["B", "A", "C"]
```

Таким образом, `unique` одновременно дает словарь категорий, частоты и код каждой
категории в исходных наблюдениях. Более глубокая работа с категориальными столбцами будет
в фазе pandas; здесь важно понять позиционный механизм.

### Значение экстремума и его позиция — разные ответы

```python
amounts = np.array([120.0, 80.0, 120.0, 95.0])

minimum = amounts.min()       # 80.0
minimum_position = amounts.argmin()  # 1
maximum_position = amounts.argmax()  # 0
```

`min` и `max` возвращают значения. `argmin` и `argmax` возвращают позиции. Позиция нужна,
чтобы получить связанное наблюдение:

```python
assert order_ids[minimum_position] == 102
assert order_ids[maximum_position] == 101
```

Если экстремум встречается несколько раз, `argmin` и `argmax` возвращают первое
вхождение в порядке обхода. Это не «единственный лучший заказ», а первый среди равных.
При наличии `NaN` используйте заранее выбранную политику, а не случайную смесь `argmax`
и `nanargmax`.

### Минимальный структурный контракт

Для сборки запишите:

```text
входные shape и dtype
предметные имена осей
существующая или новая ось
ожидаемые shape, оси и dtype результата
```

Для изменения порядка добавьте:

```text
ключ сортировки
направление и политика равенств
политика NaN
перечень всех связанных массивов
проверка обратного восстановления
```

Такой контракт не является бюрократией. Он фиксирует именно те решения, которые NumPy
не может вывести из бизнес-смысла самостоятельно.

## Соберите это

Сначала реализуем главную часть вручную на маленьком примере. Так станет видно, что
сортировка связанных данных — это работа с позициями.

```python
order_ids = [101, 102, 103, 104]
amounts = [120.0, 80.0, 120.0, 95.0]
segments = ["A", "B", "C", "A"]

permutation = sorted(range(len(amounts)), key=amounts.__getitem__)

sorted_amounts = [amounts[i] for i in permutation]
sorted_order_ids = [order_ids[i] for i in permutation]
sorted_segments = [segments[i] for i in permutation]

assert permutation == [1, 3, 0, 2]
assert sorted_amounts == [80.0, 95.0, 120.0, 120.0]
assert sorted_order_ids == [102, 104, 101, 103]
assert sorted_segments == ["B", "A", "A", "C"]
```

Обратите внимание: мы не сортируем `order_ids` и `segments` независимо. Их значения не
являются ключами сортировки. Они только следуют за общей перестановкой.

Теперь повторим на NumPy:

```python
order_ids = np.array(order_ids)
amounts = np.array(amounts)
segments = np.array(segments)

permutation = np.argsort(amounts, kind="stable")

assert np.array_equal(amounts[permutation], sorted_amounts)
assert np.array_equal(order_ids[permutation], sorted_order_ids)
assert np.array_equal(segments[permutation], sorted_segments)
```

Добавьте обратную перестановку и проверьте восстановление каждого массива:

```python
inverse = np.empty_like(permutation)
inverse[permutation] = np.arange(permutation.size)

assert np.array_equal(amounts[permutation][inverse], amounts)
assert np.array_equal(order_ids[permutation][inverse], order_ids)
assert np.array_equal(segments[permutation][inverse], segments)
```

## Сломайте это

### Ошибка 1. Перепутать продолжение оси и создание оси

```python
np.stack([amount_batch_1, amount_batch_2], axis=0).shape
# (2, 2): (batch, order_in_batch)

np.concatenate([amount_batch_1, amount_batch_2], axis=0).shape
# (4,): (order)
```

Обе операции выполняются, но отвечают на разные предметные вопросы. Назовите оси до
выбора функции.

### Ошибка 2. Соединить матрицы по неверной оси

Два пакета `(order, metric)` нужно продолжить по оси `order`. Соединение по `metric`
либо завершится ошибкой формы, либо создаст правдоподобную матрицу с неверным смыслом.

### Ошибка 3. Не заметить продвижение dtype

Соедините `int32` и `float64`. Затем попробуйте числа и строки. После каждого опыта
проверьте `dtype` и объясните, допустим ли он для следующей операции.

### Ошибка 4. Сортировать связанные массивы независимо

```python
np.sort(amounts)
np.sort(order_ids)
```

Результаты имеют одинаковую длину, но больше не описывают одни заказы. Структурная
проверка должна использовать единую перестановку, а не только shape.

### Ошибка 5. Перевернуть устойчивую сортировку целиком

```python
np.argsort(amounts, kind="stable")[::-1]
```

Проверьте порядок заказов `101` и `103` с одинаковой суммой `120`. Простое обращение
массива меняет порядок внутри группы равных значений.

### Ошибка 6. Считать порядок `unique` порядком появления

Для `['B', 'A', 'B']` результат `np.unique` начинается с `'A'`. Если нужен порядок
первого появления, используйте `return_index` и явно восстановите его.

### Ошибка 7. Считать `argmax` доказательством уникального лидера

При двух максимумах `argmax` возвращает первую позицию. Сначала посчитайте число
вхождений максимума, если уникальность лидера важна для решения.

### Ошибка 8. Собирать массив по одному элементу

Повторный `concatenate` в цикле постоянно создает новые буферы. Сравните этот подход с
одной сборкой списка частей или предварительным выделением массива.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Существующая ось | Любое место, куда добавились значения | Ось входного массива, длина которой увеличивается при `concatenate` |
| Новая ось | Другое название reshape | Новая предметная координата результата, которую добавляет `stack` |
| Перестановка | Отсортированные значения | Массив исходных позиций в новом порядке |
| Связанные массивы | Массивы одинаковой длины | Массивы, у которых одна позиция относится к одному наблюдению |
| Устойчивая сортировка | Сортировка, которая не падает | Сортировка, сохраняющая исходный взаимный порядок равных ключей |
| Обратная перестановка | Сортировка по убыванию | Индексный массив, возвращающий отсортированный результат в исходный порядок |
| `unique` | Категории в порядке появления | Обычно отсортированные уникальные значения с опциональными позициями, inverse-кодами и частотами |
| `argmin` / `argmax` | Минимальное или максимальное значение | Позиция первого экстремума вдоль выбранной оси |
