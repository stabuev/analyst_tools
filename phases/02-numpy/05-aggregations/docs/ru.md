# Агрегации и оси расчета

> Перед агрегацией назовите смысл осей, которые исчезнут, и детализацию результата, который останется.

**Тип:** Build
**Треки:** Core
**Пререквизит:** `02-numpy/04-indexing-and-masks`
**Время:** ~75 минут
**Результат:** по предметным именам осей выбирает осмысленную редукцию, предсказывает детализацию и `shape` результата, явно задает обработку пропусков и проверяет расчет.

## Цели обучения

- объяснить агрегацию как изменение детализации данных, а не только как вызов функции;
- выбирать `axis` по предметному смыслу и заранее называть оставшиеся оси;
- различать размер группы, число валидных наблюдений, сумму, среднее, границы и разброс;
- предсказывать `shape` для одной, нескольких и всех осей, с `keepdims` и без него;
- явно выбирать политику пропусков и распознавать полностью пустую группу;
- проверять агрегат ручным расчетом, формой результата и простыми инвариантами.

## Проблема

После уроков о форме, `dtype`, индексах и масках мы умеем описать массив и выбрать из него
нужные элементы. Теперь нужно перейти от отдельных наблюдений к итогам.

Представим ежедневные показатели двух магазинов за три дня:

```python
import numpy as np

daily_metrics = np.array(
    [
        [[10, 1000], [12, 1440], [8, 880]],
        [[7, 840], [9, 990], [11, 1320]],
    ]
)

assert daily_metrics.shape == (2, 3, 2)
```

Предметный контракт массива:

```text
axis 0: store  — магазин
axis 1: day    — день
axis 2: metric — показатель

metric 0: orders  — число заказов, шт.
metric 1: revenue — выручка, руб.
```

Из одного массива можно получить разные корректные результаты:

- итоги каждого магазина за все дни;
- итоги каждого дня по всем магазинам;
- средний уровень каждого показателя по всем магазинам и дням;
- один итог по всему массиву.

Последний вариант NumPy рассчитает без возражений, но сложит заказы с рублями. Число будет
технически корректным и предметно бессмысленным. Поэтому недостаточно знать синтаксис
`sum(axis=...)`. Аналитик должен определить:

1. что обозначает каждая ось;
2. какие наблюдения объединяются в одну группу;
3. какие оси и какая детализация останутся в результате;
4. допустимо ли применять выбранный агрегат к этому показателю;
5. как обрабатываются пропуски и сколько значений участвовало в расчете.

## Концепция

### Редукция меняет детализацию данных

Агрегации `sum`, `mean`, `min`, `max` и `std` относятся к редукциям: несколько исходных
значений они сворачивают в один результат. Аргумент `axis` называет ось, которая
сворачивается и обычно исчезает.

Для `daily_metrics` формы `(store=2, day=3, metric=2)`:

| Вызов | Что объединяем | Что остается | `shape` |
|---|---|---|---:|
| `sum(axis=0)` | магазины | `day, metric` | `(3, 2)` |
| `sum(axis=1)` | дни | `store, metric` | `(2, 2)` |
| `mean(axis=(0, 1))` | магазины и дни | `metric` | `(2,)` |
| `sum(axis=None)` | все оси | ничего | `()` |

Это правило надежнее фраз «по строкам» и «по столбцам». Такие фразы описывают только
двумерное изображение массива и часто создают путаницу. Если все же переводить их в
`axis` для матрицы `(row, column)`:

- итог для каждого столбца сворачивает строки: `axis=0`;
- итог для каждой строки сворачивает столбцы: `axis=1`.

Но в рабочем рассуждении лучше говорить: «сворачиваем магазины, оставляем дни и
показатели».

### Сначала назовите grain результата

В табличной аналитике `grain` означает, что представляет одна строка. Для массива ту же
роль играет сочетание оставшихся предметных осей.

До вычисления `daily_metrics.sum(axis=1)` можно сформулировать контракт:

```text
сворачиваемая ось: day
оставшиеся оси: store, metric
один элемент результата: итог одного показателя одного магазина за все дни
ожидаемый shape: (2, 2)
```

Если невозможно одним предложением назвать смысл элемента результата, расчет еще не
готов к выполнению.

### Агрегат должен отвечать вопросу

Функции похожи синтаксически, но отвечают на разные вопросы:

| Агрегат | Аналитический вопрос | Важное ограничение |
|---|---|---|
| `count` | сколько наблюдений участвовало | нужно определить, считаются ли пропуски |
| `sum` | каков общий объем | показатель должен быть аддитивным по выбранной оси |
| `mean` | каков средний уровень | наблюдения получают одинаковый вес |
| `min`, `max` | каковы наблюдаемые границы | пустая группа не имеет границ |
| `std` | насколько значения различаются | нужен явный статистический контракт `ddof` |

Число заказов и выручку обычно можно складывать по дням или магазинам. Средний чек,
процент конверсии и долю отмен нельзя бездумно суммировать или усреднять: часто их нужно
пересчитать из числителя и знаменателя. NumPy знает тип числа, но не знает смысл метрики.

### Размер группы и число наблюдений — не одно и то же

Пусть в группе запланировано три дня, но один день не загрузился:

```text
[10, missing, 8]
```

Тогда:

```text
group_size    = 3
valid_count   = 2
missing_count = 1
```

Если в отчете оставить только `count=3`, среднее по двум фактическим наблюдениям будет
выглядеть как среднее по трем дням. Поэтому проверяемый артефакт урока разделяет эти три
понятия.

### Пропуски требуют политики

Обычный `np.mean` не игнорирует `NaN`:

```python
values = np.array([10.0, np.nan, 8.0])
assert np.isnan(np.mean(values))
```

`np.nanmean(values)` исключит пропуск и вернет `9.0`. Ни один вариант не является всегда
правильным. Выбор зависит от договора о данных:

- `error` — остановить расчет и сначала разобраться с неполными данными;
- `omit` — исключить пропуски, но обязательно показать `valid_count`;
- восстановление значения — отдельная методология, а не скрытая часть агрегации.

Особенно опасна полностью пустая группа. Например, `np.nansum([np.nan, np.nan])` возвращает
`0.0`. Этот ноль является техническим нейтральным элементом суммы, а не доказательством
нулевого результата. В артефакте урока группа без валидных наблюдений считается ошибкой.

### `keepdims` сохраняет место свернутой оси

Обычная редукция удаляет выбранную ось:

```text
(store=2, day=3, metric=2)
    --mean(axis=1)-->
(store=2, metric=2)
```

С `keepdims=True` ось остается длины один:

```text
(store=2, day=3, metric=2)
    --mean(axis=1, keepdims=True)-->
(store=2, summarized_day=1, metric=2)
```

Значения не меняются — меняется только контракт формы. Единичная ось сообщает, где раньше
располагались дни. В следующем уроке она позволит применить агрегат обратно к исходному
массиву по правилам broadcasting. Здесь мы лишь подготавливаем форму, не разбирая эти
правила заранее.

### `ddof` — часть определения стандартного отклонения

Стандартное отклонение использует число наблюдений в делителе дисперсии:

```text
N - ddof
```

- `ddof=0` описывает разброс именно имеющегося набора;
- `ddof=1` часто используют, когда по выборке оценивают разброс более широкой
  совокупности.

Это не переключатель «больше или меньше точности». Выбор должен следовать методологии.
Кроме того, `ddof` должен быть меньше числа **валидных** наблюдений в каждой группе. Для
одного наблюдения `std(..., ddof=1)` не определен.

### `dtype` результата тоже входит в проверку

Редукция меняет не только значения и форму. Например, среднее целых чисел становится
вещественным:

```python
integers = np.array([1, 2, 3], dtype=np.int32)

assert integers.dtype == np.dtype("int32")
assert np.mean(integers).dtype == np.dtype("float64")
```

Правила аккумулятора для `sum` зависят от входного `dtype` и параметра `dtype`. Мы уже
разбирали диапазоны в уроке о типах, поэтому здесь не повторяем всю тему. Но артефакт
показывает `dtype` входа и каждого результата, чтобы преобразование не было скрытым.

## Соберите это

Сначала воспроизведем механизм на маленькой двумерной матрице без NumPy. Возьмем только
число заказов:

```python
orders = [
    [10, 12, 8],
    [7, 9, 11],
]

# axis 0: store
# axis 1: day
```

### Шаг 1. Явно составьте группы

Если сворачиваем `store`, каждый результат собирает один день из всех магазинов:

```python
def matrix_width(values):
    if not values or not values[0]:
        raise ValueError("expected a non-empty matrix")
    width = len(values[0])
    if any(len(row) != width for row in values):
        raise ValueError("expected a rectangular matrix")
    return width


def sum_axis_0(values):
    width = matrix_width(values)
    return [
        sum(row[column] for row in values)
        for column in range(width)
    ]


assert sum_axis_0(orders) == [17, 21, 19]
```

Группы выглядят так:

```text
day 0: 10 + 7  = 17
day 1: 12 + 9  = 21
day 2:  8 + 11 = 19
```

Если сворачиваем `day`, один результат собирает все дни одного магазина:

```python
def sum_axis_1(values):
    matrix_width(values)
    return [sum(row) for row in values]


assert sum_axis_1(orders) == [30, 27]
```

Ручные функции обязаны отклонять пустую или рваную матрицу: иначе группы по второй оси не
определены. Готовая функция `manual_sum_2d` с такими проверками находится в артефакте.

### Шаг 2. Предскажите контракт результата

Механизм предсказания не должен считать значения. Он работает только с формой, именами
осей и выбранной редукцией:

```python
def reduced_contract(shape, axis_names, reduced_axes, keepdims=False):
    reduced_axes = set(reduced_axes)
    if keepdims:
        output_shape = tuple(
            1 if index in reduced_axes else length
            for index, length in enumerate(shape)
        )
        output_names = tuple(
            f"summarized_{name}" if index in reduced_axes else name
            for index, name in enumerate(axis_names)
        )
    else:
        output_shape = tuple(
            length
            for index, length in enumerate(shape)
            if index not in reduced_axes
        )
        output_names = tuple(
            name
            for index, name in enumerate(axis_names)
            if index not in reduced_axes
        )
    return output_shape, output_names


assert reduced_contract(
    (2, 3, 2),
    ("store", "day", "metric"),
    (1,),
) == ((2, 2), ("store", "metric"))
```

Этот шаг отделяет вопрос «какие числа получатся» от вопроса «что будет представлять
результат». Артефакт реализует ту же идею в функции `aggregation_contract` и дополнительно
проверяет отрицательные, повторяющиеся и отсутствующие оси.

### Шаг 3. Разделите размер группы и валидные значения

Для одной небольшой группы политику можно сделать видимой обычным Python:

```python
values = [10, None, 8]
valid = [value for value in values if value is not None]

group_size = len(values)
valid_count = len(valid)
missing_count = group_size - valid_count
mean = sum(valid) / valid_count

assert group_size == 3
assert valid_count == 2
assert missing_count == 1
assert mean == 9
```

Этот расчет допустим только после явного решения исключать `None`. Если `valid` пуст,
делить нельзя и публиковать ноль тоже нельзя. Именно эти проверки затем берет на себя
исполняемый артефакт.

## Используйте это

### Одна предметная ось

Вернемся к трехмерному массиву. Итог каждого магазина по дням сворачивает `day`, то есть
ось `1`:

```python
store_totals = daily_metrics.sum(axis=1)

assert store_totals.shape == (2, 2)
np.testing.assert_array_equal(
    store_totals,
    [
        [30, 3320],
        [27, 3150],
    ],
)
```

Оставшиеся оси — `(store, metric)`. Первый столбец содержит заказы, второй — выручку.

Чтобы получить дневные итоги сети, сворачиваем `store`:

```python
day_totals = daily_metrics.sum(axis=0)

assert day_totals.shape == (3, 2)
np.testing.assert_array_equal(
    day_totals,
    [
        [17, 1840],
        [21, 2430],
        [19, 2200],
    ],
)
```

Теперь оставшиеся оси — `(day, metric)`.

### Несколько осей

Средний уровень каждого показателя по всем магазинам и дням требует свернуть две оси:

```python
metric_means = daily_metrics.mean(axis=(0, 1))

assert metric_means.shape == (2,)
np.testing.assert_allclose(metric_means, [9.5, 1078.3333333333333])
```

Осталась только ось `metric`. Порядок значений по-прежнему определяется ее контрактом:
сначала заказы, затем выручка.

Отрицательная ось отсчитывается справа. Для массива с тремя осями `axis=-1` означает
`axis=2`, то есть `metric`. Это удобно, но предметный смысл все равно нужно назвать:

```python
assert daily_metrics.sum(axis=-1).shape == (2, 3)
```

В данном примере такой вызов складывает заказы с рублями и потому не должен использоваться,
даже если его `shape` предсказан правильно.

### Все оси

Если `axis` не указан или равен `None`, NumPy сворачивает весь массив:

```python
technical_total = daily_metrics.sum(axis=None)
assert technical_total.shape == ()
```

Скаляр `6527` здесь не является аналитическим итогом: в нем смешаны разные единицы
измерения. `axis=None` подходит, когда все элементы массива выражают одну величину,
например для матрицы заказов по магазинам и дням. Значение по умолчанию не освобождает от
проверки смысла.

### Сохранение осей

Подготовим среднее и стандартное отклонение каждого показателя для следующего урока:

```python
metric_means = daily_metrics.mean(axis=(0, 1), keepdims=True)
metric_std = daily_metrics.std(axis=(0, 1), keepdims=True, ddof=0)

assert metric_means.shape == (1, 1, 2)
assert metric_std.shape == (1, 1, 2)
```

Контракт осей результата:

```text
(summarized_store=1, summarized_day=1, metric=2)
```

Вычисления уже готовы, но применять их обратно к массиву пока не будем. Почему форма
`(1, 1, 2)` совместима с `(2, 3, 2)` и как проверить такую операцию, разберем в следующем
уроке о broadcasting.

### Маски и пропуски

Из предыдущего урока мы знаем, как построить маску:

```python
measurements = np.array(
    [
        [10.0, np.nan, 8.0],
        [7.0, 9.0, 11.0],
    ]
)

valid = np.isfinite(measurements)
valid_count = valid.sum(axis=1)

np.testing.assert_array_equal(valid_count, [2, 3])
np.testing.assert_allclose(np.nanmean(measurements, axis=1), [9.0, 9.0])
```

Маска здесь считает фактические наблюдения. Префикс `nan` в `np.nanmean` задает политику
исключения `NaN`, но не формирует полноценный отчет. Для публикуемого результата полезно
сохранять среднее вместе с `group_size`, `valid_count` и `missing_count`.

Артефакт различает пропуск и бесконечность: JSON `null` и `NaN` подчиняются выбранной
политике пропусков, а `inf` считается ошибкой вычисления и отклоняется в обоих режимах.

### Исполняемый артефакт

Артефакт принимает значения, предметные имена осей и оси редукции:

```bash
uv run --locked python phases/02-numpy/05-aggregations/outputs/axis_aggregates.py \
  --values '[[[10, 1000], [12, 1440], [8, 880]], [[7, 840], [9, 990], [11, 1320]]]' \
  --axis-names store day metric \
  --axis 0 1 \
  --keepdims
```

В отчете видно:

- исходные `shape`, `dtype` и имена осей;
- номера и имена свернутых осей;
- оставшиеся оси и ожидаемый `shape`;
- `group_size`, `valid_count` и `missing_count`;
- значения, формы и `dtype` суммы, среднего, границ и стандартного отклонения.

Для JSON пропуск обозначается `null`. Исключать его можно только явно:

```bash
uv run --locked python phases/02-numpy/05-aggregations/outputs/axis_aggregates.py \
  --values '[[10, null, 8], [7, 9, 11]]' \
  --axis-names store day \
  --axis 1 \
  --missing-policy omit
```

Без `--missing-policy omit` тот же запуск завершится ошибкой, а не молча изменит
знаменатель среднего.

## Сломайте это

### Выбрать неверную ось и получить правдоподобный результат

Если нужны два итога магазинов, ожидается форма `(store=2, metric=2)`. Редукция
`axis=0` вернет `(day=3, metric=2)`:

```python
wrong = daily_metrics.sum(axis=0)
assert wrong.shape == (2, 2)  # проверка упадет
```

Форма помогает обнаружить ошибку до интерпретации значений.

### Смешать показатели разных единиц

Редукция оси `metric` дает совершенно ожидаемую форму `(store=2, day=3)`:

```python
meaningless = daily_metrics.sum(axis=2)
assert meaningless.shape == (2, 3)
```

Shape-проверка проходит, но каждое число складывает штуки с рублями. Поэтому нужны обе
проверки: технический контракт формы и предметный смысл агрегата.

### Назвать размер группы количеством наблюдений

Для `[10, NaN, 8]` размер равен трем, но в среднем участвуют два значения. Публикация
только `mean=9` и `count=3` создает ложное представление о полноте данных. Храните
`group_size=3`, `valid_count=2`, `missing_count=1` отдельно.

### Получить ноль из полностью пустой группы

```python
empty_group = np.array([np.nan, np.nan])
assert np.nansum(empty_group) == 0.0
```

Технический ноль не доказывает отсутствие заказов или выручки. Артефакт отклонит группу,
у которой `valid_count == 0`.

### Использовать невозможный `ddof`

```python
two_values = np.array([10.0, 12.0])
np.std(two_values, ddof=2)  # делитель N - ddof равен нулю
```

Артефакт проверяет `ddof` по `valid_count` каждой итоговой группы. Это особенно важно при
разном количестве пропусков.

### Случайно потерять оси

```python
means_vector = daily_metrics.mean(axis=(0, 1))
means_contract = daily_metrics.mean(axis=(0, 1), keepdims=True)

assert means_vector.shape == (2,)
assert means_contract.shape == (1, 1, 2)
```

Значения совпадают после удаления единичных осей, но только второй результат сохраняет
позиции свернутых осей. Выбирайте форму по следующей операции, а не по тому, какой вывод
короче выглядит в консоли.

### Скрыть изменение `dtype`

Среднее целого массива имеет вещественный тип. Сумма использует правила аккумулятора
NumPy. Если downstream-контракт ожидает конкретный диапазон или точность, проверяйте
`dtype` результата явно. Артефакт показывает его рядом с каждым значением.

report = aggregate(

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Редукция | «Любое изменение `shape`» | Сворачивание нескольких значений в итог вдоль выбранных осей |
| `axis` | «Ось, которую хотим увидеть» | Ось или оси, которые агрегируются и обычно исчезают |
| Grain результата | «Только размер массива» | Предметный смысл одного элемента, определяемый оставшимися осями |
| Аддитивность | «Любые числа можно сложить» | Возможность осмысленно суммировать показатель по выбранной оси |
| `group_size` | «Число измеренных значений» | Число предусмотренных позиций в группе, включая пропуски |
| `valid_count` | «Всегда равен размеру группы» | Число фактически участвовавших в расчете значений |
| `axis=None` | «Первая ось» | Сворачивание всех осей массива |
| `keepdims` | «Сохранить исходные значения» | Оставить свернутые оси длины один в форме результата |
| `ddof` | «Точность `float`» | Поправка числа степеней свободы в делителе дисперсии |
