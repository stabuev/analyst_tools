# Диагностика Python и SQL

> Диагностика нужна не для ярлыка «знаю», а для короткого списка конкретных пробелов.

**Тип:** Case  
**Треки:** Core  
**Пререквизиты:** 01 — Карта профессии аналитика  
**Время:** ~60 минут  
**Результат:** решает короткие задачи без подсказок и фиксирует конкретные пробелы перед стартом

## Цели обучения

- Проверить базовые операции Python и SQL на исполняемых задачах.
- Отличать общий балл от профиля отдельных навыков.
- Использовать граничные случаи: `None`, отмененный заказ и клиент без заказов.
- Превратить ошибки в ограниченный план повторения, а не в повтор всего курса.

## Проблема

Фраза «я знаю Python и SQL» почти ничего не сообщает. Один человек уверенно пишет
функции, но теряется на `LEFT JOIN`; другой помнит синтаксис запроса, но не замечает
пропуски и изменение входных данных. Самооценка особенно неточна до первой практической
задачи.

Входная диагностика должна измерять наблюдаемое действие, локализовать пробел и не
превращаться в собеседование на скорость. Ее результат — не сертификат, а решение:
продолжать курс, повторить одну тему или сначала укрепить базу.

## Концепция

У каждой задачи есть отдельный навык и контрольный пример:

| Проверка | Навык | Граничный случай |
|---|---|---|
| `select_paid_order_ids` | Фильтрация коллекции | Порядок строк и отсутствие мутации |
| `revenue_by_customer` | Функции и агрегация | Отмена и `None` в сумме |
| `CUSTOMER_REVENUE_SQL` | `WHERE`, `GROUP BY`, `SUM` | Отмененные заказы и `NULL` |
| `CUSTOMER_ACTIVITY_SQL` | Grain и `LEFT JOIN` | Клиент без заказов |

Общий балл удобен как резюме, но решение принимается по профилю. Результат `3/4` может
означать один небольшой пробел, а два одинаковых результата `2/4` — совершенно разные
планы подготовки.

## Соберите это

Не открывая `code/main.py`, создайте файл решения с четырьмя именами из таблицы выше.

### Шаг 1: решите Python-задачи

Обе функции получают список словарей заказов. Первая возвращает идентификаторы оплаченных
заказов в исходном порядке. Вторая возвращает словарь `customer_id -> revenue`, учитывая
только оплаченные строки с известной суммой.

Важный контракт: функция читает входные строки, но не меняет их. Аналитический код часто
переиспользует один DataFrame или список на нескольких шагах, поэтому скрытая мутация
создает зависимость от порядка запуска.

### Шаг 2: решите SQL-задачи

`CUSTOMER_REVENUE_SQL` агрегирует оплаченную выручку по клиенту. Результат должен иметь
grain «одна строка на клиента с оплаченной выручкой».

`CUSTOMER_ACTIVITY_SQL` начинает расчет с таблицы `customers` и сохраняет клиента без
заказов. Условия на статус и сумму должны отфильтровать факты, не превращая соединение
случайно во внутреннее.

## Используйте это

Запустите проверку своего файла:

```bash
python3 outputs/diagnostic_runner.py \
  --submission path/to/my_solution.py \
  --format markdown
```

Сохраните машиночитаемый отчет:

```bash
python3 outputs/diagnostic_runner.py \
  --submission path/to/my_solution.py \
  --format json \
  --output diagnostic-report.json
```

После первой попытки сравните решение с `code/main.py`. Не заменяйте свой отчет результатом
эталона: полезная часть диагностики — именно первая самостоятельная попытка.

## Сломайте это

Проверьте четыре характерные ошибки:

1. Удалите элемент из входного списка внутри Python-функции.
2. Сложите `amount` без проверки `None`.
3. Перенесите условие `orders.status = 'paid'` из `ON` в `WHERE` в запросе с `LEFT JOIN`.
4. Используйте обычный `JOIN` и посмотрите, почему исчезает клиент без заказов.

Раннер показывает фактический и ожидаемый результат. Это помогает отличить синтаксическую
ошибку от методологической: запрос может выполняться без исключения и все равно менять
смысл выборки.

## Проверьте это

Диагностика считается полезной, если:

- каждая проверка связана с одним понятным навыком;
- контрольные данные содержат хотя бы один релевантный граничный случай;
- повторный запуск дает тот же результат;
- отчет перечисляет темы, а не ставит человеку общий ярлык;
- успешный контрольный пример не интерпретируется как владение всем Python или SQL.

Тесты урока проверяют эталон, неполное решение, мутацию входа и ошибочный внутренний join.

## Поставьте результат

Артефакт `outputs/diagnostic_runner.py` можно использовать отдельно от урока. Он принимает
путь к Python-файлу, запускает четыре проверки и печатает Markdown или JSON. Manifest и
команда запуска находятся в `outputs/artifact.json`.

Сохраните отчет первой попытки рядом со своим учебным планом. После повторения тем
перезапустите ту же команду: изменение профиля важнее субъективного ощущения прогресса.

## Упражнения

1. Добавьте второй контрольный набор, в котором все заказы отменены.
2. Расширьте отчет временем выполнения, не превращая скорость в основной критерий.
3. Добавьте задачу на оконную функцию и объясните, для какого маршрута она обязательна на входе.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Диагностика | Экзамен, который нужно пройти идеально | Инструмент выбора стартовой точки и тем повторения |
| Контрольный пример | Любой набор данных, где код запускается | Вход с заранее известным правильным результатом |
| Grain | Количество строк в таблице | Сущность, которую представляет одна строка результата |
| Граничный случай | Редкая ситуация, которой можно пренебречь | Вход, на котором часто проявляется неверное предположение |
