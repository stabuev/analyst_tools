# Терминал и файловая система

> Терминал нужен не для «сложных команд», а для понятных и повторяемых действий с
> проектом.

**Тип:** Build
**Треки:** Core
**Пререквизиты:** нет
**Время:** ~60 минут
**Результат:** ориентируется в проекте, безопасно работает с файлами и сохраняет простой
pipeline в Bash-скрипт

## Цели обучения

- Различать терминал, shell и командную строку приглашения.
- Понимать, где находится текущий каталог и откуда вычисляется относительный путь.
- Создавать, копировать, перемещать и удалять файлы внутри учебной папки.
- Соединять две команды через pipe и направлять результат или ошибку в файл.
- Сохранять несколько понятных команд в короткий Bash-скрипт и запускать его повторно.

## Проблема

В аналитическом проекте обычно есть исходные данные, промежуточные выгрузки, ноутбуки и
отчёты. Через графический интерфейс можно открыть каждый каталог вручную, но такую
последовательность трудно повторить и передать коллеге.

Представьте простую задачу: найти все CSV-файлы в учебной выгрузке, посчитать общее число
файлов и сохранить список в отчёт. В терминале решение состоит из нескольких команд. Их
можно проверить по одной, а затем сохранить в скрипт.

Главный риск — выполнить правильную команду не в том каталоге. Поэтому первое действие в
этом уроке — не `rm` и не сложный pipeline, а ответ на вопрос: «Где я сейчас и с какой
папкой работаю?»

## Концепция

**Терминал** — окно, через которое вы вводите текстовые команды. **Shell** — программа,
которая читает эти команды и запускает другие программы. В поддерживаемой среде курса это
Bash или Zsh.

Строка перед курсором называется приглашением, или prompt. Она может выглядеть по-разному:

```text
anna@laptop analyst_tools $
```

Знак `$` здесь показывает место ввода. Его не нужно копировать вместе с командой.

У команды обычно есть три части:

```text
ls -la data
│  │   └─ аргумент: с чем работать
│  └───── опция: как изменить поведение
└──────── команда: что сделать
```

### Текущий каталог и пути

Каждый процесс работает из текущего каталога. Его показывает `pwd`, а команда `cd`
меняет его для текущего shell.

| Запись | Значение |
|---|---|
| `.` | текущий каталог |
| `..` | родительский каталог |
| `~` | домашний каталог пользователя |
| `data/orders.csv` | относительный путь от текущего каталога |
| `/home/anna/project/data/orders.csv` | абсолютный путь от корня файловой системы |

Если относительный путь «перестал работать», сначала выполните `pwd`. Не добавляйте `../`
наугад.

### Результат, ошибка и код завершения

Команда может отправить обычный результат в `stdout`, а сообщение об ошибке — в
`stderr`:

```text
stdin -> команда -> stdout
                 -> stderr
```

- `|` передаёт `stdout` команды слева на вход команды справа;
- `>` записывает `stdout` в файл, заменяя прежнее содержимое;
- `>>` дописывает `stdout` в конец файла;
- `2>` записывает `stderr` в отдельный файл;
- `$?` содержит код завершения последней команды: `0` обычно означает успех.

Кавычки сохраняют путь с пробелами одним аргументом:

```bash
wc -c "reports/analysis plan.md"
```

В этом уроке все изменения выполняются только внутри `work/00-01-terminal/`. Реальные
данные и системные каталоги не используются.

## Соберите это

Начните из корня репозитория. Убедитесь, что команда `pwd` заканчивается на
`analyst_tools`.

### Шаг 1. Создайте безопасную учебную папку

```bash
mkdir -p "work/00-01-terminal/lab/data" "work/00-01-terminal/lab/reports"
printf 'order_id,amount\n101,120\n102,75\n' \
  > "work/00-01-terminal/lab/data/orders.csv"
printf 'customer_id,segment\n1,new\n2,returning\n' \
  > "work/00-01-terminal/lab/data/customers.csv"
printf '# План анализа\n' \
  > "work/00-01-terminal/lab/reports/analysis plan.md"
```

Посмотрите, что получилось:

```bash
find "work/00-01-terminal/lab" -print
```

Ожидается два каталога внутри `lab` и три файла. Если путь не найден, проверьте `pwd`.

### Шаг 2. Потренируйте навигацию

```bash
cd "work/00-01-terminal/lab"
pwd
ls
ls "data"
cd "data"
pwd
cd ..
pwd
```

После последней команды текущим снова должен быть каталог `lab`. Перед выполнением каждой
`cd` попробуйте сначала предсказать результат `pwd`.

### Шаг 3. Выполните безопасные операции с файлами

Создайте копию CSV, переименуйте её и удалите только эту учебную копию:

```bash
cp "data/orders.csv" "data/orders-copy.csv"
mv "data/orders-copy.csv" "data/orders-snapshot.csv"
rm -i "data/orders-snapshot.csv"
```

Подтвердите удаление только созданной копии. `rm` не переносит файл в корзину. В основном
курсе не используйте `rm -rf` для пути из переменной. Сначала ограничьте рабочую область
и распечатайте путь, с которым собираетесь работать.

### Шаг 4. Соедините команды

Найдите все CSV-файлы:

```bash
find . -type f -name '*.csv'
```

Передайте список в сортировку:

```bash
find . -type f -name '*.csv' | sort
```

Посчитайте все обычные файлы:

```bash
find . -type f | wc -l
```

Сохраните отсортированный список CSV в отчёт:

```bash
find . -type f -name '*.csv' | sort > "reports/csv-files.txt"
```

Проверьте отчёт:

```bash
cat "reports/csv-files.txt"
```

Теперь файлов стало четыре: сам отчёт тоже является файлом. Это важная привычка — после
операции проверять не только команду, но и изменившееся состояние каталога.

### Шаг 5. Сохраните решение в скрипт

Вернитесь в корень репозитория:

```bash
cd ../../..
pwd
```

В любом редакторе создайте `work/00-01-terminal/folder_summary.sh`:

```bash
#!/usr/bin/env bash

set -eu

if [[ $# -ne 1 ]]; then
    printf 'usage: folder_summary.sh DIRECTORY\n' >&2
    exit 2
fi

folder=$1

if [[ ! -d "$folder" ]]; then
    printf 'folder-summary: directory does not exist: %s\n' "$folder" >&2
    exit 2
fi

printf '# Folder summary\n\n'
printf 'Folder: %s\n' "$folder"
file_count=$(find "$folder" -type f | wc -l)
printf 'Files: %d\n' "$file_count"
printf '\nCSV files:\n'
find "$folder" -type f -name '*.csv' | sort
```

Запустите собственный скрипт:

```bash
bash "work/00-01-terminal/folder_summary.sh" \
  "work/00-01-terminal/lab"
```

Конструкция `$(...)` сохраняет `stdout` команды в переменную. Скрипт намеренно небольшой.
Он корректно принимает каталог с пробелами, но использует
построчный вывод `find` и не рассчитан на имена файлов с переводом строки. Продвинутая
обработка произвольных Unix-имён не входит в минимальный результат этого урока.

## Сломайте это

### Неверный текущий каталог

```bash
cd "work/00-01-terminal/lab"
ls "work/00-01-terminal/lab/data"
```

Команда ищет вложенный путь внутри `lab` и завершается ошибкой. Сначала выполните `pwd`,
затем используйте путь `data` или вернитесь в корень репозитория.

### Путь без кавычек

```bash
file="reports/analysis plan.md"
wc -c $file
wc -c "$file"
```

Первый вызов передаёт два аргумента, второй — один путь. После эксперимента вернитесь в
корень репозитория командой `cd ../../..`.

### Случайное затирание отчёта

```bash
printf 'важная заметка\n' > "work/00-01-terminal/check.txt"
printf 'новый отчёт\n' > "work/00-01-terminal/check.txt"
cat "work/00-01-terminal/check.txt"
```

Оператор `>` заменил содержимое. Если нужно дописать строку, используйте `>>`, но сначала
решите, действительно ли старые и новые данные должны находиться в одном файле.

### Ошибка и код завершения

```bash
ls "missing-folder" 2> "work/00-01-terminal/error.txt"
printf 'exit status: %s\n' "$?"
cat "work/00-01-terminal/error.txt"
```

Ошибка не исчезла: её поток был записан в файл, а ненулевой код завершения остался
наблюдаемым.

## Проверьте это

Сначала проверьте результат вручную:

- `pwd` позволяет объяснить, откуда вычисляется каждый относительный путь;
- после `cp`, `mv` и `rm -i` состояние папки совпадает с ожиданием;
- `folder-summary.md` содержит число файлов и список CSV;
- запуск скрипта для отсутствующего каталога завершается понятной ошибкой;
- вы можете объяснить разницу между `|`, `>`, `>>` и `2>`.

Проверки reference-артефакта:

```bash
python3 -m unittest discover \
  -s phases/00-entry-and-tools/01-terminal-and-filesystem/tests \
  -p 'test_*.py' -v
```

Чтобы теми же тестами проверить свой скрипт, передайте его путь:

```bash
FOLDER_SUMMARY_SCRIPT="$PWD/work/00-01-terminal/folder_summary.sh" \
python3 -m unittest discover \
  -s phases/00-entry-and-tools/01-terminal-and-filesystem/tests \
  -p 'test_*.py' -v
```

Тесты проверяют интерфейс, каталог с пробелом, число файлов, сортировку CSV и ошибки
аргументов. Они не заменяют объяснение команд своими словами.

## Ключевые термины

| Термин | Распространённое заблуждение | Точное значение |
|---|---|---|
| Терминал | Сам язык команд | Окно для взаимодействия с shell |
| Shell | Любая команда Unix | Программа, которая читает команды и запускает процессы |
| Текущий каталог | Каталог, где лежит скрипт | Каталог, от которого процесс вычисляет относительные пути |
| Аргумент | Необязательная настройка | Значение, с которым команда должна работать |
| Pipe | Запись результата в файл | Передача `stdout` одной команды на вход другой |
| Перенаправление | Копирование файла | Настройка назначения `stdout` или `stderr` |
| Код завершения | Текст ошибки | Число, которым процесс сообщает об успехе или ошибке |

### На английском

- [MDN: Command line crash course](https://developer.mozilla.org/en-US/docs/Learn_web_development/Getting_started/Environment_setup/Command_line) — пройдите части о навигации, базовых файловых операциях и соединении команд; они дают второй практический пример того же минимального набора.
