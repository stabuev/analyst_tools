# Карта профессии аналитика

> Выбирайте маршрут по типу решений и артефактов, а не по модному названию должности.

**Тип:** Case  
**Треки:** Core  
**Пререквизиты:** Нет  
**Время:** ~45 минут  
**Результат:** различает основные аналитические роли и выбирает обоснованный маршрут курса

## Цели обучения

- Различать роли по ежедневным вопросам, ответственности и итоговым артефактам.
- Отделять обязательное аналитическое ядро от профессиональной специализации.
- Выбирать маршрут по желаемой работе, не выдавая интерес за уже освоенный навык.
- Объяснять, почему соседние роли пересекаются, но не являются взаимозаменяемыми.

## Проблема

Названия аналитических ролей нестабильны. В одной компании product analyst проектирует
эксперименты, в другой строит дашборды, а в третьей пишет production SQL. Data analyst
может означать и начинающего специалиста по отчетности, и сильного исследователя. Если
выбирать обучение только по названию вакансии, легко потратить месяцы на инструменты,
которые не приближают к желаемой работе.

Надежнее смотреть на три вещи: какой вопрос человек получает на входе, какое решение
принимается по результату и какой артефакт должен остаться после работы. Библиотеки идут
после этого выбора.

## Концепция

Все маршруты курса опираются на общее ядро: воспроизводимую среду, NumPy, pandas, SQL,
визуализацию и проверки. Специализация начинается там, где меняется основной тип решения.

| Маршрут | Главный вопрос | Типичный артефакт | Характерная ошибка |
|---|---|---|---|
| Базовый аналитик | Что произошло и что делать дальше? | Проверенный отчет или регулярный расчет | Красивый результат без контроля данных |
| Продуктовый аналитик | Почему изменилась метрика и сработало ли изменение? | Система метрик, исследование, эксперимент | Причинный вывод из наблюдаемой корреляции |
| Analytics Engineer | Как сделать надежный слой данных для других аналитиков? | Витрина, тесты, lineage, контракт | Быстрый SQL без grain и владельца |
| ML-аналитик | Можно ли предсказывать исход и использовать прогноз в решении? | Baseline, pipeline, model card | Leakage и оптимизация метрики без стоимости ошибки |

Это не четыре уровня старшинства. Сильный analytics engineer не является «следующей
ступенью» продуктового аналитика, а ML не является обязательным финалом аналитической
карьеры.

## Соберите это

Сначала постройте решение без рекомендательной системы.

### Шаг 1: выпишите желаемые результаты работы

Возьмите пять будущих задач и для каждой заполните:

| Задача | Кто принимает решение | Итоговый артефакт | Как проверить качество |
|---|---|---|---|
| Почему снизилась активация? | Product manager | Исследование и рекомендация | Метрика воспроизводится, альтернативы проверены |
| Подготовить слой заказов | Аналитики команды | Витрина и контракт | Ключи, свежесть и суммы проходят тесты |

Не записывайте библиотеки. «Использовать CatBoost» не является бизнес-результатом.

### Шаг 2: присвойте задачам маршруты

Для каждой строки выберите маршрут по артефакту:

- отчет, исследование, автоматизированный расчет: базовый;
- метрики, поведение, эксперимент, причинный вопрос: продуктовый;
- витрина, контракт, lineage, производительность: analytics engineering;
- прогноз, ранжирование, классификация, model card: ML.

Подсчитайте голоса. Ничья означает пересечение интересов, а не ошибку. В этом случае
выберите основной маршрут по ближайшему рабочему результату, а вторую ветку оставьте
дополнительной.

## Используйте это

Артефакт `outputs/route_advisor.py` автоматизирует ту же прозрачную схему. Он задает пять
вопросов, показывает баллы всех маршрутов и не скрывает ничью.

Интерактивный запуск:

```bash
python3 outputs/route_advisor.py
```

Воспроизводимый запуск:

```bash
python3 outputs/route_advisor.py \
  --answers product,product,basic,product,data \
  --json
```

CLI не оценивает текущие знания. Для placement-диагностики используется
`/find-your-level`: интерес определяет маршрут, а знания определяют стартовую фазу.

## Сломайте это

Проверьте четыре неудачных сценария:

1. Выберите ML только потому, что хотите изучать сложные модели, но укажите итоговым
   результатом регулярную витрину.
2. Дайте по одному голосу всем маршрутам и посмотрите, как CLI показывает неопределенность.
3. Передайте неизвестный код ответа и убедитесь, что программа останавливается с понятной
   ошибкой.
4. Попробуйте пропустить Core, хотя желаемый маршрут использует pandas, SQL и тесты.

Главная поломка здесь методологическая: рекомендатор не должен путать желаемую роль с
готовностью начать ее продвинутые фазы.

## Проверьте это

Хорошая рекомендация отвечает на вопросы:

- Совпадает ли маршрут с желаемыми артефактами?
- Показаны ли альтернативы при близких баллах?
- Отделен ли выбор специализации от входной диагностики?
- Включает ли маршрут все пререквизиты?
- Можно ли объяснить рекомендацию без ссылки на «алгоритм так решил»?

Тесты в `tests/test_main.py` проверяют устойчивость выбора, ничью и ошибочный ввод.

## Поставьте результат

Урок поставляет CLI `outputs/route_advisor.py`. Он использует только стандартную библиотеку,
работает интерактивно и принимает ответы через аргумент командной строки. Его можно
встроить в onboarding курса или использовать как основу для формы выбора маршрута.

Manifest и инструкция находятся в `outputs/artifact.json`.

## Упражнения

1. Заполните ручную таблицу для трех задач из своей текущей работы.
2. Добавьте в CLI шестой вопрос так, чтобы он различал исследование временных рядов и ML.
3. Измените scoring: один вопрос может давать баллы двум маршрутам. Проверьте, не стала ли
   рекомендация менее объяснимой.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Роль | Название набора библиотек | Повторяющийся тип ответственности и решений |
| Маршрут | Жесткая профессия на всю карьеру | Рекомендуемая последовательность фаз под ближайший результат |
| Артефакт | Файл, оставшийся после упражнения | Результат, который можно применить вне урока |
| Пререквизит | Формальное ограничение | Знание, без которого следующий материал становится механическим |
