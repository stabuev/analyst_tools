# Пример персональной карты маршрута

Пример показывает качество связи между задачей и маршрутом. Он не является рекомендуемым
ответом для всех студентов.

## Три задачи

### Исследовать снижение активации

- **Получатель:** product manager.
- **Решение:** откатывать ли onboarding и какую гипотезу проверять следующей.
- **Артефакт:** воспроизводимое исследование с сегментами, альтернативными объяснениями и рекомендацией.
- **Критерии качества:** единое определение активации, сверка источников, отсутствие причинного вывода без дизайна.
- **Цена ошибки:** команда откатит полезное изменение из-за дефекта данных.
- **Недостающий контекст:** какое действие доступно команде и когда нужно принять решение.

### Спроектировать эксперимент нового onboarding

- **Получатель:** product manager и разработчики onboarding.
- **Решение:** запускать ли изменение на всех пользователей.
- **Артефакт:** контракт эксперимента с primary metric, guardrails и планом анализа.
- **Критерии качества:** единица рандомизации согласована, метрики объявлены заранее, SRM проверяется.
- **Цена ошибки:** наблюдаемая корреляция будет выдана за эффект изменения.

### Сделать расчет активации переиспользуемым

- **Получатель:** продуктовые аналитики команды.
- **Решение:** использовать ли одну витрину во всех исследованиях и экспериментах.
- **Артефакт:** витрина активации с grain, контрактом, freshness и тестами.
- **Критерии качества:** уникальный ключ, сверка с событиями, мониторинг свежести и владелец.
- **Цена ошибки:** разные отчеты будут принимать решения по несовместимым определениям.

## Выбор

- **Основной маршрут:** продуктовый аналитик.
- **Обоснование:** ближайшие решения связаны с поведением пользователей и проверкой продуктового изменения; первый портфельный артефакт — исследование активации.
- **Названия вакансий для поиска:** product analyst, growth analyst и data analyst в продуктовой команде.
- **Обязанности, которые должны присутствовать:** анализ поведения пользователей, развитие системы метрик, проверка продуктовых изменений и формулирование рекомендаций.
- **Дополнительный маршрут:** Analytics Engineer.
- **Зависимость:** качество исследования ограничено надежностью общей витрины активации.

## Старт

- **Диагностика:** проверить SQL-агрегации, joins и базовую статистику до продуктовых фаз.
- **Следующий шаг:** пройти диагностику 00/02 и выбрать стартовую фазу по фактическим пробелам.
- **Условие пересмотра:** основная рабочая ответственность сместится с исследований на поддержку моделей данных команды.
