## Проблема

В уроке `00/03` вы создали локальную историю расчёта дневной выручки. В ней видно, когда
появился запрос и когда в выручке остались только оплаченные заказы. Это уже лучше папки
`final_really_final`, но пока историю проверял только её автор.

Теперь заказчик просит добавить к дневной выручке количество оплаченных заказов. Правка
кажется небольшой: одна строка в SQL и несколько строк в определении метрики. Но даже в
таком изменении можно:

- случайно считать строки после размножающего `JOIN`, а не заказы;
- изменить SQL и забыть изменить документацию;
- включить в коммит несвязанный черновик;
- написать «всё работает» без контрольного расчёта;
- отправить изменение в общую ветку до независимой проверки.

Ветка позволяет собрать изменение отдельно от основной линии истории. Pull request, или
PR, превращает эту ветку в проверяемое предложение: показывает цель, diff, способ
проверки, ограничения и обсуждение. Ревью нужно не для оценки автора, а для снижения риска
ошибочного решения.

Для аналитического проекта особенно важно, что успешно выполнившийся SQL ещё не является
доказательством корректности. Запрос может вернуть таблицу и одновременно завысить сумму
из-за неверного grain. Поэтому в этом уроке недостаточно создать ветку: вы проведёте
отдельное ревью намеренно дефектного изменения.

## Концепция

### На что опирается этот урок

Мы не начинаем Git заново. Вы уже умеете выполнять цикл:

```text
status → diff → add → diff --cached → commit → log
```

У вас уже есть репозиторий:

```text
work/00-03-git/revenue-project/
├── .gitignore
├── README.md
├── data/raw/orders.csv          # локальный игнорируемый файл
├── docs/metric-definition.md
└── queries/revenue_by_day.sql
```

В обязательной практике предыдущего урока появились три решения:

```text
Initialize revenue project
Add daily revenue calculation
Count only paid orders in revenue
```

Если вы выполнили дополнительные упражнения и создали ещё несколько понятных коммитов,
это нормально. Новый урок продолжает текущее чистое состояние проекта и не требует
точного количества коммитов.

### Ветка — не копия папки

Ветка — имя, указывающее на один из коммитов. Представьте текущую историю так:

```text
A --- B --- C  main
```

После создания ветки обе метки сначала указывают на один коммит:

```text
A --- B --- C  main, feature/paid-order-count
```

Новый коммит, сделанный в рабочей ветке, перемещает только её метку:

```text
A --- B --- C  main
             \
              D  feature/paid-order-count
```

Ветка не создаёт вторую физическую папку проекта. Вы по-прежнему работаете в одном
`revenue-project`. Команда `git switch` выбирает линию истории и приводит tracked-файлы в
состояние выбранного коммита.

Отсюда следует важная граница: untracked и игнорируемые файлы не принадлежат отдельной
ветке. Локальный `data/raw/orders.csv` останется на диске при переключении между `main` и
feature-веткой. Ветка изолирует историю tracked-файлов, но не является отдельным
окружением или защищённой копией данных.

### Основная и рабочая ветка

В этом курсе основная ветка называется `main`. Она содержит состояние проекта, которое
команда считает общим и пригодным для дальнейшей работы.

Рабочая, или feature-ветка, создаётся для одного ограниченного изменения:

```text
feature/paid-order-count
fix/order-join-cardinality
docs/revenue-definition
```

Имя должно описывать задачу, а не автора и не календарную дату. Ветка обычно живёт
недолго: её создают от актуального `main`, проверяют через PR, объединяют и удаляют.

### Локальный Git, remote и GitHub

До этого момента Git работал только на вашем компьютере. GitHub добавляет удалённую копию
репозитория и интерфейс совместной работы.

```text
локальный репозиторий                  GitHub

main ---------------------- push ----> main
feature/paid-order-count --- push ----> feature/paid-order-count
                                      │
                                      └─ pull request в main
```

**Remote** — сохранённый в локальном Git адрес другого репозитория. `origin` — обычное,
но не обязательное имя первого remote. Команда `git remote -v` показывает известные
адреса.

`git push` отправляет коммиты и ветку в remote. Он не создаёт PR автоматически. Pull
request существует на GitHub: там выбирают ветку назначения, ветку с изменением,
приглашают reviewer и принимают решение о merge.

В PR используются два направления:

- **base** — куда предлагается добавить изменение; в уроке это `main`;
- **head** — откуда берётся изменение; это `feature/paid-order-count`.

Полное устройство remote-tracking веток, протоколов, fetch, rebase и конфликтов не входит
в минимальный результат занятия.

### Что показывает diff будущего PR

Пока вы редактируете файлы, обычный `git diff` показывает незавершённую работу. После
коммита рабочая папка снова чиста, поэтому обычный diff может быть пустым. Чтобы увидеть
всё предложение ветки относительно `main`, используйте:

```bash
git diff main...HEAD
```

Здесь `HEAD` означает выбранный сейчас коммит. На практическом уровне команда отвечает на
вопрос: «Что текущая ветка предлагает добавить в `main`?»

Список коммитов предложения показывает:

```bash
git log --oneline main..HEAD
```

Почему в одной команде три точки, а в другой две, станет важно при расходящейся истории.
На этом уроке достаточно запомнить две целые команды и проверяемый вопрос каждой. Команды
`git merge-base`, rebase и подробная алгебра диапазонов остаются за пределами обязательной
практики.

### Pull request — не замена коммита

Коммит и PR отвечают на разные вопросы:

| Объект | Главный вопрос |
|---|---|
| Коммит | Какое законченное изменение сохранено в истории? |
| Ветка | Какие связанные коммиты пока отделены от `main`? |
| Pull request | Почему это изменение нужно объединить и чем доказана его корректность? |
| Review | Какие риски проверены и можно ли безопасно выполнить merge? |

Описание PR не должно пересказывать список файлов. Reviewer и так увидит его в diff.
Полезное описание содержит:

1. задачу и ожидаемое решение;
2. что изменилось на уровне поведения;
3. как получено проверяемое доказательство;
4. какие ограничения остаются.

### Ревью аналитического изменения

Проверка стиля не заменяет проверку расчёта. Для аналитического SQL порядок вопросов
обычно такой:

1. Какова единица одной строки до и после изменения?
2. Какие ключи используются и уникальны ли они там, где ожидается уникальность?
3. Какова кардинальность каждого `JOIN`?
4. Могут ли строки размножиться или исчезнуть?
5. Что происходит с `NULL`, дубликатами и фильтрами?
6. Сходится ли результат с маленьким ручным примером?
7. Обновлены ли определение метрики и ограничения вывода?

Хорошее замечание reviewer содержит четыре части:

```text
наблюдение → риск → требуемое исправление → способ повторной проверки
```

Например:

```text
После JOIN одна строка заказа повторяется для каждого товара. Поэтому SUM(orders.amount)
завышает выручку. Нужно агрегировать order_items до order_id или считать выручку до JOIN,
а затем сверить сумму с контролем на исходной таблице orders.
```

### Три решения reviewer

- **Comment** — вопрос, уточнение или необязательное улучшение. Merge не обязательно
  блокировать.
- **Approve** — обязательные риски проверены, изменение можно объединить.
- **Request changes** — найден дефект или отсутствует доказательство, без которого merge
  небезопасен.

`Request changes` относится к состоянию изменения, а не к способностям автора. После
исправления reviewer должен повторно проверить diff и изменить решение.

### Шаг 3. Добавьте количество оплаченных заказов

Откройте `queries/revenue_by_day.sql`. Добавьте `COUNT(*)` рядом с выручкой:

```sql
SELECT
    order_date,
    SUM(amount) AS revenue,
    COUNT(*) AS paid_order_count
FROM orders
WHERE status = 'paid'
GROUP BY order_date;
```

Такой расчёт корректен при предположении, что таблица `orders` содержит одну строку на
заказ. Зафиксируйте это предположение в `docs/metric-definition.md`:

```markdown
# Дневная выручка

Grain результата — одна строка на календарный день.

- `revenue` — сумма `amount` оплаченных заказов за день;
- `paid_order_count` — число строк оплаченных заказов за день.

Расчёт предполагает одну строку на `order_id` в таблице `orders`.
```

Если вы выполнили упражнения прошлого урока и уже расширили запрос, не удаляйте полезные
изменения. Добавьте новый показатель в текущую версию и сохраните только новый законченный
шаг.

Повторите знакомый цикл:

```bash

git commit -m "Add paid order count to daily revenue"
```

Перед коммитом ответьте себе:

- в diff есть только изменение текущей задачи;
- SQL и документация согласованы;
- `COUNT(*)` считает заказы только при заявленном grain источника;
- локальная выгрузка не вошла в staging area.

### Шаг 6. Разберите связь с GitHub

Посмотрите, настроен ли remote:

```bash
git -C work/00-03-git/revenue-project remote -v
```

В локальном учебном проекте команда, скорее всего, ничего не напечатает. Это ожидаемо:
предыдущий урок не создавал удалённый репозиторий.

Для необязательной реальной публикации сначала создайте пустой репозиторий на GitHub и
добавьте его адрес. Не выполняйте команды с буквальным `<USER>` и `<REPOSITORY>`:

```bash
cd work/00-03-git/revenue-project
git remote add origin https://github.com/<USER>/<REPOSITORY>.git
git push -u origin main
git push -u origin feature/paid-order-count
cd ../../..
```

Первая команда сохраняет адрес под именем `origin`. Первый push публикует base, второй —
рабочую ветку. Флаг `-u` связывает локальную ветку с одноимённой удалённой, чтобы следующие
push могли быть короче.

Публикация не является обязательной частью проверки урока. Не отправляйте рабочие данные,
чужой репозиторий или проект, который вы не имеете права публиковать. Используйте только
синтетический `revenue-project` и сначала перечитайте `git status` и историю tracked-файлов.

После push на GitHub создайте pull request:

```text
base: main
compare/head: feature/paid-order-count
```

Вставьте подготовленное описание. Сам `git push` PR не создаёт.

### Шаг 7. Проведите настоящее аналитическое ревью

Откройте учебный кейс:

```bash
cat phases/00-entry-and-tools/04-branches-and-review/outputs/review_case_context.md
cat phases/00-entry-and-tools/04-branches-and-review/outputs/review_case.sql
```

Задача автора — добавить число товаров к дневному отчёту, не изменив выручку. Исходные
данные:

```text
orders:                         order_items:
order_id  amount               order_id  item_id
101       120                  101       A
102        75                  101       B
                                102       C
```

Контрольная выручка на grain заказа равна `120 + 75 = 195`. После `JOIN` в предложенном
SQL заказ `101` появляется дважды, поэтому `SUM(orders.amount)` даёт
`120 + 120 + 75 = 315`.

Скопируйте шаблон ревью:

```bash
cp \
  phases/00-entry-and-tools/04-branches-and-review/outputs/analytics_review_template.md \
  work/00-04-git/review.md
```

Заполните его. Для данного дефекта решение должно быть блокирующим:

```markdown
## Решение

Request changes

## Файл и строки

`review_case.sql`, выражение `SUM(orders.amount)` после `JOIN order_items`.

## Наблюдение

Один заказ повторяется для каждого товара. Заказ `101` после JOIN представлен двумя
строками.

## Риск

Выручка завышается: контрольное значение 195 превращается в 315. Любой отчёт с разным
числом товаров в заказе будет систематически искажён.

## Что исправить

Посчитать выручку на grain `order_id` до JOIN либо сначала агрегировать `order_items` до
одной строки на заказ и только затем соединять таблицы.

## Как проверить

Сверить итоговую выручку с независимой суммой из `orders` и добавить случай с заказом из
двух товаров. После исправления результат должен остаться равным 195.
```

Фраза «неправильный JOIN» была бы недостаточна: она не показывает механизм ошибки, её
масштаб и критерий исправления.

## Используйте это

Проверяющий артефакт собирает локальный пакет будущего PR. Запускайте его до merge, пока
выбрана feature-ветка:

```bash
python3 \
  phases/00-entry-and-tools/04-branches-and-review/outputs/pr_review_packet.py \
  work/00-03-git/revenue-project \
  --base main \
  --body work/00-04-git/pull-request.md \
  --review work/00-04-git/review.md
```

Сохраните отчёт:

```bash
python3 \
  phases/00-entry-and-tools/04-branches-and-review/outputs/pr_review_packet.py \
  work/00-03-git/revenue-project \
  --base main \
  --body work/00-04-git/pull-request.md \
  --review work/00-04-git/review.md \
  --output work/00-04-git/pr-review-packet.md
```

CLI проверяет только наблюдаемую структуру:

- выбрана отдельная ветка с коммитами и изменёнными файлами;
- рабочая папка не содержит забытых изменений;
- PR описывает задачу, изменение, проверку и ограничения;
- ревью содержит решение, точное наблюдение, риск, исправление и повторную проверку;
- решение reviewer принадлежит допустимому набору.

Программа не может доказать, что замечание методологически верно. Именно поэтому учебный
кейс содержит ручное контрольное значение 195.

Готовая демонстрация создаёт временное продолжение `revenue-project` и печатает такой же
пакет:

```bash
python3 phases/00-entry-and-tools/04-branches-and-review/code/main.py
```

Python здесь служит только воспроизводимой проверкой Git-workflow и не является темой
занятия.

### Что происходит после замечания

В настоящем PR автор не создаёт новый PR для каждого исправления. Он меняет файлы в той
же ветке, повторяет проверки, создаёт новый коммит и делает push:

```bash
git status
git diff
git add путь/к/исправленному-файлу
git diff --cached
git commit -m "Fix order grain before item join"
git push
```

PR обновится автоматически, потому что его head остаётся той же веткой. Reviewer должен:

1. прочитать новый diff и ответ автора;
2. повторить контрольный расчёт;
3. убедиться, что исправление не создало новый риск;
4. выбрать `Approve`, только если блокирующее замечание действительно закрыто.

### Как завершается ветка

При работе на GitHub merge выполняют через интерфейс PR после обязательных проверок и
одобрений. Затем локальный `main` обновляют и удаляют уже объединённую рабочую ветку:

```bash
git switch main
git pull --ff-only
git branch -d feature/paid-order-count
```

Для полностью локальной практики без GitHub после сохранения PR-пакета имитируйте
одобренный merge:

```bash
cd work/00-03-git/revenue-project
git switch main
git merge --ff-only feature/paid-order-count
git branch -d feature/paid-order-count
git log --oneline -4
cd ../../..
```

`--ff-only` разрешает простой перенос `main` вперёд, если основная ветка не получила
других коммитов. Расходящиеся ветки и конфликты требуют отдельного решения и не входят в
этот урок.

## Сломайте это

### Создайте ветку с незавершёнными изменениями

Незакоммиченные изменения принадлежат рабочей папке, а не ветке. Если изменить файл на
`main`, а затем выполнить `git switch -c feature/...`, правка обычно перейдёт вместе с
рабочей папкой. Это может быть полезно, но для новичка часто смешивает задачи.

Безопасный порядок перед ответвлением:

```bash
git switch main
git status
git switch -c feature/task-name
```

Если `git status` не чист, сначала объясните каждую запись и решите, к какой задаче она
относится.

### Примите ветку за отдельную копию данных

Переключение ветки меняет tracked-файлы. Оно не создаёт второй `.env`, вторую выгрузку или
отдельное виртуальное окружение. Не используйте ветки как механизм разделения
чувствительных данных или локальной конфигурации.

### Решите, что `push` уже создал PR

После `git push` ветка появилась в remote, но обсуждение ещё не создано. Проверьте GitHub:
PR должен иметь правильные base и head, описание, reviewer и видимый diff.

### Напишите «всё работает»

Эта фраза не называет команду, вход, ожидаемый результат и границу проверки. Сравните:

```text
Слабо: Всё работает.

Проверяемо: git diff main...HEAD содержит изменение SQL и определения метрики.
На трёх оплаченных строках paid_order_count равен 3. Уникальность order_id отдельно
не проверяется.
```

### Одобрите зелёный SQL без проверки grain

CI проверяет только формализованные условия. Если теста на размножение строк нет, зелёный
статус не заметит завышенную выручку. Reviewer обязан проверить методологию и маленький
контрольный пример, а не только наличие зелёной отметки.

### Оставьте замечание без требуемого действия

Фраза «кажется, здесь проблема с JOIN» заставляет автора угадывать. Укажите точное место,
механизм ошибки, последствия, ожидаемое исправление и способ повторной проверки.

## Проверьте это

До локального или удалённого merge пакет должен проходить CLI-проверку:

```bash
python3 \
  phases/00-entry-and-tools/04-branches-and-review/outputs/pr_review_packet.py \
  work/00-03-git/revenue-project \

  --body work/00-04-git/pull-request.md \
```

Затем вручную ответьте:

1. По имени ветки понятно назначение изменения?
2. В `main...HEAD` находятся только файлы текущей задачи?
3. В PR названо проверяемое изменение поведения, а не только файлы?
4. Ограничение про уникальность `order_id` не скрыто?
5. Ревью воспроизводит контроль 195 → 315?
6. `Request changes` содержит конкретное условие повторной проверки?

Запустите тесты артефакта:

Тесты создают настоящие временные Git-репозитории и проверяют готовую feature-ветку,

После сохранения пакета и merge проверьте завершённое состояние:

```bash

git -C work/00-03-git/revenue-project log --oneline -4
```

Должна остаться основная ветка `main`, чистая рабочая папка и новый коммит в истории.

## Поставьте результат

Результат урока состоит из двух связанных частей.

В `work/00-03-git/revenue-project`:

- изменение создано в отдельной ветке;
- SQL и определение метрики изменены одним связанным коммитом;
- перед merge проверен полный diff относительно `main`;
- после локальной практики изменение объединено в `main`, а завершённая ветка удалена.

В `work/00-04-git/`:

```text
pull-request.md        # проверяемое описание собственного изменения
review.md              # доказательное ревью дефектного SQL
pr-review-packet.md    # машинная проверка структуры пакета до merge
```

Переиспользуемый артефакт `outputs/pr_review_packet.py` можно запускать перед публикацией
небольшого учебного PR. Он не оценивает автора, не ограничивает число файлов или строк и
не заменяет содержательное ревью.

## Упражнения

1. Создайте новую ветку только для уточнения README. Подготовьте PR, в котором явно
   объяснено, почему документационное изменение не меняет расчёт.
2. Измените учебный review-кейс так, чтобы один заказ не имел товаров. Добавьте в ревью
   вопрос о выборе `LEFT JOIN` или `INNER JOIN` и последствии для популяции заказов.
3. Классифицируйте три замечания как `Comment` или `Request changes`: опечатка в описании,
   размножение выручки после JOIN, отсутствие контрольного расчёта для нового знаменателя.
   Для каждого объясните, блокирует ли оно безопасный merge.

## Ключевые термины

| Термин | Распространённое заблуждение | Точное значение |
|---|---|---|
| Ветка | Отдельная копия папки и данных | Имя, указывающее на коммит в общей истории репозитория |
| `main` | Специальная встроенная ветка Git | Принятое имя основной линии проекта |
| Remote | Облачная версия любого файла | Сохранённый адрес другого Git-репозитория |
| `origin` | Обязательный сервер Git | Обычное имя первого remote |
| `push` | Создание pull request | Отправка локальных коммитов и ссылки ветки в remote |
| Base | Ветка автора | Ветка назначения PR, обычно `main` |
| Head | Основная ветка | Ветка с предлагаемыми изменениями |
| Pull request | Команда Git | Объект платформы для предложения, проверки и обсуждения diff |
| Review | Проверка стиля | Независимая оценка корректности, риска и доказательств |
| Request changes | Негативная оценка автора | Решение не выполнять merge до устранения блокирующего риска |
| Merge | Сохранение незакоммиченного файла | Объединение проверенной линии истории с base |
