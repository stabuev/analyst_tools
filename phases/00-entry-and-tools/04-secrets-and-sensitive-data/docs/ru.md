# Секреты и безопасная работа с данными

> В Git сохраняют воспроизводимый контракт проекта, но не ключи доступа и не рабочие
> выгрузки, которые нельзя безопасно показать всем получателям репозитория.

**Тип:** Build
**Треки:** Core
**Пререквизиты:** 00/03 — Ветки, pull request и ревью
**Время:** ~75 минут
**Результат:** отделяет публикуемые файлы аналитического проекта от локальных секретов и
чувствительных данных, проверяет эту границу перед merge и объясняет порядок действий при
утечке

## Цели обучения

- Различать секрет, чувствительные данные и обычный публикуемый файл проекта.
- Решать, можно ли получать и сохранять данные, до создания локальной выгрузки.
- Объяснять назначение `.env`, `.env.example` и переменной окружения без магии и жаргона.
- Проверять одновременно два свойства: путь игнорируется и файл не отслеживается Git.
- Готовить безопасный синтетический sample вместо фрагмента реальной выгрузки.
- Читать staged diff как последнюю границу перед коммитом и pull request.
- Ставить отзыв или ротацию раскрытого секрета раньше удаления строки из Git.

## Проблема

В двух предыдущих уроках вы создали `revenue-project`, а затем провели изменение SQL через
ветку и ревью. Теперь проект нужно подключить к хранилищу. Для этого появляется строка
подключения `WAREHOUSE_DSN`, а для локальной проверки — выгрузка заказов.

Оба объекта нужны для работы, но это не значит, что оба должны попасть в репозиторий:

- строка подключения может давать доступ к рабочей системе;
- выгрузка может содержать пользовательские, финансовые или другие непубличные поля;
- commit сохраняет содержимое в истории, откуда его недостаточно удалить следующим
  коммитом;
- pull request показывает diff другим людям и может передать его в проверки, логи и
  сторонние интеграции.

Опасный проект часто выглядит вполне аккуратно:

```text
revenue-project/
├── queries/revenue_by_day.sql
├── config.py                   # внутри рабочая строка подключения
└── data/orders-small.csv       # первые десять реальных заказов
```

Название `small` не делает данные безопасными. Перенос строки подключения в другой файл
тоже ничего не решает, если этот файл отслеживает Git.

В этом занятии мы не будем разрабатывать secret scanner и не будем изучать Python.
Нужный результат проще и важнее: провести понятную границу между тем, что можно показать
reviewer, и тем, что должно остаться вне Git.

## Концепция

### Три разных объекта

Словом «секрет» иногда называют всё непубличное. Для принятия решений этого недостаточно.

| Объект | Что это | Пример | Что произойдёт при раскрытии |
|---|---|---|---|
| Секрет доступа | Значение, подтверждающее полномочия | пароль, API token, private key, DSN с паролем | посторонний может получить доступ или выполнить действие |
| Чувствительные данные | Информация, которую нельзя свободно распространять | email, телефон, платежи, обращения в поддержку, внутренние результаты | возможен вред человеку, компании или исследованию |
| Публикуемый файл проекта | Материал, разрешённый для аудитории репозитория | SQL, документация, пустой env-пример, проверенный synthetic sample | получатель понимает и воспроизводит проект без защищаемых значений |

Секрет после раскрытия обычно нужно **отозвать или ротировать**: прекратить действие
старого значения и выпустить новое. Чувствительные данные нельзя «ротировать» — для них
нужны разрешение, минимизация, контроль доступа и правила хранения.

Поэтому вопрос «есть ли здесь пароль?» слишком узкий. Перед публикацией нужно спросить:

1. Что именно находится в файле?
2. Имеем ли мы право передать это всем получателям репозитория?
3. Нужны ли эти строки и поля для воспроизведения результата?
4. Можно ли заменить их агрегатом или синтетическим примером?
5. Где рабочая версия должна храниться по правилам организации?

### Что хранится в Git, а что остаётся снаружи

Для учебного проекта граница выглядит так:

```text
Git                                    Вне Git

queries/revenue_by_day.sql             .env с рабочим значением
docs/metric-definition.md              data/raw/orders.csv
.env.example без значений              локальные credentials
data/sample/orders.csv                 временные чувствительные результаты
docs/data-handling.md                  секреты в одобренном хранилище
```

Левая сторона объясняет проект и может пройти через commit, PR и clone. Правая сторона
нужна во время работы, но не является частью переносимой истории.

Private repository уменьшает число людей, которые видят Git, но не превращает правую
сторону в левую. Участники, CI, резервные копии и интеграции всё равно получают доступ к
tracked-файлам. Кроме того, разрешение читать репозиторий не означает разрешение видеть
любую пользовательскую выгрузку.

### `.env`, `.env.example` и переменная окружения

Эти три понятия связаны, но не равны.

**Переменная окружения** — именованное значение, которое операционная система передаёт
запущенному процессу. Проект просит значение по имени, например `WAREHOUSE_DSN`, вместо
того чтобы хранить его в коде.

**`.env`** — распространённое имя локального текстового файла с парами `ИМЯ=значение`.
Сам Python, SQL или Git не загружает этот файл автоматически. Его может прочитать
отдельный инструмент. `.env` удобен локально, но сам по себе не является защищённым
хранилищем.

**`.env.example`** — публикуемая подсказка о необходимых именах:

```text
WAREHOUSE_DSN=
```

В нём нет рабочего значения. Коллега видит, что нужно настроить, но получает сам секрет
через одобренный канал: secret manager, CI secrets, оркестратор или другой принятый в
организации механизм.

Полезная формула:

```text
репозиторий хранит имя и назначение → защищённая система хранит значение
```

Environment variable решает вопрос передачи значения процессу, но не отвечает на вопрос,
где оно безопасно хранится до запуска.

### `.gitignore` — фильтр для новых файлов, а не ластик истории

Правило `.gitignore` помогает Git не предлагать подходящий untracked-файл для добавления.
Оно не удаляет файл с диска и не прекращает отслеживание файла, который уже попал в
commit.

Нужны две разные проверки:

```bash
git check-ignore -v .env
git ls-files -- .env
```

- `git check-ignore` отвечает: «Какое правило должно игнорировать этот путь?»;
- `git ls-files` отвечает: «Хранит ли Git этот путь в своей истории и index?».

Для безопасного локального `.env` первая команда должна показать правило, а вторая — не
напечатать ничего. Одна успешная команда не заменяет другую.

### Три момента, когда аналитик принимает решение

Безопасность не начинается перед push. Решение принимается раньше.

1. **До выгрузки.** Нужны ли все строки и поля? Можно ли агрегировать данные в источнике?
2. **До `git add`.** Какие файлы видит `git status` и почему каждый из них существует?
3. **До commit или PR.** Что находится в `git diff --cached` и полном diff ветки?

Scanner может найти известный формат токена, но не поймёт, что столбец `comment` содержит
личные обращения, а десять строк принадлежат реальным пользователям. Эта часть остаётся
профессиональным решением автора и reviewer.

### Если секрет уже опубликован

Удалить строку и сделать новый commit недостаточно. Старое значение могло остаться в
истории, clone, PR, логе или уведомлении.

Базовый порядок действий:

1. Прекратить дальнейшую публикацию и не копировать значение в новые сообщения.
2. Отозвать или ротировать секрет у системы, которая его выдала.
3. Сообщить ответственному владельцу или security-команде по правилам организации.
4. Определить, где значение было доступно: commit, PR, fork, CI log, notebook output.
5. Убрать его из текущего состояния и согласовать необходимость очистки истории и логов.
6. Добавить профилактику: правило, review-вопрос, scanner или изменение процесса.

Отзыв стоит раньше очистки Git, потому что красивой истории недостаточно: уже увиденное
значение продолжает работать, пока система его принимает.

## Соберите это

Практика продолжает тот же `revenue-project`. Новый репозиторий создавать не нужно.

### Шаг 1. Вернитесь к завершённому проекту

Начните из корня курса:

```bash
python3 \
  phases/00-entry-and-tools/02-git-foundations/outputs/git_project_check.py \
  work/00-02-git/revenue-project
```

После предыдущего урока проект должен находиться на чистой ветке `main`. Проверьте это и
создайте отдельную ветку текущего изменения:

```bash
cd work/00-02-git/revenue-project
git switch main
git status --short
git switch -c feature/security-boundary
```

Если `git status --short` что-то показывает, сначала объясните каждую строку. Не создавайте
security-ветку поверх забытой работы.

### Шаг 2. Дополните существующие правила

В первом Git-уроке `.gitignore` уже получил правила `.env` и `data/raw/`. Не заменяйте
файл новым шаблоном: добавьте только недостающие варианты локального env-файла.

Итоговый фрагмент должен выглядеть так:

```gitignore
.env
.env.*
!.env.example
data/raw/
```

Правило `.env.*` закрывает `.env.local`, `.env.dev` и похожие файлы. Следующая строка с
`!` возвращает `.env.example` в видимую часть проекта, потому что этот файл хранит только
контракт.

Создайте контракт:

```bash
printf '# Required warehouse connection\nWAREHOUSE_DSN=\n' > .env.example
```

Значение после `=` должно оставаться пустым. Не вставляйте рабочий DSN даже на минуту и
даже в локальной ветке: содержимое может попасть в editor backup, staged diff или историю.

### Шаг 3. Докажите, что локальные значения остаются снаружи

Создайте только учебное, заведомо недействующее значение:

```bash
printf 'WAREHOUSE_DSN=demo://local-only\n' > .env
```

Локальная raw-выгрузка `data/raw/orders.csv` уже осталась после первого Git-урока.
Проверьте оба пути:

```bash
git check-ignore -v .env
git check-ignore -v data/raw/orders.csv
git ls-files -- .env data/raw/orders.csv
git status --short
```

Ожидаемая интерпретация:

- две команды `check-ignore` называют правила из `.gitignore`;
- `git ls-files` ничего не печатает;
- `.env` и `data/raw/orders.csv` не появляются в статусе;
- `.env.example` виден как безопасный новый файл.

Пустой вывод здесь является результатом проверки, а не признаком сломанной команды.

Чтобы увидеть принцип environment variable без программирования, можно безопасно
проверить только факт наличия значения:

```bash
read -rs WAREHOUSE_DSN
export WAREHOUSE_DSN
printf '\n'
if [ -n "${WAREHOUSE_DSN:-}" ]; then echo 'WAREHOUSE_DSN configured'; fi
unset WAREHOUSE_DSN
```

Введите учебное `demo://local-only`. Команда печатает только факт настройки, а не само
значение. В рабочем проекте способ передачи секрета определяет инфраструктура вашей
организации.

### Шаг 4. Создайте действительно синтетический sample

Raw-файл нельзя просто переименовать в sample. Создайте маленькие вымышленные строки, не
копируя пользователей из источника:

```bash
mkdir -p data/sample
printf '%s\n' \
  'order_id,order_date,status,amount' \
  'TEST-001,2026-01-10,paid,120' \
  'TEST-002,2026-01-10,pending,80' \
  'TEST-003,2026-01-11,paid,75' \
  > data/sample/orders.csv
```

Префикс `TEST-`, вымышленные даты и вручную выбранные значения делают происхождение
примера видимым. В реальной работе одного необычного ID недостаточно: убедитесь, что
остальные поля тоже не скопированы из production.

Почему sample полезен в Git:

- reviewer понимает ожидаемую структуру;
- команды и тесты можно повторить без доступа к хранилищу;
- граничные случаи выбраны намеренно;
- проект не распространяет рабочую выгрузку.

### Шаг 5. Заполните карту границы

Скопируйте самостоятельный шаблон в документацию проекта:

```bash
cp \
  ../../../phases/00-entry-and-tools/04-secrets-and-sensitive-data/outputs/data-handling-template.md \
  docs/data-handling.md
```

Заполните его своими словами. Для карты файлов достаточный уровень конкретики выглядит
так:

| Путь или объект | Что находится внутри | Где хранится рабочее значение | Можно в Git? | Почему |
|---|---|---|---|---|
| `.env.example` | имя `WAREHOUSE_DSN` без значения | Git | да | документирует настройку, но не раскрывает доступ |
| `.env` | локальный DSN | одобренное хранилище и окружение процесса | нет | значение может давать доступ к хранилищу |
| `data/raw/orders.csv` | локальная рабочая выгрузка | разрешённая локальная зона | нет | происхождение и разрешение на публикацию не подтверждены |
| `data/sample/orders.csv` | вручную созданные строки `TEST-*` | Git | да | пример синтетический и нужен для воспроизведения |

В разделе контракта окружения укажите назначение имени, но не значение:

| Имя переменной | Для чего нужна | Откуда должно приходить значение | Что нельзя делать со значением |
|---|---|---|---|
| `WAREHOUSE_DSN` | подключение к учебному хранилищу | одобренный secret storage или локальное окружение | коммитить, печатать в лог, вставлять в PR |

Заполните и план реакции на утечку. Первый пункт должен прекращать действие секрета, а не
просто обещать удалить файл.

### Шаг 6. Соберите безопасный коммит

Сначала посмотрите незавершённую работу:

```bash
git status --short
git diff
```

Подготовьте только публикуемую сторону границы:

```bash
git add .gitignore .env.example docs/data-handling.md data/sample/orders.csv
git diff --cached --name-only
git diff --cached
```

В staged diff должны находиться:

- новые ignore-правила без локальных значений;
- пустой env-контракт;
- синтетический sample;
- документ с решениями и доказательствами.

В нём не должно быть `.env`, `data/raw/orders.csv`, реального DSN, email, телефонных
номеров и фрагментов production-строк.

После просмотра заполните в `docs/data-handling.md` оставшиеся доказательства: перечислите
файлы из staged diff и кратко запишите, что проверили вручную. Документ изменился после
первого `git add`, поэтому подготовьте его новую версию и ещё раз прочитайте итоговый
staged diff:

```bash
git add docs/data-handling.md
git diff --cached --name-only
git diff --cached
git ls-files -- .env data/raw/orders.csv
git commit -m "Document secure data boundary"
git status --short
git diff main...HEAD
```

Так в одном связанном коммите сохраняются сама граница и доказательства, на которых она
основана.

## Используйте это

### Проверьте изменение глазами reviewer

До merge прочитайте полный diff ветки и ответьте:

1. Нужен ли каждый публикуемый файл для понимания или воспроизведения проекта?
2. `.env.example` содержит только имена и безопасные комментарии?
3. Sample действительно создан вручную, а не скопирован из источника?
4. Для raw-данных названо допустимое место, а не только правило `.gitignore`?
5. Доказательства проверяют и ignore, и отсутствие tracking?
6. План инцидента начинается с прекращения действия секрета?

Если вы публиковали предыдущий урок на GitHub, отправьте эту же feature-ветку и создайте
PR обычным способом:

```bash
git push -u origin feature/security-boundary
```

В описании PR объясните границу поведения: какие файлы теперь разрешены в Git, какие
остаются локальными и какими командами это проверено. Само значение секрета reviewer не
нужно.

### Примените вопросы к новому источнику

Представьте, что для отчёта по нагрузке поддержки вам предлагают выгрузку:

```text
ticket_id, created_at, category, customer_email, message
```

Для дневного числа обращений по категориям нужны `created_at` и `category`. Email и полный
текст сообщения не нужны для заявленного расчёта и создают дополнительный риск.

Лучшее первое действие — не скачать все пять полей и потом удалить два, а запросить
минимальную выборку или агрегат в источнике. Для воспроизводимого примера можно вручную
создать synthetic sample с категориями и датами.

Это и есть перенос метода: граница строится не по имени `orders.csv`, а по задаче,
содержанию и разрешённой аудитории.

### Завершите ветку

После проверки по рубрике используйте тот же workflow, что в предыдущем уроке. При
реальном PR выполните merge в интерфейсе GitHub, обновите локальный `main` и удалите
ветку.

Для полностью локальной практики:

```bash
git switch main
git merge --ff-only feature/security-boundary
git branch -d feature/security-boundary
git status --short
git log --oneline -5
```

Локальный `.env` и `data/raw/orders.csv` останутся на диске после переключения ветки,
потому что они не являются частью Git-истории.

## Сломайте это

Эти сценарии нужно разобрать по diff и описанию. Не коммитьте настоящие секреты или
чувствительные данные ради упражнения.

### Положитесь только на private repository

Ошибочная мысль: «Репозиторий закрытый, поэтому raw-выгрузку можно добавить».

Private repository отвечает только на часть вопроса о доступе. Он не подтверждает право
копировать данные, не ограничивает CI и backups и не задаёт срок хранения. Сначала нужны
правила владельца данных и минимизация.

### Добавьте правило после того, как файл уже отслеживается

Ошибочная мысль: «Я добавил `.env` в `.gitignore`, значит Git его забыл».

Обычный `git check-ignore` не считает уже отслеживаемый файл ignored. Чтобы отдельно
проверить подходящее правило, используют `git check-ignore -v --no-index .env`; при этом
`git ls-files -- .env` всё ещё напечатает путь. Для реального раскрытого секрета сначала
отзовите значение, затем согласуйте удаление из tracking и истории.

### Превратите `.env.example` в командный `.env`

Ошибочная мысль: «Коллеге будет удобнее, если пример уже заполнен».

Версионируемый пример разойдётся во все clone и PR. Он должен содержать имя,
нечувствительное описание и при необходимости безопасный синтетический placeholder, но не
рабочий credential.

### Возьмите первые десять production-строк как sample

Малый размер уменьшает объём, но не меняет происхождение. Один email или внутренний
идентификатор всё равно относится к реальному человеку или системе. Synthetic sample
создаётся независимо либо проходит отдельную процедуру разрешения публикации.

### Напечатайте значение для диагностики

Секрет может не попасть в Git и всё равно утечь через terminal history, notebook output,
traceback или CI log. Проверяйте факт наличия и безопасные метаданные, а не само значение.

### Удалите строку, но не отзовите секрет

Новый commit исправляет текущее состояние, но старый credential продолжает работать.
Это блокирующая ошибка порядка действий: сначала ротация или отзыв, затем очистка и
профилактика.

## Проверьте это

Проверьте `docs/data-handling.md` по отдельной рубрике:

```bash
cd ../../..
cat \

```

Рубрика требует доказать безопасность решения, а не просто заполнить все строки шаблона.
Особенно проверьте четыре условия:

1. Пустой `.env.example` виден Git, локальный `.env` — нет.
2. Raw-выгрузка существует локально, но отсутствует среди tracked-файлов.
3. Публикуемый sample имеет объяснимое синтетическое происхождение.
4. Реакция на утечку начинается с прекращения действия секрета.

Из корня курса выполните финальные наблюдаемые проверки:

```bash
git -C work/00-02-git/revenue-project check-ignore -v .env
git -C work/00-02-git/revenue-project check-ignore -v data/raw/orders.csv
git -C work/00-02-git/revenue-project ls-files -- .env data/raw/orders.csv
```

`ls-files` и `status` должны быть пустыми. Первые две команды должны назвать правила,

Автоматический scanner здесь намеренно не является экзаменатором. Он может найти
известный шаблон токена, но не может вместо вас решить, разрешено ли публиковать конкретную
таблицу, текст обращения или внутреннюю метрику.

## Ключевые термины

| Термин | Распространённое заблуждение | Точное значение |
|---|---|---|
| Секрет | Любые непубличные данные | Значение, предоставляющее доступ или полномочие |
| Чувствительные данные | Только пароль и паспорт | Информация, раскрытие которой может причинить вред человеку или организации |
| `.env` | Защищённое командное хранилище | Распространённый локальный файл со значениями; сам по себе не является vault |
| `.env.example` | Копия рабочего `.env` | Версионируемый контракт имён без рабочих значений |
| Environment variable | Место постоянного хранения секрета | Способ передать именованное значение запущенному процессу |
| `.gitignore` | Удаление файла из Git | Правило, не предлагающее подходящий untracked-путь для добавления |
| Tracked file | Любой файл на диске | Путь, содержимое которого Git уже хранит и отслеживает |
| Ротация | Переименование переменной | Выпуск нового секрета и прекращение действия старого |
| Минимизация данных | Удаление лишнего после выгрузки | Получение и хранение только того, что нужно для заявленной задачи |
| Synthetic sample | Первые несколько production-строк | Независимо созданный пример без реальных субъектов и защищаемых значений |
