# Терминал и файловая система

> Терминал становится рабочим инструментом, когда путь, поток данных и ошибка остаются
> явными.

**Тип:** Build  
**Треки:** Core  
**Пререквизиты:** 02 — Диагностика Python и SQL  
**Время:** ~60 минут  
**Результат:** навигирует по проекту, управляет файлами и связывает команды через pipes

## Цели обучения

- Объяснять разницу между текущим каталогом, относительным и абсолютным путём.
- Создавать, копировать, перемещать и удалять файлы в ограниченной рабочей области.
- Соединять команды через стандартные потоки, pipes и перенаправления.
- Безопасно обходить имена с пробелами и другими специальными символами.
- Превращать одноразовую команду в воспроизводимый Bash-скрипт с проверкой ошибок.

## Проблема

Аналитический проект быстро обрастает выгрузками, ноутбуками, отчётами и временными
файлами. Пока файлов мало, можно открывать Finder или проводник и искать нужное глазами.
Через месяц непонятно, какая выгрузка самая большая, где лежит результат расчёта и почему
команда коллеги работает только из его домашнего каталога.

Терминал решает эту задачу не потому, что печатать команды «профессиональнее». Его
преимущество — наблюдаемая последовательность операций. Команду можно повторить,
проверить, положить в Git и запустить на другом каталоге. Но некавыченный путь или
непроверенный `rm` превращают ту же скорость в риск потери данных.

В этом уроке вы построите аудит каталога: посчитаете файлы и байты, найдёте самые большие
файлы и сгруппируете их по расширениям.

## Концепция

Shell — одновременно интерпретатор команд и небольшой язык программирования. Он связывает
утилиты через три стандартных потока:

```text
stdin (0)  -> команда -> stdout (1)
                     \-> stderr (2)
```

- `stdin` содержит входные данные.
- `stdout` содержит обычный результат.
- `stderr` содержит диагностику и ошибки.
- `|` передаёт `stdout` команды слева в `stdin` команды справа.
- `>` записывает `stdout` в файл с заменой содержимого.
- `>>` дописывает `stdout` в конец файла.

Путь является аргументом команды, а не свойством файла.

| Вид пути | Пример | Откуда начинается |
|---|---|---|
| Относительный | `data/orders.csv` | От текущего каталога процесса |
| Абсолютный | `/home/anna/project/data/orders.csv` | От корня файловой системы |
| Домашний | `~/project` | После раскрытия `~` shell |

Команда `pwd` показывает текущий каталог, `cd` меняет его для текущего shell, а `ls`
показывает содержимое. Переменная `PATH` решает другую задачу: в ней shell ищет
исполняемую команду, когда вы вводите `python3` или `git` без пути к файлу.

## Соберите это

Работайте в отдельной учебной папке. Не повторяйте команды удаления на каталоге с
реальными данными.

Из корня репозитория перейдите в каталог урока:

```bash
cd phases/00-entry-and-tools/03-terminal-and-filesystem
```

### Шаг 1: создайте небольшое дерево проекта

```bash
mkdir -p "terminal-lab/data/raw" "terminal-lab/reports"
printf 'order_id,amount\n101,120\n102,75\n' > "terminal-lab/data/raw/orders.csv"
printf '# План анализа\n' > "terminal-lab/reports/analysis plan.md"
printf 'temporary\n' > "terminal-lab/data/raw/to-delete.txt"
```

Проверьте своё положение и дерево:

```bash
pwd
find "terminal-lab" -maxdepth 3 -print
```

Кавычки вокруг пути обязательны, когда имя содержит пробел. Полезнее выработать привычку
кавычить подстановки всегда:

```bash
project="terminal-lab"
cd "$project"
```

Без кавычек результат `$project` проходит word splitting и filename expansion. Сегодня
значение простое, завтра переменная содержит `client reports`, и команда получает два
аргумента вместо одного.

### Шаг 2: выполните управляемые операции

Находясь внутри `terminal-lab`, создайте копию и переименуйте её:

```bash
cp "data/raw/orders.csv" "data/raw/orders.backup.csv"
mv "data/raw/orders.backup.csv" "data/raw/orders.snapshot.csv"
```

Удалите только заранее созданный временный файл:

```bash
rm -i "data/raw/to-delete.txt"
```

`rm` не переносит файл в корзину. Перед рекурсивным удалением сначала распечатайте путь и
содержимое цели. Конструкцию `rm -rf "$variable"` нельзя запускать, пока не проверены
пустое значение, текущий каталог и границы рабочей области.

Вернитесь в родительский каталог:

```bash
cd ..
```

### Шаг 3: соберите первый pipeline

Найдите обычные файлы и посчитайте байты каждого:

```bash
find "terminal-lab" -type f -exec wc -c {} \;
```

Передайте строки в числовую сортировку:

```bash
find "terminal-lab" -type f -exec wc -c {} \; | sort -nr
```

Сохраните первые строки отчёта:

```bash
find "terminal-lab" -type f -exec wc -c {} \; |
  sort -nr |
  sed -n '1,5p' > "largest-files.txt"
```

Каждая команда выполняет одну операцию. Это и есть прозрачная модель будущего скрипта:
источник записей, вычисление размера, сортировка и поставка результата.

### Шаг 4: назовите ограничения одноразовой команды

Pipeline выше полезен для исследования, но ещё не является надёжным артефактом:

- формат `wc` предназначен для человека и зависит от состава аргументов;
- имя с переносом строки разрушит построчный отчёт;
- ошибка ранней команды может быть скрыта успешной последней командой;
- повторная запись отчёта внутрь каталога может изменить следующий аудит;
- команда не проверяет аргументы и не объясняет способ использования.

## Используйте это

Артефакт `outputs/file_audit.sh` превращает pipeline в Bash CLI.

Выведите отчёт в терминал:

```bash
bash outputs/file_audit.sh --top 5 "terminal-lab"
```

Запишите его атомарно в файл:

```bash
bash outputs/file_audit.sh \
  --top 10 \
  --output "terminal-lab/audit.md" \
  "terminal-lab"
```

Скрипт сначала строит отчёт во временном файле и только затем перемещает его в указанное
место. Если `audit.md` уже находится внутри проверяемого каталога, он исключается из
подсчёта. Поэтому повторный запуск на неизменившемся дереве даёт тот же результат.

Внутри используются четыре защитных решения:

1. `set -euo pipefail` останавливает выполнение при ошибке, неизвестной переменной и
   неуспешной команде внутри pipeline.
2. Все раскрытия путей заключены в двойные кавычки.
3. `find -print0` и `read -d ''` используют NUL как разделитель имён.
4. `LC_ALL=C` фиксирует правила сортировки и преобразования регистра.

Запустите готовую демонстрацию:

```bash
python3 code/main.py
```

Python здесь только создаёт временное дерево и запускает поставляемый Bash-артефакт.

## Сломайте это

Проверьте характерные поломки в копии учебного каталога.

### Пробел в имени

```bash
touch "terminal-lab/data/raw/client export.csv"
for file in $(find "terminal-lab" -type f); do printf '%s\n' "$file"; done
```

Цикл разделит имя на части. Надёжный вариант не превращает вывод `find` в слова:

```bash
while IFS= read -r -d '' file; do
  printf '%q\n' "$file"
done < <(find "terminal-lab" -type f -print0)
```

### Перенос строки в имени

На Unix имя файла может содержать перевод строки:

```bash
touch $'terminal-lab/data/raw/line\nbreak.csv'
bash outputs/file_audit.sh "terminal-lab"
```

Артефакт читает имя через NUL и печатает shell-escaped представление на одной строке.

### Скрытая ошибка pipeline

Сравните статусы:

```bash
bash -c 'false | true; echo "$?"'
bash -o pipefail -c 'false | true; echo "$?"'
```

Без `pipefail` успешная последняя команда маскирует ошибку слева.

### Неверная рабочая директория

```bash
cd "terminal-lab"
wc -c "terminal-lab/data/raw/orders.csv"
```

Команда ищет вложенный `terminal-lab/terminal-lab`. Исправление — понимать текущий каталог,
а не добавлять `../` наугад:

```bash
pwd
wc -c "data/raw/orders.csv"
```

## Проверьте это

Запустите тесты урока:

```bash
python3 -m unittest discover \
  -s phases/00-entry-and-tools/03-terminal-and-filesystem/tests \
  -p "test_*.py" -v
```

Проверки подтверждают:

- `.git` не попадает в отчёт;
- файлы с пробелами и переносами строк остаются одной записью;
- число файлов и сумма байтов совпадают с контрольным деревом;
- `--top` ограничивает таблицу крупнейших файлов;
- отчёт внутри проверяемого каталога не считает сам себя;
- ошибочные аргументы завершаются кодом `2` и понятным сообщением.

Для ручной проверки выполните скрипт дважды с `--output` и сравните checksum:

```bash
bash outputs/file_audit.sh --output "/tmp/audit-1.md" "terminal-lab"
bash outputs/file_audit.sh --output "/tmp/audit-2.md" "terminal-lab"
cmp -s "/tmp/audit-1.md" "/tmp/audit-2.md" && printf 'identical\n'
```

Одинаковый checksum подтверждает детерминированность только при неизменном дереве и
одинаковом аргументе пути. Это не доказывает полноту аудита или корректность содержимого
файлов.

## Поставьте результат

Урок поставляет `outputs/file_audit.sh` — независимый Bash CLI без Python-зависимостей.
Скопируйте его в аналитический проект и запускайте перед очисткой каталога, ревью большой
выгрузки или настройкой `.gitignore`.

```bash
bash outputs/file_audit.sh \
  --top 20 \
  --output "/tmp/project-audit.md" \
  "/path/to/project"
```

Manifest и готовая команда использования находятся в `outputs/artifact.json`. Скрипт
считает обычные файлы, не следует по символическим ссылкам и исключает каталоги `.git`.
Он измеряет размер и структуру дерева, но не определяет, можно ли безопасно удалить файл.

## Упражнения

1. Добавьте флаг `--include-git` и тест, подтверждающий оба режима.
2. Добавьте таблицу каталогов первого уровня с количеством файлов, сохранив обработку
   пробелов в именах.
3. Добавьте `--max-bytes N`, который завершает скрипт ненулевым кодом при превышении
   бюджета, и объясните, как использовать его в CI.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Текущий каталог | Каталог, где лежит скрипт | Рабочий каталог конкретного процесса |
| Относительный путь | Сокращённый абсолютный путь | Путь, вычисляемый от текущего каталога |
| Pipe | Запись результата в файл | Передача stdout одной команды в stdin другой |
| Перенаправление | То же, что копирование файла | Настройка источника или назначения стандартного потока |
| Exit status | Текст ошибки | Числовой код завершения: `0` обычно означает успех |
| Quoting | Необязательное оформление | Управление раскрытием переменных и специальных символов shell |
| NUL-разделитель | Пустая строка | Байт `0`, который не может входить в Unix-имя файла |

## Дополнительное чтение

- [GNU Findutils: safe file name handling](https://www.gnu.org/software/findutils/manual/html_mono/find.html#Safe-File-Name-Handling) — прочитайте, почему `-print0` надёжнее построчной передачи произвольных имён.
- [ShellCheck SC2044](https://www.shellcheck.net/wiki/SC2044) — изучите поломку цикла `for file in $(find ...)` и безопасные варианты обхода.
