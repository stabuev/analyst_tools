# CSV и неоднозначность типов

> CSV становится данными только после явного договора о байтах, разделителях, пропусках и типах.

**Тип:** Learn  
**Треки:** Core  
**Пререквизиты:** 04/12  
**Время:** ~75 минут  
**Результат:** объявляет диалект, кодировку, схему и правила пропусков CSV и обнаруживает
поврежденные строки до анализа.

## Цели обучения

- Разделять байтовую кодировку, CSV-диалект и табличную схему.
- Проверять одинаковую ширину строк до типизации значений.
- Задавать маркеры пропуска и числовой формат как часть контракта источника.
- Читать CSV в pandas без молчаливого пропуска строк и необратимой ранней инференции.

## Проблема

Поставщик прислал файл заказов с расширением `.csv`. На одной машине он открывается, на
другой падает с `UnicodeDecodeError`, а при `pd.read_csv(path)` вся строка оказывается в
одном столбце. После указания `sep=";"` суммы всё равно остаются строками:

```text
1 200,50
950,00
```

В комментарии встречается `скидка; май`, а два пропуска записаны по-разному: пустой
строкой и словом `NULL`. Одна поврежденная строка содержит неэкранированный `;`.

Если решить проблемы набором случайных параметров, файл может загрузиться, но результат
не станет проверяемым. Опасные варианты:

- декодировать с `errors="replace"` и потерять исходные символы;
- использовать неверный delimiter и получить один широкий столбец;
- включить `on_bad_lines="skip"` и незаметно потерять заказ;
- разрешить parser inference превратить идентификатор или маркер пропуска не в тот тип;
- удалить запятые, смешав разделитель полей и десятичный знак.

## Концепция

CSV не хранит единую самодостаточную схему. Для воспроизводимого чтения нужны четыре
отдельных договора.

| Слой | Вопрос | Пример |
|---|---|---|
| Encoding | Как байты превращаются в символы? | `cp1251` |
| Dialect | Как символы делятся на поля и строки? | delimiter `;`, quotechar `"` |
| Null policy | Какие токены означают отсутствие значения? | `""`, `NULL` |
| Schema | Какие столбцы и типы ожидаются? | `amount: decimal`, non-null |

**Encoding** применяется раньше CSV-парсера. Если байты CP1251 попытаться строго
декодировать как UTF-8, ошибка полезна: она останавливает порчу данных. Символ замены
`�` не является исправлением.

**Dialect** определяет delimiter, quotechar, правила двойных кавычек и пробелов. Символ
delimiter внутри quoted field не создаёт новое поле:

```csv
O2002;U002;2026-05-02T08:30:00Z;950,00;RUB;"скидка; май"
```

**Null policy** является семантикой источника. Токен `NULL` иногда означает пропуск, а
иногда является настоящей строкой. Автоматический словарь NA нельзя считать универсальным.

**Schema** применяется после структурной проверки. Сначала нужно доказать, что заголовок
и ширина строк согласованы, затем приводить `amount` и `ordered_at`.

`csv.Sniffer` и parser inference полезны для разведки, но эвристика не заменяет контракт:
результат угадывания нужно проверить и зафиксировать.

## Соберите это

Committed fixture использует CP1251, `;`, десятичную запятую и пробел тысяч.

### Шаг 1. Декодируйте строго

```python
raw = path.read_bytes()
text = raw.decode("cp1251", errors="strict")
```

`strict` превращает неверную кодировку в наблюдаемую ошибку. Не используйте
`errors="ignore"` или `errors="replace"` в quality gate.

### Шаг 2. Сравните наивный и явный dialect

Парсер по умолчанию ожидает запятую:

```python
naive_header = next(csv.reader(io.StringIO(text)))
```

В учебном файле он видит один field. Явный delimiter восстанавливает шесть:

```python
reader = csv.DictReader(
    io.StringIO(text),
    delimiter=";",
    quotechar='"',
)
```

### Шаг 3. Проверьте ширину каждой строки

До создания словаря сравните фактическое число полей с заголовком:

```python
if len(row) != len(header):
    raise ValueError(
        f"line {reader.line_num}: expected {len(header)}, got {len(row)}"
    )
```

Номер строки и обе ширины важнее сообщения «CSV сломан»: они позволяют найти источник
дефекта, не удаляя наблюдение.

### Шаг 4. Примените числовой договор

Для `1 200,50`:

```python
normalized = value.replace(" ", "").replace(",", ".")
amount = Decimal(normalized)
```

Это не универсальная очистка строк, а преобразование, разрешённое конкретным контрактом.

Запустите прозрачный пример:

```bash
uv run --locked python code/main.py
```

Он показывает, что наивное чтение видит один столбец, явное — шесть, quoted delimiter
остаётся внутри комментария, а сумма после договорного преобразования равна `3226.59`.

## Используйте это

pandas читает тот же файл только после фиксации структуры:

```python
frame = pd.read_csv(
    path,
    encoding="cp1251",
    sep=";",
    quotechar='"',
    dtype="string",
    keep_default_na=False,
    na_filter=False,
    on_bad_lines="error",
)
```

Первый проход намеренно сохраняет исходные токены как строки:

- `dtype="string"` откладывает типизацию;
- `keep_default_na=False` и `na_filter=False` не расширяют null policy;
- `on_bad_lines="error"` не разрешает молчаливую потерю строк;
- encoding и dialect не зависят от локали машины.

После структурного gate можно отдельно применить `null_values`, `decimal`, `thousands`,
timestamp parser и domain checks.

Самостоятельный аудитор выполняет оба слоя:

```bash
uv run --locked python outputs/csv_audit.py \
  --input ../data/tiny/orders_semicolon_cp1251.csv \
  --contract ../data/contract.json
```

JSON-отчёт содержит:

- размер и SHA-256 исходных байтов;
- объявленную кодировку и обнаруженный BOM;
- фактический и ожидаемый заголовок;
- malformed rows с номером строки и шириной;
- null и invalid counts по каждому столбцу;
- результат строгого чтения pandas;
- итоговый `valid` и число проваленных проверок.

Коды возврата:

- `0` — файл соответствует контракту;
- `1` — файл прочитан, но данные или структура нарушают контракт;
- `2` — файл или контракт нельзя проверить;
- `0` с `--allow-failures` — учебный просмотр известного дефекта.

## Сломайте это

Второй fixture содержит строку:

```text
O2002;U002;2026-05-02T08:30:00Z;950,00;RUB;скидка; май
```

Комментарий не заключён в кавычки, поэтому parser видит семь полей вместо шести.

```bash
uv run --locked python outputs/csv_audit.py \
  --input ../data/tiny/orders_broken_cp1251.csv \
  --contract ../data/contract.json \
  --allow-failures
```

Отчёт должен показать line `3`, expected fields `6` и actual fields `7`.

Проверьте другие failure modes:

1. замените encoding в копии контракта на `utf-8`;
2. удалите `currency` из заголовка;
3. запишите `not-a-number` в `amount`;
4. сделайте пустым non-null `order_id`;
5. поменяйте `NULL` на `N/A`, не меняя null policy.

Каждый дефект должен менять конкретную проверку, а не приводить к неуправляемому
traceback.

## Ключевые термины

| Термин | Распространенное заблуждение | Точное значение |
|---|---|---|
| Encoding | Настройка pandas | Правило преобразования байтов в символы до CSV-парсинга |
| CSV dialect | Всегда запятая | Набор правил delimiter, quotechar и экранирования |
| Header | Первая строка с любыми словами | Объявленные имена и порядок полей |
| Null policy | Все похожие слова являются NA | Явный набор токенов отсутствующего значения |
| Type inference | Надёжное определение схемы | Эвристика по наблюдаемой выборке значений |
| Malformed row | Строка, которую можно пропустить | Запись, не соответствующая структурному контракту |
