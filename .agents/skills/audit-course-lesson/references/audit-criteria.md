# Критерии педагогического аудита урока

Используй этот справочник после чтения текущего урока. Он помогает не пропустить важное,
но не требует выдавать пользователю длинный чек-лист.

## 1. Роль в программе

| Вопрос | Сильный признак | Тревожный признак |
|---|---|---|
| Результат | Измеримое действие студента | «Познакомится», «узнает о библиотеке» |
| Место | Нужен для следующего шага | Появился в порядке справочника API |
| Масштаб | Один связный учебный переход | Несколько независимых тем в одном уроке |
| Граница | Ясно, что будет позже | Урок забирает темы будущих фаз |
| Профессия | Видна рабочая задача аналитика | Инструмент изучается без применения |

Удаление или объединение — нормальный результат аудита. Количество уроков не является
метрикой качества программы.

## 2. Пререквизиты

Извлеки предпосылки не только из `lesson.json`, но и из объяснений, кода, практики,
артефакта и вопросов квиза. Проверь три слоя:

1. **Термины:** каждое новое слово объяснено до использования.
2. **Механизмы:** студент уже строил операции, на которые опирается урок.
3. **Предметная модель:** понятны единица наблюдения, ключи, оси, метрика или решение.

Для каждой существенной предпосылки найди доказательство в завершенных уроках:

| Статус | Доказательство | Решение |
|---|---|---|
| Готово | Опора объяснена и применена студентом | Использовать без повторного мини-урока |
| Нужен мост | Опора пройдена, но новый контекст или термин неочевиден | Дать короткое напоминание, контраст или пример |
| Пробел | Механизм лишь упомянут, оставлен в чтении либо появляется впервые | Перестроить порядок, добавить опору или сузить outcome |

Смотри дальше ближайшего соседа, если урок опирается на более раннюю тему, но не
перечитывай всю программу: сначала назови конкретную предпосылку, затем найди место, где
она должна была быть введена и отработана. Запись prerequisite в метаданных подтверждает
порядок, а не фактическую педагогическую готовность.

Отдельно проверь обратную ошибку: текущий урок не должен требовать знания, которое
программа обещает объяснить только позже. Если недостает небольшой опоры, добавь
локальное объяснение и ссылку на будущее углубление. Если недостает самостоятельного
механизма, сначала исправь порядок программы.

## 3. Понятность

- Первый пример должен быть понятен без знания результата урока.
- Незнакомый термин раскрывается через обычный язык и конкретную ситуацию.
- Код следует после предсказания смысла, формы или результата.
- Каждая таблица или схема отвечает на вопрос, который трудно увидеть в прозе.
- Повторение допустимо, если меняется уровень: интуиция, точный контракт, перенос.
- Англоязычный термин сохраняется, когда нужен в документации, но получает русское
  объяснение.

## 4. Достаточность без энциклопедичности

Обязательная часть включает:

- центральную модель;
- рабочий минимальный API;
- типичные ошибки с высокой ценой;
- проверку результата;
- границу применимости.

Обычно откладываются:

- редкие параметры и оптимизации до появления рабочей потребности;
- темы следующей фазы;
- расширенные математические детали, не меняющие решение аналитика;
- альтернативные библиотеки без отдельного выбора между ними.

Полезную, но необязательную тему обозначь коротко и отправь в дополнительное чтение или
будущий урок. Не превращай факультатив в скрытый prerequisite.

## 5. Связность разделов

Проверь, решают ли все разделы одну задачу:

| Переход | Контрольный вопрос |
|---|---|
| Problem → Concept | Концепция объясняет именно показанную проблему? |
| Concept → Build It | Ручной механизм делает скрытую абстракцию наблюдаемой? |
| Build It → Use It | Библиотека решает ту же задачу и дает сопоставимый результат? |
| Use It → Break It | Ошибки относятся к реальному применению этого механизма? |
| Break It → Verify It | Проверки ловят показанные ошибки? |
| Verify It → Ship It | Артефакт сохраняет проверки и пригоден вне урока? |

Если «Соберите это» вводит новую предметную область или требует неизвестного термина,
оно стало вторым уроком внутри первого.

## 6. Практика и артефакт

Хорошая практика заставляет принять решение, предсказать результат или диагностировать
ошибку. Слабая практика просит перепечатать пример.

Для executable-практики проверь:

- существует прозрачный baseline до production API;
- код запускается без скрытого состояния;
- тесты проверяют значения, форму, тип, порядок или иной предметный инвариант;
- хотя бы один тест воспроизводит правдоподобную ошибку;
- CLI добавлен только если делает артефакт переиспользуемым.

Для guided-artifact проверь:

- шаблон можно заполнить без чтения reference-ответа;
- рубрика оценивает качество решения, а не наличие заголовков;
- неоднозначный сценарий требует обоснования;
- результат можно передать коллеге.

Артефакт не обязан быть большим. Он обязан сохранять новый способ работы студента.

## 7. Квиз и проверки

Вопросы должны различать понимание и типичную ошибку. Хороший дистрактор выглядит
правдоподобно для студента, который перепутал ось, grain, тип, причинность или границу
метода.

Избегай:

- проверки готовности к курсу;
- вопросов о синтаксисе, который легко посмотреть;
- абсурдных вариантов;
- подсказок длиной или грамматикой;
- одинаковой позиции правильного ответа;
- утверждения, что квиз доказывает профессиональную готовность.

## 8. Дополнительное чтение

Каждая ссылка выполняет одну из ролей:

1. точный API-контракт;
2. концептуальная модель или первичный источник;
3. практический разбор failure mode;
4. осознанное углубление за границей урока.

Аннотация отвечает «что открыть» и «зачем читать». Проверяй актуальность API и не
советуй источник только из-за высокого места в поиске.

## 9. Сигналы лишнего содержания

- Тема не влияет на outcome, практику, failure mode или следующий урок.
- Код существует только потому, что соседние уроки содержат код.
- Новый термин нужен лишь для объяснения упражнения, которое само не нужно.
- Один тезис повторяется в Problem, Concept, Build It и Use It без нового уровня.
- Артефакт сложнее профессиональной задачи, которую он должен поддержать.
- Полный набор проверок запускается после каждой маленькой текстовой правки.

## 10. Перед передачей результата

- Назови педагогическое решение и его причину.
- Отдели обязательное от сознательно отложенного.
- Проверь только затронутый объем.
- Не объявляй всю фазу готовой после проверки одного урока.
- Не переходи к следующему уроку и не создавай коммит без просьбы пользователя.
